Usando /init para gerar um CLAUDE.md base

O comando /init analisa seu codebase e gera automaticamente um arquivo CLAUDE.md com as convenções, comandos e estrutura do projeto. É o ponto de partida mais rápido — mas nunca o resultado final.

O que o comando /init faz

Quando você digita /init no Claude Code, ele não simplesmente cria um arquivo vazio. O Claude recebe acesso ao seu codebase e explora ativamente a estrutura do projeto para construir um CLAUDE.md que reflita a realidade do repositório. Ele age como um agente: lê arquivos de configuração, examina a árvore de diretórios, identifica padrões de código e sintetiza tudo em instruções persistentes.

Se um CLAUDE.md já existir no projeto, o /init não sobrescreve o arquivo existente — ele sugere melhorias e adições. Isso torna seguro rodar o comando mesmo em projetos que já possuem configuração.

Novidade: Com a variável de ambiente CLAUDE_CODE_NEW_INIT=1, o /init entra em um fluxo interativo multi-fase: pergunta quais artefatos configurar (CLAUDE.md, skills, hooks), explora o codebase com um subagente, faz perguntas de follow-up e apresenta uma proposta revisável antes de escrever qualquer arquivo.

O que o /init analisa

O Claude Code percorre o repositório buscando sinais sobre a stack, a arquitetura e as convenções do projeto. Os principais alvos de análise incluem:

P Manifesto do projeto

package.json, pyproject.toml, Cargo.toml, go.mod — identifica linguagem, dependências e scripts disponíveis.

C Arquivos de configuração

tsconfig.json, .eslintrc, Dockerfile, Makefile — detecta tooling de build, lint e deploy.

D Estrutura de diretórios

Identifica pastas como src/, tests/, lib/, api/ — mapeia a arquitetura e os módulos principais.

F Frameworks e padrões

Detecta React, Next.js, Express, Django, Spring Boot e outros — ajusta convenções ao ecossistema.

T Testes e CI

Encontra test runners (jest, pytest, vitest), pipelines CI e scripts de verificação.

A Configs de outros agentes

Lê AGENTS.md, .cursorrules, .windsurfrules e incorpora as partes relevantes ao CLAUDE.md gerado.

Exemplo de CLAUDE.md gerado pelo /init

Abaixo está um exemplo realista do que o /init gera para um projeto Next.js com TypeScript. Note que o conteúdo é objetivo, focado em comandos e estrutura — exatamente o tipo de informação que o Claude precisa para ser produtivo:

CLAUDE.md (gerado pelo /init) ~45 linhas 
# Project: meu-app

## Commands
- Build: `npm run build`
- Dev server: `npm run dev`
- Test all: `npm run test`
- Test single: `npx vitest run path/to/test.ts`
- Lint: `npm run lint`
- Type check: `npx tsc --noEmit`
- Format: `npx prettier --write .`

## Architecture
- Framework: Next.js 14 with App Router
- Language: TypeScript (strict mode)
- Styling: Tailwind CSS
- State management: Zustand
- Database: Prisma ORM with PostgreSQL

## Project Structure
- src/app/         → App Router pages and layouts
- src/components/  → Reusable React components
- src/lib/         → Shared utilities and helpers
- src/server/      → Server-side logic (actions, API)
- src/types/       → TypeScript type definitions
- prisma/          → Database schema and migrations
- tests/           → Test files (mirrors src/ structure)

## Code Style
- Use named exports (not default exports)
- Prefer `interface` over `type` for object shapes
- Use absolute imports with `@/` prefix

## Testing
- Test runner: Vitest
- Test files: `*.test.ts` or `*.test.tsx`
- Coverage: `npm run test:coverage`
Observe o padrão: O output é curto, factual e orientado a ação. Ele lista comandos que o Claude pode executar, descreve a estrutura para navegação e registra convenções para manter consistência. Não há texto narrativo ou explicações longas — exatamente como a documentação oficial recomenda.

O que fazer depois do /init

O CLAUDE.md gerado é um primeiro rascunho, não um produto final. O Claude consegue descobrir informações técnicas (comandos, frameworks, estrutura), mas não consegue adivinhar decisões de design, regras de negócio ou preferências do time. Depois de rodar o /init, você deve:

1

Revisar o conteúdo gerado

Abra o CLAUDE.md e verifique se os comandos estão corretos, se a estrutura reflete a realidade e se nenhuma informação sensível foi incluída (variáveis de ambiente, paths internos, etc.).

2

Adicionar o que o Claude não descobriu

Regras de negócio, convenções do time, padrões de commit, arquivos que não devem ser tocados, decisões arquiteturais que não estão óbvias no código.

3

Remover o que é redundante

Se o Claude já consegue inferir algo direto do código (ex: "usa TypeScript"), não precisa repetir. Foque no que ele não consegue descobrir sozinho.

4

Commitar e iterar

Commite o CLAUDE.md no repositório para que o time todo se beneficie. Atualize conforme o projeto evolui — regras mudam, dependências são atualizadas, convenções novas surgem.

Armadilha comum: Rodar o /init e nunca mais tocar no CLAUDE.md. O arquivo envelhece junto com o projeto. Trate-o como documentação viva — atualize quando mudar padrões, adicionar dependências ou alterar processos de build/test.

O /init como ponto de partida

O /init é uma ferramenta de bootstrap, não de conclusão. Ele resolve o problema do "papel em branco" — a dificuldade de começar a escrever um CLAUDE.md do zero. Mas o valor real do arquivo vem das informações que você adiciona: o conhecimento que está na cabeça do time e não no código.

O que o /init descobre O que você precisa adicionar
Comandos de build, test, lint Ordem de execução dos comandos, pré-requisitos
Framework e linguagem Decisões de design: por que esse framework, quais alternativas foram descartadas
Estrutura de diretórios Regras de organização: onde cada tipo de arquivo deve ficar
Dependências instaladas Dependências proibidas, alternativas preferidas
Padrões de naming detectados Convenções de naming explícitas, exceções
Existência de testes Cobertura mínima esperada, padrões de mock, fixtures

Quando usar /init vs. criar manualmente

A Use /init quando...

Está trazendo o Claude Code para um projeto existente pela primeira vez
O projeto tem configurações padrão que o Claude pode detectar automaticamente
Quer um ponto de partida rápido para iterar sobre
O repositório já tem AGENTS.md ou .cursorrules que podem ser incorporados
vs

M Crie manualmente quando...

O projeto é muito novo e ainda não tem configurações ou código significativo
Você precisa de um CLAUDE.md altamente específico com regras de negócio complexas
Quer começar minimal e só adicionar regras conforme problemas aparecem
Já existe um CLAUDE.md maduro e você só quer ajustes finos

Workflow recomendado

O fluxo ideal combina a velocidade do /init com a curadoria humana. O resultado é um CLAUDE.md que evolui junto com o projeto:

O /init é o passo 1 de um processo contínuo. O CLAUDE.md evolui com o projeto.

Dicas para aproveitar melhor o /init

1 Rode na raiz do projeto

O /init analisa a partir do diretório atual. Se rodar em uma subpasta, o CLAUDE.md será parcial. Sempre rode na raiz do repositório.

2 Use a tecla # para updates rápidos

Depois do /init, use a tecla # durante o trabalho para instruir o Claude a adicionar novas convenções diretamente ao CLAUDE.md.

3 Mantenha abaixo de 200 linhas

A documentação oficial recomenda manter cada CLAUDE.md abaixo de 200 linhas. Use .claude/rules/ para regras específicas por tipo de arquivo.

4 Revise com o time

O CLAUDE.md é compartilhado via git. Antes de commitar, peça feedback do time — as instruções afetam a experiência de todos que usam Claude Code no projeto.

Dica avançada: Use /memory para ver todos os arquivos CLAUDE.md e regras carregados na sessão atual. É útil para auditar se o /init gerou algo que conflita com instruções existentes em subdiretórios ou no ~/.claude/CLAUDE.md pessoal.
