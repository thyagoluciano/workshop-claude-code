Modulo 2 . Topico 5

Skills vs. CLAUDE.md: quando usar cada um

O Claude Code oferece dois mecanismos para injetar contexto e instrucoes: o CLAUDE.md (sempre presente) e as Skills (sob demanda). Saber quando usar cada um e a chave para manter o contexto enxuto e os workflows eficientes.

Dois mecanismos, dois propositos

O Claude Code carrega contexto de formas diferentes dependendo do mecanismo. O CLAUDE.md e lido automaticamente no inicio de toda conversa — ele contem as regras permanentes do projeto. As Skills sao carregadas apenas quando invocadas (via /comando) ou quando o Claude decide autonomamente que sao relevantes para a tarefa atual.

Essa distincao e fundamental: tudo que esta no CLAUDE.md consome tokens em todas as conversas, mesmo quando nao e relevante. Skills, por outro lado, so consomem tokens quando acionadas. A regra pratica e simples: se a instrucao se aplica a toda conversa, vai no CLAUDE.md. Se e um workflow especifico, vira uma skill.

Comparacao lado a lado

📄 CLAUDE.md

Carregado automaticamente em toda conversa
Contem convencoes que se aplicam sempre
Estilo de codigo, comandos de build, regras de commit
Deve ser conciso (recomendado < 200 linhas)
Consome tokens em toda interacao
Equivale ao que voce repetiria a todo novo colega
vs

⚡ Skills

Carregadas sob demanda via /comando
Contem workflows especificos e repetiveis
Review de PR, geracao de testes, debugging, deploy
Podem ser extensas — so carregam quando necessario
Consomem tokens apenas quando acionadas
Equivale a um runbook ou checklist do time

Como o Claude Code carrega cada um

O CLAUDE.md consome tokens em toda conversa. Skills so entram no contexto quando invocadas.

Quando usar CLAUDE.md

O CLAUDE.md e o lugar para informacoes que o Claude precisa ter em toda interacao, independentemente da tarefa. Pense nele como o onboarding que voce daria a um novo desenvolvedor no primeiro dia: convencoes do projeto, comandos essenciais e regras inegociaveis.

🔧 Comandos de build e teste

pnpm test, pytest -x, make build — comandos que o Claude precisa saber em qualquer tarefa de desenvolvimento.

📑 Convencoes de codigo

Estilo de imports, naming conventions, patterns obrigatorios (ex: "sempre use Result type para erros", "componentes React com arrow functions").

📦 Estrutura do projeto

Descricao breve da arquitetura, pastas principais e onde cada tipo de codigo vive. Contexto que o Claude nao consegue inferir so do codigo.

⚠️ Regras e restricoes

"Nunca edite arquivos gerados", "sempre use commits convencionais", "nao adicione dependencias sem perguntar". Regras que se aplicam a toda tarefa.

CLAUDE.md (exemplo enxuto) ~30 linhas 
# Projeto: API de Pagamentos

## Build & Test
- Use `pnpm` (nao npm ou yarn)
- Rodar testes: `pnpm test`
- Lint: `pnpm lint`
- Build: `pnpm build`

## Convencoes
- TypeScript strict mode
- Erros com Result<T, E> (nunca throw)
- Testes com Vitest, co-locados com o codigo
- Imports absolutos via @/

## Arquitetura
- src/domain/ — entidades e regras de negocio
- src/infra/ — adaptadores externos (DB, APIs)
- src/api/ — controllers e rotas Express

## Regras
- Commits com Conventional Commits (feat:, fix:, etc.)
- Nunca edite arquivos em src/generated/
- Nao adicione dependencias sem perguntar
Regra de ouro: Se voce se pega repetindo a mesma instrucao em varias conversas com o Claude Code, ela pertence ao CLAUDE.md. Se e algo que so faz sentido em um tipo especifico de tarefa, e candidata a skill.

Quando usar Skills

Skills sao workflows repetiveis que voce invoca com /nome-da-skill ou que o Claude carrega automaticamente quando detecta que sao relevantes. Elas encapsulam procedimentos especificos — como um runbook que o time segue para review, deploy ou debugging.

🔍 Review de Pull Request

Checklist de review: verificar tipos, testes, performance, seguranca, convencoes do projeto. Procedimento complexo demais para o CLAUDE.md.

🧪 Geracao de testes

Template de como gerar testes no padrao do projeto: setup, mocks, assertions, coverage minima. Carregado so quando voce pede para criar testes.

🐛 Debugging estruturado

Passos padrao para investigar bugs: reproduzir, isolar, identificar causa raiz, corrigir, adicionar teste de regressao.

🚀 Deploy e release

Checklist de release: bump de versao, changelog, tag git, validacao de CI. Workflow que so roda em momentos especificos.

Como criar uma Skill

Uma skill e um diretorio contendo um arquivo SKILL.md como ponto de entrada. O SKILL.md tem duas partes: um frontmatter YAML (entre marcadores ---) que informa ao Claude quando usar a skill, e o conteudo markdown com as instrucoes que o Claude segue quando a skill e acionada.

Estrutura de diretorios

~/.claude/skills/ .............. skills pessoais (todos os projetos)
  |-- review/
  |   |-- SKILL.md .............. ponto de entrada
  |   |-- checklist.md .......... arquivo de apoio (opcional)
  |-- debug/
      |-- SKILL.md

.claude/skills/ ................. skills do projeto (compartilhadas via git)
  |-- create-migration/
  |   |-- SKILL.md
  |   |-- template.sql ......... template para Claude preencher
  |-- create-endpoint/
      |-- SKILL.md
      |-- example-output.ts .. exemplo de saida esperada
Pessoal vs. Projeto: Skills em ~/.claude/skills/ sao suas e funcionam em qualquer repositorio. Skills em .claude/skills/ (na raiz do projeto) sao compartilhadas com o time via git — o ideal para padronizar workflows da equipe.

Anatomia de um SKILL.md

.claude/skills/review/SKILL.md Skill de projeto 
---
name: review
description: Faz code review seguindo o padrao do time.
  Usar quando o usuario pedir para revisar codigo,
  PR ou changeset.
---

# Code Review

Ao revisar codigo, siga este checklist:

## 1. Corretude
- O codigo faz o que se propoe?
- Existem edge cases nao tratados?
- Os tipos estao corretos (sem any desnecessario)?

## 2. Testes
- Existem testes para o caso feliz?
- Existem testes para os edge cases?
- Os testes passam? Rode `pnpm test` para verificar.

## 3. Seguranca
- Inputs sao validados/sanitizados?
- Dados sensiveis nao sao logados?

## 4. Performance
- Queries N+1?
- Operacoes desnecessarias em loops?

## Formato da saida
Apresente os achados em categorias:
critico, sugestao, elogio.

Campos do frontmatter

Campo Obrigatorio? Funcao
name Nao Nome do /slash-command. Se omitido, usa o nome da pasta.
description Recomendado Ajuda o Claude a decidir quando carregar a skill automaticamente. Quanto mais descritivo, melhor o matching.
disable-model-invocation Nao Se true, a skill so e acionada manualmente via /comando. O Claude nao a invoca sozinho.
allowed-tools Nao Lista de ferramentas que a skill pode usar (ex: Read, Grep, Bash).
Dica: O Claude Code monitora os diretorios de skills em tempo real. Adicionar, editar ou remover uma skill em ~/.claude/skills/ ou .claude/skills/ tem efeito imediato na sessao corrente, sem necessidade de reiniciar.

Skills com arquivos de apoio

Skills podem incluir arquivos auxiliares alem do SKILL.md: templates para o Claude preencher, exemplos de saida esperada, scripts auxiliares, ou documentacao de referencia. Isso permite criar skills mais sofisticadas sem sobrecarregar o arquivo principal.

.claude/skills/create-endpoint/SKILL.md Skill com template 
---
name: create-endpoint
description: Cria um novo endpoint REST seguindo o padrao
  do projeto. Usar quando pedirem para criar rota,
  endpoint ou API nova.
---

# Criar novo endpoint

Use o template em `./template-controller.ts` como base.
Veja o exemplo de saida em `./example-output.ts`.

## Passos
1. Crie o schema Zod em `src/schemas/`
2. Crie o controller baseado no template
3. Registre a rota em `src/api/routes.ts`
4. Crie testes unitarios co-locados
5. Rode `pnpm test` para verificar

Exemplo pratico: migrando do CLAUDE.md para uma Skill

Um sinal claro de que algo deve sair do CLAUDE.md e virar uma skill: voce tem um bloco grande de instrucoes que so se aplica a um tipo especifico de tarefa. Veja como migrar um procedimento de "criacao de migration SQL" que estava inchando o CLAUDE.md.

Antes: tudo no CLAUDE.md

CLAUDE.md (antes — 45+ linhas so de migrations) Problema: ocupa tokens sempre 
# Projeto: API de Pagamentos

## Build & Test
- pnpm test, pnpm lint, pnpm build

## Convencoes
- TypeScript strict, Result<T,E>, Vitest

## Como criar migrations (!!!)
- Use o formato: YYYYMMDDHHMMSS_descricao.sql
- Sempre crie UP e DOWN
- Use transacoes (BEGIN/COMMIT)
- Nunca use DROP sem IF EXISTS
- Adicione comentarios em cada ALTER
- Valide com: pnpm db:migrate:dry-run
- Teste rollback: pnpm db:migrate:down
- Template:
  BEGIN;
  -- UP: descricao da mudanca
  ALTER TABLE ...
  COMMIT;
  -- DOWN
  BEGIN;
  ALTER TABLE ...
  COMMIT;
  ... (+20 linhas de regras e exemplos)

Nota sobre .claude/commands (formato legado)

Antes das Skills, o Claude Code usava o diretorio .claude/commands/ com arquivos .md simples (sem frontmatter YAML). Esse formato ainda funciona, mas o formato recomendado agora e .claude/skills/nome/SKILL.md, que suporta tanto invocacao via /comando quanto invocacao autonoma pelo Claude.

Precedencia: Se uma skill e um comando legado tiverem o mesmo nome, a skill tem precedencia. Se voce tem arquivos em .claude/commands/, considere migra-los para o formato de skills para aproveitar o frontmatter YAML e a deteccao automatica.

Arvore de decisao: CLAUDE.md ou Skill?

Fluxo de decisao: se se aplica a toda conversa, vai no CLAUDE.md. Se e um workflow repetivel, crie uma skill. Caso contrario, instrua diretamente no prompt.

Dicas para manter o equiibrio

📊 Audite periodicamente

Revise o CLAUDE.md a cada sprint. Se uma secao so foi relevante em 1 de cada 10 conversas, migre para skill.

👥 Skills de projeto no git

Commite skills em .claude/skills/ para que todo o time use os mesmos workflows. PRs para skills tambem merecem review.

🔑 Descriptions descritivas

Escreva descriptions claras no frontmatter. O Claude usa esse texto para decidir quando carregar a skill automaticamente.

📦 Arquivos de apoio

Use templates e exemplos de saida como arquivos separados na pasta da skill. Mantém o SKILL.md focado em instrucoes.

Anti-pattern: Colocar tudo no CLAUDE.md "porque e mais facil". Um CLAUDE.md com 500+ linhas degrada a aderencia do Claude as instrucoes e desperica tokens em toda conversa. Se voce tem mais de 200 linhas, e hora de extrair skills.
