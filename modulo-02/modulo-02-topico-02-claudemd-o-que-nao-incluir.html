O que nunca incluir no CLAUDE.md

O Claude ja sabe programar. Ele conhece linguagens, frameworks, APIs populares e padroes de design. Incluir esse tipo de informacao no CLAUDE.md nao ajuda — atrapalha. Cada token desperdicado com o obvio e um token a menos para o que realmente importa.

O que o Claude ja sabe (e voce nao precisa ensinar)

O Claude foi treinado em vastos conjuntos de dados que incluem documentacao oficial, repositorios open-source, artigos tecnicos e livros. Isso significa que ele ja possui conhecimento profundo sobre uma enorme gama de tecnologias. Repetir esse conhecimento no CLAUDE.md consome tokens sem agregar valor.

L Linguagens

Python, JavaScript/TypeScript, Java, Go, Rust, C/C++, Ruby, PHP, Swift, Kotlin, SQL e dezenas de outras.

F Frameworks

React, Next.js, Vue, Angular, Django, Flask, Spring Boot, Rails, Express, FastAPI, NestJS e muitos mais.

A APIs e Ferramentas

REST, GraphQL, gRPC, Docker, Kubernetes, AWS/GCP/Azure, Git, CI/CD, bancos SQL e NoSQL.

P Padroes de Design

SOLID, MVC, Repository, Factory, Observer, Strategy, Dependency Injection, Clean Architecture.

T Testes

Jest, Pytest, JUnit, Mocha, Vitest, Testing Library, Cypress, Playwright — e como escrever bons testes.

D DevOps & Infra

Terraform, Ansible, GitHub Actions, GitLab CI, Dockerfile best practices, networking basics.

Regra de ouro: Se a informacao pode ser encontrada na documentacao oficial de uma tecnologia amplamente usada, o Claude provavelmente ja sabe. Nao repita no CLAUDE.md.

Anti-patterns: o que nao colocar

A documentacao da Anthropic sobre best practices para Claude Code e context engineering enfatiza que contexto e um recurso finito com retornos decrescentes. Cada token adicionado ao contexto dilui a atencao do modelo sobre os tokens que realmente importam. Eis os anti-patterns mais comuns:

Anti-pattern Por que e ruim O que fazer em vez disso
Colar documentacao de bibliotecas O Claude ja conhece React, Zod, Prisma etc. Duplicar a docs gasta tokens. Documente apenas convencoes do seu projeto com essa lib.
Tutoriais genericos "Como criar um componente React" e conhecimento base do modelo. Documente como o seu time cria componentes (naming, pasta, testes).
Codigo boilerplate Templates longos poluem o contexto; o modelo ja sabe gerar boilerplate. Referencie arquivos-exemplo: "siga o padrao de src/components/Button.tsx".
Regras que o modelo ja segue Dizer "use const em vez de var" e instruir o obvio. So documente se o Claude erra repetidamente sem a instrucao.
Explicacoes de conceitos "O que e REST", "como funciona async/await" nao e informacao de projeto. Foque em decisoes arquiteturais especificas do seu sistema.
Lista exaustiva de edge cases O modelo se perde em listas longas de regras. Itens importantes se diluem. Cubra os casos criticos e use exemplos canonicos para o resto.

O problema do CLAUDE.md muito longo: context rot

A Anthropic documenta o conceito de context rot no artigo sobre context engineering: a medida que o numero de tokens no contexto aumenta, a capacidade do modelo de recordar informacoes com precisao diminui. Isso acontece porque a arquitetura transformer exige que cada token "preste atencao" a todos os outros tokens — com n tokens, sao n² relacoes para gerenciar.

Context rot: quanto mais tokens no contexto, menor a precisao de recall. Um CLAUDE.md inchado empurra o modelo para a zona de risco.

Impacto pratico: O CLAUDE.md e carregado no inicio de cada conversa. Se ele tem 5.000 tokens de informacao redundante, isso significa 5.000 tokens a menos disponiveis para o seu codigo-fonte, outputs de ferramentas e historico de conversa — justamente onde o modelo mais precisa de atencao.

Regra pratica: o teste da remocao

Antes de adicionar qualquer linha ao CLAUDE.md, faca a si mesmo uma pergunta simples:

"Se eu remover essa linha, o Claude produziria um output errado?"

Se a resposta for nao → remova a linha.
Se a resposta for sim → mantenha.

Essa heuristica simples funciona porque filtra exatamente o que importa: informacao que o modelo nao consegue inferir sozinho. Se o Claude ja faz algo corretamente sem a instrucao, a instrucao e redundante e so adiciona ruido ao contexto.

X Falha no teste

"Use TypeScript para novos arquivos" — se o projeto ja e 100% TS, o Claude infere isso ao ler o codebase.

✓ Passa no teste

"Nunca use any; use unknown e faca type narrowing" — e uma convencao do time que o Claude nao infere sozinho.

X Falha no teste

"Componentes React devem retornar JSX" — isso e a definicao literal de um componente React.

✓ Passa no teste

"Componentes de pagina vao em app/(routes)/, componentes reutilizaveis em src/ui/" — e a estrutura especifica deste projeto.

Antes e depois: CLAUDE.md inchado vs. enxuto

Veja a transformacao de um CLAUDE.md real. A versao "antes" tem informacao redundante que o Claude ja sabe; a versao "depois" mantem apenas o que e especifico do projeto.

Exemplo: secao sobre estilo de codigo

X Antes (inchado) — ~350 tokens

CLAUDE.mdruim
# Estilo de Codigo

- Use TypeScript para todo o codigo
- Use const para variaveis que nao mudam
- Use let apenas quando necessario
- Nunca use var
- Use arrow functions para callbacks
- Use async/await em vez de .then()
- Use template literals em vez de concatenacao
- Sempre trate erros com try/catch
- Use interfaces para definir contratos
- Use enums para valores constantes
- Componentes React devem ser funcionais
- Use hooks em vez de class components
- Importe React no topo do arquivo
- Exporte componentes como default export
- Prefixe interfaces com I (ex: IUser)
- Use barrel exports em cada pasta
- Erros de dominio estendem AppError
- Use Zod para validacao, nunca yup
vs

✓ Depois (enxuto) — ~80 tokens

CLAUDE.mdbom
# Estilo de Codigo

- Prefixe interfaces com I (ex: IUser)
- Use barrel exports (index.ts) em cada pasta
- Erros de dominio devem estender AppError
  (ver src/core/errors.ts)
- Validacao: sempre Zod, nunca yup ou joi
- named exports apenas (sem default export)

Resultado: de ~350 tokens para ~80 tokens — uma reducao de 77% — sem perder nenhuma informacao que o Claude precisaria. As 14 linhas removidas descrevem praticas que sao padrao moderno de TypeScript/React e que o modelo ja segue por default.

O principio por tras: context engineering

A Anthropic define context engineering como a pratica de curar e manter o conjunto otimo de tokens durante a inferencia do LLM. O principio guia e encontrar o menor conjunto possivel de tokens de alto sinal que maximize a probabilidade do resultado desejado.

Aplicado ao CLAUDE.md, isso significa:

1 Alto sinal

Cada linha deve conter informacao que o modelo nao consegue descobrir sozinho lendo o codebase ou usando suas ferramentas.

2 Baixo ruido

Remova tudo que e conhecimento geral, documentacao de libs, ou praticas que o modelo ja segue por padrao.

3 Token-eficiente

Use frases curtas e diretas. Prefira bullet points a paragrafos. Referencie arquivos em vez de colar codigo.

4 Documento vivo

Revise periodicamente. Se o Claude acerta sem a instrucao, remova-a. Se erra repetidamente, adicione.

Dica da Anthropic: O Claude Code usa um modelo hibrido: o CLAUDE.md e carregado no contexto no inicio, enquanto primitivos como glob e grep permitem ao modelo navegar pelo codebase e recuperar arquivos just-in-time. Isso significa que o CLAUDE.md nao precisa (e nao deve) conter tudo — o agente sabe buscar o que precisa.

Resumo: como manter o CLAUDE.md saudavel

Pratica Descricao
Aplique o teste da remocao "Se eu remover essa linha, o Claude erraria?" — Se nao, remova.
Referencie, nao copie Em vez de colar 50 linhas de boilerplate, aponte para um arquivo-exemplo no codebase.
Prune regularmente Modelos melhoram com o tempo. Instrucoes que eram necessarias antes podem ser redundantes agora.
Mexa no minimo viavel Comece com pouco e adicione apenas quando encontrar falhas reais, nao antecipadas.
Use exemplos canonicos Um bom exemplo vale mais que 10 regras escritas. Aponte para "o arquivo modelo" do projeto.
Lembre-se: O objetivo do CLAUDE.md nao e ensinar o Claude a programar. E dar a ele o contexto especifico do seu projeto que ele nao teria de outra forma. Menos e mais — desde que o "menos" seja altamente relevante.
