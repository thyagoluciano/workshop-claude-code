O que incluir no CLAUDE.md

O CLAUDE.md é o arquivo que dá memória ao Claude Code entre sessões. Ele define comandos de build, estilo de código, convenções de branch e tudo aquilo que o Claude erraria sem instruções explícitas.

O que é o CLAUDE.md?

O CLAUDE.md é um arquivo Markdown que o Claude Code lê automaticamente no início de toda sessão. Ele funciona como uma "cola" persistente: tudo que você escreveria num onboarding de novo dev — comandos de build, convenções do projeto, armadilhas conhecidas — vai nesse arquivo. O conteúdo é injetado no context window como uma mensagem de usuário, logo após o system prompt.

Não existe um formato obrigatório. O arquivo é Markdown puro, e o Claude o interpreta como instruções. Quanto mais específico e conciso, maior a aderência. A documentação oficial recomenda manter o arquivo com menos de 200 linhas por CLAUDE.md.

Comando rápido: Execute /init dentro do Claude Code para gerar um CLAUDE.md inicial automaticamente. O Claude analisa seu codebase e cria um arquivo com comandos de build, instruções de teste e convenções que ele descobriu. Se já existir um, o /init sugere melhorias sem sobrescrever.

Onde o CLAUDE.md pode ficar

O Claude Code suporta CLAUDE.md em vários locais, cada um com um escopo diferente. Todos são carregados e concatenados — nenhum sobrescreve o outro. A tabela abaixo lista na ordem de carregamento, do escopo mais amplo ao mais específico:

Escopo Local Compartilhado com
Política gerenciada /Library/Application Support/ClaudeCode/CLAUDE.md (macOS) Todos os usuários da organização
Usuário ~/.claude/CLAUDE.md Só você (todos os projetos)
Projeto ./CLAUDE.md ou ./.claude/CLAUDE.md Time (via controle de versão)
Local ./CLAUDE.local.md Só você (projeto atual) — adicione ao .gitignore
Subdiretório ./src/api/CLAUDE.md (qualquer subdir) Time — carregado sob demanda
Detalhe importante: Arquivos CLAUDE.md na raiz do projeto e nos diretórios acima são carregados integralmente no início da sessão. Arquivos em subdiretórios são carregados sob demanda, apenas quando o Claude lê arquivos naquele diretório. Isso permite escalar instruções em monorepos sem poluir o contexto.

Hierarquia de carregamento

O Claude Code percorre a árvore de diretórios de baixo para cima (do cwd até a raiz) buscando arquivos CLAUDE.md e CLAUDE.local.md. Todos são concatenados em sequência, do mais genérico ao mais específico. Dentro de cada diretório, CLAUDE.local.md é adicionado depois de CLAUDE.md.

Instruções mais específicas (mais perto do cwd) são lidas por último e têm maior prioridade contextual.

O que incluir no CLAUDE.md

A documentação oficial recomenda tratar o CLAUDE.md como o lugar para registrar aquilo que você teria que re-explicar ao Claude em toda sessão. A regra de ouro: se o Claude erra algo pela segunda vez, adicione a correção ao CLAUDE.md.

1 Comandos de build, test e lint

Comandos exatos para compilar, rodar testes (incluindo testes de um único arquivo) e verificar estilo. O Claude usa esses comandos para auto-verificação.

2 Estilo de código

Indentação, aspas, convenções de import, padrões de nomenclatura. Coisas que o linter/formatter cobram mas que o Claude pode gerar diferente.

3 Convenções de branch e commit

Formato de mensagem de commit, estratégia de branching (trunk-based, GitFlow), regras de PR. O Claude pode criar commits e branches automaticamente.

4 Arquitetura e layout do projeto

Onde ficam os handlers de API, componentes, schemas, testes. Evita que o Claude crie arquivos em locais errados.

5 Workflows e padrões do projeto

Como criar um novo endpoint, como rodar migrations, como fazer deploy. Padrões que o time segue e que o Claude deve replicar.

6 Armadilhas e gotchas

Comportamentos inesperados do projeto, workarounds conhecidos, dependências com versões fixas. Tudo que um dev novo erraria.

Critério de inclusão: Adicione ao CLAUDE.md quando uma dessas situações ocorrer: (1) o Claude comete o mesmo erro pela segunda vez, (2) um code review pega algo que o Claude deveria saber, (3) você digita a mesma correção que já digitou na sessão anterior, (4) um novo membro do time precisaria do mesmo contexto.

Exemplo: um bom CLAUDE.md de projeto

Abaixo está um exemplo realista de CLAUDE.md para um projeto backend em Node.js/TypeScript. Note como cada seção é curta, específica e verificável:

CLAUDE.md Exemplo de projeto backend 
# Projeto: API de Pedidos

## Comandos
- Build: `npm run build`
- Testes (todos): `npm test`
- Teste unico: `npm test -- --grep "nome do teste"`
- Lint: `npm run lint`
- Lint fix: `npm run lint:fix`
- Dev server: `npm run dev` (porta 3001)

## Estilo de Codigo
- TypeScript strict mode, sem `any`
- Usar ES modules (import/export), nunca CommonJS (require)
- Indentacao: 2 espacos
- Aspas simples para strings
- Sempre desestruturar imports quando possivel
- Tipos em arquivos separados: `src/types/<dominio>.ts`

## Convencoes de Git
- Commits: Conventional Commits (feat:, fix:, chore:, docs:)
- Branches: `feat/JIRA-123-descricao-curta`
- Sempre rodar `npm test` antes de commitar
- PRs devem ter descricao com "O que" e "Por que"

## Estrutura do Projeto
- Handlers de API: `src/api/handlers/`
- Schemas Zod: `src/schemas/`
- Servicos de dominio: `src/services/`
- Testes: colocados ao lado do arquivo (`*.test.ts`)
- Migrations: `src/db/migrations/`

## Padroes Importantes
- Novos endpoints seguem o padrao de `src/api/handlers/orders.ts`
- Toda rota precisa de schema Zod para validação de input
- Erros usam a classe `AppError` de `src/lib/errors.ts`
- Logs via `src/lib/logger.ts`, nunca `console.log`

## Gotchas
- O Redis local roda na porta 6380 (nao 6379)
- Migrations precisam de `DATABASE_URL` no .env
- O ESLint tem regra customizada: no-floating-promises
Por que funciona: Cada instrução é concreta e verificável. "Indentação: 2 espaços" é melhor que "formate o código direito". "Handlers em src/api/handlers/" é melhor que "mantenha os arquivos organizados". O Claude consegue seguir instruções específicas com muito mais consistência.

Exemplo: CLAUDE.md para projeto frontend React

CLAUDE.md Exemplo de projeto frontend 
# Portal do Cliente

## Comandos
- Dev: `pnpm dev` (porta 5173)
- Build: `pnpm build`
- Testes: `pnpm test`
- Teste unico: `pnpm test -- OrderForm.test.tsx`
- Storybook: `pnpm storybook`

## Estilo de Codigo
- React 18 com function components (sem classes)
- Styled Components para estilos (nao CSS modules)
- Zustand para estado global, React Query para server state
- Formularios com React Hook Form + Zod
- Imports absolutos via alias `@/` → `src/`

## Estrutura de Componentes
- Componentes em `src/components/<NomeComponente>/`
- Cada pasta contem: index.tsx, styles.ts, types.ts, *.test.tsx
- Paginas em `src/pages/` seguindo rotas do React Router
- Hooks customizados em `src/hooks/`

## Convencoes
- Nomes de componentes: PascalCase
- Nomes de hooks: camelCase com prefixo `use`
- Sem default exports (usar named exports)
- Testes com Testing Library, sem enzyme

Organizando regras com .claude/rules/

Para projetos grandes, você pode dividir instruções em vários arquivos dentro do diretório .claude/rules/. Cada arquivo cobre um tópico e tem um nome descritivo. Regras podem ser condicionais — ativadas apenas quando o Claude trabalha com arquivos que correspondem a um padrão de path.

Estrutura de diretório .claude/rules/ 
your-project/
├── .claude/
│   ├── CLAUDE.md           # Instruções principais do projeto
│   └── rules/
│       ├── code-style.md   # Diretrizes de estilo
│       ├── testing.md      # Convenções de testes
│       ├── security.md     # Requisitos de segurança
│       └── api-design.md   # Regras de design de API

Regras condicionais por path

Regras podem ser restritas a arquivos específicos usando frontmatter YAML com o campo paths. Essas regras só entram no contexto quando o Claude lê arquivos que correspondem aos padrões — economizando espaço no context window:

.claude/rules/api-design.md Regra condicional 
---
paths:
  - "src/api/**/*.ts"
---

# Regras de API

- Todo endpoint deve ter validação de input com Zod
- Usar o formato padrão de resposta de erro
- Incluir comentários OpenAPI nos handlers
- Novos endpoints seguem o padrão de orders.ts
Padrão glob O que corresponde
**/*.ts Todos os arquivos TypeScript em qualquer diretório
src/**/* Todos os arquivos dentro de src/
src/components/*.tsx Componentes React em um diretório específico
**/*.{ts,tsx} Todos os arquivos TS e TSX (brace expansion)

Importando outros arquivos com @

Arquivos CLAUDE.md suportam imports com a sintaxe @path/to/file. Os arquivos importados são expandidos e carregados no contexto junto com o CLAUDE.md que os referência. Paths relativos são resolvidos em relação ao arquivo que contém o import.

CLAUDE.md Usando imports 
# Meu Projeto

Veja @README.md para visão geral do projeto.
Consulte @package.json para comandos npm disponíveis.

# Workflow de Git
Siga as instruções em @docs/git-workflow.md

# Instruções adicionais
- Sempre rodar testes antes de commitar
- Usar Conventional Commits
Import vs. rules/: Imports com @ carregam o conteúdo no início da sessao (aumentam o contexto). Regras em .claude/rules/ com paths carregam sob demanda. Para projetos grandes, prefira rules condicionais para economizar contexto.

CLAUDE.md vs. Auto Memory

Além do CLAUDE.md (que você escreve), existe o auto memory — notas que o proprio Claude escreve para si mesmo ao longo das sessões. São dois sistemas complementares:

CLAUDE.md Auto Memory
Quem escreve Voce O Claude
Conteúdo Instruções e regras Padrões e aprendizados
Armazenamento No repositório do projeto ~/.claude/projects/<projeto>/memory/
Carregamento Integral na sessão Primeiras 200 linhas do MEMORY.md
Usar para Padrões de código, workflows, arquitetura Comandos de build, insights de debug, preferências

Dicas práticas para começar

1 Comece com /init

Execute /init no seu projeto. O Claude analisa o codebase e gera um CLAUDE.md inicial. Refine a partir dai.

2 Itere como um prompt

O CLAUDE.md é injetado como contexto, então trate-o como um prompt que você refina. Teste, observe o comportamento e ajuste.

3 Mantenha conciso

Menos de 200 linhas por arquivo. Se estiver crescendo demais, mova instruções específicas para .claude/rules/ com path scoping.

4 Seja específico e verificável

"Indentação: 2 espaços" funciona. "Formate o código direito" não funciona. O Claude segue instruções concretas com muito mais consistência.

5 Adicione ao .gitignore o local

Adicione CLAUDE.local.md ao .gitignore. Use-o para preferências pessoais que não devem ser compartilhadas com o time.

6 Elimine o que é redundante

Se o Claude já faz algo correto sem a instrução, remova-a. Instruções desnecessárias consomem contexto e diluem as importantes.

Erro comum: Arquivos CLAUDE.md com mais de 200 linhas onde metade das instruções são redundantes ou vagas. O Claude começa a ignorar regras quando o arquivo é muito longo porque as instruções importantes se perdem no ruído. A solução é podar impiedosamente e mover instruções específicas para .claude/rules/.

Compatibilidade com outros agentes

Se o seu repositório já usa AGENTS.md para outros agentes de código, você pode criar um CLAUDE.md que importa o arquivo existente com a sintaxe @, evitando duplicação:

CLAUDE.md Importando AGENTS.md existente 
@AGENTS.md

## Claude Code

- Usar plan mode para mudanças em `src/billing/`
- Sempre rodar `make lint` antes de commitar

Alternativamente, um symlink também funciona se você não precisa adicionar instruções específicas do Claude: ln -s AGENTS.md CLAUDE.md. O comando /init em um repo que já tem AGENTS.md lê o arquivo e incorpora as partes relevantes no CLAUDE.md gerado. Ele também lê configs como .cursorrules e .windsurfrules.
