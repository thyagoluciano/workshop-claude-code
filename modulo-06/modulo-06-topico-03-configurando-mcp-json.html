Modulo 6 . Topico 3

Configurando servidores MCP com .mcp.json

O arquivo .mcp.json e o mecanismo central para declarar quais servidores MCP o Claude Code deve conectar. Entender sua estrutura, escopos e o CLI de gerenciamento e essencial para integrar ferramentas externas de forma consistente -- tanto individualmente quanto em equipe.

O que e o .mcp.json

O .mcp.json e um arquivo JSON posicionado na raiz do seu projeto que declara servidores MCP de escopo de projeto. Quando commitado no repositorio, ele garante que todos os membros da equipe tenham acesso aos mesmos servidores e ferramentas MCP, funcionando como uma especie de "package.json para integracao de ferramentas de IA".

Alem do .mcp.json no projeto, o Claude Code suporta configuracoes em ~/.claude.json (para escopos local e user) e em arquivos gerenciados pela organizacao (managed-mcp.json). A combinacao desses niveis cria uma hierarquia flexivel de configuracao.

Ponto-chave: Por razoes de seguranca, o Claude Code solicita aprovacao antes de usar servidores de escopo project vindos de um .mcp.json. Isso impede que um repositorio malicioso execute servidores automaticamente ao ser clonado. Use claude mcp reset-project-choices para resetar essas aprovacoes.

Estrutura do arquivo .mcp.json

O schema e direto: um objeto raiz contendo a chave mcpServers, onde cada entrada e um servidor nomeado com sua configuracao de transporte.

Servidor stdio (processo local)

.mcp.json -- servidor stdio
{
  "mcpServers": {
    "database": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@bytebase/dbhub", "--dsn", "postgresql://user:pass@localhost:5432/mydb"],
      "env": {
        "CACHE_DIR": "/tmp/dbhub"
      }
    }
  }
}

Servidor HTTP (remoto)

.mcp.json -- servidor HTTP remoto
{
  "mcpServers": {
    "sentry": {
      "type": "http",
      "url": "https://mcp.sentry.dev/mcp"
    },
    "github": {
      "type": "http",
      "url": "https://api.githubcopilot.com/mcp/",
      "headers": {
        "Authorization": "Bearer ${GITHUB_PAT}"
      }
    }
  }
}

Schema completo dos campos

Campo Tipo Descricao
type "stdio" | "http" | "sse" Transporte. stdio e default para comandos locais. "streamable-http" aceito como alias de "http". SSE esta deprecated.
command string Executavel do servidor (apenas stdio). Ex: "npx", "python", "node".
args string[] Argumentos passados ao comando (apenas stdio).
env Record<string, string> Variaveis de ambiente injetadas no processo do servidor (apenas stdio).
url string URL do servidor remoto (apenas http/sse).
headers Record<string, string> Headers HTTP enviados na conexao (apenas http/sse). Util para auth.
headersHelper string Comando shell que retorna headers em JSON via stdout. Para auth dinamica (Kerberos, SSO).
oauth object Configuracao OAuth: clientId, callbackPort, scopes, authServerMetadataUrl.
alwaysLoad boolean Se true, ferramentas deste servidor sao carregadas no contexto imediatamente (ignora Tool Search deferral).

Expansao de variaveis de ambiente

O .mcp.json suporta expansao de variaveis de ambiente, permitindo que equipes compartilhem configuracoes sem expor segredos no repositorio. A sintaxe segue o padrao ${VAR} com suporte a valores default via ${VAR:-default}.

.mcp.json -- variaveis de ambiente com defaults
{
  "mcpServers": {
    "api-server": {
      "type": "http",
      "url": "${API_BASE_URL:-https://api.example.com}/mcp",
      "headers": {
        "Authorization": "Bearer ${API_KEY}"
      }
    }
  }
}

A expansao funciona nos campos command, args, env, url e headers. Se uma variavel obrigatoria (sem default) nao estiver definida no ambiente, o Claude Code falhara ao parsear a configuracao.

Padrao recomendado para equipes: Commite o .mcp.json com ${VAR} para segredos e documente as variaveis necessarias no README do projeto. Cada desenvolvedor define os valores no seu .env ou shell profile. Assim o arquivo e seguro para versionamento e cada membro usa suas proprias credenciais.

Os tres escopos de configuracao

O Claude Code organiza servidores MCP em tres escopos, cada um com proposito e local de armazenamento distintos. Quando um mesmo servidor esta definido em mais de um escopo, o Claude Code conecta apenas a definicao de maior precedencia.

Escopo Carrega em Compartilhado? Armazenado em
Local (default) Projeto atual apenas Nao ~/.claude.json (sob o path do projeto)
Project Projeto atual apenas Sim, via VCS .mcp.json na raiz do projeto
User Todos os projetos Nao ~/.claude.json (secao global)

Quando o mesmo servidor aparece em multiplos escopos, o Claude Code usa a definicao de maior precedencia e ignora as demais.

L Local

Servidores experimentais, credenciais pessoais, testes. Nao poluem o repo nem afetam outros projetos.

P Project

Servidores que toda a equipe precisa: banco de dados do projeto, issue tracker, monitoramento. Commitado no VCS.

U User

Ferramentas pessoais usadas em todos os projetos: servidor de notas, utilitarios de produtividade.

Gerenciando servidores via CLI

O Claude Code oferece um conjunto completo de subcomandos claude mcp para adicionar, listar, inspecionar e remover servidores sem editar JSON manualmente.

Adicionando servidores

Terminal -- adicionar servidor HTTP remoto
# Servidor HTTP remoto (transporte recomendado)
claude mcp add --transport http notion https://mcp.notion.com/mcp

# Com header de autenticacao
claude mcp add --transport http github https://api.githubcopilot.com/mcp/ \
  --header "Authorization: Bearer YOUR_GITHUB_PAT"

# Servidor HTTP com escopo project (grava no .mcp.json)
claude mcp add --transport http --scope project sentry https://mcp.sentry.dev/mcp
Terminal -- adicionar servidor stdio local
# Servidor stdio local (processo filho)
# IMPORTANTE: opcoes ANTES do nome, -- separa args do servidor
claude mcp add --transport stdio --env AIRTABLE_API_KEY=YOUR_KEY airtable \
  -- npx -y airtable-mcp-server

# PostgreSQL via dbhub
claude mcp add --transport stdio db \
  -- npx -y @bytebase/dbhub \
  --dsn "postgresql://readonly:pass@prod.db.com:5432/analytics"

# Servidor user-scope (disponivel em todos os projetos)
claude mcp add --transport http --scope user hubspot https://mcp.hubspot.com/anthropic
Ordem dos argumentos: Todas as flags (--transport, --env, --scope, --header) devem vir antes do nome do servidor. O -- (double dash) separa o nome do servidor do comando e argumentos que serao passados ao processo MCP. Isso evita conflitos entre flags do Claude e flags do servidor.

Adicionando via JSON direto

Terminal -- adicionar via JSON
# Util quando voce ja tem o JSON de configuracao
claude mcp add-json weather-api \
  '{"type":"http","url":"https://api.weather.com/mcp","headers":{"Authorization":"Bearer token"}}'

# Importar servidores do Claude Desktop
claude mcp add-from-claude-desktop

Listando, inspecionando e removendo

Terminal -- gerenciamento de servidores
# Listar todos os servidores configurados
claude mcp list

# Detalhes de um servidor especifico
claude mcp get github

# Remover um servidor
claude mcp remove github

# Dentro do Claude Code: ver status de conexao
/mcp

# Resetar aprovacoes de servidores project-scope
claude mcp reset-project-choices

Referencia rapida: flags do claude mcp add

Flag Descricao Exemplo
--transport Tipo de transporte: http, sse, stdio --transport http
--scope local (default), project, user --scope project
--env Variavel de ambiente para o processo (stdio) --env API_KEY=abc123
--header Header HTTP para conexao (http/sse) --header "Authorization: Bearer ..."
--callback-port Porta fixa para callback OAuth --callback-port 8080
--client-id Client ID para OAuth pre-configurado --client-id your-id
--client-secret Solicita o client secret (input mascarado) --client-secret

Exemplos praticos: .mcp.json para equipes

A seguir, um exemplo completo de .mcp.json que uma equipe poderia commitar na raiz do repositorio. Ele combina servidores remotos (HTTP) e locais (stdio), usando variaveis de ambiente para segredos.

.mcp.json -- configuracao de equipe completa
{
  "mcpServers": {
    // Monitoramento de erros -- autenticacao via OAuth no /mcp
    "sentry": {
      "type": "http",
      "url": "https://mcp.sentry.dev/mcp"
    },

    // GitHub -- cada dev usa seu proprio PAT via variavel de ambiente
    "github": {
      "type": "http",
      "url": "https://api.githubcopilot.com/mcp/",
      "headers": {
        "Authorization": "Bearer ${GITHUB_PAT}"
      }
    },

    // Banco de dados -- servidor stdio local
    "database": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@bytebase/dbhub", "--dsn", "${DATABASE_URL}"]
    },

    // Servidor interno da empresa -- auth via helper script
    "internal-api": {
      "type": "http",
      "url": "https://mcp.internal.company.com",
      "headersHelper": "./scripts/get-mcp-auth.sh"
    }
  }
}
Nota sobre comentarios: JSON puro nao suporta comentarios. O exemplo acima usa comentarios para fins didaticos. Na pratica, remova os comentarios ou use uma ferramenta que suporte JSONC. O Claude Code interpreta o arquivo como JSON estrito.

Variaveis de ambiente especiais

O Claude Code injeta automaticamente variaveis uteis no ambiente dos servidores stdio:

$ CLAUDE_PROJECT_DIR

Caminho absoluto para a raiz do projeto. Permite que o servidor resolva paths relativos sem depender do working directory. Disponivel em process.env ou os.environ.

$ MCP_TIMEOUT

Timeout de conexao do servidor em milissegundos. Default e 5 segundos. Use MCP_TIMEOUT=10000 claude para servidores lentos para iniciar.

$ MAX_MCP_OUTPUT_TOKENS

Limite maximo de tokens na saida de ferramentas MCP. Default: 25.000 tokens. Warning a partir de 10.000. Aumente para ferramentas que retornam datasets grandes.

$ CLAUDE_PLUGIN_ROOT

Para servidores bundled em plugins: aponta para o diretorio raiz do plugin. Use em command e args para referenciar binarios do plugin.

Autenticacao de servidores remotos

Servidores HTTP frequentemente requerem autenticacao. O Claude Code suporta multiplas estrategias, da mais simples a mais sofisticada:

Metodo Quando usar Como configurar
Header estatico API keys simples, tokens de longa duracao --header "Authorization: Bearer ..." ou headers no JSON
OAuth 2.0 Servicos como Sentry, Slack, Notion Autentique via /mcp no Claude Code; tokens armazenados com seguranca
headersHelper Auth dinamica (Kerberos, SSO, tokens curtos) Script shell que retorna JSON com headers para stdout
Variavel de ambiente Segredos no .mcp.json compartilhado ${API_KEY} no JSON, valor definido no ambiente de cada dev

Tool Search: escalando com muitos servidores

Quando voce conecta muitos servidores MCP, a quantidade de ferramentas disponiveis pode consumir contexto desnecessario. O Tool Search (habilitado por default) resolve isso adiando o carregamento das definicoes de ferramentas ate que o Claude realmente precise delas.

Na pratica, apenas os nomes das ferramentas sao carregados na inicializacao da sessao. Quando uma tarefa precisa de uma ferramenta especifica, o Claude usa um mecanismo de busca para encontrar e carregar o schema completo sob demanda. Isso permite conectar dezenas de servidores MCP com impacto minimo na janela de contexto.

Terminal -- configurar Tool Search
# Default: todas as ferramentas MCP sao deferred (recomendado)
claude

# Modo auto: carrega upfront se couber em 10% do contexto
ENABLE_TOOL_SEARCH=auto claude

# Threshold customizado: 5% do contexto
ENABLE_TOOL_SEARCH=auto:5 claude

# Desabilitar completamente (carrega tudo upfront)
ENABLE_TOOL_SEARCH=false claude

Para servidores cujas ferramentas devem estar sempre visiveis (sem busca), use o campo alwaysLoad: true na configuracao do servidor:

.mcp.json -- alwaysLoad para ferramentas criticas
{
  "mcpServers": {
    "core-tools": {
      "type": "http",
      "url": "https://mcp.example.com/mcp",
      "alwaysLoad": true
    }
  }
}

Configuracao gerenciada (Enterprise)

Para organizacoes que precisam de controle centralizado, o Claude Code suporta o arquivo managed-mcp.json, implantado em diretorio de sistema. Quando presente, ele assume controle exclusivo sobre todos os servidores MCP -- usuarios nao podem adicionar, modificar ou usar servidores fora dos definidos nesse arquivo.

Plataforma Caminho do managed-mcp.json
macOS /Library/Application Support/ClaudeCode/managed-mcp.json
Linux / WSL /etc/claude-code/managed-mcp.json
Windows C:\Program Files\ClaudeCode\managed-mcp.json
Atencao: Esses sao caminhos de sistema (nao do diretorio home do usuario) e requerem privilegios de administrador. Sao projetados para implantacao por equipes de TI/DevOps. O formato do arquivo e identico ao .mcp.json padrao.

Ciclo de vida e reconexao

Entender como o Claude Code gerencia o ciclo de vida dos servidores ajuda a diagnosticar problemas de conexao:

↻ HTTP/SSE: reconexao automatica

Ate 5 tentativas com backoff exponencial (1s, 2s, 4s, 8s, 16s). Servidor aparece como "pending" no /mcp durante reconexao. Apos 5 falhas, marcado como "failed".

⚠ Stdio: sem reconexao

Servidores stdio sao processos locais e nao sao reconectados automaticamente. Se o processo morrer, reinicie a sessao ou reconecte via /mcp.

⚙ Atualizacao dinamica

Servidores podem enviar notificacoes list_changed para atualizar ferramentas disponiveis sem desconectar. O Claude Code recarrega automaticamente.

⌛ Startup nao-bloqueante

Servidores conectam em background por default. Excecao: servidores com alwaysLoad: true bloqueiam ate conectar (max 5s).

Boas praticas

1 Commite o .mcp.json

Trate servidores MCP como dependencias do projeto. Commite o .mcp.json com variaveis de ambiente para segredos, nao valores hardcoded.

2 Use o escopo certo

Project para ferramentas da equipe, local para experimentos pessoais, user para utilitarios cross-projeto. Nao polua o escopo errado.

3 Prefira HTTP sobre SSE

O transporte SSE esta deprecated. Use --transport http para servidores remotos. streamable-http e aceito como alias.

4 Verifique confianca

Servidores MCP que buscam conteudo externo podem expor voce a riscos de prompt injection. Verifique a procedencia antes de conectar.

5 Nao nomeie "workspace"

O nome workspace e reservado para uso interno do Claude Code. Servidores com esse nome serao ignorados com um warning.

6 Documente as variaveis

Se o .mcp.json usa ${VAR}, documente quais variaveis cada dev precisa configurar. Inclua defaults com ${VAR:-fallback} quando possivel.
