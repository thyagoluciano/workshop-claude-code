Modulo 6 . Topico 5

Casos de uso: Git, banco de dados, APIs externas, automacoes

O MCP transforma o Claude Code de um assistente de codigo em um agente integrado ao ecossistema de desenvolvimento. Este topico mapeia os casos de uso mais impactantes, com configuracoes reais e criterios para decidir quando usar MCP vs. as ferramentas nativas.

O ecossistema MCP em perspectiva

A documentacao oficial do Claude Code resume bem o criterio de quando conectar um servidor MCP: quando voce se pega copiando dados de outra ferramenta para colar no chat -- seja de um issue tracker, de um dashboard de monitoramento ou de um banco de dados -- e hora de conectar um MCP server. Com ele, o Claude le e age diretamente no sistema de origem.

O ecossistema MCP cresceu rapidamente desde o lancamento do protocolo. Os servidores oficiais de referencia vivem no repositorio modelcontextprotocol/servers (filesystem, git, fetch, memory, sequential-thinking, time). Servidores que eram oficiais mas migraram para manutencao por terceiros (PostgreSQL, Puppeteer, Redis, Sentry, Slack, SQLite) foram movidos para modelcontextprotocol/servers-archived. Alem disso, a comunidade mantém milhares de servidores no ecossistema, catalogados em listas como awesome-mcp-servers.

Dica: Voce pode navegar servidores MCP revisados diretamente no Anthropic Directory. Servidores listados la usam a mesma infraestrutura MCP do Claude Code -- basta executar claude mcp add para conecta-los.

Diagrama: Claude Code como hub de integracao

Claude Code atua como hub central, conectando-se a multiplos sistemas via MCP. Cada servidor expoe ferramentas especificas que o Claude descobre e usa sob demanda.

Caso 1: Git e GitHub

O que o Claude Code ja faz nativamente

O Claude Code possui ferramentas nativas de Git integradas. Ele ja consegue criar commits, branches, ler diffs, executar git log, git status e outros comandos diretamente via Bash. Para operacoes locais de repositorio, voce nao precisa de nenhum servidor MCP.

O que o MCP server do GitHub adiciona

O servidor MCP do GitHub (mantido pelo proprio GitHub em github/github-mcp-server) adiciona acesso direto a API do GitHub: criar e revisar PRs, gerenciar issues, buscar repositorios, visualizar actions/workflows, e interagir com o ecossistema GitHub sem precisar copiar/colar URLs ou outputs do gh CLI.

Configuracao: GitHub MCP Server (HTTP remoto)
// Terminal: adicionar o servidor GitHub via CLI
claude mcp add --transport http github \
  https://api.githubcopilot.com/mcp/ \
  --header "Authorization: Bearer YOUR_GITHUB_PAT"

// Ou via .mcp.json no projeto:
{
  "mcpServers": {
    "github": {
      "type": "http",
      "url": "https://api.githubcopilot.com/mcp/",
      "headers": {
        "Authorization": "Bearer ${GITHUB_PAT}"
      }
    }
  }
}
Capacidade Nativo (Git/Bash) MCP GitHub Server
Commits e branches locais Sim Desnecessario
Diffs e log do repositorio Sim Desnecessario
Criar/revisar PRs na API Via gh CLI Direto, sem CLI
Gerenciar issues Via gh CLI Direto, com busca
Listar PRs/reviews atribuidos Requer scripting Nativo
Buscar codigo cross-repo Nao Sim
Quando usar MCP GitHub: Quando voce precisa interagir com a plataforma GitHub (PRs, issues, reviews, actions) e nao apenas com o repositorio local. Se o workflow e puramente local (commit, branch, merge), as ferramentas nativas bastam.

Caso 2: Banco de dados

Conectar o Claude Code a um banco de dados via MCP e um dos casos de uso mais transformadores. Em vez de copiar schemas e resultados de queries para o chat, o Claude consulta diretamente, inspeciona a estrutura e correlaciona dados com o codigo.

PostgreSQL com DBHub

A documentacao oficial do Claude Code recomenda o @bytebase/dbhub como servidor para PostgreSQL. Ele suporta leitura de schema, execucao de queries e inspecao de tabelas.

Configuracao: PostgreSQL via DBHub
// Terminal: adicionar servidor PostgreSQL
claude mcp add --transport stdio db \
  -- npx -y @bytebase/dbhub \
  --dsn "postgresql://readonly:pass@localhost:5432/myapp"

// .mcp.json equivalente:
{
  "mcpServers": {
    "db": {
      "command": "npx",
      "args": [
        "-y", "@bytebase/dbhub",
        "--dsn", "postgresql://readonly:pass@localhost:5432/myapp"
      ]
    }
  }
}

SQLite para desenvolvimento local

Configuracao: SQLite
// Terminal:
claude mcp add --transport stdio app-db \
  -- npx -y @modelcontextprotocol/server-sqlite \
  "./data/app.db"

// .mcp.json equivalente:
{
  "mcpServers": {
    "app-db": {
      "command": "npx",
      "args": [
        "-y", "@modelcontextprotocol/server-sqlite",
        "./data/app.db"
      ]
    }
  }
}
Seguranca critica: Sempre use credenciais read-only ao conectar bancos de dados de producao. Nunca habilite escrita contra producao a partir de um loop de agente. Aponte para uma read replica ou um snapshot local. Para ambientes de desenvolvimento, escrita pode ser aceitavel com as devidas precaucoes.

Com o banco conectado, voce pode fazer perguntas em linguagem natural:

Q Inspecao de schema

"Mostra o schema da tabela orders e suas relacoes com outras tabelas"

Q Analise de dados

"Qual o total de receita deste mes agrupado por categoria de produto?"

Q Debug de dados

"Encontre usuarios que nao fizeram compra nos ultimos 90 dias"

Q Correlacao codigo-dados

"Compare o schema real com os models do Prisma e identifique divergencias"

Caso 3: APIs de workflow (Slack, Sentry, Linear, Jira)

Ferramentas de workflow do dia a dia sao um dos maiores ganhos com MCP. Em vez de alternar entre abas no navegador, o Claude Code interage diretamente com issue trackers, plataformas de monitoramento e canais de comunicacao.

Sentry -- monitoramento de erros

Configuracao: Sentry (HTTP remoto com OAuth)
// Terminal: adicionar o Sentry via HTTP
claude mcp add --transport http sentry \
  https://mcp.sentry.dev/mcp

// Autenticar via OAuth no Claude Code:
/mcp
// Siga o fluxo de autenticacao no browser

// .mcp.json equivalente:
{
  "mcpServers": {
    "sentry": {
      "type": "http",
      "url": "https://mcp.sentry.dev/mcp"
    }
  }
}

Slack -- comunicacao

Configuracao: Slack (HTTP remoto com OAuth e scopes restritos)
// .mcp.json com restricao de scopes:
{
  "mcpServers": {
    "slack": {
      "type": "http",
      "url": "https://mcp.slack.com/mcp",
      "oauth": {
        "scopes": "channels:read chat:write search:read"
      }
    }
  }
}

Notion, Linear, Jira e outros

Configuracao: Notion (HTTP remoto)
// Terminal:
claude mcp add --transport http notion \
  https://mcp.notion.com/mcp
Fluxo tipico com APIs de workflow: O Claude Code pode receber um prompt como "Implemente a feature descrita na issue JIRA ENG-4521 e crie um PR no GitHub". Com os servidores MCP de Jira e GitHub conectados, ele le a issue, implementa o codigo, e abre o PR -- tudo em uma unica sessao, sem copiar/colar.

Caso 4: Browser e Web

Playwright -- automacao de browser

O servidor MCP do Playwright permite ao Claude Code controlar um browser headless: navegar paginas, interagir com elementos, tirar screenshots e executar testes automatizados. A recomendacao atual e usar o Playwright MCP em vez do Puppeteer (que foi depreciado como servidor oficial).

Configuracao: Playwright MCP
claude mcp add --transport stdio playwright \
  -- npx -y @playwright/mcp@latest

Brave Search -- busca na web

O Brave Search MCP permite ao Claude Code buscar informacoes atualizadas na web durante a conversa. Util para consultar documentacao, verificar APIs, e pesquisar solucoes para bugs. Possui tier gratuito com 2.000 queries/mes.

Configuracao: Brave Search
{
  "mcpServers": {
    "brave-search": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-brave-search"],
      "env": {
        "BRAVE_API_KEY": "${BRAVE_API_KEY}"
      }
    }
  }
}

Caso 5: Servidores MCP customizados

Um dos maiores diferenciais do MCP e a capacidade de criar servidores para APIs internas e proprietarias. Se sua empresa possui um sistema de deploy, um monolito legado, ou qualquer API interna, voce pode expor essas capacidades ao Claude Code via MCP.

Scaffold com o plugin oficial

O Claude Code oferece um plugin oficial para criar servidores MCP interativamente:

Criando um servidor MCP com o plugin oficial
// Instalar o plugin de desenvolvimento MCP
/plugin install mcp-server-dev@claude-plugins-official

// Recarregar plugins na sessao atual
/reload-plugins

// Executar o skill de build
/mcp-server-dev:build-mcp-server
// O Claude pergunta sobre o caso de uso e gera o servidor

Exemplo: API interna de deploy

.mcp.json: servidor customizado para API interna
{
  "mcpServers": {
    "deploy-api": {
      "command": "node",
      "args": ["./tools/mcp-deploy-server/index.js"],
      "env": {
        "DEPLOY_API_URL": "${DEPLOY_API_URL}",
        "DEPLOY_TOKEN": "${DEPLOY_TOKEN}"
      }
    }
  }
}
Variavel util: O Claude Code define automaticamente CLAUDE_PROJECT_DIR no ambiente do servidor MCP spawned, apontando para a raiz do projeto. Seu servidor pode usar essa variavel para resolver caminhos relativos sem depender do diretorio de trabalho.

Quando usar MCP vs. ferramentas nativas

Nem tudo precisa de MCP. O Claude Code ja possui ferramentas nativas poderosas (Read, Write, Edit, Bash, Grep, Glob, etc.). A regra geral: se a ferramenta nativa resolve, use-a. MCP e para quando voce precisa ir alem do terminal e do filesystem local.

Arvore de decisao para escolher entre ferramentas nativas, servidores MCP existentes, scripts Bash ou criacao de servidor customizado.

Cenarios concretos: combinando multiplos servidores

O verdadeiro poder do MCP aparece quando voce combina multiplos servidores em uma unica sessao. Abaixo, cinco cenarios reais que demonstram esse potencial.

1 Triage de bug em producao

Servidores: Sentry + PostgreSQL + GitHub
O Claude consulta os erros recentes no Sentry, correlaciona com dados do banco de producao para entender o impacto, localiza o codigo responsavel, e abre uma issue no GitHub com toda a analise.

"Verifique os erros criticos das ultimas 24h no Sentry, identifique quantos usuarios foram afetados consultando o banco, e abra uma issue no GitHub com a analise completa."

2 Feature end-to-end a partir de uma issue

Servidores: Jira/Linear + GitHub + PostgreSQL
O Claude le os requisitos na issue do tracker, inspeciona o schema do banco para entender a modelagem, implementa a feature, cria migrations se necessario, roda testes e abre o PR.

"Implemente a feature descrita na issue LINEAR ENG-234, incluindo migration no banco, testes, e abra um PR no GitHub."

3 Documentacao a partir de dados reais

Servidores: PostgreSQL + Notion/Slack
O Claude inspeciona o schema real do banco, compara com a documentacao existente no Notion, identifica discrepancias e atualiza a documentacao. Opcionalmente notifica o time no Slack.

"Compare o schema atual do banco com a documentacao no Notion e atualize o que estiver desatualizado. Avise no canal #eng do Slack."

4 Validacao visual de deploy

Servidores: Playwright + Sentry
Apos um deploy, o Claude navega pelas paginas criticas da aplicacao via Playwright, tira screenshots, verifica se ha erros novos no Sentry, e reporta o status.

"Navegue pela homepage, pagina de login e dashboard no staging. Tire screenshots e verifique se surgiram erros novos no Sentry apos o deploy."

5 Onboarding de desenvolvedor

Servidores: GitHub + PostgreSQL + Memory
O Claude le a estrutura do repositorio, inspeciona o banco, busca PRs recentes para entender padroes do time, e gera um guia de onboarding personalizado usando o servidor Memory para persistir o conhecimento adquirido.

"Analise este repositorio, o schema do banco, e os ultimos 20 PRs mergeados. Gere um guia de onboarding para um novo dev."

Exemplo de .mcp.json completo para um time

Abaixo, um exemplo realista de .mcp.json de projeto que combina multiplos servidores para um time de desenvolvimento. Este arquivo deve ser commitado no repositorio (scope project).

.mcp.json -- configuracao completa de time
{
  "mcpServers": {
    // Plataforma de codigo -- PRs, issues, reviews
    "github": {
      "type": "http",
      "url": "https://api.githubcopilot.com/mcp/",
      "headers": {
        "Authorization": "Bearer ${GITHUB_PAT}"
      }
    },

    // Banco de dados -- read-only em producao
    "db-prod": {
      "command": "npx",
      "args": [
        "-y", "@bytebase/dbhub",
        "--dsn", "${DATABASE_READONLY_URL}"
      ]
    },

    // Monitoramento de erros
    "sentry": {
      "type": "http",
      "url": "https://mcp.sentry.dev/mcp"
    },

    // Comunicacao do time
    "slack": {
      "type": "http",
      "url": "https://mcp.slack.com/mcp",
      "oauth": {
        "scopes": "channels:read chat:write search:read"
      }
    },

    // API interna de deploy (servidor customizado do time)
    "deploy": {
      "command": "node",
      "args": ["${CLAUDE_PROJECT_DIR:-.}/tools/mcp-deploy/index.js"],
      "env": {
        "DEPLOY_API_URL": "${DEPLOY_API_URL}"
      }
    }
  }
}
Variaveis de ambiente em .mcp.json: O Claude Code suporta expansao de variaveis no formato ${VAR} e ${VAR:-default}. Isso permite commitar o .mcp.json no repositorio sem expor credenciais. Cada desenvolvedor define suas variaveis localmente.

Impacto no contexto: Tool Search

Uma preocupacao natural e: "se eu conectar 10 servidores MCP, isso nao vai poluir o contexto?" A resposta e nao, gracas ao mecanismo de Tool Search, habilitado por padrao no Claude Code.

Com Tool Search ativo, apenas os nomes das ferramentas MCP sao carregados na inicializacao da sessao. Os schemas completos sao carregados sob demanda, somente quando o Claude determina que precisa deles para a tarefa atual. Isso significa que adicionar mais servidores MCP tem impacto minimo na janela de contexto.

1 Sem Tool Search

Todos os schemas de todas as ferramentas MCP sao carregados no contexto desde o inicio. 10 servidores com 50 ferramentas = centenas de tokens consumidos.

2 Com Tool Search (padrao)

So nomes sao carregados. Schemas entram sob demanda. 10 servidores com 50 ferramentas = impacto minimo ate o momento do uso.

Para autores de servidores MCP: Com Tool Search ativo, o campo instructions do servidor se torna ainda mais importante. Ele ajuda o Claude a entender quando buscar as ferramentas do seu servidor. Mantenha as descricoes concisas (limite de 2KB) e coloque informacoes criticas no inicio.

Boas praticas para uso de MCP em projetos

A Credentials via variaveis

Nunca coloque tokens diretamente no .mcp.json. Use ${VAR} para referenciar variaveis de ambiente. O arquivo pode ser commitado com seguranca.

B Read-only em producao

Bancos de dados de producao devem ser acessados com credenciais somente leitura. Use read replicas ou snapshots. Escrita so em ambientes de dev.

C Scopes de projeto

Use --scope project para servidores que o time inteiro deve usar. O .mcp.json e commitado e garante consistencia entre devs.

D Verifique a confianca

Servidores que buscam conteudo externo podem expor voce a riscos de prompt injection. So conecte servidores em que voce confia (Topico 4: Allowlist).

E OAuth com scopes restritos

Ao conectar servidores remotos com OAuth, restrinja os scopes ao minimo necessario usando oauth.scopes na configuracao.

F Teste com /mcp

Use o comando /mcp dentro do Claude Code para verificar o status de todos os servidores conectados, numero de ferramentas expostas e eventuais erros.

Bonus: Channels -- MCP servers que enviam mensagens

Alem de expor ferramentas, um servidor MCP pode funcionar como um channel, enviando mensagens diretamente para a sessao do Claude Code. Isso permite que o Claude reaja a eventos externos em tempo real: resultados de CI, alertas de monitoramento, mensagens de Telegram/Discord, ou webhooks.

Para habilitar, o servidor declara a capability claude/channel e voce inicia o Claude Code com a flag --channels. Esse mecanismo transforma o Claude Code de um assistente reativo (responde quando voce pergunta) em um agente proativo (age quando algo acontece).
