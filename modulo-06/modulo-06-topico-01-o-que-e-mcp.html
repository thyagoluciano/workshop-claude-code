Modulo 6 . Topico 1

O que e MCP e como ampliar as capacidades do Claude Code

O Model Context Protocol (MCP) e um padrao aberto que permite ao Claude Code se conectar a ferramentas externas, bancos de dados e APIs de forma padronizada. Pense nele como o USB-C da inteligencia artificial -- um conector universal que elimina integracoes ad-hoc.

O problema: N x M integracoes

Antes do MCP, cada aplicacao de IA precisava construir integracoes customizadas para cada ferramenta que quisesse acessar. Se voce tem N aplicacoes (Claude Code, Claude Desktop, IDEs com copilots, etc.) e M ferramentas (GitHub, Postgres, Sentry, Jira, etc.), o resultado e N x M integracoes -- cada uma com seu proprio protocolo, formato de dados e mecanismo de autenticacao.

O MCP resolve isso introduzindo uma camada de abstracoes padronizada. Cada aplicacao implementa o protocolo MCP uma unica vez (como client), e cada ferramenta expoe um MCP server. O resultado: N + M implementacoes em vez de N x M.

MCP transforma o problema combinatorio N x M em um problema linear N + M, assim como USB-C padronizou conexoes fisicas.

Definicao oficial: O Model Context Protocol (MCP) e um padrao aberto que permite integracao perfeita entre aplicacoes de LLM e fontes de dados e ferramentas externas. Ele fornece uma forma padronizada de conectar LLMs ao contexto que precisam.
-- modelcontextprotocol.io

Arquitetura: Host, Client, Server

O MCP segue uma arquitetura host-client-server. O host (como o Claude Code) coordena e gerencia multiplos clients MCP, onde cada client mantem uma conexao dedicada com um MCP server especifico. Os servers, por sua vez, expoem ferramentas, recursos e prompts para os clients consumirem.

O host (Claude Code) cria um client MCP para cada server configurado. Cada client mantem uma conexao dedicada via stdio ou HTTP.

Papeis na arquitetura

Componente Papel Exemplo
MCP Host Aplicacao de IA que coordena e gerencia multiplos MCP clients. Mantem limites de seguranca e isolamento entre clients. Claude Code, Claude Desktop, IDEs com suporte MCP
MCP Client Componente interno ao host que mantem conexao 1:1 com um server. Obtem contexto (tools, resources, prompts) do server e o disponibiliza ao host. Um client para GitHub, outro para Postgres, outro para Sentry
MCP Server Programa que expoe capacidades via o protocolo MCP. Pode ser local (processo no seu computador) ou remoto (URL na nuvem). npx @modelcontextprotocol/server-github, URL remota do Sentry
Detalhe tecnico: A comunicacao entre client e server usa JSON-RPC 2.0. O protocolo e stateful -- client e server negociam capacidades durante a inicializacao (initialize handshake) e mantem sessao ativa durante toda a interacao. As capacidades declaradas determinam quais primitivas e funcionalidades estarao disponiveis.

As duas camadas do protocolo

O MCP opera em duas camadas distintas que separam o que e transmitido de como e transmitido:

D Camada de Dados (Data Layer)

Define o protocolo JSON-RPC para comunicacao client-server: gerenciamento de ciclo de vida, negociacao de capacidades e as primitivas centrais (tools, resources, prompts, notifications). E o "contrato" do que pode ser feito.

T Camada de Transporte (Transport Layer)

Define os mecanismos de comunicacao e canais de troca de dados: como a conexao e estabelecida, como as mensagens sao enquadradas (framing) e como a autorizacao funciona. Dois transportes principais: stdio e Streamable HTTP.

As tres primitivas do MCP

Servers MCP expoem capacidades atraves de tres primitivas fundamentais. Cada uma tem um papel distinto no ecossistema e determina como o modelo interage com o mundo externo:

T Tools

Funcoes executaveis que o modelo pode chamar para realizar acoes ou obter informacoes. Equivalente a funcoes/endpoints que o LLM invoca. Controladas pelo modelo (model-controlled).

Exemplos: query_database, create_issue, send_email

R Resources

Dados estruturados ou conteudo que fornecem contexto adicional ao modelo. Semelhante a endpoints GET -- expoe dados para leitura sem efeitos colaterais. Controlados pela aplicacao (application-controlled).

Exemplos: repo://files, db://schema, docs://api-reference

P Prompts

Templates pre-definidos de instrucoes que guiam a interacao com o modelo. Servers podem expor prompts parametrizaveis que o usuario seleciona. Controlados pelo usuario (user-controlled).

Exemplos: review_code, explain_error, generate_migration
Primitiva Controlado por Analogia REST Direcao
Tools Modelo (LLM decide quando chamar) POST endpoints -- executa acoes Server expoe, modelo invoca
Resources Aplicacao (host/client decide o que carregar) GET endpoints -- le dados Server expoe, aplicacao consome
Prompts Usuario (seleciona qual template usar) Templates parametrizaveis Server define, usuario seleciona
Na pratica no Claude Code: Quando voce configura um MCP server (ex: GitHub), as tools desse server aparecem como funcoes que o Claude pode chamar autonomamente durante a conversa. Voce vera no output algo como mcp__github__create_issue -- o prefixo mcp__ identifica que a ferramenta veio de um server MCP.

Claude Code como MCP Host

O Claude Code atua como MCP host, o que significa que ele:

1 Gerencia multiplos clients

Cria e mantem um MCP client para cada server configurado. Cada client tem sua propria conexao e isolamento de seguranca, impedindo que um server acesse dados de outro.

2 Negocia capacidades

Na inicializacao, faz handshake com cada server para descobrir quais tools, resources e prompts estao disponiveis. So usa o que o server declarou suportar.

3 Injeta tools no contexto

Disponibiliza as tools dos servers configurados para o modelo. O LLM ve essas tools como funcoes que pode chamar, junto com as tools nativas do Claude Code (Read, Write, Bash, etc.).

4 Controla permissoes

Permite que voce defina allowlists para tools MCP via /permissions ou .claude/settings.json, controlando o que o modelo pode executar sem confirmacao.

Capacidades nativas vs. MCP

Sem MCP, o Claude Code ja possui capacidades poderosas: ler e escrever arquivos, executar comandos no terminal, buscar na web. O MCP expande essas capacidades para qualquer sistema que tenha um server MCP:

Capacidade Nativa (sem MCP) Com MCP
Arquivos Ler/escrever arquivos locais Acessar arquivos em repos remotos, Google Drive, S3
Banco de dados Executar queries via Bash + CLI do banco Query tipada via server MCP com schema awareness
Issue tracking Ler issues via web ou CLI Criar, atualizar, fechar issues diretamente no Jira/Linear/GitHub
Monitoramento Ler logs via terminal Consultar Sentry, Datadog, PagerDuty com contexto estruturado
Design Nao disponivel Ler componentes e tokens do Figma para gerar codigo fiel

Configuracao basica: claude mcp add

Adicionar um MCP server ao Claude Code e feito pelo comando claude mcp add. Veja a sintaxe e exemplos:

Terminal -- Sintaxe do comando
# Sintaxe geral
claude mcp add [options] <name> -- <command> [args...]

# Exemplo: adicionar server do GitHub (local, via stdio)
claude mcp add --transport stdio github -- \
  npx -y @modelcontextprotocol/server-github

# Exemplo: server com variavel de ambiente
claude mcp add --transport stdio --env AIRTABLE_API_KEY=sua_key \
  airtable -- npx -y airtable-mcp-server

# Exemplo: server remoto via URL (Streamable HTTP)
claude mcp add --transport http sentry -- \
  https://mcp.sentry.dev/sse

Apos adicionar, o server fica registrado e sera conectado automaticamente quando o Claude Code iniciar. Voce tambem pode configurar servers via arquivo .mcp.json na raiz do projeto, o que permite compartilhar a configuracao com o time:

.mcp.json -- Configuracao versionada no repositorio
{
  "mcpServers": {
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_TOKEN": "${GITHUB_TOKEN}"
      }
    },
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres", "postgresql://localhost:5432/mydb"]
    }
  }
}
Dica de debug: Use a flag --mcp-debug ao iniciar o Claude Code para ver logs detalhados da comunicacao MCP. Isso ajuda a identificar problemas de configuracao, falhas de conexao ou erros de handshake com os servers.
claude --mcp-debug

Escopos de configuracao

O Claude Code permite configurar MCP servers em tres escopos diferentes, controlando visibilidade e compartilhamento:

L Local (padrao)

Armazenado em ~/.claude.json por projeto. Privado para voce, nao compartilhado. Use para servers pessoais ou com credenciais sensiveis.

P Projeto

Armazenado em .mcp.json na raiz do repo. Versionado no Git, disponivel para todo o time. Ideal para servers compartilhados do projeto.

U Usuario

Armazenado em ~/.claude.json globalmente. Disponivel em todos os projetos do usuario. Use para servers que voce quer em todo lugar (ex: Notion pessoal).

Ciclo de vida de uma chamada MCP

Quando voce pede ao Claude Code algo que requer um MCP server, o fluxo segue estas etapas:

Etapa O que acontece Quem faz
1. Discover Na inicializacao, o client faz handshake com o server e recebe a lista de tools/resources/prompts disponiveis Client + Server
2. Plan O modelo analisa sua mensagem e decide qual tool MCP usar, com quais argumentos LLM (modelo)
3. Approve Claude Code apresenta a chamada para sua aprovacao (a menos que esteja na allowlist) Host + Usuario
4. Execute O client envia a chamada JSON-RPC para o server, que executa a acao e retorna o resultado Client + Server
5. Integrate O resultado e injetado no contexto do modelo, que o utiliza para formular a resposta Host + LLM

Servers locais vs. remotos

O MCP suporta dois modelos de deployment para servers, cada um com trade-offs distintos:

L Server Local (stdio)

Roda como processo local na sua maquina. Comunicacao via stdin/stdout. Latencia minima, sem dependencia de rede. Serve um unico client. Voce gerencia instalacao e atualizacoes.

R Server Remoto (HTTP)

Roda em uma URL na nuvem. Comunicacao via Streamable HTTP. Pode servir multiplos clients. O vendor gerencia infraestrutura. Suporte nativo a OAuth para autenticacao.

Tendencia: Servers remotos estao se tornando a opcao preferida para ferramentas SaaS. O vendor (ex: Sentry, Linear, Notion) hospeda o server MCP e voce so precisa adicionar a URL ao Claude Code. Sem instalacao, sem atualizacoes manuais, sem processos locais consumindo recursos.

Modelo de seguranca

O MCP implementa seguranca em multiplas camadas:

1 Isolamento de clients

Cada MCP client roda isolado. Um server malicioso nao consegue acessar dados ou tools de outro server. O host mantem fronteiras de seguranca entre clients.

2 Negociacao de capacidades

Servers so podem oferecer o que declararam no handshake. Clients so aceitam primitivas que o server explicitamente suporta. Nao ha escalacao implicita de permissoes.

3 Aprovacao do usuario

O Claude Code pede sua aprovacao antes de executar tools MCP (a menos que estejam na allowlist). Voce controla exatamente o que o modelo pode fazer.

4 Permissoes granulares

Via /permissions ou .claude/settings.json, voce define allowlists por tool especifica: mcp__github__create_issue pode ser permitido enquanto mcp__github__delete_repo requer aprovacao explicita.

Resumo

O que voce precisa lembrar:
  • MCP e um padrao aberto (JSON-RPC 2.0) que padroniza como aplicacoes de IA se conectam a ferramentas externas
  • A arquitetura e host → client → server, onde o Claude Code e o host
  • Tres primitivas: Tools (modelo executa), Resources (aplicacao le), Prompts (usuario seleciona)
  • Resolve o problema N x M de integracoes, reduzindo para N + M
  • Servers podem ser locais (stdio) ou remotos (HTTP)
  • Configuracao via claude mcp add ou .mcp.json no repositorio
  • Seguranca por isolamento de clients, negociacao de capacidades e aprovacao do usuario
