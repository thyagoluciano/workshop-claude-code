Modulo 6 . Topico 4

Allowlist de servidores: tratar MCP como dependencia de software

Cada servidor MCP que voce adiciona ao Claude Code e codigo de terceiros com acesso ao seu ambiente. O mesmo rigor que voce aplica a dependencias npm ou pip -- versionamento, auditoria, lockfiles -- deve ser aplicado a servidores MCP. Neste topico, exploramos como governar esse ecossistema.

MCP servers sao dependencias de software

Quando voce roda npm install algum-pacote, esta confiando que aquele pacote nao vai exfiltrar dados, corromper arquivos ou injetar codigo malicioso. A mesma logica se aplica a servidores MCP: cada servidor que voce configura ganha acesso a ferramentas que o Claude Code pode invocar, incluindo leitura de arquivos, execucao de comandos e chamadas de rede.

A diferenca critica e que um pacote npm roda no seu processo com permissoes limitadas, enquanto um servidor MCP opera como uma ponte bidirecional entre o LLM e recursos externos. Pesquisas recentes identificaram vulnerabilidades fundamentais no protocolo MCP, incluindo ausencia de atestacao de capacidades, amostragem bidirecional sem autenticacao de origem e propagacao implicita de confianca em configuracoes multi-servidor.

Risco real: Pesquisadores documentaram que tool poisoning -- onde instrucoes maliciosas sao embutidas em metadados de ferramentas MCP -- e a vulnerabilidade mais prevalente e impactante do lado do cliente. Um servidor malicioso pode influenciar o comportamento do agente sem tocar no codigo da aplicacao, apenas manipulando descricoes de ferramentas que o LLM interpreta como texto livre.
-- "Breaking the Protocol: Security Analysis of MCP" (arXiv, Jan/2025)

Diagrama: cadeia de suprimentos MCP

O diagrama abaixo ilustra como um servidor MCP malicioso ou comprometido pode impactar a seguranca do seu ambiente de desenvolvimento, comparando com a cadeia de dependencias tradicional.

Sem governanca, servidores MCP pulam todas as camadas de protecao que aplicamos a dependencias tradicionais.

Vetores de ataque de servidores MCP nao confiados

Entender os riscos especificos ajuda a justificar o investimento em governanca. Estes sao os principais vetores documentados por pesquisadores de seguranca:

1 Tool Poisoning

Instrucoes maliciosas embutidas nas descricoes de ferramentas. O LLM interpreta essas descricoes como texto livre e pode ser manipulado a executar acoes nao intencionadas.

2 Exfiltracao de dados

Um servidor malicioso pode capturar dados sensiveis do contexto (variaveis de ambiente, tokens, codigo proprietario) e envia-los para endpoints externos.

3 Prompt Injection via respostas

As respostas de ferramentas MCP sao injetadas no contexto do LLM. Um servidor pode retornar payloads que manipulam o comportamento do Claude nas proximas acoes.

4 Escalacao de privilegios

Em configuracoes multi-servidor, um servidor comprometido pode explorar a propagacao implicita de confianca para acessar recursos de outros servidores.

Dado alarmante: Pesquisadores encontraram mais de 1.800 servidores MCP na internet publica sem autenticacao habilitada. O suporte a OAuth no MCP so foi adicionado em marco de 2025, e muitos servidores existentes ainda nao o implementam.
-- "Securing the Model Context Protocol" (arXiv, Nov/2025)

Modelo de permissoes do Claude Code para MCP

O Claude Code implementa multiplas camadas de protecao para servidores MCP. Antes de usar qualquer ferramenta MCP, o Claude solicita permissao explicita ao usuario. Voce pode gerenciar isso de tres formas:

Abordagem Como funciona Quando usar
Permissao interativa Claude pede confirmacao antes de cada chamada a uma ferramenta MCP Exploracao, primeiros usos, servidores nao familiares
/permissions allowlist Adicione ferramentas especificas a allowlist via /permissions, ex: mcp__github__create_issue Ferramentas confiadas que voce usa frequentemente
Auto Mode + Sandbox Um classificador avalia riscos automaticamente; sandbox limita o escopo de danos Automacao em CI/CD, pipelines nao interativos

A combinacao de permissoes granulares por ferramenta com sandboxing no nivel do sistema operacional cria uma defesa em profundidade. Mas essas protecoes sao reativas -- a allowlist de servidores e a medida preventiva que impede servidores nao aprovados de sequer carregar no ambiente.

Fluxo de decisao: allowlist de servidores

O fluxo de verificacao e: denylist primeiro (bloqueio absoluto), depois allowlist (aprovacao necessaria), e por fim permissao por ferramenta.

Configurando allowlists com managed-settings e managed-mcp

O Claude Code oferece dois mecanismos complementares para governanca corporativa de servidores MCP. Ambos requerem privilegios de administrador e sao implantados em diretorios de sistema (nao no home do usuario), garantindo que nao possam ser sobrescritos por configuracoes locais.

Opcao 1: Controle exclusivo com managed-mcp.json

Quando voce implanta um arquivo managed-mcp.json, ele assume controle exclusivo sobre todos os servidores MCP. Usuarios nao podem adicionar, modificar ou usar nenhum servidor alem dos definidos neste arquivo.

/Library/Application Support/ClaudeCode/managed-mcp.json (macOS) JSON 
{
  "mcpServers": {
    "github": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github@1.2.0"],
      "env": {
        "GITHUB_TOKEN": "${GITHUB_TOKEN}"
      }
    },
    "postgres": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres@0.6.1"],
      "env": {
        "DATABASE_URL": "${DATABASE_URL}"
      }
    },
    "internal-api": {
      "type": "sse",
      "url": "https://mcp.internal.company.com/sse"
    }
  }
}
Caminhos por sistema operacional:
macOS: /Library/Application Support/ClaudeCode/managed-mcp.json
Linux: /etc/claude-code/managed-mcp.json
Windows: %ProgramData%\ClaudeCode\managed-mcp.json
Estes sao diretorios de sistema que requerem privilegios de administrador.

Opcao 2: Allowlist/denylist com managed-settings.json

Quando voce quer permitir que desenvolvedores adicionem seus proprios servidores, mas dentro de limites definidos, use allowedMcpServers e deniedMcpServers no arquivo de configuracoes gerenciadas. Cada entrada filtra por um de tres criterios: serverName, serverCommand ou serverUrl.

/Library/Application Support/ClaudeCode/managed-settings.json (macOS) JSON 
{
  // Servidores que os devs podem usar (allowlist)
  "allowedMcpServers": [
    { "serverName": "github" },
    { "serverName": "postgres" },
    { "serverName": "linear" },
    { "serverCommand": "npx" },
    { "serverUrl": "https://mcp.internal.company.com/*" }
  ],

  // Servidores bloqueados (denylist - precedencia absoluta)
  "deniedMcpServers": [
    { "serverCommand": "docker" },
    { "serverUrl": "http://*" },
    { "serverName": "*-unofficial" }
  ]
}
Regra critica: A denylist tem precedencia absoluta. Se um servidor casa com uma entrada na denylist (por nome, comando ou URL), ele sera bloqueado mesmo que esteja na allowlist. Isso garante que restricoes de seguranca nunca sejam acidentalmente sobrescritas.

Escopos de configuracao e precedencia

Servidores MCP no Claude Code podem ser configurados em multiplos escopos. O escopo escolhido controla em quais projetos o servidor e carregado e se a configuracao e compartilhada com o time.

Escopo Arquivo Compartilhado? Precedencia
Managed managed-mcp.json ou managed-settings.json Sim (IT deploy) Mais alta (nao pode ser sobrescrito)
Projeto .mcp.json na raiz do repo Sim (via Git) Media
Usuario ~/.claude.json Nao (local) Mais baixa

Configuracoes de escopos superiores (Managed) sobrescrevem escopos inferiores (Usuario/Projeto) para valores escalares. Arrays sao concatenados entre escopos. Isso significa que um admin pode adicionar servidores obrigatorios via managed sem impedir os que o projeto define via .mcp.json -- desde que estejam na allowlist.

Principio do menor privilegio aplicado a MCP

Assim como voce nao da permissao de root a uma aplicacao web, cada servidor MCP deveria operar com o minimo de acesso necessario. Aplique estas praticas:

✎ Revise as ferramentas expostas

Antes de habilitar um servidor, inspecione quais ferramentas ele expoe. Use claude mcp list para ver todos os servidores configurados e suas ferramentas. Desconfie de servidores que expoem ferramentas excessivas ou nao relacionadas ao seu proposito.

🔒 Limite permissoes por ferramenta

Use /permissions para conceder acesso apenas as ferramentas especificas que voce precisa, nao ao servidor inteiro. Ex: permita mcp__github__search_repositories mas nao mcp__github__delete_repository.

🛡 Use sandbox para isolamento

O sandboxing do Claude Code cria fronteiras no nivel do sistema operacional, limitando acesso ao filesystem e a rede. Combine allowlist de servidores com sandbox para defesa em profundidade.

👁 Monitore inputs e outputs

Preste atencao ao que o Claude esta enviando para ferramentas MCP e o que esta recebendo de volta. A Anthropic recomenda: conecte apenas servidores confiados e revise os inputs e outputs das ferramentas.

Analogia completa: MCP servers vs. dependencias tradicionais

A tabela abaixo mapeia conceitos familiares de gestao de dependencias para o ecossistema MCP, ajudando a aplicar praticas que voce ja conhece.

Conceito Dependencias (npm/pip) Servidores MCP

Best practices para governanca de MCP servers

Para desenvolvedores individuais

Checklist antes de adicionar um servidor MCP Texto 
# 1. Verificar a origem
- E um servidor oficial (@modelcontextprotocol/* ou mantido pelo vendor)?
- O repositorio tem manutencao ativa (commits recentes, issues respondidas)?
- Quantas stars/forks? Existe comunidade ao redor?

# 2. Auditar o codigo
- Revise o codigo-fonte, especialmente handlers de ferramentas
- Verifique se faz chamadas de rede nao documentadas
- Procure por acesso a variaveis de ambiente alem das necessarias

# 3. Pinar a versao
- Use @x.y.z em vez de @latest nos args do .mcp.json
- Atualize intencionalmente, nao automaticamente

# 4. Aplicar menor privilegio
- Limite tokens/credenciais ao minimo escopo necessario
- Use read-only tokens quando possivel
- Revise as ferramentas expostas com `claude mcp list`

Para times e organizacoes

1 Crie um catalogo aprovado

Mantenha uma lista curada de servidores MCP aprovados. Implante via managed-mcp.json ou allowedMcpServers para garantir que apenas servidores revisados sejam usados.

2 Padronize via .mcp.json no repo

Cada projeto deve ter um .mcp.json versionado no Git com os servidores que o time usa, com versoes pinadas. Isso funciona como um "lockfile" de MCP servers para o projeto.

3 Implante managed-settings via MDM

Use ferramentas de MDM (Jamf, Intune, Chef, Ansible) para distribuir managed-settings.json com allowlists e denylists para todas as maquinas do time.

4 Processo de aprovacao

Estabeleca um processo formal (ex: PR no repo de configuracao) para solicitar novos servidores MCP. Inclua revisao de seguranca antes de adicionar a allowlist.

Dica pratica: Trate o managed-mcp.json como voce trata o package-lock.json: versionado, revisado em PRs, e nunca editado manualmente em producao. A diferenca e que o managed-mcp tem um poder ainda maior -- ele controla o que um agente de IA pode acessar no seu ambiente.

Exemplo completo: governanca em uma organizacao

O exemplo abaixo mostra como uma organizacao pode combinar os dois mecanismos para ter controle total sobre servidores MCP, permitindo flexibilidade controlada:

managed-settings.json -- politica da organizacao JSON 
{
  // Permitir apenas servidores oficiais e internos
  "allowedMcpServers": [
    { "serverCommand": "npx" },
    { "serverUrl": "https://*.company.com/*" }
  ],

  // Bloquear servidores HTTP (sem TLS) e Docker
  "deniedMcpServers": [
    { "serverUrl": "http://*" },
    { "serverCommand": "docker" }
  ]
}
managed-mcp.json -- servidores obrigatorios para todos JSON 
{
  "mcpServers": {
    "company-tools": {
      "type": "sse",
      "url": "https://mcp.internal.company.com/sse"
    }
  }
}
.mcp.json -- configuracao do projeto (versionada no Git) JSON 
{
  "mcpServers": {
    "github": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github@1.2.0"],
      "env": {
        "GITHUB_TOKEN": "${GITHUB_TOKEN}"
      }
    },
    "postgres": {
      "type": "stdio",
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres@0.6.1"],
      "env": {
        "DATABASE_URL": "${DATABASE_URL}"
      }
    }
  }
}

Neste cenario, o resultado e: todos os desenvolvedores recebem automaticamente o servidor company-tools (via managed-mcp); o projeto define github e postgres com versoes pinadas (via .mcp.json); servidores HTTP sem TLS sao bloqueados (via denylist); e qualquer servidor que use npx ou URLs internas e permitido (via allowlist).

Resumo: o mindset de dependencia

Pratica Sem governanca Com governanca MCP
Adicionar servidor Qualquer dev instala qualquer servidor Apenas servidores na allowlist
Versionamento @latest (muda sem aviso) Versao pinada (@x.y.z)
Auditoria Nenhuma Revisao de codigo antes da aprovacao
Inventario Ninguem sabe o que esta rodando managed-mcp.json como SBOM
Remocao Servidores abandonados permanecem Revisao periodica da allowlist
Credenciais Tokens com escopo amplo Menor privilegio, read-only quando possivel
Principio central: Todo servidor MCP e uma dependencia de software que ganha acesso privilegiado ao seu agente de IA. Aplique o mesmo rigor que voce ja aplica a dependencias tradicionais: versione, audite, mantenha um inventario, e restrinja o acesso ao minimo necessario. A allowlist nao e burocracia -- e higiene de seguranca.
