Modulo 6 . Topico 2

Servidores locais (stdio) vs. remotos (HTTP/SSE)

O MCP define dois mecanismos de transporte padrao para comunicacao cliente-servidor: stdio (processos locais via stdin/stdout) e Streamable HTTP (servidores remotos via HTTP). Entender as diferencas entre eles e fundamental para escolher a arquitetura certa para cada caso de uso no Claude Code.

Os dois transportes oficiais do MCP

A especificacao MCP (versao 2025-03-26) define exatamente dois mecanismos de transporte padrao. Ambos usam JSON-RPC 2.0 codificado em UTF-8 como formato de mensagem, mas diferem radicalmente em como o cliente e o servidor se comunicam.

L stdio (Local)

O cliente lanca o servidor MCP como um subprocesso local. Comunicacao via stdin/stdout. Zero rede, zero latencia de rede. Ideal para ferramentas que precisam de acesso direto ao sistema de arquivos.

R Streamable HTTP (Remoto)

O servidor opera como um processo independente acessivel via HTTP. Aceita multiplas conexoes simultaneas. Usa POST/GET com SSE opcional para streaming. Ideal para servicos cloud e APIs compartilhadas.

Recomendacao da spec: A especificacao MCP afirma que clientes SHOULD suportar stdio sempre que possivel. O Claude Code suporta ambos os transportes nativamente, alem do transporte SSE legado (deprecated).

Transporte stdio: como funciona

No transporte stdio, o Claude Code lanca o servidor MCP como um subprocesso na sua maquina. O servidor le mensagens JSON-RPC do seu stdin e escreve respostas no seu stdout. Cada mensagem e delimitada por uma quebra de linha e nao pode conter quebras de linha internas.

O transporte stdio usa comunicacao direta entre processos via pipes do sistema operacional. Zero overhead de rede.

Caracteristicas do stdio

+ Vantagens

Latencia minima (IPC local). Sem configuracao de rede. Acesso direto ao filesystem. Seguranca por isolamento de processo. Simples de debugar. O servidor recebe CLAUDE_PROJECT_DIR no environment automaticamente.

- Limitacoes

Um cliente por servidor (relacao 1:1). Nao compartilhavel entre maquinas ou usuarios. Servidor morre se o cliente morrer. Sem reconexao automatica -- se o processo cair, precisa ser relancado manualmente.

Configuracao stdio no Claude Code

Terminal -- adicionando servidor stdio via CLI
# Sintaxe basica
claude mcp add --transport stdio <nome> -- <comando> [args...]

# Exemplo real: servidor de filesystem
claude mcp add --transport stdio filesystem -- \
  npx -y @modelcontextprotocol/server-filesystem /home/user/projeto

# Exemplo com variavel de ambiente
claude mcp add --transport stdio --env AIRTABLE_API_KEY=abc123 airtable -- \
  npx -y airtable-mcp-server
.mcp.json -- configuracao stdio em JSON
{
  "mcpServers": {
    "database": {
      "command": "npx",
      "args": ["-y", "@bytebase/dbhub", "--dsn", "postgresql://user:pass@localhost:5432/mydb"],
      "env": {
        "NODE_ENV": "production"
      }
    }
  }
}
Detalhe importante: Quando o type e omitido na configuracao JSON, o Claude Code assume stdio como padrao. Isso explica por que muitos exemplos de .mcp.json nao incluem "type": "stdio" explicitamente.

Transporte Streamable HTTP: como funciona

No transporte Streamable HTTP, o servidor opera como um processo independente (geralmente em nuvem ou em outro host) que expoe um unico endpoint HTTP. O cliente envia mensagens JSON-RPC via POST e pode opcionalmente receber respostas como Server-Sent Events (SSE) para streaming, ou como JSON simples para respostas diretas.

O transporte Streamable HTTP permite que servidores remotos atendam multiplos clientes simultaneamente, com sessoes e reconexao.

Caracteristicas do Streamable HTTP

+ Vantagens

Multiplos clientes simultaneos. Compartilhavel entre equipes. Reconexao automatica com backoff exponencial. Gerenciamento de sessao nativo. Autenticacao via OAuth 2.0. Resumabilidade de streams via Last-Event-ID.

- Limitacoes

Latencia de rede. Requer infraestrutura de hosting. Mais complexo para configurar. Precisa de HTTPS em producao. Necessita validacao de header Origin para prevenir ataques de DNS rebinding.

Configuracao HTTP no Claude Code

Terminal -- adicionando servidor HTTP via CLI
# Sintaxe basica
claude mcp add --transport http <nome> <url>

# Exemplo real: Notion
claude mcp add --transport http notion https://mcp.notion.com/mcp

# Exemplo com autenticacao via header
claude mcp add --transport http github https://api.githubcopilot.com/mcp/ \
  --header "Authorization: Bearer YOUR_GITHUB_PAT"

# Exemplo com OAuth (servidor pede login no navegador)
claude mcp add --transport http sentry https://mcp.sentry.dev/mcp
# Depois: /mcp para autenticar via browser
.mcp.json -- configuracao HTTP em JSON
{
  "mcpServers": {
    "notion": {
      "type": "http",
      "url": "https://mcp.notion.com/mcp"
    },
    "github": {
      "type": "http",
      "url": "https://api.githubcopilot.com/mcp/",
      "headers": {
        "Authorization": "Bearer ${GITHUB_PAT}"
      }
    }
  }
}
Alias na config JSON: O campo type aceita tanto "http" quanto "streamable-http" como valores equivalentes. O nome streamable-http e o nome oficial na spec MCP, enquanto http e o atalho suportado pelo Claude Code. Configuracoes copiadas de documentacao de servidores funcionam sem modificacao.

A evolucao: de HTTP+SSE para Streamable HTTP

A versao original do MCP (protocolo 2024-11-05) definia um transporte chamado HTTP+SSE que usava dois endpoints separados: um endpoint SSE para o servidor enviar mensagens ao cliente, e um endpoint POST para o cliente enviar mensagens ao servidor. Esse design era funcional, mas apresentava complexidades desnecessarias.

A partir da versao 2025-03-26 da spec, esse transporte foi substituido pelo Streamable HTTP, que unifica tudo em um unico endpoint. O SSE agora e opcional -- o servidor pode responder com JSON simples ou abrir um stream SSE conforme necessario.

O Streamable HTTP simplifica o transporte remoto mantendo retrocompatibilidade com servidores SSE antigos.

SSE no Claude Code: O transporte SSE legado ainda e suportado via claude mcp add --transport sse e "type": "sse" no JSON, mas a documentacao oficial marca-o como deprecated. Para novos servidores, use sempre http. Para clientes conectando a servidores antigos, o Claude Code tenta automaticamente o fallback: primeiro tenta POST (Streamable HTTP); se falhar com 4xx, tenta GET esperando SSE.

Comparacao detalhada: stdio vs. Streamable HTTP

Aspecto stdio (Local) Streamable HTTP (Remoto)
Comunicacao stdin/stdout (pipes do OS) HTTP POST/GET + SSE opcional
Latencia Minima (IPC local, microsegundos) Variavel (rede, milissegundos+)
Clientes simultaneos 1 (relacao 1:1) Multiplos
Ciclo de vida Morre com o cliente Independente do cliente
Reconexao Sem reconexao automatica Backoff exponencial (5 tentativas)
Acesso ao filesystem Direto (mesmo usuario do OS) Via rede (API necessaria)
Autenticacao Env vars, credenciais locais OAuth 2.0, headers, tokens
Sessoes N/A (conexao unica) Mcp-Session-Id, resumabilidade
Deployment Zero (roda local) Requer hosting (cloud, container)
Compartilhamento Apenas a maquina local Equipes, organizacoes
Config no Claude Code --transport stdio --transport http
type no JSON "stdio" (ou omitido) "http" ou "streamable-http"

Implicacoes de seguranca

A escolha do transporte tem impacto direto no modelo de seguranca. Cada abordagem traz riscos e mitigacoes diferentes.

Seguranca stdio

Superficie de ataque reduzida: sem portas de rede expostas. Comunicacao confinada a pipes do OS entre processos pai/filho.

Risco principal: o servidor roda com as mesmas permissoes do seu usuario. Se o codigo do servidor for malicioso, ele tem acesso total ao filesystem, rede e processos.

Mitigacao: A especificacao recomenda exibir dialogo de consentimento mostrando o comando exato antes de executar. O Claude Code pede aprovacao para servidores de projeto (.mcp.json) na primeira execucao.

Seguranca HTTP

Autenticacao robusta: suporte nativo a OAuth 2.0, headers customizados, tokens bearer, e headersHelper para esquemas proprietarios.

Risco principal: servidores devem validar o header Origin para prevenir ataques de DNS rebinding. Servidores locais devem fazer bind apenas em 127.0.0.1, nunca em 0.0.0.0.

Mitigacao: Usar HTTPS em producao. Implementar autenticacao adequada. Validar Origin em todas as conexoes. Gerenciar sessoes com IDs criptograficamente seguros.

Alerta de seguranca da spec MCP: Sem protecoes adequadas (validacao de Origin, bind em localhost, autenticacao), atacantes podem usar DNS rebinding para interagir com servidores MCP locais a partir de websites remotos. Isso se aplica especialmente a servidores HTTP rodando localmente durante desenvolvimento.

Quando usar cada transporte

Use stdio quando...

1 Acesso local

O servidor precisa ler/escrever arquivos locais, executar comandos shell, ou acessar recursos da maquina (banco local, Docker, etc).

2 Uso individual

Apenas voce vai usar o servidor. Nao precisa compartilhar com a equipe ou rodar em infra centralizada.

3 Performance critica

A latencia e critica e voce nao pode ter overhead de rede. Ex: ferramentas de build, linters, formatters.

Use Streamable HTTP quando...

1 Servico cloud

O servidor conecta a uma API SaaS (GitHub, Notion, Sentry, Slack). Esses servicos ja oferecem endpoints MCP remotos prontos.

2 Uso em equipe

Multiplos desenvolvedores precisam acessar o mesmo servidor MCP. Um servidor HTTP central atende toda a equipe.

3 Resiliencia

Voce precisa de reconexao automatica, gerenciamento de sessao, e o servidor deve sobreviver a desconexoes do cliente.

Configuracao hibrida: o melhor dos dois mundos

Na pratica, a maioria dos projetos usa uma combinacao de servidores locais e remotos. O .mcp.json do projeto pode definir servidores compartilhados (remotos), enquanto cada desenvolvedor adiciona seus servidores locais pessoais via --scope local.

.mcp.json -- configuracao hibrida de projeto
{
  "mcpServers": {
    // Servidor remoto: compartilhado com toda a equipe
    "github": {
      "type": "http",
      "url": "https://api.githubcopilot.com/mcp/",
      "headers": {
        "Authorization": "Bearer ${GITHUB_PAT}"
      }
    },
    // Servidor remoto: monitoramento compartilhado
    "sentry": {
      "type": "http",
      "url": "https://mcp.sentry.dev/mcp"
    },
    // Servidor local: banco de dados do projeto
    "postgres": {
      "command": "npx",
      "args": ["-y", "@bytebase/dbhub", "--dsn", "${DATABASE_URL}"]
    }
  }
}
Dica de escopo: Servidores em .mcp.json (scope project) sao versionados no repositorio e compartilhados com a equipe. Servidores pessoais adicionados com --scope local ficam em ~/.claude.json e sao privados. Servidores com --scope user ficam disponiveis em todos os seus projetos. A precedencia e: local > project > user > plugins > conectores do claude.ai.

Arvore de decisao rapida

Na duvida: se o servidor precisa tocar no disco local, use stdio. Se conecta a um servico externo, use http.
