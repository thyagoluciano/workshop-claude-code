Quando Prompts Vagos São Úteis

Contra-intuitivo: nem sempre mais contexto é melhor. Para tarefas de exploração aberta, prompts vagos permitem que o Claude Code navegue livremente pelo codebase e descubra coisas que você não saberia perguntar.

O paradoxo: menos direção, mais descoberta

Ao longo deste workshop, enfatizamos a importância de ser específico nos prompts: descrever o sintoma, localizar o problema, fornecer contexto. Mas existe um cenário onde essa regra se inverte. A documentação oficial de best practices da Anthropic reconhece explicitamente que prompts vagos podem ser úteis quando você está explorando e pode corrigir o rumo ao longo do caminho.

Um prompt como "o que você melhoraria neste arquivo?" pode revelar problemas que você nem pensaria em perguntar. Quando você não sabe o que não sabe, restringir demais o prompt é restringir também as descobertas.

Insight da Anthropic: Prompts vagos são úteis quando você está explorando e pode corrigir o rumo. Um prompt aberto pode surfar coisas que você não teria pensado em perguntar. A chave é saber quando usar cada abordagem.
— Anthropic, "Best Practices for Claude Code"

O espectro de especificidade

Prompts não são binários (vagos ou específicos) — eles existem em um espectro. O segredo é calibrar o nível de especificidade de acordo com o tipo de tarefa.

O nível de especificidade deve refletir o quanto você já sabe sobre o problema. Menos conhecimento = prompt mais aberto.

Quando prompts vagos funcionam

Prompts vagos não são preguiça — são uma estratégia deliberada para cenários onde restringir o escopo significaria perder insights importantes. A Anthropic recomenda que para tarefas exploratórias, você pode pular o planejamento e deixar o Claude descobrir o caminho.

1 Codebase desconhecido

Você acabou de entrar em um projeto. Não sabe a estrutura, os padrões, nem onde as coisas vivem. Um prompt vago como "explique a arquitetura deste projeto" deixa o Claude explorar e mapear tudo.

2 Investigacao sem pista

O bug existe mas você não tem ideia de onde. "Encontre possíveis bugs neste projeto" permite que o Claude vasculhe livremente em vez de se limitar a um módulo que talvez não seja o problema.

3 Brainstorming de arquitetura

Você quer ideias novas, não implementação. "Como você redesenharia este sistema?" abre espaço para sugestoes que você não consideraria se pedisse algo pontual.

4 Auditoria de qualidade

"O que pode ser melhorado neste código?" — sem restringir a segurança, performance ou estilo. Deixa o Claude priorizar o que ele considera mais crítico.

Regra prática: Se você consegue descrever o problema em uma frase específica, seja específico. Se você está tentando descobrir qual é o problema, seja vago.

Exemplos de prompts vagos eficazes

Prompts vagos não significam prompts de uma palavra. Os melhores prompts vagos são abertos no escopo mas claros na intenção. Veja a diferenca:

Prompts vagos eficazes para exploração Claude Code 
# Entender um projeto novo
"Explique a arquitetura deste projeto. Quais são os módulos
principais e como eles se conectam?"

# Descobrir problemas que você não sabe que existem
"O que você melhoraria neste arquivo?"

# Mapear dependências
"Trace as dependências do módulo de pagamento.
Quem depende dele e de quem ele depende?"

# Auditoria aberta
"Encontre possíveis problemas de segurança neste projeto."

# Entender decisoes de design
"Por que este projeto usa esse padrão? Quais seriam
as alternativas?"
Prompts vagos INEFICAZES (vagos demais) Evite estes 
# Vago demais — sem direção nenhuma
"Olhe o código"
  → Claude não sabe o que procurar. Resposta genérica.

# Ambiguo — múltiplas interpretações
"Melhore isso"
  → Melhorar o que? Performance? Legibilidade? Seguranca?

# Amplo demais sem ponto de partida
"Analise tudo"
  → Claude le centenas de arquivos, enche o contexto,
     e a qualidade da resposta cai.
Atenção ao contexto: Quando você pede ao Claude para "investigar" algo sem delimitar o escopo, ele pode ler centenas de arquivos e encher a context window. A documentacao da Anthropic recomenda delimitar investigações ou usar subagentes para que a exploração não consuma o contexto principal.

Combinando vago com restrição

A técnica mais poderosa é o prompt semi-aberto: vago no escopo, mas com uma restrição que direciona a exploração. Isso da liberdade ao Claude enquanto evita respostas completamente genéricas.

Padrão Exemplo Por que funciona
Explore + foco "Explore o projeto mas foque em segurança" Livre para navegar, mas com lente definida
Aberto + arquivo "O que pode melhorar em src/auth/" Aberto no tipo de melhoria, fechado no escopo
Investigar + sintoma "Investigue por que a API está lenta" Não diz onde procurar, mas define o que procurar
Revisar + criterio "Revise este PR pensando em manutenibilidade" Liberdade para apontar qualquer issue, com prioridade clara
Mapear + aspecto "Mapeie o fluxo de dados do login ate o dashboard" Aberto na profundidade, fechado no caminho
Exemplos de prompts semi-abertos na prática Claude Code 
# Exploracao com lente de segurança
"Analise este projeto procurando vulnerabilidades.
Não precisa cobrir tudo — foque nos pontos mais críticos
que você encontrar."

# Exploracao com restrição de escopo
"Entenda como o sistema de cache funciona neste projeto.
Quais estratégias são usadas e onde ha riscos de inconsistência?"

# Brainstorming com guardrails
"Sugira melhorias de performance, mas que não exijam
mudanca de infraestrutura. Foque no código da aplicacao."

Prompts vagos como primeiro passo

A melhor estrategia não é escolher entre vago e específico — e usar os dois em sequência. Comece vago para explorar e entender, depois refine com especificidade para executar. A Anthropic descreve isso como a progressão natural de trabalhar com o Claude Code: desenvolver intuição e iterar.

O fluxo ideal: use prompts vagos para descobrir, depois prompts específicos para implementar — com /clear entre as fases.

Dica da Anthropic: Se você corrigiu o Claude mais de duas vezes no mesmo problema em uma sessão, o contexto está poluido com tentativas falhas. Use /clear e comece uma nova sessão com um prompt melhor que incorpore o que você aprendeu. Uma sessão limpa quase sempre supera uma sessão longa com correções acumuladas.

Decision tree: quando ser vago vs. quando ser específico

Use está árvore de decisão para calibrar a especificidade do seu prompt:

?

Você sabe qual arquivo ou módulo tem o problema?

Não → Use um prompt vago para exploração: "Investigue o que pode estar causando lentidão no sistema"

Sim → Próximo passo abaixo

?

Você sabe qual é a causa raiz?

Não → Use um prompt semi-aberto com restrição: "Analise src/api/ procurando problemas de performance"

Sim → Próximo passo abaixo

?

Você sabe exatamente o que precisa ser feito?

Não → Use um prompt com contexto mas aberto na solução: "A query N+1 está no UserRepository. Sugira formas de resolver"

Sim → Use um prompt específico e direto

!

Prompt específico — você tem todas as informações

"Refatore UserRepository.findAll() para usar JOIN em vez de queries separadas. Mantenha a interface publica igual. Rode os testes depois."

Guia rápido: cenário vs. tipo de prompt

Cenário Tipo de prompt Exemplo
Onboarding em projeto novo Vago "Explique a arquitetura e os padrões deste projeto"
Bug sem pista da causa Vago "Usuarios reportam lentidão. Investigue possíveis causas"
Code review geral Semi-aberto "Revise este PR focando em manutenibilidade"
Auditoria de segurança Semi-aberto "Procure vulnerabilidades no módulo de autenticacao"
Bug com stacktrace Contextualizado "TypeError no AuthService.validate() quando token expira"

O risco: quando prompts vagos saem errado

Prompts vagos não são uma carta branca. Usados nos cenários errados, eles levam a problemas concretos:

! Context window esgotada

Sem escopo, o Claude le dezenas de arquivos tentando entender "tudo". O contexto enche com informação irrelevante e a qualidade das respostas seguintes despenca.

! Respostas genéricas

Sem direção, o Claude pode cair em conselhos genéricos ("use TypeScript strict mode", "adicione testes") que não trazem valor para o seu caso específico.

! Escopo infinito

Um prompt como "melhore este projeto" pode levar o Claude a tentar refatorar tudo, gerando mudanças massivas e difíceis de revisar.

! Perda de foco na sessão

Se a exploração vaga não for seguida de um /clear, todo o contexto acumulado atrapalha os prompts específicos que vem depois.

Mitigação: Use subagentes (Task tool) para exploração vaga. O subagente le centenas de arquivos e reporta de volta um resumo, sem consumir o contexto da sua conversa principal. Isso e exatamente o que a Anthropic recomenda para investigações de escopo aberto.

Resumo: o espectro da especificidade

Prompts vagos são para...

  • • Explorar codebases desconhecidos
  • • Investigar sem pistas
  • • Brainstorming e ideação
  • • Auditorias abertas de qualidade
  • • Mapear dependências e arquitetura
  • • O primeiro passo antes de ser específico

Prompts específicos são para...

  • • Implementação de features
  • • Correcao de bugs com causa conhecida
  • • Refatorações pontuais
  • • Tarefas com criterio de sucesso claro
  • • Modificações em arquivos específicos
  • • Tudo onde você sabe exatamente o que quer
Takeaway: A especificidade do prompt deve refletir o quanto você já sabe sobre o problema. Pouco conhecimento → prompt aberto para descobrir. Muito conhecimento → prompt preciso para executar. E entre as fases: /clear.
