Escopo da Tarefa: Arquivo, Cenário, Preferências de Teste

A qualidade do output do Claude Code é diretamente proporcional à precisão do seu prompt. Definir onde atuar, o quê fazer e como validar transforma interações vagas em entregas precisas e confiáveis.

Por que definir escopo claro importa?

A documentação oficial da Anthropic sobre best practices do Claude Code enfatiza que quanto mais precisas as instruções, menos correções serão necessárias. O Claude pode inferir intenção, mas não pode ler sua mente. Referenciar arquivos específicos, mencionar restrições e apontar para padrões existentes faz toda a diferença.

! Reduz Ambiguidade

Sem escopo, o agente precisa adivinhar onde atuar, qual framework usar e que padrão seguir. Cada suposição é um ponto de falha potencial.

$ Economiza Tokens

Um prompt vago faz o Claude ler centenas de arquivos buscando contexto. Escopar a investigação evita consumir a context window com informação irrelevante.

^ Melhora Qualidade

Quando o agente sabe exatamente o alvo, ele gera código que respeita as convenções do projeto, trata edge cases e passa nos testes de primeira.

Armadilha clássica: Pedir ao Claude para "investigar" algo sem escopar a busca. Ele pode ler centenas de arquivos, enchendo a context window. Sempre escopar a investigação ou usar subagentes para que a exploração não consuma seu contexto principal.

As 3 dimensões do escopo

Todo prompt bem escopado responde a três perguntas fundamentais. Pense nelas como coordenadas que localizam a tarefa com precisão no espaço do seu codebase:

As três dimensões convergem para formar um prompt preciso: ONDE atuar, O QUE fazer, COMO validar.

Dimensão 1: ONDE (arquivo / diretório)

Apontar para arquivos específicos é a forma mais eficaz de reduzir ambiguidade. Em vez de deixar o agente buscar no codebase inteiro, você pode referenciar diretamente os arquivos relevantes. Isso economiza tokens e evita que o Claude leia arquivos desnecessários.

Apontar para arquivos vs. deixar o agente buscar

Abordagem Vantagens Desvantagens Quando usar
Apontar arquivo Rápido, economiza tokens, sem ambiguidade Pode perder dependências relacionadas Você conhece bem o codebase
Deixar buscar Descobre dependências, ve o contexto completo Consome tokens, pode ler arquivos irrelevantes Codebase novo, refactoring amplo
Híbrido Equilíbrio entre precisão e contexto Requer conhecimento parcial do projeto Maioria dos cenários do dia a dia
Exemplos de escopo de localização Prompt do usuário 
// Específico: aponta para o arquivo exato
Em src/components/LoginForm.tsx, adicione validação
de email no campo de input usando o schema Zod
que já existe em src/schemas/auth.schema.ts

// Semi-específico: aponta para o diretório
Nos componentes de src/features/auth/, corrija o
bug que causa redirect infinito após login com SSO

// Por padrão: quando você não sabe o arquivo exato
Encontre o componente de signup (deve estar em
src/features/auth/ ou src/components/) e adicione
validação de telefone com máscara brasileira
Dica: Quando componentes tem nomes ou funções similares, seja extremamente específico nas suas requisições. Um prompt detalhado evita que o Claude faça alterações inesperadas na parte errada do codebase.

Dimensão 2: O QUE (cenário / funcionalidade)

Descreva o cenário ou funcionalidade com clareza suficiente para que um colega com contexto mínimo sobre a tarefa consiga segui-lo. A documentação da Anthropic sugere este teste: mostre seu prompt a um colega e pergunte se ele conseguiria executar a tarefa. Se ele ficaria confuso, o Claude também ficara.

X Prompt vago

"Arrume o login"
"Melhore a performance"
"Adicione testes"
"Refatore esse código"
"Tem um bug no checkout"
vs

OK Prompt específico

"O login retorna 401 quando o email tem letras maiúsculas. Normalize o email antes de enviar ao backend."
"A lista de produtos re-renderiza a cada keystroke no filtro. Adicione debounce de 300ms no onChange."
"Adicione testes unitários para o useAuth hook, cobrindo login com sucesso, credenciais inválidas e token expirado."
"Extraia a lógica de cálculo de preço do OrderService para uma classe PriceCalculator, mantendo a mesma interface pública."
"No checkout, o cupom de desconto aplica o percentual sobre o frete em vez de sobre o subtotal. Corrija em src/services/discount.ts."

Dimensão 3: COMO (preferências de implementação e teste)

Especificar preferências de implementação e teste elimina a necessidade do agente adivinhar seu stack e suas convenções. Isso é especialmente importante para testes, onde frameworks, padrões de nomeação e níveis de cobertura variam entre projetos.

T Preferências de Teste

"Use Jest com React Testing Library"
"Teste com Vitest e @testing-library/vue"
"Use pytest com fixtures, não mocks"
"Garanta cobertura de edge cases"
"Inclua testes de erro e boundary"

I Preferências de Implementação

"Siga o padrão de error handling do projeto"
"Use o pattern Repository que já existe"
"Mantenha compatibilidade com a API v2"
"Prefira composição sobre herança"
"Use tipos explicitados, não any"

Prompt com preferências explícitas Exemplo completo 
Em src/hooks/useCart.ts, adicione lógica para aplicar
cupom de desconto percentual.

Preferências de implementação:
- Siga o mesmo pattern do useDiscount hook existente
- Valide que o cupom não está expirado antes de aplicar
- Retorne um erro tipado (DiscountError) se o cupom for inválido

Preferências de teste:
- Use Jest com React Testing Library
- Cubra: cupom válido, cupom expirado, cupom já usado, valor negativo
- Siga o padrão de nomeação "should [verbo] when [condição]"

Verificação:
- Rode npm test -- --grep useCart e garanta que todos passam
- Rode npm run typecheck para confirmar que não há erros de tipo
Best practice: Sempre fornecer meios de verificação: testes, scripts ou builds. A documentação da Anthropic destaca que sem verificação, o Claude pode produzir uma implementação que parece plausível mas não lida com edge cases. Critérios de verificação alimentam o loop de auto-correção do agente.

Escopo progressivo: comece específico, expanda conforme necessário

Nem sempre você sabe o escopo completo de antemão. A estratégia recomendada e começar com um escopo estreito e expandir iterativamente. Isso é mais eficiente do que começar amplo e refinar, porque evita o problema de context window cheia com informação irrelevante.

Estratégia em 3 passos

Passo 1: Comece cirúrgico

Primeiro promptEscopo estreito
Em src/services/auth.ts, a função validateToken()
não trata o caso de token expirado. Adicione o
tratamento retornando { valid: false, reason: 'expired' }.

Passo 2: Avalie o resultado e expanda

Follow-upEscopo expandido
Agora verifique se os consumers de validateToken()
em src/middleware/authMiddleware.ts e
src/hooks/useAuth.ts tratam o novo campo reason
corretamente. Ajuste se necessário.

Passo 3: Valide e feche

FechamentoVerificação
Rode npm test -- --grep "auth|token" e garanta
que todos os testes passam. Se algum falhar por
causa da mudança, corrija o teste ou a implementação.
Sobre /clear: Use /clear entre tarefas não relacionadas para resetar o contexto. A "sessão pia de cozinha" — onde você acumula tarefas desconectadas na mesma conversa — é um dos erros mais comuns e enche a context window com informação irrelevante para a tarefa atual.

Templates de prompt para cenários comuns

Estes templates aplicam as três dimensões do escopo (ONDE, O QUE, COMO) para os cenários mais frequentes no dia a dia de um desenvolvedor:

B Bug Fix 

## Contexto
[Descreva o bug: o que acontece vs. o que deveria acontecer]

## Localização
O bug está em [arquivo/diretório]. Veja a função [nome].

## Correção esperada
[Descreva o comportamento correto esperado]

## Verificação
Rode [comando de teste] e confirme que o cenário [X] funciona.

F Nova Feature 

## Contexto
[Por que essa feature existe? Que problema do usuário resolve?]

## Escopo
Implemente em [arquivo/módulo].
Siga o padrão de [referência/componente similar].

## Requisitos
- [Requisito funcional 1]
- [Requisito funcional 2]
- [Restrição ou constraint]

## Testes
Use [framework de teste]. Cubra: [cenários].
Rode [comando] para validar.

R Refactoring 

## Objetivo
[O que melhorar: legibilidade, performance, separação de concerns?]

## Escopo
Refatore [função/classe/módulo] em [arquivo].
[Extraia para / Consolide / Renomeie para]

## Restrições
- Mantenha a mesma interface pública (não quebre consumers)
- [Outras restrições]

## Verificação
Rode [suite de testes] — todos devem passar sem mudanças.
Rode [typecheck/build] para garantir integridade.

T Adicionar Testes 

## Alvo
Adicione testes para [função/componente/hook] em [arquivo].

## Framework e padrões
Use [Jest/Vitest/pytest] com [RTL/Enzyme/etc].
Siga o padrão de nomeação: "should [verbo] when [condição]".

## Cenários obrigatórios
- Caminho feliz (happy path)
- [Edge case 1: input inválido, vazio, null]
- [Edge case 2: erro de rede, timeout]
- [Boundary: limites numéricos, strings longas]

## Verificação
Rode [comando] e confirme 100% pass. Verifique cobertura com [--coverage].
