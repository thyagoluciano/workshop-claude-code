Modulo 4 . Topico 4

Sintoma + Localizacao + Definicao de Resolvido

O framework SLD e a tecnica mais eficaz para transformar prompts vagos em instrucoes que o Claude Code consegue resolver de primeira. Descreva o que esta errado, onde acontece e quando considerar feito.

Por que seus prompts precisam de estrutura

A documentacao oficial da Anthropic sobre best practices do Claude Code destaca que especificidade gera alinhamento -- quanto mais preciso voce explica o que quer, melhor o resultado. A regra de ouro: se um colega com contexto minimo ficaria confuso lendo seu prompt, o Claude tambem ficara.

O problema e que a maioria dos desenvolvedores descreve tarefas de forma incompleta. Dizem o que querem ("corrige esse bug") mas omitem informacoes criticas que o agente precisa para tomar boas decisoes: qual e o comportamento atual? Onde exatamente acontece? Como saber se a correcao funcionou?

O framework SLD resolve isso com tres componentes obrigatorios:

O framework SLD: cada componente elimina um tipo de ambiguidade do seu prompt.

S -- Sintoma: descreva O QUE esta errado

O sintoma e a descricao do comportamento observado comparado ao comportamento esperado. Nao e a sua teoria sobre a causa -- e o que voce ve acontecendo. A documentacao da Anthropic recomenda descrever o problema sem prescrever a solucao, porque o agente pode encontrar uma causa raiz diferente da que voce imagina.

! Comportamento atual

O que esta acontecendo agora? Qual o erro, output incorreto ou comportamento inesperado?

+ Comportamento esperado

O que deveria acontecer? Qual e o resultado correto, a resposta esperada ou o estado desejado?

~ Delta (a diferenca)

Qual e o gap entre os dois? Dados incorretos? Erro silencioso? Performance degradada? Crash?

Dica: Se voce consegue colar a mensagem de erro exata, faca isso. A Anthropic recomenda usar cat error.log | claude para enviar logs diretamente. Dados concretos sao sempre melhores que descricoes vagas do erro.

L -- Localizacao: descreva ONDE acontece

A localizacao ajuda o agente a comecar a busca no lugar certo. Sem ela, o Claude precisa varrer o projeto inteiro com Grep e Glob antes de entender onde atuar. Com ela, o agente vai direto ao ponto, economizando tokens e tempo.

A documentacao da Anthropic sugere referenciar arquivos com @ ao inves de descrever onde o codigo mora. Quanto mais especifico, melhor: nome do arquivo, funcao, classe, endpoint, ou fluxo do usuario.

Nivel de especificidade Exemplo Eficacia
Vago "no backend" O agente precisa buscar em todo o projeto
Razoavel "no controller de usuarios" Reduz o escopo, mas ainda precisa buscar o arquivo
Especifico "em src/controllers/UserController.ts, funcao createUser()" Agente vai direto ao ponto, sem buscas extras
Otimo "@src/controllers/UserController.ts na linha ~142, dentro do try/catch do createUser()" Maximo de precisao, minimo de tokens gastos

D -- Definicao de Resolvido: descreva QUANDO esta feito

Este e o componente mais negligenciado -- e o mais poderoso. A documentacao da Anthropic enfatiza que o Claude performa dramaticamente melhor quando consegue verificar seu proprio trabalho. Sem criterios claros de sucesso, ele pode produzir algo que parece correto mas nao funciona. A definicao de resolvido alimenta diretamente o loop de auto-correcao do agente.

A Criterios automatizaveis

Testes passam, build compila, linter nao reclama, endpoint retorna 200, migration roda sem erro. O agente pode verificar sozinho rodando comandos.

V Criterios visuais/manuais

Screenshot do componente renderizado, output formatado de um certo jeito, comportamento especifico na UI. O agente nao verifica sozinho mas sabe o alvo.

Regra de ouro: Sempre que possivel, de ao Claude um comando que ele possa rodar para verificar se a tarefa esta completa. Exemplos: npm test -- --grep "user validation", cargo build 2>&1 | head -20, python -m pytest tests/test_auth.py -v. Isso transforma a definicao de resolvido em algo executavel.

Por que o SLD funciona: a mecanica por tras

1 Reduz ciclos de tentativa e erro

Sem SLD, o agente adivinha o que voce quer, implementa, voce corrige, ele tenta de novo -- e o contexto fica poluido com tentativas falhas. Com SLD, acerta de primeira (ou quase).

2 Guia o loop agentico

O Sintoma diz o que investigar, a Localizacao diz onde comecar, e a Definicao de Resolvido diz quando parar. Cada componente alimenta uma fase diferente do loop.

3 Economiza tokens e dinheiro

Menos buscas com Grep/Glob, menos tentativas falhas, menos correcoes. O agente gasta tokens resolvendo o problema ao inves de descobrindo qual e o problema.

4 Habilita auto-verificacao

A definicao de resolvido e o que permite ao Claude rodar testes, conferir o build e se auto-corrigir sem depender de voce. Sem ela, o agente nao sabe se terminou.

Antes e depois: prompt vago vs. prompt com SLD

Cenario 1: Bug fix

X Prompt vago 

"O login nao funciona. Corrige."
vs

+ Prompt com SLD 

# Sintoma
Ao fazer login com email valido,
a API retorna 401 ao inves de 200.
O token JWT nao esta sendo gerado.

# Localizacao
src/auth/login.service.ts, funcao
authenticate(). O bcrypt.compare()
parece retornar false sempre.

# Definicao de resolvido
npm test -- --grep "auth" passa
e POST /api/login retorna 200
com um JWT valido no body.

Cenario 2: Feature request

X Prompt vago

vs

+ Prompt com SLD

src/products/products.controller.ts

Templates prontos por tipo de tarefa

Use estes templates como ponto de partida. Copie, preencha os campos e envie ao Claude Code. A estrutura ja garante que os tres componentes do SLD estejam presentes.

Template: Bug Fix

# Sintoma
[Comportamento atual] ao inves de [comportamento esperado].
Erro exato: [mensagem de erro ou stack trace]

# Localizacao
[arquivo:funcao] -- [como reproduzir o bug]

# Resolvido quando
[comando de teste] passa e [comportamento esperado] funciona.

Template: Nova Feature

# Sintoma (estado atual)
[O que o sistema faz hoje] -- [por que isso e insuficiente].

# Localizacao
[arquivos que serao modificados] e [arquivos de referencia para seguir o padrao]

# Resolvido quando
[endpoint/funcao] retorna [resultado esperado].
Testes existentes continuam passando + novos testes cobrem [cenarios].

Template: Performance

# Sintoma
[Metrica atual: Xs, Nms, N re-renders] -- [metrica aceitavel: Ys, Nms].
Profiling mostra: [gargalo identificado].

# Localizacao
[arquivo:funcao onde o gargalo esta]

# Resolvido quando
[Metrica] cai de [valor atual] para [valor alvo].
[comando de build/test] passa sem regressao.

Template: Refactoring

# Sintoma (problema estrutural)
[Descricao do code smell: arquivo grande, duplicacao, acoplamento, etc.]

# Localizacao
[arquivos afetados] -- testes em [diretorio de testes]

# Resolvido quando
[Padrao alvo: Strategy, Repository, etc.] implementado.
Todos os [N] testes existentes passam SEM modificacao.
Nenhuma mudanca de comportamento externo.

Combinando SLD com Plan Mode

Para tarefas complexas onde o escopo nao e claro, voce pode usar o SLD junto com o Plan Mode (shift+tab para ativar). A ideia e: voce fornece o SLD, e pede ao Claude para planejar antes de agir. Isso e especialmente util quando a mudanca envolve multiplos arquivos ou voce nao conhece bem o codigo.

SLD + Plan Mode Prompt completo 
# Sintoma
O endpoint POST /api/orders retorna 500 quando
o carrinho tem mais de 50 itens. O erro no log e
"Maximum call stack size exceeded".

# Localizacao
src/orders/order.service.ts, funcao calculateTotal().
Suspeito de recursao infinita no calculo de descontos
cumulativos em src/pricing/discount.engine.ts.

# Definicao de resolvido
POST /api/orders com 100 itens retorna 201.
npm test -- --grep "order" passa.
Nenhuma regressao nos testes de pricing.

# Instrucao
Antes de implementar, use o Plan Mode para:
1. Confirmar a causa raiz lendo os arquivos
2. Listar todos os arquivos que precisam mudar
3. Propor a abordagem de correcao
Depois implemente e rode os testes.
Quando usar Plan Mode com SLD: A regra da Anthropic e simples -- se voce consegue descrever a mudanca em uma frase, pule o plano. Se a mudanca envolve multiplos arquivos, ou voce esta pouco familiarizado com o codigo, peca um plano primeiro.

Anti-patterns: o que evitar

Anti-pattern Exemplo Problema
Prescrever a solucao "Mude a linha 42 para usar Promise.all" Voce tira a autonomia do agente. Ele pode encontrar uma solucao melhor se voce descrever o problema, nao a solucao.
Ser generico demais "Melhora o codigo" Sem sintoma, sem localizacao, sem criterio. O agente nao sabe o que e "melhor" no seu contexto.
Omitir a verificacao "Corrige o bug no auth" (sem dizer como testar) O agente implementa e para. Sem criterio de verificacao, ele nao usa o loop de auto-correcao.
Descrever a teoria, nao o sintoma "Acho que tem um race condition" Pode ser race condition ou nao. Descreva o que voce observa (ex: "dados inconsistentes apos requests simultaneos") e deixe o agente diagnosticar.
Kitchen sink de contexto Colar 500 linhas de codigo no prompt O agente pode ler o arquivo sozinho com Read. Aponte para o arquivo e deixe ele explorar.
Armadilha comum: Corrigir o Claude varias vezes quando ele erra, poluindo o contexto com tentativas falhas. A documentacao da Anthropic recomenda: apos duas correcoes sem sucesso, use /clear e reescreva um prompt SLD melhor incorporando o que voce aprendeu.

Exemplo completo: do prompt ao resultado

Veja como um prompt SLD bem escrito se traduz em acoes concretas do agente:

Prompt enviado ao Claude Code SLD completo 
# Sintoma
O formulario de signup aceita emails invalidos como "user@" e "@@test".
O campo email nao tem validacao -- qualquer string e aceita.
Esperado: emails invalidos mostram mensagem de erro inline.

# Localizacao
@src/components/SignupForm.tsx -- o campo email na linha ~28
@src/schemas/signup.schema.ts -- schema Zod sem .email()

# Resolvido quando
1. npm test -- --grep "signup" passa
2. Emails invalidos ("user@", "@@test", "sem-arroba") sao rejeitados
3. Emails validos ("user@example.com") sao aceitos
4. Mensagem de erro: "Informe um email valido"
O que o Claude Code faz internamente Sequencia de tool calls 
// 1. Le os arquivos indicados na Localizacao
Read { file: "src/components/SignupForm.tsx" }
Read { file: "src/schemas/signup.schema.ts" }
  -> Confirma: schema Zod com email: z.string() sem .email()

// 2. Implementa a correcao (Sintoma -> acao)
Edit { file: "src/schemas/signup.schema.ts",
       old: "email: z.string()",
       new: "email: z.string().email('Informe um email valido')" }

// 3. Verifica (Definicao de Resolvido -> auto-check)
Bash { command: "npm test -- --grep signup" }
  -> 1 teste falhou (snapshot desatualizado)

// 4. Auto-corrige e verifica de novo
Bash { command: "npm test -- --grep signup -u" }
  -> Todos os testes passaram

// 5. Entrega: todas as condicoes do "Resolvido" atendidas
Observe: Cada componente do SLD guiou uma fase do trabalho do agente. O Sintoma disse o que corrigir, a Localizacao disse onde comecar (sem buscas extras), e a Definicao de Resolvido permitiu que ele se auto-verificasse e auto-corrigisse o snapshot falho -- tudo sem intervencao humana.
