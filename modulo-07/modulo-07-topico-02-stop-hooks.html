Stop Hooks: Rodar Build, Typecheck e Testes ao Fim de Cada Resposta

Como usar Stop hooks para garantir automaticamente que o codigo gerado pelo Claude compila, passa no typecheck e nos testes antes de devolver o controle ao desenvolvedor.

O que sao Stop Hooks

Stop hooks sao hooks que disparam toda vez que o Claude termina de responder - ou seja, a cada turno de conversa, nao apenas quando a tarefa inteira e concluida. Esse comportamento e fundamental: se o Claude faz 5 respostas durante uma tarefa, o Stop hook executa 5 vezes.

Atencao: Stop hooks nao disparam quando o usuario interrompe a resposta (Ctrl+C). Em caso de erro de API, o evento disparado e StopFailure, nao o Stop hook comum.

Diferente de PreToolUse e PostToolUse, Stop hooks nao suportam matchers. Eles sempre disparam em toda ocorrencia do evento Stop. Isso simplifica a configuracao, mas exige que a logica de decisao esteja dentro do proprio hook.

Tres tipos de Stop hook

$ Command

Executa um comando shell. O exit code determina se o Claude pode parar ou deve continuar trabalhando.

? Prompt

Envia a conversa para um modelo (Haiku por padrao) que avalia se a tarefa esta completa.

A Agent

Cria um subagente com acesso a ferramentas (Read, Grep, Glob) para inspecionar o estado do projeto.

Fluxo de Execucao do Stop Hook

Fluxo completo do Stop hook: cada resposta do Claude dispara o evento, e o hook decide se permite a parada ou forca o Claude a continuar.

Stop Hook tipo Command

O tipo command executa um comando shell quando o Claude termina de responder. O comportamento e controlado pelo exit code:

Exit CodeComportamentoDescricao
0Permite paradaO Claude para normalmente e devolve o controle ao usuario.
2Bloqueia paradaO Claude continua trabalhando. O conteudo de stderr e enviado como instrucao para o Claude.

Esse mecanismo e perfeito para validar automaticamente o codigo gerado. Se o build quebra, o typecheck falha ou os testes nao passam, o Claude recebe o erro e tenta corrigir sem intervencao humana.

Exemplo: rodar testes ao fim de cada resposta

.claude/settings.jsonJSON
{
  "hooks": {
    "Stop": [
      {
        "type": "command",
        "command": "npm test 2>&1 || (echo 'Testes falharam. Corrija os erros acima.' >&2 && exit 2)"
      }
    ]
  }
}

Neste exemplo, npm test e executado. Se os testes passam (exit 0), o Claude para. Se falham, o script redireciona a mensagem de erro para stderr e retorna exit 2, forcando o Claude a analisar a saida e corrigir o codigo.

Exemplo: typecheck com TypeScript

.claude/settings.jsonJSON
{
  "hooks": {
    "Stop": [
      {
        "type": "command",
        "command": "tsc --noEmit 2>&1 || (echo 'Erros de tipo encontrados. Corrija-os.' >&2 && exit 2)"
      }
    ]
  }
}
Dica: Voce pode encadear multiplas verificacoes em um unico comando usando &&. Por exemplo: tsc --noEmit && npm test && npm run lint. Se qualquer etapa falhar, o Claude recebe o erro e tenta corrigir.

Exemplo: build + typecheck + testes combinados

.claude/settings.jsonJSON
{
  "hooks": {
    "Stop": [
      {
        "type": "command",
        "command": "bash -c 'ERRORS=\"\"; tsc --noEmit 2>&1 || ERRORS=\"$ERRORS\nTypecheck falhou.\"; npm test 2>&1 || ERRORS=\"$ERRORS\nTestes falharam.\"; if [ -n \"$ERRORS\" ]; then echo \"$ERRORS\" >&2; exit 2; fi'"
      }
    ]
  }
}

Este hook mais sofisticado executa tanto o typecheck quanto os testes, acumula todos os erros e, se houver qualquer falha, envia todos de uma vez para o Claude corrigir.

Stop Hook tipo Prompt

O tipo prompt envia o contexto da conversa para um modelo auxiliar (Haiku por padrao) que avalia se a tarefa esta completa. O modelo retorna um objeto JSON com a decisao:

✓ Tarefa completa

{"ok": true} — O Claude para normalmente.

✗ Tarefa incompleta

{"ok": false, "reason": "descricao do que falta"} — O Claude continua usando o reason como proxima instrucao.

Exemplo: verificar se todas as tarefas estao completas

.claude/settings.jsonJSON
{
  "hooks": {
    "Stop": [
      {
        "type": "prompt",
        "prompt": "Verifique se o Claude completou TODAS as tarefas solicitadas pelo usuario. Considere: 1) Todos os arquivos mencionados foram criados ou modificados? 2) O codigo parece funcional e sem erros obvios? 3) Todas as instrucoes explicitas foram seguidas? Retorne {\"ok\": true} se tudo foi feito, ou {\"ok\": false, \"reason\": \"descricao do que falta\"} caso contrario."
      }
    ]
  }
}
Modelo padrao: O prompt-type Stop hook usa Haiku por padrao para manter baixa a latencia e o custo. Como Haiku avalia a cada turno, a escolha do modelo mais leve e intencional. Voce pode alterar o modelo se necessario, mas considere o impacto no tempo de resposta.

Stop Hook tipo Agent

O tipo agent e o mais poderoso. Ele cria um subagente completo com acesso a ferramentas de leitura do projeto (Read, Grep, Glob). O subagente pode inspecionar arquivos, verificar a existencia de testes, validar conteudo e tomar decisoes complexas.

ParametroPadraoDescricao
type"agent"Define o hook como subagente.
prompt—Instrucao para o subagente avaliar.
Timeout60 segundosTempo maximo de execucao do subagente.
Tool-use turnsAte 50Numero maximo de turnos de uso de ferramentas.
Resposta—{"ok": true} ou {"ok": false, "reason": "..."}

Exemplo: subagente que verifica se testes existem e passam

.claude/settings.jsonJSON
{
  "hooks": {
    "Stop": [
      {
        "type": "agent",
        "prompt": "Verifique se existem arquivos de teste correspondentes para cada arquivo de codigo-fonte modificado nesta sessao. Use Glob para encontrar arquivos *.test.ts ou *.spec.ts. Para cada arquivo de codigo modificado, confirme que existe um arquivo de teste correspondente. Se algum arquivo de teste estiver faltando, retorne {\"ok\": false, \"reason\": \"Faltam testes para: [lista de arquivos]\"}. Se todos os testes existem, retorne {\"ok\": true}."
      }
    ]
  }
}
Vantagem do Agent: Diferente do tipo command (que so roda shell) e do tipo prompt (que so analisa texto), o agent pode navegar pelo projeto, ler arquivos especificos, buscar padroes com Grep e tomar decisoes baseadas no estado real do codigo.

Block Cap: Evitando Loops Infinitos

Um risco real de Stop hooks e o loop infinito: o hook bloqueia, o Claude tenta corrigir mas nao consegue, o hook bloqueia novamente, e assim por diante. Para evitar isso, o Claude Code implementa um mecanismo de seguranca chamado block cap.

Comportamento padrao do Block Cap

Apos 8 bloqueios consecutivos sem progresso, o Claude Code ignora o Stop hook e forca a parada. Isso evita que o agente fique preso em um loop eterno tentando corrigir algo que nao consegue resolver.

O limite pode ser ajustado pela variavel de ambiente CLAUDE_CODE_STOP_HOOK_BLOCK_CAP.

Verificando stop_hook_active no script

Dentro de um Stop hook do tipo command, voce pode verificar o campo stop_hook_active presente no JSON de entrada do hook. Quando stop_hook_active e true, significa que o Claude ja esta em um ciclo de correcao disparado por um bloqueio anterior. Isso permite que o script decida ser mais tolerante para evitar loops.

stop-hook.shBash
#!/bin/bash
# Le o JSON de entrada do hook via stdin
INPUT=$(cat)

# Verifica se o hook ja esta ativo (evita loops)
HOOK_ACTIVE=$(echo "$INPUT" | jq -r '.stop_hook_active // false')

if [ "$HOOK_ACTIVE" = "true" ]; then
  # Ja estamos em um ciclo de correcao.
  # Ser mais tolerante para evitar loop infinito.
  echo "Stop hook ativo - permitindo parada para evitar loop." >&2
  exit 0
fi

# Execucao normal: roda testes
npm test 2>&1
TEST_EXIT=$?

if [ $TEST_EXIT -ne 0 ]; then
  echo "Testes falharam. Corrija os erros antes de finalizar." >&2
  exit 2
fi

exit 0
.claude/settings.jsonJSON
{
  "hooks": {
    "Stop": [
      {
        "type": "command",
        "command": "bash ./stop-hook.sh"
      }
    ]
  }
}
Cuidado: Se voce aumentar o block cap com CLAUDE_CODE_STOP_HOOK_BLOCK_CAP, esteja ciente de que isso pode resultar em consumo significativo de tokens e tempo. Um valor muito alto pode fazer o Claude gastar dezenas de milhares de tokens tentando resolver algo irresolvivel.

Estrategias Avancadas

Combinando multiplos Stop hooks

Voce pode definir multiplos Stop hooks na mesma configuracao. Todos sao executados, e qualquer um que retorne bloqueio impede a parada. Isso permite separar responsabilidades:

.claude/settings.jsonJSON
{
  "hooks": {
    "Stop": [
      {
        "type": "command",
        "command": "tsc --noEmit 2>&1 || (echo 'Typecheck falhou.' >&2 && exit 2)"
      },
      {
        "type": "command",
        "command": "npm test 2>&1 || (echo 'Testes falharam.' >&2 && exit 2)"
      },
      {
        "type": "command",
        "command": "npm run lint 2>&1 || (echo 'Lint falhou.' >&2 && exit 2)"
      }
    ]
  }
}

Stop hook seletivo por tipo de arquivo

Como Stop hooks nao tem matchers, voce pode implementar a logica de selecao dentro do proprio script. Por exemplo, so rodar testes se arquivos TypeScript foram modificados:

stop-hook-seletivo.shBash
#!/bin/bash
# Verifica se houve mudancas em arquivos .ts ou .tsx
CHANGED=$(git diff --name-only HEAD 2>/dev/null | grep -E '\.(ts|tsx)$')

if [ -z "$CHANGED" ]; then
  # Nenhum arquivo TypeScript alterado, pula verificacao
  exit 0
fi

# Roda typecheck e testes
tsc --noEmit 2>&1
if [ $? -ne 0 ]; then
  echo "Typecheck falhou em arquivos TypeScript modificados." >&2
  exit 2
fi

npm test 2>&1
if [ $? -ne 0 ]; then
  echo "Testes falharam." >&2
  exit 2
fi

exit 0

Boas Praticas

1 Mensagens claras no stderr

Quando o hook bloqueia, o conteudo de stderr e a instrucao que o Claude recebe. Seja explicito e objetivo sobre o que falhou e o que precisa ser corrigido.

2 Verifique stop_hook_active

Sempre considere verificar esse campo em scripts complexos para evitar loops infinitos quando o Claude nao consegue resolver o problema.

3 Prefira command para CI

Para build, typecheck e testes, o tipo command e o mais eficiente: zero custo adicional, execucao local rapida e feedback direto.

4 Limite a saida

Redirecione apenas os erros relevantes para stderr. Saidas muito longas podem confundir o Claude e desperdicar tokens.

Padrao recomendado para projetos TypeScript: Combine tsc --noEmit (typecheck) + npm test (testes) em um unico Stop hook command. Isso garante que o Claude so para quando o codigo compila e os testes passam, sem custo adicional de API e com feedback rapido.
