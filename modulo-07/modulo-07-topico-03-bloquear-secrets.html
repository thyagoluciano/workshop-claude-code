Hooks para Bloquear Commits com Secrets Expostos

Implementacao de uma pipeline de seguranca multicamada usando hooks do Claude Code para prevenir vazamento de credenciais, tokens e chaves em cada etapa do fluxo de desenvolvimento.

O Problema: Secrets no Codigo-fonte

Vazamento de secrets e a causa raiz de alguns dos incidentes de seguranca mais caros da industria. Uma unica chave AWS commitada acidentalmente pode resultar em centenas de milhares de dolares em custos de mineracao de criptomoedas em questao de minutos. O problema se agrava porque secrets commitados persistem no historico do Git mesmo apos remocao — exigindo reescrita completa do historico ou rotacao imediata da credencial.

O sistema de hooks do Claude Code oferece uma oportunidade unica: interceptar secrets antes que cheguem ao repositorio, em multiplos pontos do fluxo. Diferente de git hooks tradicionais (pre-commit), os hooks do Claude Code operam no nivel da ferramenta de IA, bloqueando tanto a escrita quanto o commit de secrets.

Defesa em profundidade: Nenhuma camada isolada e suficiente. A estrategia correta combina PreToolUse (bloquear escrita e commits), PostToolUse (redactar output) e Stop hooks (scan final). Se um secret passar por uma camada, a proxima o captura.

Arquitetura da Pipeline de Seguranca

Pipeline multicamada: cada checkpoint intercepta secrets em um ponto diferente do fluxo

Camada 1: PreToolUse — Bloquear Escrita de Secrets

A primeira linha de defesa intercepta chamadas ao Edit e Write antes que o Claude escreva conteudo nos arquivos. O hook analisa o campo tool_input procurando patterns conhecidos de secrets. Se encontrar, retorna exit code 2 para bloquear a operacao.

Configuracao do Hook

.claude/settings.jsonJSON
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "hook": ".claude/hooks/detect-secrets-content.sh",
        "timeout": 10000
      }
    ]
  }
}

O matcher "Edit|Write" usa a sintaxe de regex para capturar ambas as ferramentas. O hook recebe via stdin um JSON contendo tool_name e tool_input — onde tool_input contem o conteudo que sera escrito.

Script de Deteccao

.claude/hooks/detect-secrets-content.shBash
#!/bin/bash
# detect-secrets-content.sh
# Analisa o conteudo que Claude esta prestes a escrever
# Exit 2 = bloquear operacao | Exit 0 = permitir

INPUT=$(cat)
TOOL_NAME=$(echo "$INPUT" | jq -r '.tool_name')

# Extrai o conteudo sendo escrito dependendo da ferramenta
if [ "$TOOL_NAME" = "Edit" ]; then
  CONTENT=$(echo "$INPUT" | jq -r '.tool_input.new_string // empty')
elif [ "$TOOL_NAME" = "Write" ]; then
  CONTENT=$(echo "$INPUT" | jq -r '.tool_input.content // empty')
else
  exit 0
fi

[ -z "$CONTENT" ] && exit 0

# Patterns de secrets conhecidos
PATTERNS=(
  # AWS Access Key ID
  'AKIA[0-9A-Z]{16}'
  # AWS Secret Access Key
  '[0-9a-zA-Z/+]{40}'
  # GitHub Personal Access Token (classic e fine-grained)
  'ghp_[0-9a-zA-Z]{36}'
  'github_pat_[0-9a-zA-Z_]{82}'
  # GitLab tokens
  'glpat-[0-9a-zA-Z\-_]{20}'
  # Slack tokens
  'xox[bporas]-[0-9a-zA-Z\-]{10,}'
  # Generic API key patterns
  '(?i)(api[_-]?key|api[_-]?secret|access[_-]?token)\s*[=:]\s*["\x27][0-9a-zA-Z\-_.]{16,}["\x27]'
  # Private keys
  '-----BEGIN (RSA |EC |DSA |OPENSSH )?PRIVATE KEY-----'
  # Password em variaveis
  '(?i)(password|passwd|pwd)\s*[=:]\s*["\x27][^\s"'\'']{8,}["\x27]'
  # Connection strings com credenciais
  '(?i)(mongodb|postgres|mysql|redis):\/\/[^:]+:[^@]+@'
  # JWT tokens
  'eyJ[0-9a-zA-Z_-]{10,}\.eyJ[0-9a-zA-Z_-]{10,}\.[0-9a-zA-Z_-]+'
)

for PATTERN in "${PATTERNS[@]}"; do
  if echo "$CONTENT" | grep -qP "$PATTERN"; then
    MATCH=$(echo "$CONTENT" | grep -oP "$PATTERN" | head -1)
    # Redacta o match para o log
    REDACTED="${MATCH:0:6}...${MATCH: -4}"
    echo "BLOQUEADO: Secret detectado no conteudo ($REDACTED)"
    echo "Pattern: $PATTERN"
    echo "Ferramenta: $TOOL_NAME"
    echo ""
    echo "Remova o secret e use variaveis de ambiente ou um gerenciador de secrets."
    exit 2
  fi
done

exit 0
Por que exit 2? Na convencao de hooks do Claude Code, exit 0 permite a operacao, exit 1 indica erro no proprio hook (Claude pode tentar novamente), e exit 2 bloqueia a operacao — o stdout do script e mostrado ao Claude como motivo do bloqueio.

Camada 2: PreToolUse — Bloquear Commits com Secrets

A segunda camada intercepta chamadas ao Bash que executam git commit ou git push. O hook faz scan dos arquivos staged antes que o commit aconteca.

Configuracao do Hook

.claude/settings.jsonJSON
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hook": ".claude/hooks/scan-staged-secrets.sh",
        "timeout": 15000
      }
    ]
  }
}
Filtragem pelo comando: O matcher captura todas as chamadas ao Bash. O script e responsavel por verificar se o comando e um git commit ou git push e ignorar outros comandos. Isso e mais robusto que filtrar no matcher porque captura variantes como git commit -am, pipelines com &&, etc.

Script de Scan dos Arquivos Staged

.claude/hooks/scan-staged-secrets.shBash
#!/bin/bash
# scan-staged-secrets.sh
# Intercepta git commit/push e faz scan de secrets nos arquivos staged

INPUT=$(cat)
COMMAND=$(echo "$INPUT" | jq -r '.tool_input.command // empty')

# So processa comandos git commit ou git push
if ! echo "$COMMAND" | grep -qE '^\s*git\s+(commit|push)'; then
  exit 0
fi

# Obtem a lista de arquivos staged
STAGED_FILES=$(git diff --cached --name-only --diff-filter=ACMR 2>/dev/null)

if [ -z "$STAGED_FILES" ]; then
  exit 0
fi

FOUND_SECRETS=0
REPORT=""

# Patterns de alta confianca (baixo false-positive)
declare -A SECRET_PATTERNS
SECRET_PATTERNS=(
  ["AWS Access Key"]='AKIA[0-9A-Z]{16}'
  ["GitHub Token"]='gh[ps]_[0-9a-zA-Z]{36}'
  ["GitHub Fine-grained PAT"]='github_pat_[0-9a-zA-Z_]{82}'
  ["GitLab Token"]='glpat-[0-9a-zA-Z\-_]{20}'
  ["Slack Token"]='xox[bporas]-[0-9a-zA-Z\-]+'
  ["Private Key"]='-----BEGIN (RSA |EC |DSA |OPENSSH )?PRIVATE KEY-----'
  ["Generic Secret Assignment"]='(?i)(secret|token|password|api_key)\s*[=:]\s*["\x27][A-Za-z0-9+/=\-_.]{20,}["\x27]'
  ["Connection String"]='(?i)(mongodb|postgres|mysql|redis):\/\/[^:]+:[^@]+@'
  ["JWT"]='eyJ[0-9a-zA-Z_-]{10,}\.eyJ[0-9a-zA-Z_-]{10,}'
)

while IFS= read -r FILE; do
  [ -f "$FILE" ] || continue

  # Ignora arquivos binarios e lockfiles
  if file "$FILE" | grep -q "binary"; then
    continue
  fi

  # Ignora lockfiles e dependencias
  case "$FILE" in
    *.lock|package-lock.json|yarn.lock|Cargo.lock|go.sum)
      continue
      ;;
  esac

  # Scan do conteudo staged (nao do working tree)
  CONTENT=$(git show ":$FILE" 2>/dev/null)
  [ -z "$CONTENT" ] && continue

  for LABEL in "${!SECRET_PATTERNS[@]}"; do
    PATTERN="${SECRET_PATTERNS[$LABEL]}"
    if echo "$CONTENT" | grep -qP "$PATTERN"; then
      MATCH=$(echo "$CONTENT" | grep -oP "$PATTERN" | head -1)
      LINE_NUM=$(echo "$CONTENT" | grep -nP "$PATTERN" | head -1 | cut -d: -f1)
      REDACTED="${MATCH:0:6}...${MATCH: -4}"
      REPORT+="  - $FILE:$LINE_NUM -> $LABEL ($REDACTED)\n"
      FOUND_SECRETS=$((FOUND_SECRETS + 1))
    fi
  done
done <<< "$STAGED_FILES"

if [ $FOUND_SECRETS -gt 0 ]; then
  echo "BLOQUEADO: $FOUND_SECRETS secret(s) encontrado(s) nos arquivos staged:"
  echo ""
  echo -e "$REPORT"
  echo ""
  echo "Acoes recomendadas:"
  echo "  1. Remova os secrets dos arquivos"
  echo "  2. Use variaveis de ambiente ou .env (adicionado ao .gitignore)"
  echo "  3. Use um gerenciador de secrets (Vault, AWS Secrets Manager, etc.)"
  echo "  4. Execute: git reset HEAD  para unstage"
  exit 2
fi

exit 0

O script usa git show ":$FILE" para ler o conteudo da staging area, nao do working tree. Isso e importante porque o desenvolvedor pode ter removido o secret do arquivo mas ele ainda estar staged para commit.

Camada 3: PostToolUse — Redacao de Output

Mesmo que secrets nao sejam escritos em codigo, eles podem aparecer no output de comandos executados pelo Claude — por exemplo, ao rodar env, cat .env, ou ler logs de aplicacao. O hook PostToolUse pode reescrever o output da ferramenta antes que o Claude o processe.

Configuracao

.claude/settings.jsonJSON
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Bash|Read",
        "hook": ".claude/hooks/redact-output.sh",
        "timeout": 10000
      }
    ]
  }
}

Script de Redacao

.claude/hooks/redact-output.shBash
#!/bin/bash
# redact-output.sh
# Redacta secrets no output de ferramentas antes que Claude os veja
# PostToolUse: stdout substitui o tool_result se nao-vazio

INPUT=$(cat)
OUTPUT=$(echo "$INPUT" | jq -r '.tool_result.stdout // .tool_result.content // empty')

[ -z "$OUTPUT" ] && exit 0

REDACTED="$OUTPUT"

# Redacta AWS keys
REDACTED=$(echo "$REDACTED" | sed -E 's/AKIA[0-9A-Z]{16}/AKIA***REDACTED***/g')

# Redacta tokens GitHub
REDACTED=$(echo "$REDACTED" | sed -E 's/gh[ps]_[0-9a-zA-Z]{36}/gh*_***REDACTED***/g')

# Redacta tokens genericos em variaveis de ambiente
REDACTED=$(echo "$REDACTED" | sed -E 's/(SECRET|TOKEN|PASSWORD|API_KEY|PRIVATE_KEY)=([^ "\x27]+)/\1=***REDACTED***/gI')

# Redacta connection strings
REDACTED=$(echo "$REDACTED" | sed -E 's|(mongodb\|postgres\|mysql\|redis)://([^:]+):([^@]+)@|\1://\2:***REDACTED***@|gI')

# Redacta JWT tokens
REDACTED=$(echo "$REDACTED" | sed -E 's/eyJ[0-9a-zA-Z_-]{10,}\.eyJ[0-9a-zA-Z_-]{10,}\.[0-9a-zA-Z_-]+/***JWT_REDACTED***/g')

# Se houve alteracao, emite o output redactado
if [ "$REDACTED" != "$OUTPUT" ]; then
  echo "[AVISO: Secrets foram redactados do output por seguranca]"
  echo ""
  echo "$REDACTED"
fi

# Se nao houve alteracao, nao emite nada (output original e mantido)
exit 0
Comportamento do PostToolUse: Se o script emitir algo no stdout, esse conteudo substitui o resultado original da ferramenta. Se o stdout estiver vazio, o resultado original e preservado. Isso permite modificar seletivamente apenas outputs que contenham secrets.

Camada 4: Stop Hook — Scan Final com Ferramentas Dedicadas

O Stop hook roda ao final de cada resposta do Claude, apos todas as ferramentas terem sido executadas. E a ultima linha de defesa — ideal para rodar ferramentas dedicadas de deteccao como gitleaks ou trufflehog, que possuem bases de regras muito mais abrangentes.

Configuracao

.claude/settings.jsonJSON
{
  "hooks": {
    "Stop": [
      {
        "matcher": "",
        "hook": ".claude/hooks/scan-final-secrets.sh",
        "timeout": 30000
      }
    ]
  }
}

O matcher vazio significa que o hook roda em toda parada do Claude. O timeout maior (30s) acomoda o tempo de scan de ferramentas externas.

Script de Integracao com gitleaks

.claude/hooks/scan-final-secrets.shBash
#!/bin/bash
# scan-final-secrets.sh
# Stop hook: roda gitleaks no working directory apos cada resposta do Claude
# Exit 2 = bloqueia (Claude continua para corrigir)

INPUT=$(cat)

# Extrai os arquivos modificados na sessao do stop_hook_input
MODIFIED_FILES=$(echo "$INPUT" | jq -r '
  .tool_results[]? |
  select(.tool_name == "Edit" or .tool_name == "Write") |
  .tool_input.file_path // .tool_input.path
' 2>/dev/null | sort -u)

# Se nenhum arquivo foi modificado, nao precisa escanear
[ -z "$MODIFIED_FILES" ] && exit 0

# Verifica se gitleaks esta instalado
if ! command -v gitleaks &>/dev/null; then
  # Fallback: usa scan com grep se gitleaks nao disponivel
  FOUND=0
  while IFS= read -r FILE; do
    [ -f "$FILE" ] || continue
    if grep -qP 'AKIA[0-9A-Z]{16}|gh[ps]_[0-9a-zA-Z]{36}|-----BEGIN .* PRIVATE KEY-----|(?i)(password|secret|token)\s*[=:]\s*["\x27][^\s]{16,}' "$FILE" 2>/dev/null; then
      FOUND=$((FOUND + 1))
      echo "Secret potencial em: $FILE"
    fi
  done <<< "$MODIFIED_FILES"

  if [ $FOUND -gt 0 ]; then
    echo ""
    echo "ATENCAO: $FOUND arquivo(s) com secrets potenciais."
    echo "Corrija antes de prosseguir."
    exit 2
  fi
  exit 0
fi

# Scan com gitleaks nos arquivos modificados
REPORT=$(mktemp)
SCAN_FAILED=0

while IFS= read -r FILE; do
  [ -f "$FILE" ] || continue
  gitleaks detect --source="$FILE" --no-git --report-format=json --report-path="$REPORT" 2>/dev/null
  if [ $? -eq 1 ]; then
    SCAN_FAILED=1
    FINDINGS=$(jq -r '.[] | "  - \(.File):\(.StartLine) \(.Description) [\(.RuleID)]"' "$REPORT" 2>/dev/null)
    echo "$FINDINGS"
  fi
done <<< "$MODIFIED_FILES"

rm -f "$REPORT"

if [ $SCAN_FAILED -eq 1 ]; then
  echo ""
  echo "BLOQUEADO: gitleaks detectou secrets nos arquivos modificados."
  echo "Claude deve remover os secrets e substituir por variaveis de ambiente."
  exit 2
fi

exit 0
Stop hook com exit 2: Quando um Stop hook retorna exit 2, o Claude nao para — ele continua executando para tentar corrigir o problema apontado no stdout. Isso permite um ciclo automatico de deteccao e correcao: o hook detecta o secret, Claude remove e substitui por variavel de ambiente, o hook roda novamente e valida.

Estrategias Avancadas

Reducao de Falsos Positivos

1 Allowlists

Mantenha um arquivo .claude/hooks/secret-allowlist.txt com hashes SHA256 de valores permitidos (ex: chaves de teste documentadas). O script compara o hash do match contra a allowlist antes de bloquear.

2 Contexto de Arquivo

Ignore patterns em arquivos de teste (*_test.go, *.spec.ts), fixtures e mocks. Secrets em testes geralmente sao valores falsos. Mas valide que nao sao secrets reais copiados por engano.

3 Entropia

Combine regex com analise de entropia de Shannon. Strings com entropia alta (> 4.5 bits/char) em contexto de atribuicao sao mais provavelmente secrets reais do que valores de teste como "test123".

4 Baseline

Rode o scan no repositorio antes de ativar hooks para criar um baseline. Secrets preexistentes devem ser tratados separadamente (rotacao + remocao do historico) para nao gerar ruido constante.

Integracao com .gitignore e .env

Verificacao adicional no hookBash
# Dentro do hook de PreToolUse para Write
# Bloqueia escrita de secrets em arquivos que NAO estao no .gitignore

FILE_PATH=$(echo "$INPUT" | jq -r '.tool_input.file_path // .tool_input.path // empty')

# Permite secrets em .env (desde que esteja no .gitignore)
if echo "$FILE_PATH" | grep -qE '\.env(\.local|\.development)?$'; then
  if git check-ignore -q "$FILE_PATH" 2>/dev/null; then
    # Arquivo esta no .gitignore, permite
    exit 0
  else
    echo "AVISO: $FILE_PATH contem secrets mas NAO esta no .gitignore!"
    echo "Adicione ao .gitignore antes de continuar."
    exit 2
  fi
fi

    scan-staged-secrets.sh     # Camada 2: PreToolUse Bash (git commit)

Importante sobre permissoes: Todos os scripts de hook devem ter permissao de execucao (chmod +x). O Claude Code executa hooks como subprocessos — se o script nao for executavel, o hook falha silenciosamente com exit 1 e a operacao prossegue sem protecao.

Consideracoes de Performance

⏱ Timeout

Configure timeouts realistas. Scan de regex em arquivos pequenos: 5-10s. gitleaks em repo medio: 15-30s. Timeout excedido = hook ignorado.

⚡ Escopo

Escaneie apenas arquivos relevantes. Ignore binarios, lockfiles e node_modules. O hook de staging so le arquivos na staging area, nao o repo inteiro.

⚙ Cache

Para repos grandes, considere cachear resultados de scan por hash do arquivo. Se o hash nao mudou, reutilize o resultado anterior.

Fluxo Completo: Da Deteccao a Correcao

Quando o Stop hook detecta um secret e retorna exit 2, o Claude recebe a mensagem de erro e automaticamente tenta corrigir. O fluxo tipico e:

1. Claude escreve codigo contendo um token hardcoded

2. PreToolUse (Camada 1) bloqueia o Write — Claude recebe a mensagem de erro

3. Claude reescreve usando os.environ.get("API_TOKEN")

4. PreToolUse permite o Write — arquivo salvo

5. Claude tenta commit

6. PreToolUse (Camada 2) escaneia staging — nenhum secret encontrado

7. Commit realizado com sucesso

8. Stop hook (Camada 4) roda gitleaks — validacao final OK

Resultado: Codigo seguro, sem secrets, com variavel de ambiente configurada corretamente.

Se um secret passar pela Camada 1 (por exemplo, um pattern novo nao coberto), a Camada 2 o captura no commit. Se o commit ja existia antes dos hooks, a Camada 4 detecta no scan final. Cada camada e independente e complementar.
