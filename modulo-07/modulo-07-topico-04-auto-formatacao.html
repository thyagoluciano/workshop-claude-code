Auto-formatacao na Escrita de Arquivos

Como usar hooks PostToolUse e Stop para garantir formatacao deterministica em todo codigo gerado pelo Claude Code, independente da linguagem ou ferramenta utilizada.

O Problema: Formatacao Inconsistente

Quando o Claude Code escreve ou edita arquivos, o LLM gera codigo com base no contexto disponivel. Mesmo com instrucoes explicitas no CLAUDE.md como "use 2 espacos de indentacao" ou "siga o estilo do projeto", nao ha garantia deterministica de que a formatacao sera consistente em toda interacao.

Instruir o modelo via prompt e uma abordagem probabilistica: funciona na maioria das vezes, mas eventualmente falha. O custo de uma falha de formatacao parece baixo isoladamente, mas acumula-se em diffs poluidos, conflitos de merge e revisoes de PR desnecessarias.

Principio fundamental: Hooks sao deterministicos. Se o matcher corresponde, o comando executa. Sempre. Isso elimina a classe inteira de problemas de formatacao inconsistente, movendo a responsabilidade do LLM para uma ferramenta dedicada.

A documentacao oficial da Anthropic apresenta exatamente esse caso de uso como exemplo canonico de PostToolUse hooks: executar Prettier automaticamente apos cada edicao de arquivo.

Anatomia do Hook de Auto-formatacao

O padrao segue uma estrutura simples: interceptar chamadas as tools Edit e Write, extrair o caminho do arquivo do JSON de input da tool, e passar esse caminho para o formatador adequado.

O exemplo canonico: Prettier

Este e o exemplo direto da documentacao oficial da Anthropic para auto-formatacao:

.claude/settings.jsonPostToolUse hook
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.file_path' | xargs npx prettier --write"
          }
        ]
      }
    ]
  }
}

Vamos decompor o comando jq -r '.tool_input.file_path' | xargs npx prettier --write:

SegmentoFuncao
jq -r '.tool_input.file_path'Recebe o JSON de input da tool via stdin e extrai o campo file_path. A flag -r remove as aspas do output.
|Pipe do caminho do arquivo para o proximo comando.
xargs npx prettier --writeExecuta Prettier no arquivo extraido, sobrescrevendo-o com a versao formatada (--write).
Sobre o JSON de input: O hook recebe via stdin um objeto JSON contendo tool_name, tool_input (os parametros passados a tool) e tool_response (a resposta da tool). Para Edit e Write, o tool_input sempre inclui file_path.

O matcher: Edit|Write

O campo matcher aceita uma regex que e testada contra o nome da tool. Edit|Write captura ambas as tools de escrita de arquivo do Claude Code. Quando o matcher corresponde, todos os hooks listados no array hooks sao executados em sequencia.

Isso significa que o formatador roda a cada edicao individual. Se o Claude faz 5 edicoes em um arquivo, Prettier roda 5 vezes. Isso e intencional: garante que o arquivo esta formatado apos cada modificacao, prevenindo estados intermediarios inconsistentes.

Diagrama: Fluxo de Auto-formatacao

Dois fluxos complementares: PostToolUse para formatacao imediata, Stop hook para cobertura completa incluindo arquivos criados via Bash.

Expandindo para Outros Formatadores

O exemplo com Prettier cobre JavaScript, TypeScript, CSS, HTML e JSON. Mas projetos reais frequentemente envolvem multiplas linguagens. Cada ecossistema tem seu formatador padrao, e todos seguem o mesmo padrao de integracao via hook.

ESLint com auto-fix

Para projetos que usam ESLint como formatador (com regras de estilo configuradas), o hook segue o mesmo padrao, substituindo Prettier por eslint --fix:

.claude/settings.jsonESLint auto-fix
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.file_path' | grep -E '\\.(js|ts|jsx|tsx)$' | xargs -I{} npx eslint --fix {}"
          }
        ]
      }
    ]
  }
}
Filtro por extensao: Note o uso de grep -E para filtrar apenas arquivos com extensoes relevantes. Sem esse filtro, ESLint tentaria processar arquivos como .json ou .md e poderia falhar. Prettier e mais tolerante nesse aspecto, ignorando arquivos que nao reconhece.

Script Multi-linguagem Inteligente

Em projetos poliglotas, manter um hook separado por linguagem e fragil e verboso. A abordagem robusta e criar um script que detecta a extensao do arquivo e despacha para o formatador correto:

scripts/format-file.shScript de despacho por extensao
#!/usr/bin/env bash
set -euo pipefail

# Recebe o path do arquivo via stdin (do jq)
FILE_PATH=$(cat)

# Se o arquivo nao existe, sai silenciosamente
[[ -f "$FILE_PATH" ]] || exit 0

# Extrai a extensao
EXT="${FILE_PATH##*.}"

case "$EXT" in
  js|jsx|ts|tsx|css|html|json|md|yaml|yml)
    npx prettier --write "$FILE_PATH" 2>/dev/null
    ;;
  py)
    black --quiet "$FILE_PATH" 2>/dev/null
    ;;
  go)
    gofmt -w "$FILE_PATH" 2>/dev/null
    ;;
  rs)
    rustfmt "$FILE_PATH" 2>/dev/null
    ;;
  rb)
    rubocop -A --fail-level=F "$FILE_PATH" 2>/dev/null
    ;;
  java)
    google-java-format -i "$FILE_PATH" 2>/dev/null
    ;;
  c|cpp|h|hpp)
    clang-format -i "$FILE_PATH" 2>/dev/null
    ;;
  *)
    # Extensao nao reconhecida: nao faz nada
    ;;
esac

exit 0

O hook no settings.json fica simples:

.claude/settings.jsonHook com script multi-linguagem
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.file_path' | bash scripts/format-file.sh"
          }
        ]
      }
    ]
  }
}
Resiliencia com 2>/dev/null e exit 0: O script redireciona stderr para /dev/null e sempre retorna exit code 0. Isso e critico porque um hook que falha (exit code diferente de 0) bloqueia a operacao do Claude Code. Se um formatador nao esta instalado, o script simplesmente ignora e continua.

O Gap do Bash: Arquivos Criados via Shell

A documentacao oficial da Anthropic destaca um ponto importante: o Claude pode criar ou modificar arquivos executando comandos shell via Bash tool. Um PostToolUse com matcher Edit|Write nao captura essas modificacoes, porque a tool utilizada e Bash, nao Edit ou Write.

✓ Capturado pelo PostToolUse

Claude usa Edit para modificar src/app.ts. Hook dispara, Prettier formata.

✗ NAO capturado

Claude usa Bash com cat > src/config.ts. PostToolUse Edit|Write nao dispara.

Cenarios comuns onde o Claude usa Bash para criar/modificar arquivos:

  • Gerar arquivos de configuracao (cat > .env, echo > config.yaml)
  • Executar scaffolding (npx create-react-app, cargo init)
  • Aplicar patches (sed -i, patch)
  • Copiar templates (cp template.ts src/new-file.ts)
Risco real: Se voce depende apenas do PostToolUse com Edit|Write, arquivos criados via Bash escapam completamente da formatacao automatica. A solucao e complementar com um Stop hook.

Stop Hook: Formatacao Final Completa

O hook Stop executa quando o Claude Code encerra uma resposta. Ele e o ponto ideal para uma varredura final de todos os arquivos modificados durante a sessao, usando git para identificar o que mudou:

scripts/format-changed.shScript de formatacao final via git
#!/usr/bin/env bash
set -euo pipefail

# Lista arquivos modificados (staged e unstaged) com git
# M = modificado, A = adicionado, ? = untracked
CHANGED_FILES=$(git status --porcelain | awk '{print $2}')

if [[ -z "$CHANGED_FILES" ]]; then
  exit 0
fi

for FILE in $CHANGED_FILES; do
  # Pula se o arquivo nao existe (pode ter sido deletado)
  [[ -f "$FILE" ]] || continue

  EXT="${FILE##*.}"

  case "$EXT" in
    js|jsx|ts|tsx|css|html|json|md|yaml|yml)
      npx prettier --write "$FILE" 2>/dev/null || true
      ;;
    py)
      black --quiet "$FILE" 2>/dev/null || true
      ;;
    go)
      gofmt -w "$FILE" 2>/dev/null || true
      ;;
    rs)
      rustfmt "$FILE" 2>/dev/null || true
      ;;
    *)
      ;;
  esac
done

exit 0
.claude/settings.jsonStop hook para formatacao final
{
  "hooks": {
    "Stop": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "bash scripts/format-changed.sh"
          }
        ]
      }
    ]
  }
}
Sobre git status --porcelain: A flag --porcelain produz output machine-readable, estavel entre versoes do git. Cada linha comeca com um codigo de status de 2 caracteres seguido do caminho do arquivo. O awk '{print $2}' extrai apenas os caminhos. Arquivos renomeados aparecem com o caminho de destino.

Configuracao Completa: PostToolUse + Stop

A configuracao ideal combina ambos os hooks para cobertura deterministica total:

.claude/settings.jsonConfiguracao completa de auto-formatacao
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "jq -r '.tool_input.file_path' | bash scripts/format-file.sh"
          }
        ]
      }
    ],
    "Stop": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "bash scripts/format-changed.sh"
          }
        ]
      }
    ]
  }
}

1 PostToolUse

Formata imediatamente apos cada Edit/Write. Feedback rapido e incremental.

2 Stop

Varredura final de TODOS os arquivos modificados, incluindo os criados via Bash.

3 Resultado

100% dos arquivos modificados sao formatados. Zero dependencia do LLM para estilo.

Consideracoes de Performance e Robustez

Tempo de execucao

Cada hook de PostToolUse adiciona latencia a cada operacao de edicao. Para projetos grandes, considere:

FormatadorTempo tipico (arquivo unico)Impacto percebido
Prettier (via npx)200-800ms (primeira execucao com cold start do npx)Noticeable na primeira chamada
Prettier (instalado local)50-150msImperceptivel
Black30-100msImperceptivel
gofmt10-50msImperceptivel
rustfmt50-200msImperceptivel
Dica de performance: Evite npx em hooks de alta frequencia. Instale o formatador como dependencia local (npm install --save-dev prettier) e use ./node_modules/.bin/prettier --write ou configure o PATH adequadamente. A diferenca pode ser de 500ms+ por invocacao.

Tratamento de erros

Hooks que retornam exit code diferente de 0 sao tratados como falha. A estrategia defensiva e:

  • Sempre redirecionar stderr: 2>/dev/null ou 2>/tmp/format-errors.log
  • Sempre garantir exit 0: Use || true ou || exit 0 apos comandos que podem falhar
  • Verificar existencia do arquivo: [[ -f "$FILE" ]] || exit 0 antes de formatar
  • Verificar existencia do formatador: command -v prettier &>/dev/null || exit 0

Interacao com .prettierignore e configuracoes locais

Formatadores respeitam seus arquivos de configuracao naturalmente. Prettier le .prettierrc e .prettierignore, Black le pyproject.toml, e assim por diante. Nenhuma configuracao adicional e necessaria no hook. O hook simplesmente invoca o formatador, que usa suas proprias regras de configuracao do projeto.

Determinismo vs. Instrucao via Prompt

Para concluir, vale reforcar a diferenca fundamental entre as duas abordagens:

≈ Via prompt (probabilistico)

Adicionar "use 2 espacos, Prettier style" no CLAUDE.md funciona na maioria das vezes, mas o LLM pode ignorar, especialmente em contextos longos ou quando esta focado em logica complexa. Nenhuma garantia formal.

✓ Via hook (deterministico)

O hook executa incondicionalmente quando o matcher corresponde. A formatacao e aplicada por uma ferramenta dedicada, usando as mesmas regras que o CI/CD. Zero variancia entre execucoes.

Hooks de auto-formatacao nao substituem instrucoes de estilo no CLAUDE.md. As instrucoes de prompt ajudam o Claude a gerar codigo mais proximo do estilo final, reduzindo o diff que o formatador precisa aplicar. O hook e a rede de seguranca que garante consistencia absoluta.
