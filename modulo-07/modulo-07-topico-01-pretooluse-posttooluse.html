PreToolUse e PostToolUse: interceptar acoes antes e depois

Hooks que interceptam chamadas de ferramenta permitem bloquear operacoes perigosas, reescrever argumentos e filtrar saidas antes que Claude as veja. Entenda o ciclo de vida completo de uma tool call e como cada ponto de interceptacao funciona.

O modelo mental: lifecycle de uma tool call

Toda vez que Claude decide invocar uma ferramenta (Bash, Edit, Write, Read, MCP tools, etc.), a execucao passa por um pipeline bem definido. Os hooks PreToolUse e PostToolUse sao pontos de interceptacao nesse pipeline — o primeiro antes da execucao, o segundo depois.

Lifecycle completo de uma tool call com pontos de interceptacao PreToolUse e PostToolUse
Ponto critico: Hooks PreToolUse disparam antes de qualquer verificacao de permission mode. Um hook que retorna "deny" bloqueia a ferramenta mesmo quando Claude Code esta rodando com --dangerously-skip-permissions (bypassPermissions). Isso torna hooks a unica camada de seguranca que nao pode ser contornada por flags de CLI.

PreToolUse em profundidade

O que o hook recebe

Quando um hook PreToolUse e acionado, Claude Code envia um JSON via stdin para o processo do hook. O payload contem todas as informacoes necessarias para decidir se a acao deve prosseguir.

JSON · stdin do hook PreToolUse
{
  "hook_event_name": "PreToolUse",
  "session_id": "abc123-def456",
  "cwd": "/home/user/my-project",
  "tool_name": "Edit",
  "tool_input": {
    "file_path": "/home/user/my-project/src/index.ts",
    "old_string": "const x = 1",
    "new_string": "const x = 2"
  }
}

Exit codes e decisoes

O hook comunica sua decisao atraves de uma combinacao de exit code e JSON em stdout. Os dois mecanismos se complementam:

Exit Code Significado Comportamento
0 Sem objecao O fluxo normal de permissoes continua (permission mode e aplicado)
2 Bloquear A tool call e cancelada imediatamente, sem passar pelo permission check
Qualquer outro Erro no hook Claude Code reporta o erro mas nao bloqueia a acao

JSON de resposta: permissionDecision

Alem do exit code, o hook pode emitir JSON em stdout com controle mais granular. O campo permissionDecision aceita tres valores:

✓ allow

Permite a execucao sem pedir confirmacao ao usuario, ignorando o permission mode.

✗ deny

Bloqueia a execucao incondicionalmente. Equivalente funcional a exit code 2.

? ask

Forca o prompt de confirmacao mesmo que o permission mode normalmente auto-aprovasse.

JSON · stdout do hook (exemplo deny com razao)
{
  "permissionDecision": "deny",
  "reason": "Edicao em .env bloqueada por politica de seguranca"
}

Reescrita de argumentos: updatedInput

Um hook PreToolUse pode modificar os argumentos da tool call antes da execucao. Isso e feito retornando o campo updatedInput no JSON de stdout:

JSON · stdout com updatedInput
{
  "updatedInput": {
    "command": "git diff --staged --no-color"
  }
}
Atencao com multiplos hooks: Quando mais de um hook PreToolUse retorna updatedInput para a mesma tool call, o ultimo hook a executar vence — mas a ordem de execucao entre hooks nao e deterministica. Evite cenarios onde multiplos hooks tentam reescrever os mesmos argumentos.

Matchers: filtrando por nome de ferramenta

O campo matcher na configuracao do hook define para quais ferramentas ele dispara. Aceita nomes exatos ou regex:

Matcher Ferramentas afetadas
Bash Apenas a tool Bash
Edit|Write Edit ou Write (regex OR)
mcp__.* Todas as ferramentas MCP (qualquer server)
mcp__github__.* Ferramentas do MCP server "github"
.* Todas as ferramentas (catch-all)

Campo if: filtragem granular por argumentos

Alem do matcher por nome de ferramenta, o campo if usa a mesma sintaxe de permission rules para filtrar com base nos argumentos da tool call. Isso permite hooks cirurgicos:

Exemplos de if patterns
// Dispara apenas para comandos git no Bash
"Bash(git *)"

// Dispara apenas para edits em arquivos TypeScript
"Edit(*.ts)"

// Dispara para Write em qualquer arquivo dentro de src/
"Write(src/**)"

// Dispara para Bash com rm
"Bash(rm *)"

PostToolUse em profundidade

O hook PostToolUse dispara apos a ferramenta ter sido executada com sucesso. A acao ja aconteceu — o arquivo ja foi editado, o comando ja rodou, a requisicao ja foi feita. Nao ha como desfazer.

O que o hook recebe

O payload e similar ao PreToolUse, mas inclui o resultado da execucao da ferramenta:

JSON · stdin do hook PostToolUse
{
  "hook_event_name": "PostToolUse",
  "session_id": "abc123-def456",
  "cwd": "/home/user/my-project",
  "tool_name": "Edit",
  "tool_input": {
    "file_path": "/home/user/my-project/src/index.ts",
    "old_string": "const x = 1",
    "new_string": "const x = 2"
  },
  "tool_result": "File edited successfully"
}

Capacidades do PostToolUse

✎ Reescrever output

Modifica o resultado que Claude vera. Util para redactar secrets, normalizar diffs ou filtrar output ruidoso.

⚡ Side-effects

Executar acoes como formatar codigo, rodar linters, enviar notificacoes ou logar metricas.

📊 Logging

Registrar todas as tool calls em um arquivo de auditoria, banco de dados ou servico externo.

📋 Validacao pos-acao

Checar se o resultado atende criterios (ex: testes passando, build ok) e notificar se nao.

Para reescrever o output, o hook emite JSON em stdout com o campo result:

JSON · stdout do PostToolUse (reescrita de output)
{
  "result": "[output redacted by security policy]"
}

Exemplos praticos de configuracao

1. PreToolUse: bloquear edits em arquivos protegidos

Este hook impede qualquer modificacao em arquivos sensiveis como .env, .git/ e package-lock.json:

.claude/settings.json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "python3 .claude/hooks/block-protected-files.py"
          }
        ]
      }
    ]
  }
}
.claude/hooks/block-protected-files.py
import sys, json

data = json.load(sys.stdin)
file_path = data["tool_input"].get("file_path", "")

# Padroes de arquivos protegidos
BLOCKED = [".env", ".git/", "package-lock.json", "yarn.lock", "pnpm-lock.yaml"]

for pattern in BLOCKED:
    if pattern in file_path:
        result = {
            "permissionDecision": "deny",
            "reason": f"Arquivo protegido: {file_path} corresponde ao padrao '{pattern}'"
        }
        json.dump(result, sys.stdout)
        sys.exit(2)

# Sem objecao - fluxo normal de permissoes
sys.exit(0)

2. PreToolUse com campo if: filtrar comandos git

Usando o campo if para restringir o hook a comandos git especificos, bloqueando operacoes destrutivas como git push --force e git reset --hard:

.claude/settings.json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "if": "Bash(git *)",
        "hooks": [
          {
            "type": "command",
            "command": "bash .claude/hooks/guard-git.sh"
          }
        ]
      }
    ]
  }
}
.claude/hooks/guard-git.sh
#!/usr/bin/env bash
# Le o JSON do stdin e extrai o comando
INPUT=$(cat)
CMD=$(echo "$INPUT" | jq -r '.tool_input.command')

# Bloqueia operacoes git destrutivas
if echo "$CMD" | grep -qE 'git\s+(push\s+.*--force|reset\s+--hard|clean\s+-fd)'; then
  echo '{"permissionDecision":"deny","reason":"Operacao git destrutiva bloqueada por hook"}'
  exit 2
fi

# Forca confirmacao manual para git push (qualquer variante)
if echo "$CMD" | grep -qE 'git\s+push'; then
  echo '{"permissionDecision":"ask","reason":"Confirme o push para o remote"}'
  exit 0
fi

exit 0

3. PostToolUse: auto-formatar com Prettier apos edits

Um dos usos mais comuns de PostToolUse: rodar um formatter automaticamente toda vez que Claude edita um arquivo, garantindo consistencia de estilo sem que Claude precise se preocupar com isso.

.claude/settings.json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [
          {
            "type": "command",
            "command": "bash .claude/hooks/prettier-format.sh"
          }
        ]
      }
    ]
  }
}
.claude/hooks/prettier-format.sh
#!/usr/bin/env bash
INPUT=$(cat)
FILE=$(echo "$INPUT" | jq -r '.tool_input.file_path // empty')

if [ -z "$FILE" ]; then
  exit 0
fi

# Verifica se o arquivo existe e tem extensao suportada
if [[ "$FILE" =~ \.(ts|tsx|js|jsx|json|css|md|html)$ ]]; then
  # Roda Prettier silenciosamente
  npx prettier --write "$FILE" 2>/dev/null
fi

exit 0

4. PreToolUse: bloquear rm -rf e comandos destrutivos

Um hook de seguranca que impede a execucao de comandos de remocao recursiva e outros padroes perigosos no Bash:

.claude/settings.json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "python3 .claude/hooks/block-destructive.py"
          }
        ]
      }
    ]
  }
}
.claude/hooks/block-destructive.py
import sys, json, re

data = json.load(sys.stdin)
cmd = data["tool_input"].get("command", "")

# Padroes de comandos destrutivos
DANGEROUS = [
    r"rm\s+(-[a-zA-Z]*f[a-zA-Z]*\s+|.*--force)",   # rm com -f ou --force
    r"rm\s+-[a-zA-Z]*r[a-zA-Z]*\s+/",              # rm -r a partir da raiz
    r"chmod\s+-R\s+777",                             # permissoes perigosas
    r">\s*/dev/sda",                                 # escrita direta em disco
    r"mkfs\.",                                       # formatacao de disco
    r"dd\s+if=.*of=/dev/",                           # dd para dispositivo
]

for pattern in DANGEROUS:
    if re.search(pattern, cmd):
        result = {
            "permissionDecision": "deny",
            "reason": f"Comando destrutivo bloqueado: '{cmd}' corresponde a '{pattern}'"
        }
        json.dump(result, sys.stdout)
        sys.exit(2)

sys.exit(0)

Tabela comparativa: PreToolUse vs PostToolUse

Caracteristica PreToolUse PostToolUse
Momento de execucao Antes da tool call Apos a tool call (sucesso)
Pode bloquear a acao? Sim (deny / exit 2) Nao (acao ja executada)
Pode reescrever input? Sim (updatedInput) Nao
Pode reescrever output? Nao Sim (result)
Pode forcar confirmacao? Sim (ask) Nao
Ignora permission mode? Sim (deny supera bypassPermissions) N/A
Matcher Nome de tool (regex) Nome de tool (regex)
Campo if Sim (sintaxe de permission rules) Sim (sintaxe de permission rules)
Casos de uso tipicos Guardrails, seguranca, sanitizacao de input Formatacao, logging, validacao, redacao de secrets

Ordem de execucao e composicao de hooks

E possivel registrar multiplos hooks para o mesmo evento e matcher. Eles executam sequencialmente, mas com algumas nuances importantes:

Regras de composicao

1. Decisao mais restritiva vence: Se um hook retorna "allow" e outro retorna "deny", o resultado final e "deny". A hierarquia e: deny > ask > allow > sem decisao.

2. updatedInput: ultimo vence: Se multiplos hooks retornam updatedInput, apenas o ultimo e aplicado. Como a ordem nao e garantida, evite esse cenario.

3. Hooks com erro nao bloqueiam: Se um hook falha (exit code diferente de 0 e 2), Claude Code registra o erro mas continua o fluxo normalmente. O hook nao tem poder de veto por acidente.

4. Timeout: Hooks tem um tempo limite de execucao. Se excederem, sao tratados como erro e o fluxo continua.

Onde configurar hooks

Hooks podem ser configurados em tres niveis, cada um com escopo diferente:

P Projeto

.claude/settings.json na raiz do repositorio. Compartilhado via git. Ideal para hooks de equipe.

U Usuario

~/.claude/settings.json no home do usuario. Aplica a todos os projetos do usuario.

L Local

.claude/settings.local.json na raiz do projeto. Nao commitado (adicione ao .gitignore). Sobrescreve os outros.

Seguranca de hooks em projeto: Hooks definidos em .claude/settings.json (nivel projeto) precisam ser aprovados pelo usuario na primeira execucao. Claude Code exibe o comando que sera executado e pede confirmacao explicita. Isso evita que um repositorio malicioso execute codigo arbitrario via hooks.

Padroes avancados

Redacao de secrets no output

Um PostToolUse que intercepta a saida de comandos Bash e substitui tokens, senhas e chaves de API antes que Claude os veja:

.claude/hooks/redact-secrets.sh
#!/usr/bin/env bash
INPUT=$(cat)
TOOL_RESULT=$(echo "$INPUT" | jq -r '.tool_result // empty')

if [ -z "$TOOL_RESULT" ]; then
  exit 0
fi

# Redacta padroes de secrets comuns
REDACTED=$(echo "$TOOL_RESULT" | sed -E \
  -e 's/(AKIA[0-9A-Z]{16})/[AWS_KEY_REDACTED]/g' \
  -e 's/(ghp_[a-zA-Z0-9]{36})/[GITHUB_TOKEN_REDACTED]/g' \
  -e 's/(sk-[a-zA-Z0-9]{48})/[OPENAI_KEY_REDACTED]/g' \
  -e 's/(xoxb-[0-9-]+)/[SLACK_TOKEN_REDACTED]/g' \
)

# So emite resultado se houve mudanca
if [ "$REDACTED" != "$TOOL_RESULT" ]; then
  jq -n --arg r "$REDACTED" '{"result": $r}'
fi

exit 0

Audit log de todas as tool calls

Um PostToolUse catch-all que registra cada tool call em um arquivo de log local:

.claude/settings.json (trecho)
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": ".*",
        "hooks": [
          {
            "type": "command",
            "command": "python3 .claude/hooks/audit-log.py"
          }
        ]
      }
    ]
  }
}
.claude/hooks/audit-log.py
import sys, json
from datetime import datetime

data = json.load(sys.stdin)
entry = {
    "timestamp": datetime.utcnow().isoformat(),
    "session_id": data.get("session_id"),
    "tool": data.get("tool_name"),
    "input": data.get("tool_input"),
}

with open(".claude/audit.jsonl", "a") as f:
    f.write(json.dumps(entry) + "\n")

sys.exit(0)

Resumo do modelo mental

Fluxo linear simplificado: PreToolUse controla a entrada, PostToolUse controla a saida
Regra de ouro: Use PreToolUse para tudo que voce quer impedir que aconteca. Use PostToolUse para tudo que voce quer que aconteca como consequencia de uma acao. PreToolUse e o guarda na porta; PostToolUse e o zelador que limpa depois.
