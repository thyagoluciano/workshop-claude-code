Hooks como Quality Gates: Aprovacao Automatica de Operacoes Seguras

Como construir um sistema de camadas de seguranca usando hooks para que o Claude opere com autonomia maxima sem comprometer a integridade do projeto. PermissionRequest, PreToolUse e prompt hooks como porteiros inteligentes.

O Problema: Fadiga de Permissoes

Quando o Claude Code opera em modo interativo padrao, cada operacao potencialmente destrutiva gera um prompt de confirmacao. Para tarefas complexas que envolvem dezenas de edits, execucoes de testes e leituras de arquivos, a quantidade de interrupcoes torna o fluxo improdutivo. O desenvolvedor acaba aprovando tudo mecanicamente, o que anula o proposito do sistema de permissoes.

A solucao nao e desabilitar permissoes. E automatizar as decisoes obvias e reservar a intervencao humana apenas para operacoes que realmente exigem julgamento. Isso e exatamente o que hooks como quality gates fazem.

Principio fundamental: Hooks podem restringir alem do que as regras de permissao permitem, mas nao podem afrouxar restricoes alem do que as permission rules ja autorizam. Deny rules das settings sempre vencem.

PermissionRequest: O Hook de Decisao

O evento PermissionRequest e disparado sempre que um dialogo de permissao seria exibido ao usuario. O hook intercepta esse momento e pode tomar uma decisao automatica, eliminando a necessidade de interacao humana.

Anatomia da resposta JSON

O hook deve emitir um JSON com a estrutura hookSpecificOutput contendo o campo decision. O campo behavior aceita tres valores possiveis:

Behavior Efeito Caso de uso
"allow" Pula o prompt interativo e permite a operacao Operacoes de leitura, comandos seguros
"deny" Bloqueia a operacao silenciosamente Comandos destrutivos, acesso a secrets
"ask" Exibe o prompt normal de permissao Operacoes que requerem julgamento humano
Atencao critica: Retornar "allow" pula o prompt interativo, MAS deny rules definidas nas settings continuam sendo aplicadas. O hook nao pode sobrescrever restricoes ja configuradas no sistema de permissoes do Claude Code.

Exemplo canonico: Auto-approve ExitPlanMode

Este e o exemplo direto da documentacao oficial. Quando o Claude quer sair do modo de planejamento para comecar a executar, normalmente exige confirmacao. Com esse hook, a transicao e automatica:

.claude/settings.jsonPermissionRequest hook
{
  "hooks": {
    "PermissionRequest": [
      {
        "matcher": "ExitPlanMode",
        "hooks": [
          {
            "type": "command",
            "command": "echo '{\"hookSpecificOutput\": {\"hookEventName\": \"PermissionRequest\", \"decision\": {\"behavior\": \"allow\"}}}'"
          }
        ]
      }
    ]
  }
}

O matcher filtra qual tipo de permissao esta sendo solicitada. Neste caso, apenas ExitPlanMode e auto-aprovado. Todas as outras permissoes continuam passando pelo fluxo normal.

Configurando permissoes via updatedPermissions

Alem de decisoes pontuais, o campo updatedPermissions permite alterar o modo de permissao de forma mais ampla dentro da resposta do hook. Isso possibilita, por exemplo, elevar temporariamente o nivel de autonomia para uma sessao especifica baseado em condicoes verificadas pelo hook.

Piramide de Seguranca: Modelo Mental

O modelo ideal para quality gates e uma piramide de tres camadas. A base, mais larga, contem operacoes auto-aprovadas. O meio contem operacoes que passam pelo prompt padrao. O topo, mais estreito, contem operacoes automaticamente bloqueadas.

Piramide de seguranca: quanto mais frequente a operacao, menor deve ser a friccao

PreToolUse como Quality Gate

Enquanto PermissionRequest intercepta dialogos de permissao genericos, o evento PreToolUse oferece controle granular sobre ferramentas especificas. Ele e disparado antes de cada invocacao de ferramenta e recebe no stdin o JSON completo com o nome da ferramenta e seus parametros.

O campo chave na resposta e permissionDecision, que aceita os mesmos tres comportamentos:

✓ allow

Pula o prompt interativo. A ferramenta executa imediatamente.

✗ deny

Bloqueia a execucao. Funciona mesmo em modo bypassPermissions.

? ask

Delega a decisao ao prompt interativo padrao.

Detalhe importante sobre deny: Diferente do allow que pode ser sobrescrito por deny rules, o deny via hook e absoluto. Ele bloqueia a operacao mesmo quando o usuario esta operando em modo bypassPermissions. Hooks podem restringir, nunca afrouxar.

Auto-allow para operacoes de leitura

Ferramentas como Read, Grep e Glob nunca modificam o sistema de arquivos. Sao candidatas perfeitas para auto-aprovacao total:

.claude/settings.jsonPreToolUse - auto-allow reads
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Read|Grep|Glob",
        "hooks": [
          {
            "type": "command",
            "command": "echo '{\"decision\": \"allow\"}'"
          }
        ]
      }
    ]
  }
}

O matcher usa regex, entao Read|Grep|Glob captura qualquer uma dessas tres ferramentas. Todas as invocacoes dessas ferramentas serao auto-aprovadas sem qualquer prompt.

Auto-deny para operacoes destrutivas

O caminho inverso: bloquear automaticamente qualquer tentativa de executar comandos sabidamente perigosos. O hook recebe o JSON da ferramenta via stdin e inspeciona os parametros:

scripts/deny-destructive.shShell script para quality gate
#!/bin/bash
# Le o input JSON do stdin
INPUT=$(cat)

# Extrai o comando sendo executado
TOOL_NAME=$(echo "$INPUT" | jq -r '.tool_name // empty')
COMMAND=$(echo "$INPUT" | jq -r '.tool_input.command // empty')

# Lista de padroes perigosos
if echo "$COMMAND" | grep -qE '(rm\s+-rf|git\s+push\s+--force|DROP\s+TABLE|TRUNCATE|format\s+c:|del\s+/[sf])'; then
  echo '{"decision": "deny", "reason": "Comando destrutivo bloqueado pelo quality gate"}'
  exit 0
fi

# Tudo que nao e perigoso segue o fluxo normal
echo '{"decision": "ask"}'
.claude/settings.jsonPreToolUse - deny destructive
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "bash ./scripts/deny-destructive.sh"
          }
        ]
      }
    ]
  }
}

Script Inteligente de Categorizacao

A abordagem mais robusta combina allow, deny e ask em um unico script que categoriza operacoes dinamicamente. O script recebe o contexto completo da ferramenta e toma a decisao apropriada:

scripts/quality-gate.shCategorizador completo de operacoes
#!/bin/bash
INPUT=$(cat)

TOOL_NAME=$(echo "$INPUT" | jq -r '.tool_name // empty')
COMMAND=$(echo "$INPUT" | jq -r '.tool_input.command // empty')
FILE_PATH=$(echo "$INPUT" | jq -r '.tool_input.file_path // empty')

# ============================================
# CAMADA 1: AUTO-DENY (operacoes destrutivas)
# ============================================
if echo "$COMMAND" | grep -qiE '(rm\s+-rf|git\s+push\s+--force|DROP\s+TABLE|TRUNCATE)'; then
  echo '{"decision": "deny", "reason": "Operacao destrutiva bloqueada"}'
  exit 0
fi

# Bloquear acesso a arquivos de secrets
if echo "$FILE_PATH" | grep -qE '\.(env|pem|key)$|/secrets/|/credentials/'; then
  echo '{"decision": "deny", "reason": "Acesso a secrets bloqueado"}'
  exit 0
fi

# ============================================
# CAMADA 2: AUTO-ALLOW (operacoes seguras)
# ============================================

# Ferramentas de leitura sao sempre seguras
if echo "$TOOL_NAME" | grep -qE '^(Read|Grep|Glob|ListDir)$'; then
  echo '{"decision": "allow"}'
  exit 0
fi

# Comandos de verificacao (nao modificam nada)
if echo "$COMMAND" | grep -qE '^(npm test|npx tsc --noEmit|npx eslint|prettier --check|cargo check|go vet)'; then
  echo '{"decision": "allow"}'
  exit 0
fi

# Edits somente dentro de src/ ou tests/
if [ "$TOOL_NAME" = "Edit" ] || [ "$TOOL_NAME" = "Write" ]; then
  if echo "$FILE_PATH" | grep -qE '^(src/|tests/|test/|__tests__/)'; then
    echo '{"decision": "allow"}'
    exit 0
  fi
fi

# ============================================
# CAMADA 3: ASK (tudo o resto)
# ============================================
echo '{"decision": "ask"}'
.claude/settings.jsonConfiguracao completa com quality gate
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": ".*",
        "hooks": [
          {
            "type": "command",
            "command": "bash ./scripts/quality-gate.sh"
          }
        ]
      }
    ],
    "PermissionRequest": [
      {
        "matcher": "ExitPlanMode",
        "hooks": [
          {
            "type": "command",
            "command": "echo '{\"hookSpecificOutput\": {\"hookEventName\": \"PermissionRequest\", \"decision\": {\"behavior\": \"allow\"}}}'"
          }
        ]
      }
    ]
  }
}

Essa configuracao cobre os dois eventos simultaneamente: PreToolUse com matcher .* (todas as ferramentas) passando pelo script categorizador, e PermissionRequest auto-aprovando especificamente o ExitPlanMode.

Fluxo de Decisao do Quality Gate

Fluxo de decisao: deny primeiro, allow segundo, ask como fallback

Prompt Hooks como Quality Gate Inteligente

Para decisoes que exigem julgamento contextual alem de simples pattern matching, hooks do tipo prompt delegam a avaliacao a um modelo de linguagem. O modelo recebe o contexto da operacao e decide se ela e segura.

.claude/settings.jsonPrompt hook para decisoes contextuais
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "prompt",
            "prompt": "Avalie se o comando bash a seguir e seguro para execucao automatica. Considere: 1) O comando modifica o filesystem? 2) O comando acessa rede? 3) O comando pode causar perda de dados? Se for seguro, responda com JSON {\"decision\": \"allow\"}. Se for perigoso, responda com {\"decision\": \"deny\", \"reason\": \"\"}. Se for ambiguo, responda com {\"decision\": \"ask\"}."
          }
        ]
      }
    ]
  }
}
Vantagem dos prompt hooks: Eles conseguem avaliar contexto semantico que regex nao captura. Um comando como curl -X DELETE seria detectado pelo modelo mesmo sem um regex especifico para ele. A desvantagem e a latencia adicional de uma chamada ao modelo.

Prompt hooks sao mais adequados para situacoes onde a lista de padroes perigosos nao e facilmente enumeravel, ou quando o risco depende da combinacao de parametros em vez de um unico token.

Estrategias de Quality Gate por Cenario

1 Projeto greenfield

Auto-allow em tudo dentro de src/ e tests/. O Claude tem autonomia total para criar e editar codigo no escopo do projeto.

2 Refactoring em codebase legado

Auto-allow reads + deny em delecoes de arquivos fora do escopo. Ask para todo edit. Protege contra mudancas acidentais em areas criticas.

3 CI/CD pipeline

Auto-allow reads e testes. Deny em qualquer operacao de rede. Ask para edits em configs de deploy. Ideal para ambiente headless.

4 Code review automatizado

Auto-allow todas as ferramentas de leitura. Deny total em Write/Edit/Bash. O Claude analisa sem modificar nada.

Combinando Hooks em Cadeia

Multiplos hooks podem ser registrados para o mesmo evento. Eles executam em sequencia, e a regra e: o resultado mais restritivo vence. Se um hook retorna allow e outro retorna deny, o resultado final e deny.

.claude/settings.jsonCadeia de hooks: seguranca + compliance
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": ".*",
        "hooks": [
          {
            "type": "command",
            "command": "bash ./scripts/security-gate.sh"
          },
          {
            "type": "command",
            "command": "bash ./scripts/compliance-gate.sh"
          }
        ]
      }
    ]
  }
}

Nesse modelo, security-gate.sh verifica riscos de seguranca (comandos destrutivos, acesso a secrets) enquanto compliance-gate.sh verifica regras de compliance (padroes de codigo, restricoes de licenca). Ambos precisam aprovar para que a operacao prossiga.

Tabela de precedencia de decisoes

Hook 1 Hook 2 Resultado final Razao
allow allow allow Ambos concordam
allow deny deny Mais restritivo vence
allow ask ask Mais restritivo vence
deny ask deny Deny e absoluto
ask ask ask Nenhum hook decidiu

Interacao com o Sistema de Permissoes

E fundamental entender como hooks interagem com as permission rules ja configuradas no Claude Code. O sistema de permissoes tem sua propria hierarquia, e hooks operam dentro dessa hierarquia, nao acima dela.

Hook allow + deny rule = DENY. O sistema de permissoes tem a palavra final sobre allow.

Padroes Avancados

Gate baseado em horario

Um quality gate pode considerar fatores externos como horario. Durante horario comercial, operacoes de deploy sao permitidas. Fora do horario, sao bloqueadas:

scripts/time-gate.shQuality gate temporal
#!/bin/bash
INPUT=$(cat)
COMMAND=$(echo "$INPUT" | jq -r '.tool_input.command // empty')
HOUR=$(date +%H)

# Comandos de deploy somente em horario comercial (9-18)
if echo "$COMMAND" | grep -qE '(deploy|publish|release)'; then
  if [ "$HOUR" -ge 9 ] && [ "$HOUR" -lt 18 ]; then
    echo '{"decision": "ask"}'  # Permite mas pede confirmacao
  else
    echo '{"decision": "deny", "reason": "Deploy bloqueado fora do horario comercial"}'
  fi
  exit 0
fi

echo '{"decision": "ask"}'

Gate baseado em branch Git

Restringir operacoes destrutivas baseado na branch atual. Na main, quase tudo e bloqueado. Em branches de feature, o Claude tem mais liberdade:

scripts/branch-gate.shQuality gate por branch
#!/bin/bash
INPUT=$(cat)
TOOL_NAME=$(echo "$INPUT" | jq -r '.tool_name // empty')
BRANCH=$(git branch --show-current 2>/dev/null || echo "unknown")

# Na main: deny writes, allow somente reads
if [ "$BRANCH" = "main" ] || [ "$BRANCH" = "master" ]; then
  if echo "$TOOL_NAME" | grep -qE '^(Edit|Write|Bash)$'; then
    echo '{"decision": "deny", "reason": "Escrita bloqueada na branch main"}'
    exit 0
  fi
fi

# Em branches de feature: allow edits em src/
if echo "$BRANCH" | grep -qE '^(feature|feat|fix)/'; then
  if echo "$TOOL_NAME" | grep -qE '^(Read|Grep|Glob)$'; then
    echo '{"decision": "allow"}'
    exit 0
  fi
fi

echo '{"decision": "ask"}'

Consideracoes de Seguranca

Hooks rodam como seu usuario. Scripts de quality gate executam com as mesmas permissoes do processo do Claude Code. Um hook mal configurado que retorna allow indiscriminadamente e pior que nenhum hook, pois cria uma falsa sensacao de seguranca.

! Nao faca

Matcher .* com allow incondicional. Isso desabilita toda protecao interativa sem nenhum filtro.

✓ Faca

Default para ask com allow/deny explicitos apenas para padroes bem conhecidos. Fail-safe por padrao.

Boas praticas para quality gates seguros:

  • Default deny/ask: O fallback deve ser sempre ask ou deny, nunca allow
  • Whitelist, nao blacklist: Enumere o que e seguro em vez de tentar listar tudo que e perigoso
  • Teste seus hooks: Execute manualmente com inputs maliciosos para verificar que o gate funciona
  • Logs de auditoria: Hooks podem logar decisoes para revisao posterior via PostToolUse
  • Versionamento: Mantenha scripts de quality gate sob controle de versao junto ao codigo
