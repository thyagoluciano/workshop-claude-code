Nunca Rodar Claude Code como Root

Sandboxing e isolamento: por que o Claude Code recusa executar com privilegios elevados, como o sandbox protege seu sistema e como configurar isolamento adequado para ambientes de producao.

1. O Risco de Rodar como Root

O Claude Code implementa uma protecao hard-coded contra execucao com privilegios de root quando o bypass de permissoes esta ativo. Se voce tentar rodar com --dangerously-skip-permissions como root ou via sudo, o processo sera encerrado imediatamente com a seguinte mensagem:

Erro ao tentar rodar como root com bypass
$ sudo claude --dangerously-skip-permissions

Error: --dangerously-skip-permissions cannot be used with
root/sudo privileges for security reasons.

Esse check existe porque a combinacao de root access + sem permission checks cria o cenario mais perigoso possivel: um agente autonomo com poder total sobre o sistema operacional, sem nenhuma barreira entre uma instrucao e a execucao.

Excecao: O check de root e pulado automaticamente quando o Claude Code detecta que esta rodando dentro de um sandbox reconhecido (como um container Docker ou dev container). Nesses ambientes, root dentro do container nao significa root no host.

2. Por Que Essa Restricao Existe

Um agente de IA com root access e sem permission checks pode causar danos irreversiveis ao sistema. Os vetores de ataque incluem:

1 Arquivos de sistema

Modificar /etc/passwd, /etc/shadow, /etc/sudoers ou qualquer binario em /usr/bin. Pode criar backdoors permanentes.

2 Malware

Instalar pacotes maliciosos, rootkits ou scripts de persistencia que sobrevivem a reboot. Registrar servicos systemd para execucao automatica.

3 Rede

Alterar configuracoes de rede, DNS, firewall. Redirecionar trafego, abrir portas para acesso externo. Modificar /etc/hosts ou regras iptables.

4 Dados de terceiros

Acessar dados de outros usuarios do sistema: /home/*, bancos de dados locais, chaves SSH, tokens de API e credenciais armazenadas.

O ponto critico e que esses danos podem acontecer como resultado de prompt injection: um arquivo malicioso no repositorio ou uma resposta de API comprometida pode instruir o agente a executar comandos destrutivos. Com root e sem checks, nao ha nenhuma camada de defesa.

3. Sandboxing como Alternativa

Em vez de rodar como root com bypass de permissoes, a abordagem correta e usar sandboxing. O sandbox do Claude Code opera em duas camadas independentes que se complementam:

FS Filesystem Isolation

Controla quais diretorios e arquivos podem ser lidos e escritos. Impede acesso a areas sensiveis do sistema mesmo que um comando tente acessar.

NET Network Isolation

Controla quais dominios podem ser acessados via rede. Impede exfiltracao de dados e download de scripts maliciosos mesmo dentro do sandbox.

As duas camadas sao enforced no nivel do sistema operacional, nao apenas no nivel da aplicacao. Isso significa que mesmo scripts e subprocessos spawnados pelo Claude Code estao sujeitos as mesmas restricoes: nao ha como escapar via bash -c ou subshells.

4. Filesystem Isolation

O comportamento padrao do sandbox de filesystem e:

Operacao Escopo padrao Configuravel
Write Working directory e subdiretorios apenas Sim (allowWrite)
Read Computador inteiro (exceto denied) Sim (denyRead, allowRead)

As configuracoes de filesystem sao granulares e permitem controle fino sobre o que o sandbox pode acessar:

Opcoes de configuracao do filesystem sandbox
{
  "sandbox": {
    "filesystem": {
      "allowWrite": [
        ".",              // working directory (padrao)
        "/tmp/build",     // diretorio temporario
        "~/.kube"         // config do kubernetes
      ],
      "denyWrite": [
        "~/.ssh",         // nunca escrever em chaves SSH
        "~/.gnupg"        // nunca escrever em GPG keys
      ],
      "denyRead": [
        "~/.aws",         // bloquear leitura de credenciais AWS
        "/etc/shadow"     // bloquear leitura de senhas
      ],
      "allowRead": [
        ".",              // garantir leitura do projeto
        "/usr/local/lib"  // libs necessarias
      ]
    }
  }
}

Enforcement no nivel do OS

O sandbox de filesystem nao e implementado apenas no nivel da aplicacao. O Claude Code usa mecanismos nativos do sistema operacional:

Plataforma Mecanismo Detalhes
macOS Seatbelt (sandbox-exec) Kernel-level enforcement. Mesmo processos filhos sao restritos. Impossivel bypassar sem root no host.
Linux / WSL2 bubblewrap (bwrap) Usa namespaces do kernel para isolamento. Cria um ambiente restrito com montagens controladas.
Protecao contra escape: O enforcement no OS significa que mesmo que o Claude Code execute bash -c "cat /etc/shadow", o kernel bloqueia o acesso. Nao e uma restricao de software que pode ser contornada com subprocessos criativos.

5. Network Isolation

O sandbox de rede funciona atraves de um proxy server que roda fora do sandbox. Todo trafego de rede gerado dentro do sandbox e forcado a passar por esse proxy, que aplica as restricoes configuradas.

Como funciona:

  • Um proxy server e iniciado fora do sandbox antes do Claude Code comecar
  • Dentro do sandbox, todas conexoes de rede sao redirecionadas para o proxy
  • O proxy verifica cada request contra a allowlist de dominios
  • Dominios nao listados exigem confirmacao do usuario
  • A cobertura inclui todos scripts, programas e subprocessos executados dentro do sandbox
Configuracao de network isolation
{
  "sandbox": {
    "network": {
      "allowDomains": [
        "registry.npmjs.org",
        "api.github.com",
        "pypi.org",
        "rubygems.org"
      ]
    }
  }
}

Quando um processo dentro do sandbox tenta acessar um dominio que nao esta na allowlist, o proxy intercepta a requisicao e solicita confirmacao do usuario antes de permitir. Isso previne tanto exfiltracao de dados (envio de dados sensiveis para servidores externos) quanto download de payloads maliciosos.

6. Modos de Sandbox

O sandbox do Claude Code opera em dois modos distintos que afetam como os comandos sao aprovados:

A Auto-allow mode

Comandos bash tentam rodar dentro do sandbox e sao auto-aprovados sem prompt. O sandbox substitui a necessidade de confirmacao manual porque as restricoes de filesystem e rede ja limitam o dano possivel. Usado com --dangerously-skip-permissions.

R Regular permissions mode

Todos os comandos passam pelo fluxo normal de permissoes, mesmo estando sandboxed. O sandbox atua como camada adicional de defesa, nao como substituto. Modo padrao quando nao se usa bypass.

A diferenca pratica e significativa: no auto-allow mode, o sandbox e a unica barreira de seguranca. No regular mode, ele e uma camada adicional sobre o fluxo de permissoes. Para ambientes de CI/CD e automacao, o auto-allow mode com sandbox bem configurado e a abordagem recomendada.

Regra de ouro: Se voce precisa de --dangerously-skip-permissions, ative sempre o sandbox. A combinacao de bypass sem sandbox e root e exatamente o cenario que o Claude Code recusa executar.

7. Configuracao Pratica

Uma configuracao completa de sandbox combina filesystem e network isolation. Aqui esta um exemplo para um projeto Node.js tipico:

.claude/settings.json - Sandbox completo
{
  "sandbox": {
    "enabled": true,
    "filesystem": {
      "allowWrite": [
        ".",
        "/tmp/build",
        "~/.npm",
        "~/.kube"
      ],
      "denyWrite": [
        "~/.ssh",
        "~/.gnupg",
        "~/.aws"
      ],
      "denyRead": [
        "~/"
      ],
      "allowRead": [
        ".",
        "/usr/local/lib/node_modules",
        "~/.npmrc"
      ]
    },
    "network": {
      "allowDomains": [
        "registry.npmjs.org",
        "api.github.com",
        "*.githubusercontent.com"
      ]
    }
  }
}

Observe o padrao denyRead: ["~/"] com allowRead: ["."]. Isso cria um modelo de deny-by-default para leitura: o sandbox nao pode ler nada no home directory do usuario, exceto o diretorio do projeto atual. Essa e a configuracao mais segura para projetos que nao precisam acessar configuracoes globais.

Exemplo: projeto com Kubernetes

Sandbox para projeto que interage com k8s
{
  "sandbox": {
    "enabled": true,
    "filesystem": {
      "allowWrite": [
        ".",
        "/tmp/build",
        "~/.kube/cache"
      ],
      "allowRead": [
        ".",
        "~/.kube/config",
        "/usr/local/bin/kubectl"
      ],
      "denyRead": [
        "~/.aws",
        "~/.ssh"
      ]
    },
    "network": {
      "allowDomains": [
        "*.k8s.io",
        "registry.npmjs.org"
      ]
    }
  }
}

8. Dev Containers

Para isolamento total, dev containers sao a alternativa recomendada. Um dev container roda o Claude Code dentro de um container Docker dedicado, com seu proprio filesystem, rede e usuario. O root dentro do container nao tem nenhum privilegio no host.

Vantagens do dev container:

  • Isolamento completo do sistema host
  • Ambiente reproduzivel e descartavel
  • Root no container e seguro (nao afeta o host)
  • Network isolation nativa via Docker networking
  • Possibilidade de limitar recursos (CPU, memoria, disco)
.devcontainer/devcontainer.json
{
  "name": "claude-code-sandbox",
  "image": "mcr.microsoft.com/devcontainers/base:ubuntu",
  "remoteUser": "vscode",
  "features": {
    "ghcr.io/devcontainers/features/node:1": {}
  },
  "postCreateCommand": "npm install",
  "customizations": {
    "claude-code": {
      "sandbox": {
        "enabled": true
      }
    }
  }
}
Non-root user: A pratica recomendada e usar um usuario nao-root dentro do container (como vscode ou node). Isso adiciona uma camada extra de defesa: mesmo dentro do container isolado, o agente nao tem privilegios de root. Para detalhes, consulte a documentacao de dev containers.

9. Protecao Contra Prompt Injection com Sandbox

O sandbox e uma defesa critica contra prompt injection. Mesmo que um atacante consiga injetar instrucoes maliciosas (via arquivo no repositorio, resposta de API comprometida, ou conteudo de pagina web), o sandbox limita drasticamente o que o agente consegue fazer.

Cenario: injection bem-sucedida

Imagine que um arquivo malicioso no repositorio instrui o Claude Code a executar comandos destrutivos. Sem sandbox, esses comandos seriam executados normalmente. Com sandbox ativo:

Acao maliciosa Sem sandbox Com sandbox
Modificar ~/.bashrc Executa Bloqueado pelo filesystem
Alterar binarios em /bin/ Executa (com root) Bloqueado pelo filesystem
Exfiltrar dados via curl Executa Bloqueado pelo network proxy
Baixar script malicioso Executa Bloqueado pelo network proxy
Ler chaves SSH Executa Bloqueado pelo filesystem
Instalar pacote malicioso Executa Bloqueado (rede + filesystem)
Defense in depth: O sandbox nao impede a injection em si (o agente ainda pode ser enganado), mas impede que a injection cause dano real. E uma camada de contencao, nao de prevencao. Combine com regras de deny no CLAUDE.md e hooks de validacao para multiplas camadas de defesa.

10. Limitacoes do Sandbox

O sandbox do Claude Code tem limitacoes praticas que voce precisa conhecer antes de adotar em producao:

Performance

O overhead de performance e minimo na maioria dos casos. O Seatbelt (macOS) e o bubblewrap (Linux) operam no nivel do kernel com custo negligivel. O proxy de rede adiciona uma pequena latencia em cada request, mas e imperceptivel na pratica.

Incompatibilidades

Ferramenta Problema Solucao
watchman Incompativel com sandbox de filesystem Desabilitar watchman ou usar polling
docker CLI Nao consegue acessar Docker socket dentro do sandbox Adicionar docker aos excludedCommands
WSL1 Nao suportado (requer namespaces do kernel) Migrar para WSL2
Excluindo comandos incompativeis do sandbox
{
  "sandbox": {
    "enabled": true,
    "excludedCommands": [
      "docker",
      "docker-compose",
      "watchman"
    ]
  }
}

Sandbox aninhado em Docker

Quando o Claude Code roda dentro de um container Docker e tenta criar seu proprio sandbox, existe um conflito de namespaces. A opcao enableWeakerNestedSandbox resolve isso, mas com uma reducao na seguranca:

Sandbox aninhado (Linux/Docker)
{
  "sandbox": {
    "enabled": true,
    "enableWeakerNestedSandbox": true
  }
}
Atencao: enableWeakerNestedSandbox enfraquece as garantias de isolamento. Use apenas quando necessario (Claude Code dentro de Docker) e combine com restricoes adicionais do proprio container. A alternativa preferida e usar dev containers com isolamento nativo.

11. Escape Hatch: dangerouslyDisableSandbox

Quando um comando falha por restricoes do sandbox, o Claude Code pode tentar novamente sem sandbox usando o mecanismo de escape hatch. Isso nao e automatico: o comando passa pelo fluxo normal de permissoes e requer aprovacao do usuario.

Fluxo do escape hatch:

  1. Comando e executado dentro do sandbox
  2. Comando falha por restricao de filesystem ou rede
  3. Claude Code detecta que a falha e relacionada ao sandbox
  4. Claude pode fazer retry com dangerouslyDisableSandbox: true
  5. O retry passa pelo fluxo normal de permissoes
  6. Usuario decide se permite a execucao fora do sandbox

Para ambientes onde voce quer impedir completamente que comandos escapem do sandbox, desabilite essa opcao:

Desabilitando o escape hatch
{
  "sandbox": {
    "enabled": true,
    "allowUnsandboxedCommands": false
  }
}

Com allowUnsandboxedCommands: false, nenhum comando pode escapar do sandbox, mesmo com aprovacao do usuario. Essa e a configuracao mais restritiva e recomendada para ambientes de CI/CD e automacao.

12. Camadas de Seguranca

Camadas de seguranca do Claude Code: do kernel do OS ate a aprovacao do usuario

O modelo de seguranca do Claude Code segue o principio de defense in depth. Um atacante precisaria comprometer todas as quatro camadas simultaneamente para causar dano. O sandbox sozinho ja bloqueia a maioria dos vetores de ataque, mas combinado com permission rules e aprovacao do usuario, o sistema se torna extremamente resiliente.

13. Boas Praticas

1 Comece restritivo

Inicie com denyRead: ["~/"] e allowRead: ["."]. Adicione permissoes conforme necessario, nunca o contrario.

2 Monitore violations

Acompanhe os logs de sandbox para identificar comandos bloqueados. Isso ajuda a ajustar a configuracao sem abrir brechas desnecessarias.

3 Configs por ambiente

Use configuracoes diferentes para dev, staging e CI. O ambiente de CI deve ser o mais restritivo, dev pode ser mais permissivo.

4 Combine com permissions

Sandbox e permission rules sao complementares. Use ambos: sandbox para enforcement no OS, permissions para controle granular no Claude Code.

5 Use containers

Para tarefas sensiveis (rodar codigo de terceiros, processar dados confidenciais), use dev containers com usuario nao-root para isolamento total.

6 Nunca desabilite tudo

Evite allowUnsandboxedCommands: true em ambientes de CI/CD. Se o sandbox atrapalha, ajuste a configuracao em vez de desabilitar.

Resumo da regra de ouro: Nunca rode Claude Code como root com --dangerously-skip-permissions. Use sandbox para automacao, dev containers para isolamento total, e combine todas as camadas de seguranca disponiveis. O custo de configurar o sandbox e minimo comparado ao risco de um agente comprometido com acesso irrestrito ao sistema.
