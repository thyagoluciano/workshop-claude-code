Regras de Deny: Bloquear Acesso a .env, ~/.ssh/, secrets

Como usar deny rules para impedir que o Claude Code acesse arquivos sensiveis como variaveis de ambiente, chaves SSH e credenciais - a primeira linha de defesa no sistema de permissoes.

1. Por que Deny Rules sao Essenciais

O Claude Code tem acesso de leitura e escrita a qualquer arquivo dentro do working directory e, dependendo das permissoes, a todo o filesystem. Isso significa que arquivos como .env, chaves SSH e credenciais ficam expostos por padrao.

🔒 Secrets e API Keys

Arquivos .env com tokens de API, database URLs e senhas em texto plano.

🔑 Chaves SSH

Chaves privadas em ~/.ssh/ que dao acesso a servidores e repositorios.

📄 Credenciais

Arquivos como credentials.json, .npmrc com tokens e keystores.

Sem deny rules, nao ha barreira. Mesmo no modo plan, o Claude pode ler arquivos sensiveis. Deny rules sao a unica forma declarativa de bloquear acesso a paths especificos, independente do modo de permissao ativo.

Deny rules funcionam em todos os modos de permissao - inclusive quando o usuario aceita uma acao manualmente. A unica excecao e o modo bypassPermissions via SDK, que ignora todas as regras de permissao (e por isso so deve ser usado em ambientes isolados como containers CI).

2. Sintaxe de Regras de Permissao

Regras de permissao seguem o formato Tool(specifier), onde Tool e o nome da ferramenta e specifier define o escopo (path, comando, dominio, etc).

Formato geralSintaxe
Tool              // bare - afeta a tool inteira
Tool(specifier)    // scoped - afeta apenas chamadas que match o specifier

Ordem de avaliacao

O sistema avalia regras na seguinte ordem de prioridade. O primeiro match determina o resultado:

Prioridade Tipo Efeito
1 (maior) deny Bloqueia a operacao silenciosamente. Nenhum prompt exibido.
2 ask Exige aprovacao manual do usuario antes de executar.
3 (menor) allow Executa automaticamente sem prompt.
Deny sempre prevalece. Se uma regra de deny faz match, nenhuma regra de allow ou ask pode sobrescreve-la. Isso e valido mesmo entre diferentes escopos de configuracao - se um managed policy define deny, nem a configuracao local pode permitir.

3. Deny Bare Tool vs Scoped

Existe uma diferenca fundamental entre bloquear uma tool inteiramente (bare) e bloquear apenas chamadas especificas (scoped).

X Bare deny

"Bash" remove a ferramenta Bash completamente do contexto do Claude. Ele nem sabe que ela existe.

~ Scoped deny

"Bash(rm *)" mantem o Bash disponivel, mas bloqueia chamadas que match o pattern rm *.

Exemplos de bare vs scopedJSON
{
  "deny": [
    // Bare: remove a tool inteira
    "Bash",           // sem terminal nenhum
    "Edit",           // sem edicao de arquivos

    // Scoped: bloqueia patterns especificos
    "Bash(rm -rf *)", // bloqueia rm destrutivo, permite outros comandos
    "Read(.env)",     // bloqueia leitura de .env, permite outros reads
    "Edit(.env)"      // bloqueia edicao de .env, permite outros edits
  ]
}

4. Padroes de Path para Read/Edit

Os specifiers de Read e Edit usam um sistema de paths com quatro tipos de referencia:

Prefixo Significado Exemplo
//path Caminho absoluto do filesystem root Read(//etc/passwd)
~/path Relativo ao home do usuario Read(~/.ssh/id_rsa)
/path Relativo ao project root Read(/config/db.yml)
path ou ./path Relativo ao diretorio atual Read(.env)

Wildcards

* - Um nivel

Match em qualquer nome dentro de um unico diretorio. Read(config/*.json) match config/db.json mas nao config/sub/db.json.

** - Recursivo

Match em qualquer profundidade de diretorios. Read(config/**) match config/a.json e config/sub/deep/b.json.

Equivalencia importante: Read(.env) e Read(**/.env) sao equivalentes. Um path sem prefixo de diretorio faz match em qualquer profundidade do projeto. Ou seja, Read(.env) ja protege src/.env, deploy/.env, etc.

5. Exemplos Essenciais de Deny Rules

Estas sao as deny rules que todo projeto deveria ter como baseline de seguranca:

.claude/settings.jsonJSON
{
  "permissions": {
    "deny": [
      // Variaveis de ambiente (match em qualquer profundidade)
      "Read(.env)",
      "Read(.env.local)",
      "Read(.env.production)",

      // Chaves SSH - todo o diretorio recursivamente
      "Read(~/.ssh/**)",

      // Diretorio de secrets do projeto
      "Read(./secrets/**)",

      // Arquivo de credenciais especifico
      "Read(./config/credentials.json)",

      // Prevenir edicao de .env em qualquer lugar do filesystem
      "Edit(//**/.env)",

      // Prevenir edicao de .env.* (production, staging, etc)
      "Edit(**/.env.*)"
    ]
  }
}
Dica pratica: Coloque essas regras em .claude/settings.json no repositorio para que toda a equipe compartilhe as mesmas protecoes. Este arquivo deve ser versionado no VCS.

6. Symlinks e Deny Rules

O tratamento de symlinks difere entre allow e deny rules, seguindo o principio de seguranca por padrao:

✓ Allow rules

Ambos o symlink e o target devem dar match na regra de allow. Se apenas um deles faz match, o acesso e negado. Isso impede bypass via symlinks para arquivos fora do escopo.

✗ Deny rules

Qualquer um (symlink OU target) que de match na regra de deny e suficiente para bloquear. Isso impede bypass criando symlinks para arquivos protegidos.

Atencao: Se um atacante criar um symlink ./innocent.txt -> ~/.ssh/id_rsa, a deny rule Read(~/.ssh/**) ainda bloqueia o acesso porque o target do symlink da match na regra.

7. Bash Deny Rules

Deny rules para Bash usam wildcards com * para match de argumentos e reconhecem operadores de shell para avaliar compound commands.

Exemplos de Bash denyJSON
{
  "permissions": {
    "deny": [
      "Bash(git push *)",        // bloqueia qualquer git push
      "Bash(rm -rf *)",          // bloqueia rm destrutivo
      "Bash(curl *)",            // bloqueia requests HTTP via curl
      "Bash(wget *)",            // bloqueia downloads via wget
      "Bash(ssh *)",             // bloqueia conexoes SSH
      "Bash(scp *)"              // bloqueia copia remota
    ]
  }
}

Compound commands

O sistema reconhece operadores de shell e avalia cada subcommand independentemente:

Operador Descricao Exemplo
&& AND logico npm test && git push origin main
|| OR logico make build || echo "falhou"
; Sequencial ls; rm -rf /tmp/*
| Pipe cat secret.key | curl -X POST

Se qualquer subcommand dentro de um compound command fizer match com uma deny rule, o comando inteiro e bloqueado. Nao e possivel contornar uma deny rule encadeando com && ou ;.

8. WebFetch Deny Rules

O specifier de WebFetch usa o prefixo domain: para controlar quais dominios podem ser acessados:

Bloqueando dominiosJSON
{
  "permissions": {
    "deny": [
      "WebFetch(domain:evil-site.com)",
      "WebFetch(domain:pastebin.com)",
      "WebFetch(domain:requestbin.net)"
    ],
    "allow": [
      "WebFetch(domain:docs.company.com)",
      "WebFetch(domain:api.github.com)"
    ]
  }
}

Isso e util para impedir exfiltracao de dados via requests HTTP para dominios nao confiados, mantendo o acesso a documentacao e APIs necessarias.

9. MCP Deny Rules

Deny rules para servidores MCP permitem bloquear servidores inteiros ou ferramentas individuais:

Bloqueando MCP toolsJSON
{
  "permissions": {
    "deny": [
      // Bloqueia TODAS as tools de um servidor MCP
      "mcp__untrusted_server",

      // Bloqueia uma tool especifica de um servidor
      "mcp__database__drop_table",
      "mcp__deploy__production_deploy"
    ]
  }
}

🔒 Servidor inteiro

mcp__server_name bloqueia todas as ferramentas expostas por aquele servidor MCP. Util para servidores nao auditados.

⚙ Tool especifica

mcp__server__tool_name bloqueia apenas a ferramenta indicada, mantendo o restante do servidor acessivel.

10. Combinando com Sandboxing

Deny rules e sandbox operam em camadas diferentes, formando uma estrategia de defense-in-depth:

🛡 Deny Rules

Bloqueiam no nivel do Claude Code (aplicacao). O Claude nao tenta acessar o recurso.

🗃 Sandbox

Bloqueia no nivel do OS (kernel). Mesmo que o Claude tente, o sistema operacional nega.

As configuracoes de filesystem do sandbox se mesclam com as deny rules de Read e Edit. Se o sandbox restringe o acesso a um diretorio, a deny rule reforça isso no nivel da aplicacao. Se a deny rule bloqueia um path, o sandbox impede no nivel do OS como camada adicional.

Defense-in-depth: Use deny rules para declarar a intencao de seguranca e sandbox para enforcement no OS. Mesmo que um bug no Claude Code ignore a deny rule, o sandbox captura a tentativa.

11. Fluxo de Avaliacao de Permissoes

Fluxo de avaliacao: deny tem prioridade absoluta, seguido de ask e allow.

12. Configuracoes por Escopo de Prioridade

Regras de permissao podem ser definidas em multiplos niveis. Deny rules sao cumulativas - se qualquer nivel define deny, nenhum outro pode sobrescrever com allow.

Prioridade Escopo Origem
1 (maior) Managed settings (enterprise) Policy da organizacao via MDM
2 CLI arguments Flags --deny na linha de comando
3 Local project .claude/settings.local.json
4 Shared project .claude/settings.json (versionado)
5 (menor) User settings ~/.claude/settings.json
Regra de ouro: Uma deny rule em qualquer nivel nao pode ser anulada por um allow em outro nivel. Deny e final e irrevogavel dentro de uma sessao. Isso garante que politicas de seguranca da organizacao nao possam ser relaxadas por desenvolvedores individuais.

13. Comandos Read-Only Reconhecidos

O Claude Code classifica automaticamente certos comandos como read-only, permitindo execucao sem prompt em modos mais restritivos. Esta lista nao e configuravel, mas deny rules podem sobrescrever esse comportamento:

Listagem

ls, find, du, stat

Leitura

cat, head, tail, grep

Informacao

echo, pwd, wc, which

Comparacao

diff

Navegacao

cd

Override

Deny rules podem bloquear qualquer um desses

Bloqueando comando read-onlyJSON
{
  "permissions": {
    "deny": [
      // Mesmo cat sendo read-only, deny rule prevalece
      "Bash(cat ~/.ssh/*)",
      "Bash(cat .env)",
      "Bash(find ~/.ssh *)"
    ]
  }
}

14. Boas Praticas

1 Comece restritivo

Defina deny rules abrangentes primeiro. E mais facil relaxar permissoes do que lembrar de adicionar restricoes depois de um incidente.

2 Versione no VCS

Coloque deny rules em .claude/settings.json e faca commit. Toda a equipe deve compartilhar as mesmas restricoes de seguranca.

3 Monitore com /permissions

Use o comando /permissions durante sessoes interativas para verificar quais regras estao ativas e diagnosticar bloqueios inesperados.

4 Combine com sandbox

Deny rules sao a camada declarativa. Sandbox e a camada de enforcement do OS. Use ambos para defense-in-depth real.

.claude/settings.json - Template completo recomendadoJSON
{
  "permissions": {
    "deny": [
      // Secrets e envs
      "Read(.env)",
      "Read(.env.*)",
      "Edit(.env)",
      "Edit(.env.*)",

      // Chaves SSH
      "Read(~/.ssh/**)",

      // Credenciais do projeto
      "Read(./secrets/**)",
      "Read(./config/credentials.json)",

      // Comandos perigosos
      "Bash(rm -rf *)",
      "Bash(git push *)",

      // Exfiltracao de dados
      "Bash(curl *)",
      "Bash(wget *)"
    ],
    "ask": [
      "Bash(git commit *)",
      "Bash(npm publish *)"
    ],
    "allow": [
      "Read",
      "Bash(npm test)",
      "Bash(npm run lint)"
    ]
  }
}
