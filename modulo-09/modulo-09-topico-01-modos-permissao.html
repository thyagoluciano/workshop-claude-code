Modulo 9 • Topico 1

Modos de Permissao: plan, default, acceptEdits, auto, dontAsk, bypass

O Claude Code oferece seis modos de permissao que controlam quanta autonomia o agente tem para executar acoes. Do planejamento read-only ate a execucao irrestrita, cada modo atende a um cenario especifico de risco e produtividade.

Visao Geral dos Modos

Os modos de permissao formam um espectro de autonomia. No extremo conservador, plan so le e propoe. No extremo oposto, bypassPermissions executa tudo sem nenhum check. Escolher o modo certo depende do contexto: quanto voce confia no ambiente, quao critico e o codigo, e se ha supervisao humana disponivel.

Modo Reads Edits Bash Prompt? Best for
plan Auto Nao Read-only N/A Investigar antes de agir, entender scope
default Auto Prompt Prompt Sim Uso geral com supervisao humana
acceptEdits Auto Auto FS auto, resto prompt Parcial Desenvolvimento ativo, refatoracao
auto Auto Auto Classifier Raro Fluxo autonomo com guardrails
dontAsk Auto Deny* Deny* Nunca CI/CD, pipelines automatizados
bypassPermissions Auto Auto Auto Nunca Containers isolados, VMs descartaveis
* dontAsk: Somente acoes listadas em permissions.allow sao executadas. Todo o resto e automaticamente negado sem prompt. Ideal quando nenhum humano esta disponivel para aprovar.

Default Mode

O modo default e o comportamento padrao do Claude Code. Operacoes de leitura (Read, Glob, Grep, LS) sao executadas automaticamente sem pedir aprovacao. Qualquer operacao que modifique o sistema — editar arquivos, rodar comandos bash, executar MCP tools — exige aprovacao explicita do usuario.

Comportamento de aprovacao

Quando o Claude solicita aprovacao para uma acao, voce tem quatro opcoes:

Y Yes

Aprova esta execucao especifica. O Claude prossegue e pode pedir novamente para a proxima acao similar.

A Yes, don't ask again

Aprova e adiciona a tool/comando ao permissions.allow do projeto. Persiste entre sessoes para aquele projeto.

N No

Rejeita esta execucao. O Claude recebe o feedback de que a acao foi negada e pode tentar uma abordagem alternativa.

E Edit

Permite editar o comando ou argumento antes de aprovar. Util para ajustar um comando bash antes de rodar.

Terminalbash
# Iniciar em default mode (padrao, nao precisa de flag)
claude

# Equivalente explicito
claude --permission-mode default
Dica: Quando voce seleciona "Yes, don't ask again", a regra e salva em .claude/settings.local.json. Isso significa que ela persiste entre sessoes naquele projeto especifico, mas nao se propaga para outros projetos.

acceptEdits Mode

O modo acceptEdits auto-aprova todas as operacoes de edicao de arquivos (Write, Edit) e comandos de filesystem (mkdir, touch, rm, mv, cp, chmod, sed). Comandos bash que nao sejam operacoes de filesystem continuam pedindo aprovacao.

O que e auto-aprovado

  • Todas as operacoes de leitura (como no default)
  • Write e Edit em qualquer arquivo dentro do working directory
  • Comandos de filesystem: mkdir, touch, rm, mv, cp, chmod, sed

O que ainda pede prompt

  • Comandos bash genericos (npm install, git push, curl, etc.)
  • Operacoes em paths fora do working directory
  • MCP tool calls
Terminalbash
# Iniciar em acceptEdits mode
claude --permission-mode acceptEdits

# Ou: a partir do default, pressione Shift+Tab uma vez
Shift+Tab cycling: No CLI interativo, Shift+Tab alterna entre os modos na ordem: default → acceptEdits → plan → default. O modo atual aparece no indicador na parte inferior do terminal.

Plan Mode

No modo plan, o Claude pesquisa o codebase e propoe mudancas sem executar nenhuma. Ele pode ler arquivos e rodar comandos shell read-only, mas nao pode editar arquivos, criar novos ou executar comandos destrutivos. O objetivo e gerar um plano de acao detalhado antes de qualquer modificacao.

Terminalbash
# Iniciar em plan mode
claude --permission-mode plan

# Ou: a partir do default, pressione Shift+Tab duas vezes
# Ou: use o comando /plan dentro de uma sessao ativa

Fluxo de aprovacao do plano

Depois que o Claude apresenta o plano, voce tem varias opcoes para prosseguir:

Opcao Comportamento
Aprovar → auto mode Executa o plano com autonomia total (requer elegibilidade para auto mode)
Aprovar → acceptEdits Executa com auto-approve de edits, mas pede confirmacao para bash commands
Aprovar → revisao manual Executa em default mode, pedindo aprovacao para cada acao
Continuar planejando Pede ao Claude para aprofundar o plano, investigar mais areas ou considerar alternativas
Ultraplan Planejamento extensivo e multi-passo com subagentes de pesquisa para tarefas complexas
Ctrl+G: Abre o plano no seu editor externo configurado ($EDITOR) para edicao manual. Voce pode reorganizar, remover ou adicionar passos antes de aprovar. O Claude respeita o plano editado como instrucao.

Auto Mode

O modo auto permite que o Claude execute a maioria das acoes autonomamente, com um classifier model de seguranca que avalia cada acao antes da execucao. O classifier e um modelo separado que decide se uma acao e segura no contexto atual.

Research preview: Auto mode esta em preview e tem requisitos especificos de elegibilidade.

Requisitos

P Plano elegivel

Max, Team, Enterprise ou API. O plano Pro nao e elegivel para auto mode.

M Modelo compativel

Sonnet 4.6, Opus 4.6 ou Opus 4.7. Modelos anteriores nao suportam o classifier.

A API provider

Somente Anthropic API direta. Bedrock, Vertex e Azure AI Foundry nao suportam auto mode.

F Flag ou UI

claude --permission-mode auto ou selecione via Shift+Tab / interface grafica.

O que o classifier bloqueia

O classifier de seguranca avalia cada tool call individualmente. Acoes classificadas como potencialmente perigosas sao bloqueadas e o usuario e notificado:

Categoria Exemplos bloqueados
Execucao remota curl ... | bash, download e execute de scripts remotos
Exfiltracao de dados Enviar conteudo de arquivos locais para endpoints externos
Deploy production Deploys para ambientes de producao, terraform apply, etc.
Delecao em massa rm -rf em diretorios criticos, DROP TABLE, truncate
Permissoes IAM Modificar roles, policies, service accounts
Force push git push --force para main/master

O que o classifier permite

Categoria Exemplos permitidos
Operacoes de arquivo locais Criar, editar, mover, deletar arquivos no projeto
Gerenciamento de dependencias npm install, pip install, cargo add
HTTP read-only curl -s para leitura, requests GET para APIs
Git no branch atual git push para o branch corrente (nao main/master)
Build e teste npm run build, pytest, cargo test

Boundaries da conversa

Regras definidas em CLAUDE.md, permissions.deny e managed policies sao respeitadas pelo classifier. Se voce definir "deny": ["Bash(npm publish)"], o classifier bloqueara npm publish mesmo em auto mode. As boundaries da conversa sao inputs para a decisao do classifier, nao apenas sugestoes.

Fallback automatico

Limites de bloqueio: Apos 3 bloqueios consecutivos ou 20 bloqueios totais em uma sessao, o auto mode faz fallback automatico para o modo default, exigindo aprovacao manual para todas as acoes subsequentes.

dontAsk Mode

O modo dontAsk nunca solicita aprovacao ao usuario. Acoes que estao explicitamente listadas em permissions.allow sao executadas automaticamente. Qualquer acao que nao esta na allowlist e automaticamente negada sem prompt.

Terminalbash
# Ideal para CI/CD pipelines
claude --permission-mode dontAsk \
  --allowedTools "Read,Write,Edit,Bash(npm test),Bash(npm run lint)"

Este modo e projetado para ambientes onde nenhum humano esta disponivel para responder prompts — CI/CD, scripts automatizados, cron jobs. A combinacao de dontAsk com uma allowlist restrita garante que o Claude executa apenas as acoes previstas.

Configuracao recomendada: Em pipelines, combine dontAsk com --allowedTools e timeouts. Sem uma allowlist, o Claude nao consegue fazer nada alem de ler arquivos, pois todas as acoes modificadoras serao negadas.

bypassPermissions Mode

O modo bypassPermissions pula todos os checks de permissao. Nenhuma acao e avaliada, nenhum prompt e exibido, nenhum classifier e consultado. O Claude executa tudo diretamente.

Restricoes criticas:
• Nao pode ser ativado mid-session — requer a flag --dangerously-skip-permissions ao iniciar
• Recusado com root/sudo — o Claude Code se recusa a rodar em bypass como root
• Somente em ambientes isolados — containers Docker, VMs descartaveis, sandboxes CI
Terminalbash
# Equivalente a bypassPermissions
claude --dangerously-skip-permissions

# Flag explicita de permission mode
claude --permission-mode bypassPermissions

# Em Docker (uso tipico)
docker run -it my-sandbox claude --dangerously-skip-permissions -p "fix all linting errors"

Este modo existe exclusivamente para ambientes descartaveis onde o custo de um erro e zero (container que sera destruido) e a produtividade maxima e desejada. Nunca use em maquinas de desenvolvimento com dados reais.

Protected Paths

Independente do modo de permissao (exceto bypassPermissions), certos paths nunca sao auto-aprovados. Mesmo em auto ou acceptEdits, modificacoes nesses paths exigem aprovacao manual.

Diretorios protegidos

PathsSempre pedem aprovacao
.git/              # Internals do git
.vscode/           # Configuracao do VS Code
.idea/             # Configuracao do IntelliJ/JetBrains
.husky/            # Git hooks
.claude/           # Configuracao do Claude Code
                   # (exceto: commands/, agents/, skills/, worktrees/)

Arquivos protegidos

G Git config

.gitconfig, .gitmodules

S Shell config

.bashrc, .bash_profile, .zshrc, .zprofile, .profile

T Tooling config

.ripgreprc

C Claude config

.mcp.json, .claude.json

Excecoes em .claude/: Os subdiretorios commands/, agents/, skills/ e worktrees/ dentro de .claude/ nao sao protegidos. O Claude pode criar e editar slash commands, subagentes, skills e worktree configs sem pedir aprovacao nos modos que auto-aprovam edits.

Como Trocar de Modo

No CLI interativo

TerminalAtalhos
# Shift+Tab: alterna entre default, acceptEdits e plan
# O modo atual aparece no indicador da barra inferior

# Comando /plan: entra diretamente em plan mode
/plan

# Volta para o modo anterior ao sair do plan

Via flag de inicializacao

Terminalbash
claude --permission-mode default
claude --permission-mode acceptEdits
claude --permission-mode plan
claude --permission-mode auto
claude --permission-mode dontAsk
claude --permission-mode bypassPermissions

Como padrao em settings

.claude/settings.jsonJSON
{
  "defaultMode": "acceptEdits"
}

O defaultMode define qual modo o Claude Code usa quando inicia sem a flag --permission-mode. Pode ser definido no nivel de projeto (.claude/settings.json) ou de usuario (~/.claude/settings.json).

No VS Code, Desktop e Web

Nas interfaces graficas, o indicador de modo aparece na parte inferior do painel do Claude. Clique nele para alternar entre os modos disponiveis. A disponibilidade de cada modo depende do seu plano e configuracao.

Espectro de Autonomia

Espectro de autonomia: do controle total (plan) ate execucao irrestrita (bypass)

Quando Usar Cada Modo

P plan

Codebase desconhecido. Investigar bug complexo. Entender impacto antes de refatorar. Revisar arquitetura. Gerar plano para aprovar manualmente.

D default

Primeiro contato com o projeto. Mudancas sensiveis. Producao. Quando voce quer revisar cada acao. Onboarding de novos devs no Claude Code.

A acceptEdits

Refatoracao ativa. Desenvolvimento feature em branch. Quando voce confia nos edits mas quer ver os comandos. Pair programming com Claude.

U auto

Tarefas well-scoped com baixo risco. Implementar feature inteira em branch isolado. Bug fixes com testes. Quando quer velocidade com guardrails.

K dontAsk

CI/CD pipelines. GitHub Actions. Scripts automatizados sem usuario interativo. Linting e formatacao automatica. Pre-commit hooks.

B bypass

Containers Docker descartaveis. VMs de CI efemeras. Sandboxes de teste. Nunca em maquinas com dados reais ou acesso a producao.

Comparacao Rapida: Fluxo de Decisao

Use este fluxo para escolher o modo adequado:

  1. Precisa apenas investigar sem modificar nada? → plan
  2. Quer revisar cada acao antes de executar? → default
  3. Confia nos edits de arquivo mas quer ver comandos? → acceptEdits
  4. Quer autonomia com guardrails de seguranca? → auto
  5. Nao tem usuario interativo disponivel? → dontAsk + allowlist
  6. Ambiente 100% descartavel sem dados reais? → bypassPermissions
Recomendacao pratica: Comece com default, evolua para acceptEdits quando estiver confortavel com o projeto, e use plan antes de mudancas grandes. Reserve auto para tarefas bem definidas em branches isolados.
