managed-settings.json para Politicas de Equipe/Empresa

Configuracoes centralizadas que administradores implantam e que nao podem ser sobrescritas por settings de usuario ou projeto. Controle total de seguranca, permissoes e comportamento do Claude Code em escala organizacional.

1. O que sao Managed Settings

Managed settings sao configuracoes definidas por administradores de equipe ou empresa que possuem a maior prioridade na hierarquia de settings do Claude Code. Diferente de settings de usuario ou projeto, managed settings nao podem ser sobrescritos por nenhuma outra camada de configuracao.

O objetivo e simples: garantir que politicas de seguranca, restricoes de ferramentas e regras de compliance sejam aplicadas de forma uniforme em toda a organizacao, independente do que cada desenvolvedor configure localmente. Se um admin define que .env nunca deve ser lido, nenhum usuario pode desfazer essa regra.

Regra fundamental: Se um deny existe em qualquer nivel da hierarquia de settings, nenhum outro nivel pode fazer allow dessa mesma permissao. O deny sempre vence. Managed settings apenas reforçam isso ao garantir que o deny nao pode ser removido.

Managed settings suportam todas as propriedades de settings normais (permissions, allowedTools, deniedTools, mcpServers, hooks, etc.) e adicionam propriedades exclusivas que so existem nesse nivel, como disableBypassPermissionsMode e allowManagedMcpServersOnly.

2. Hierarquia de Settings

O Claude Code resolve configuracoes em uma ordem de precedencia bem definida. Entender essa hierarquia e essencial para saber o que pode e o que nao pode ser sobrescrito.

Prioridade Camada Local Pode ser sobrescrito?
1 (maior) Managed settings MDM / servidor / arquivo de sistema Nao - camada final
2 Command line arguments Flags na invocacao do CLI Sim, por managed
3 Local project settings .claude/settings.local.json Sim, por managed e CLI
4 Shared project settings .claude/settings.json Sim, por niveis superiores
5 (menor) User settings ~/.claude/settings.json Sim, por todos acima
Deny e absoluto: Independente da hierarquia de precedencia, se uma regra deny aparece em qualquer camada, ela nao pode ser revertida por um allow em outra camada. O merge de permissoes e aditivo para allows, mas deny sempre prevalece.

3. Mecanismos de Entrega

Existem tres formas de entregar managed settings para as maquinas dos desenvolvedores. Cada uma atende cenarios diferentes de infraestrutura e escala.

1 MDM / OS-level

Politicas nativas do sistema operacional. Registry keys no Windows, plist no macOS, arquivos em /etc no Linux. Ideal para organizacoes que ja possuem infraestrutura de MDM (Jamf, Intune, etc.).

2 Arquivo no sistema

Arquivo managed-settings.json posicionado em local padrao do OS. Pode ser distribuido por scripts de provisionamento, Ansible, Chef, ou qualquer ferramenta de config management.

3 Server-managed

Entregue pela API da Anthropic no momento da autenticacao. Configurado via interface web no claude.ai. Sem necessidade de infraestrutura de endpoint ou deploy em maquinas.

Locais de arquivo por sistema operacional

Caminhos padrao do managed-settings.json
# macOS
~/Library/Application Support/claude-code/managed-settings.json

# Linux
/etc/claude-code/managed-settings.json

# Windows
%APPDATA%\claude-code\managed-settings.json

4. Server-Managed Settings

Server-managed settings sao a forma mais conveniente de aplicar politicas em escala. As configuracoes sao definidas na interface web do claude.ai e entregues automaticamente para cada instancia do Claude Code via API durante a autenticacao.

Como funciona

  • Configurado em Claude.ai > Admin Settings > Claude Code > Managed settings
  • Entregue pela API no momento da autenticacao do usuario
  • Refresh automatico a cada hora durante sessoes ativas
  • Sem necessidade de infraestrutura de endpoint ou deploy nas maquinas
  • Requisitos de versao: v2.1.38+ (Teams) ou v2.1.30+ (Enterprise)
Vantagem operacional: Com server-managed settings, voce atualiza a politica em um unico lugar (painel web) e todas as instancias do Claude Code na organizacao recebem a atualizacao automaticamente no proximo refresh, sem tocar em nenhuma maquina.

5. Settings Exclusivos de Managed

Alem de suportar todas as propriedades de settings normais, managed settings oferecem configuracoes exclusivas que so existem nesse nivel. Essas propriedades controlam aspectos criticos de seguranca e compliance.

Propriedade Descricao
allowManagedHooksOnly Quando true, so hooks de managed settings, SDK e plugins forcados em enabledPlugins sao carregados. Hooks de user/project sao bloqueados.
allowManagedMcpServersOnly Restringe MCP servers exclusivamente aos definidos em allowedMcpServers de managed settings.
allowManagedPermissionRulesOnly Previne user e project de definir regras allow, ask ou deny. So regras managed se aplicam.
disableBypassPermissionsMode Impede o uso de --dangerously-skip-permissions e bypass mode. Critico para compliance.
disableAutoMode Impede que usuarios ativem auto mode, forcando confirmacao manual de acoes.
blockedMarketplaces Lista de URLs de marketplaces de plugins que devem ser bloqueados.
strictKnownMarketplaces Controla quais marketplaces de plugins sao permitidos. Somente fontes aprovadas.
channelsEnabled Habilita ou desabilita o recurso de channels na organizacao.
allowedChannelPlugins Allowlist de plugins que podem ser usados em channels.
forceRemoteSettingsRefresh Bloqueia startup do CLI ate que managed settings sejam buscados com sucesso. Modelo fail-closed.
sandbox.filesystem.allowManagedReadPathsOnly Restringe leitura de filesystem somente a paths definidos em managed settings.
sandbox.network.allowManagedDomainsOnly Restringe acesso de rede somente a dominios definidos em managed settings.
pluginTrustMessage Mensagem customizada exibida quando o usuario e solicitado a confiar em um plugin.
wslInheritsWindowsSettings Permite que WSL herde managed settings definidos no lado Windows.

6. Exemplos Praticos de Politicas

6.1 Forcar deny rules para arquivos sensiveis

managed-settings.json - Proteger .env e .ssh
{
  "permissions": {
    "deny": [
      "Read(.env*)",
      "Read(.ssh/**)",
      "Write(.env*)",
      "Write(.ssh/**)",
      "Read(**/*.pem)",
      "Read(**/*.key)"
    ]
  }
}

6.2 Desabilitar bypass permissions mode

managed-settings.json - Bloquear bypass
{
  "disableBypassPermissionsMode": true,
  "disableAutoMode": false
}

6.3 Restringir MCP servers a lista aprovada

managed-settings.json - MCP allowlist
{
  "allowManagedMcpServersOnly": true,
  "mcpServers": {
    "internal-docs": {
      "command": "npx",
      "args": ["@company/mcp-docs-server"]
    },
    "approved-db": {
      "command": "npx",
      "args": ["@company/mcp-db-readonly"]
    }
  }
}

6.4 Forcar sandbox habilitado

managed-settings.json - Sandbox obrigatorio
{
  "sandbox": {
    "enabled": true,
    "filesystem": {
      "allowManagedReadPathsOnly": true,
      "readPaths": ["/home/*/projects", "/opt/shared"]
    },
    "network": {
      "allowManagedDomainsOnly": true,
      "allowedDomains": [
        "api.anthropic.com",
        "registry.npmjs.org",
        "github.com"
      ]
    }
  }
}

6.5 Definir defaultMode para toda equipe

managed-settings.json - Modo padrao
{
  "defaultMode": "acceptEdits",
  "disableAutoMode": true
}

7. Fail-Closed Enforcement

A propriedade forceRemoteSettingsRefresh implementa um modelo fail-closed para managed settings. Quando habilitada, o CLI do Claude Code bloqueia o startup ate que os managed settings sejam buscados com sucesso do servidor.

Comportamento fail-closed

  • Se o fetch dos managed settings falha (timeout, rede indisponivel), o CLI nao inicia
  • Garante que politicas estao sempre atualizadas antes de qualquer operacao
  • Previne cenarios onde um desenvolvedor trabalha com politicas desatualizadas ou sem politicas
  • Essencial para ambientes com requisitos rigorosos de compliance (SOC2, HIPAA, etc.)
Impacto operacional: Com forceRemoteSettingsRefresh: true, se o servidor de settings estiver fora do ar, nenhum desenvolvedor consegue usar o Claude Code. Avalie esse tradeoff entre seguranca e disponibilidade antes de habilitar.
managed-settings.json - Fail-closed
{
  "forceRemoteSettingsRefresh": true
}

8. Managed Hooks

Quando allowManagedHooksOnly e true, o Claude Code carrega somente hooks definidos em tres fontes:

1 Managed settings

Hooks definidos diretamente no arquivo managed-settings.json ou via server-managed settings.

2 SDK

Hooks registrados programaticamente via Claude Code SDK em integracoes customizadas.

3 Plugins forcados

Hooks de plugins listados em enabledPlugins dentro de managed settings.

Hooks definidos por usuario (~/.claude/settings.json), por projeto (.claude/settings.json) e por plugins nao forcados sao completamente ignorados. Isso garante que nenhum hook nao autorizado execute em maquinas da organizacao.

managed-settings.json - Hooks gerenciados
{
  "allowManagedHooksOnly": true,
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Write|Edit",
        "command": "/opt/company/hooks/audit-write.sh $TOOL_INPUT"
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Bash",
        "command": "/opt/company/hooks/log-bash-output.sh $TOOL_INPUT"
      }
    ]
  }
}

9. Managed MCP

A propriedade allowManagedMcpServersOnly restringe os MCP servers disponíveis exclusivamente aos definidos em managed settings. Isso impede que desenvolvedores adicionem MCP servers arbitrarios, potencialmente inseguros.

Merge parcial: Enquanto allowedMcpServers vem exclusivamente de managed settings quando essa flag esta ativa, deniedMcpServers ainda faz merge de todas as fontes (user, project, managed). Ou seja, qualquer camada pode bloquear um server, mas so managed pode permitir.
managed-settings.json - MCP restrito
{
  "allowManagedMcpServersOnly": true,
  "mcpServers": {
    "company-knowledge-base": {
      "command": "npx",
      "args": ["@company/mcp-kb", "--readonly"]
    },
    "jira-integration": {
      "command": "npx",
      "args": ["@company/mcp-jira"]
    }
  },
  "deniedMcpServers": ["*filesystem*", "*sqlite*"]
}

10. Managed Permissions

Com allowManagedPermissionRulesOnly habilitado, usuarios e projetos nao podem definir regras allow, ask ou deny. Somente as regras de permissao definidas em managed settings se aplicam.

Isso e particularmente util quando a organizacao quer um conjunto uniforme de permissoes que nao pode ser relaxado por projetos individuais. Um projeto nao pode fazer allow para uma ferramenta que a politica corporativa define como ask.

managed-settings.json - Permissoes centralizadas
{
  "allowManagedPermissionRulesOnly": true,
  "permissions": {
    "allow": [
      "Read",
      "Glob",
      "Grep",
      "ListDir"
    ],
    "ask": [
      "Write",
      "Edit",
      "MultiEdit"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(curl * | bash)",
      "Read(.env*)",
      "Read(**/*.key)"
    ]
  }
}

11. Exemplo JSON Completo

Uma configuracao managed realista que combina deny rules, sandbox, restricoes de MCP, controle de hooks e enforcement fail-closed.

managed-settings.json - Configuracao corporativa completa
{
  "disableBypassPermissionsMode": true,
  "disableAutoMode": false,
  "forceRemoteSettingsRefresh": true,
  "allowManagedHooksOnly": true,
  "allowManagedMcpServersOnly": true,
  "allowManagedPermissionRulesOnly": true,
  "defaultMode": "acceptEdits",
  "pluginTrustMessage": "Este plugin foi aprovado pela equipe de seguranca.",

  "permissions": {
    "allow": [
      "Read",
      "Glob",
      "Grep",
      "ListDir"
    ],
    "ask": [
      "Write",
      "Edit",
      "MultiEdit",
      "Bash"
    ],
    "deny": [
      "Read(.env*)",
      "Read(.ssh/**)",
      "Read(**/*.pem)",
      "Read(**/*.key)",
      "Write(.env*)",
      "Write(.ssh/**)",
      "Bash(rm -rf /)",
      "Bash(curl * | sh)",
      "Bash(wget * | sh)"
    ]
  },

  "sandbox": {
    "enabled": true,
    "filesystem": {
      "allowManagedReadPathsOnly": true,
      "readPaths": [
        "/home/*/projects",
        "/opt/shared/libs"
      ]
    },
    "network": {
      "allowManagedDomainsOnly": true,
      "allowedDomains": [
        "api.anthropic.com",
        "registry.npmjs.org",
        "github.com",
        "*.company-internal.com"
      ]
    }
  },

  "mcpServers": {
    "company-docs": {
      "command": "npx",
      "args": ["@company/mcp-docs-server", "--readonly"]
    },
    "approved-db": {
      "command": "npx",
      "args": ["@company/mcp-postgres", "--read-only"]
    }
  },
  "deniedMcpServers": ["*filesystem*", "*sqlite*", "*exec*"],

  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Write|Edit|MultiEdit",
        "command": "/opt/company/hooks/audit-changes.sh"
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Bash",
        "command": "/opt/company/hooks/log-bash.sh $TOOL_INPUT"
      }
    ]
  }
}

13. ConfigChange Hooks e Auditoria

Para auditar mudancas de settings durante uma sessao ativa, voce pode utilizar hooks nos eventos ConfigChange. Isso permite registrar quando um desenvolvedor tenta alterar configuracoes locais enquanto o Claude Code esta em execucao.

Alem disso, hooks SubagentStart e SubagentStop permitem monitorar quando subagentes sao criados e destruidos, garantindo visibilidade sobre toda a arvore de execucao.

managed-settings.json - Hooks de auditoria
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Write(.claude/settings*)",
        "command": "/opt/company/hooks/alert-settings-change.sh $TOOL_INPUT"
      }
    ],
    "SubagentStart": [
      {
        "matcher": "",
        "command": "/opt/company/hooks/log-subagent.sh start $TOOL_INPUT"
      }
    ],
    "SubagentStop": [
      {
        "matcher": "",
        "command": "/opt/company/hooks/log-subagent.sh stop $TOOL_INPUT"
      }
    ]
  }
}
Centralizacao de logs: Os hooks de auditoria devem enviar dados para um sistema centralizado (Splunk, Datadog, ELK stack). Os scripts em /opt/company/hooks/ atuam como shims que formatam e encaminham eventos para a infraestrutura de observabilidade da organizacao.

14. OpenTelemetry Monitoring

O Claude Code suporta OpenTelemetry (OTel) para rastrear e auditar atividade em escala organizacional. Metricas e traces permitem que equipes de seguranca e compliance monitorem o uso do Claude Code em tempo real.

T Traces

Spans para cada invocacao de ferramenta, chamada de API e execucao de hook. Permitem reconstruir a sequencia completa de acoes de uma sessao.

M Metricas

Contadores de ferramentas usadas, tokens consumidos, tempo de execucao, erros e rejeicoes de permissao. Dashboards para visibilidade organizacional.

Para habilitar OTel, configure as variaveis de ambiente padrao do OpenTelemetry. O Claude Code exporta dados no formato OTLP, compativel com qualquer backend OTel (Jaeger, Zipkin, Grafana Tempo, etc.).

Variaveis de ambiente para OpenTelemetry
export OTEL_EXPORTER_OTLP_ENDPOINT="https://otel-collector.company.com:4318"
export OTEL_EXPORTER_OTLP_HEADERS="Authorization=Bearer $OTEL_TOKEN"
export OTEL_SERVICE_NAME="claude-code"
export OTEL_TRACES_EXPORTER="otlp"
export OTEL_METRICS_EXPORTER="otlp"
Compliance e auditoria: Combinando managed settings com OpenTelemetry, voce cria uma camada completa de governanca: as politicas garantem o que pode ser feito, e OTel registra o que foi feito. Juntas, atendem requisitos de auditoria para SOC2, ISO 27001, HIPAA e similares.
