Escolha do Modelo por Tarefa: Opus, Sonnet, Haiku

Como selecionar o modelo certo para cada subagente, otimizando custo, latencia e capacidade em workflows multi-agente do Claude Code.

Por que a Escolha do Modelo Importa

Em workflows multi-agente, cada subagente executa uma tarefa diferente com requisitos distintos. Um subagente que apenas busca arquivos nao precisa da mesma capacidade de raciocinio que um orquestrador que toma decisoes arquiteturais. A escolha do modelo por tarefa permite otimizar tres dimensoes simultaneamente:

C Capacidade

Profundidade de raciocinio, compreensao de contexto complexo, qualidade das decisoes tomadas pelo agente.

$ Custo

Tokens consumidos por invocacao. Em pipelines com dezenas de subagentes, a diferenca se acumula rapidamente.

L Latencia

Tempo de resposta por invocacao. Modelos menores respondem mais rapido, critico para tarefas interativas.

Principio fundamental: Use o modelo mais barato e rapido que consiga realizar a tarefa com qualidade aceitavel. Nao faz sentido usar Opus para listar arquivos em um diretorio.

Modelos Disponiveis no Claude Code

O Claude Code suporta tres familias de modelos, cada uma acessivel tanto por alias (nome curto) quanto por model ID completo:

Alias Model ID Perfil
opus claude-opus-4-7 Maxima capacidade, mais lento e caro
sonnet claude-sonnet-4-6 Equilibrio entre capacidade e custo
haiku claude-haiku-3-5-20241022 Rapido e economico, menor capacidade
inherit — Herda o modelo da conversa principal (padrao)

O valor inherit e o padrao para subagentes. Quando nenhum modelo e especificado, o subagente usa automaticamente o mesmo modelo da sessao principal do Claude Code.

Dica: Use aliases (sonnet, opus, haiku) para simplicidade. Use model IDs completos quando precisar fixar uma versao especifica para reprodutibilidade.

Ordem de Resolucao do Modelo

Quando um subagente e invocado, o Claude Code resolve qual modelo usar seguindo uma cadeia de prioridade bem definida. A primeira fonte que fornece um valor vence:

Cadeia de prioridade (da mais alta para a mais baixa)

1
Variavel de ambiente CLAUDE_CODE_SUBAGENT_MODEL

Sobrescreve tudo. Util para forcar um modelo em todos os subagentes durante desenvolvimento ou CI.

2
Parametro model por invocacao

Passado diretamente na chamada ao subagente via CLI ou API programatica.

3
Campo model no frontmatter do subagente

Definido no arquivo YAML/Markdown do subagente. E a forma mais comum de configurar.

4
Modelo da conversa principal (inherit)

Fallback padrao. O subagente usa o mesmo modelo que o agente principal esta usando.

Exemplo: Forcar Haiku em todos os subagentes via env
export CLAUDE_CODE_SUBAGENT_MODEL=haiku
claude "refatorar o modulo de autenticacao"

Perfil de Cada Modelo

O Opus

Maxima capacidade

Melhor para:

• Orquestracao complexa

• Decisoes arquiteturais

• Coordenar equipes de subagentes

• Raciocinio multi-etapa profundo

Trade-off: Mais lento e mais caro por token. Use apenas quando a qualidade da decisao justifica o investimento.

S Sonnet

Equilibrio ideal

Melhor para:

• Implementacao de codigo

• Code review detalhado

• Analise e documentacao

• Maioria das tarefas do dia-a-dia

Trade-off: Bom custo-beneficio. E o modelo padrao recomendado para a maioria dos subagentes.

H Haiku

Velocidade e economia

Melhor para:

• Exploracao rapida de codebase

• Classificacao e triagem

• Busca e filtragem

• Tarefas simples e repetitivas

Trade-off: Menor capacidade de raciocinio complexo. Nao recomendado para decisoes criticas.

Piramide de delegacao: Opus orquestra, Sonnet implementa, Haiku explora. As setas tracejadas representam delegacao de tarefas.

Escolha Built-in dos Subagentes

O Claude Code ja vem com subagentes internos pre-configurados com modelos especificos. Esses defaults refletem as melhores praticas da equipe da Anthropic:

Subagente Modelo Razao
Explore Haiku Busca rapida em codebase. Nao precisa raciocinar, apenas localizar.
Plan inherit Planejamento requer raciocinio. Herda o modelo principal.
General-purpose inherit Subagente generico. Herda para flexibilidade.
statusline-setup Sonnet Configuracao de UI. Requer capacidade moderada.
claude-code-guide Haiku Consulta a documentacao. Tarefa simples de recuperacao.
Observe o padrao: Tarefas de busca e consulta usam Haiku. Tarefas que precisam de raciocinio herdam o modelo principal. Essa mesma logica deve guiar suas escolhas ao criar subagentes customizados.

Configurando o Modelo no Subagente

Via frontmatter YAML

A forma mais comum e declarar o campo model diretamente no frontmatter do arquivo de definicao do subagente:

agents/explorer.mdYAML frontmatter
---
name: explorer
model: haiku
description: Explora o codebase buscando arquivos e padroes relevantes
allowed_tools:
  - Read
  - Glob
  - Grep
---

Voce e um agente de exploracao de codebase.
Sua funcao e localizar arquivos, padroes e trechos
de codigo relevantes para a tarefa solicitada.

Retorne apenas os caminhos e trechos encontrados,
sem tentar modificar nenhum arquivo.
agents/architect.mdYAML frontmatter
---
name: architect
model: opus
description: Analisa arquitetura e toma decisoes de design
allowed_tools:
  - Read
  - Glob
  - Grep
---

Voce e um arquiteto de software senior.
Analise o codebase, identifique padroes arquiteturais
e proponha decisoes de design fundamentadas.

Considere trade-offs de manutenibilidade,
performance e escalabilidade.

Via CLI com --agents

Tambem e possivel especificar subagentes e seus modelos diretamente na linha de comando usando JSON:

TerminalCLI
# Invocacao direta com modelo especifico
claude --model opus "analisar a arquitetura do projeto"

# Variavel de ambiente para forcar modelo em subagentes
CLAUDE_CODE_SUBAGENT_MODEL=haiku claude "refatorar modulo X"

Via model ID completo

Para fixar uma versao especifica do modelo (util em pipelines de CI/CD onde reprodutibilidade importa):

agents/reviewer.mdYAML frontmatter
---
name: reviewer
model: claude-sonnet-4-6
description: Revisa codigo com criterios de qualidade
allowed_tools:
  - Read
  - Glob
  - Grep
---

Voce e um revisor de codigo.
Analise o diff fornecido e reporte problemas
de qualidade, seguranca e performance.

Effort Levels

Alem do modelo, o Claude Code permite configurar o nivel de esforco do subagente atraves do campo effort no frontmatter. Esse campo controla o quanto de processamento o modelo dedica a cada resposta:

Effort Descricao Caso de uso
low Respostas rapidas, menos reflexao Classificacao, triagem, lookups simples
medium Nivel padrao de processamento Maioria das tarefas do dia-a-dia
high Mais reflexao e verificacao Implementacao, code review
xhigh Processamento extensivo Analise arquitetural, decisoes criticas
max Maximo de esforco possivel Problemas extremamente complexos
agents/quick-classifier.mdYAML frontmatter
---
name: quick-classifier
model: haiku
effort: low
description: Classifica rapidamente o tipo de tarefa
allowed_tools:
  - Read
---

Classifique a tarefa recebida em uma das categorias:
bug-fix, feature, refactor, docs, test.
Responda apenas com a categoria.
Interacao model + effort: Combinar um modelo mais capaz com effort baixo pode ser mais eficiente do que usar um modelo menos capaz com effort alto. Por exemplo, sonnet com effort: low pode superar haiku com effort: high em certas tarefas, com latencia comparavel.

Estrategias de Otimizacao de Custo

Haiku para Exploracao

Use Haiku para subagentes que apenas leem e buscam: listar arquivos, grep por padroes, navegar na estrutura do projeto. Essas tarefas nao exigem raciocinio profundo.

Sonnet para Implementacao

Subagentes que escrevem codigo, fazem review ou geram documentacao se beneficiam do equilibrio do Sonnet. E o sweet spot para a maioria das tarefas de desenvolvimento.

Opus para Orquestracao

Reserve Opus para o agente principal que coordena os demais, toma decisoes arquiteturais e precisa manter coerencia em fluxos complexos multi-etapa.

Env Var para Controle Global

Use CLAUDE_CODE_SUBAGENT_MODEL para forcar um modelo economico durante desenvolvimento/testes, e depois remova para usar os defaults em producao.

Exemplo: Impacto no custo

Considere um workflow com 5 subagentes executando em sequencia. A tabela abaixo ilustra a diferenca de custo ao escolher modelos adequados vs usar Opus para tudo:

Subagente Tarefa Todos Opus Modelo Otimizado
Explorer Buscar arquivos Opus $$$$ Haiku $
Analyzer Analisar dependencias Opus $$$$ Sonnet $$
Implementer Escrever codigo Opus $$$$ Sonnet $$
Reviewer Revisar mudancas Opus $$$$ Sonnet $$
Orchestrator Coordenar tudo Opus $$$$ Opus $$$$
Economia estimada: Ao usar modelos adequados por tarefa, voce pode reduzir o custo total do workflow em 60-70% comparado a usar Opus para tudo, sem perda perceptivel de qualidade nas tarefas que nao exigem raciocinio profundo.

Exemplo Pratico Completo: Pipeline Multi-Modelo

Vamos montar um pipeline real onde um orquestrador Opus coordena um implementador Sonnet e um explorador Haiku:

agents/orchestrator.mdOrquestrador (Opus)
---
name: orchestrator
model: opus
effort: high
description: Coordena o pipeline de refatoracao
allowed_tools:
  - Read
  - Glob
  - Grep
  - Write
  - dispatch_agent
---

Voce e o orquestrador de um pipeline de refatoracao.

## Seu fluxo de trabalho:

1. Use o agente "explorer" para mapear os arquivos relevantes
2. Analise os resultados e defina a estrategia de refatoracao
3. Use o agente "implementer" para aplicar as mudancas
4. Valide o resultado final

## Regras:
- Nao implemente codigo diretamente, delegue
- Foque em decisoes arquiteturais e coordenacao
- Verifique a coerencia entre as mudancas
agents/explorer.mdExplorador (Haiku)
---
name: explorer
model: haiku
effort: low
description: Explora rapidamente o codebase
allowed_tools:
  - Read
  - Glob
  - Grep
---

Voce e um explorador de codebase ultra-rapido.
Busque arquivos, padroes e dependencias conforme solicitado.
Retorne resultados de forma concisa e estruturada.
Nao tente analisar ou modificar nada.
agents/implementer.mdImplementador (Sonnet)
---
name: implementer
model: sonnet
effort: high
description: Implementa mudancas de codigo com qualidade
allowed_tools:
  - Read
  - Write
  - Glob
  - Grep
---

Voce e um implementador de codigo.
Receba instrucoes especificas do orquestrador e
aplique as mudancas com qualidade de producao.

## Requisitos:
- Siga os padroes existentes do codebase
- Mantenha backward compatibility
- Adicione comentarios quando necessario
- Trate edge cases

Executando o pipeline

TerminalExecucao
# O orquestrador (Opus) delega para explorer (Haiku)
# e implementer (Sonnet) automaticamente
claude --model opus \
  --agents-dir ./agents \
  "refatorar o modulo de pagamentos para usar o padrao Strategy"
Fluxo resultante: Opus analisa a tarefa e delega a exploracao para Haiku (rapido e barato). Com os resultados, Opus define a estrategia e delega a implementacao para Sonnet (bom custo-beneficio). Opus valida o resultado final. Cada modelo trabalha no que faz melhor.

Boas Praticas e Armadilhas

Faca

• Comece com inherit e otimize depois

• Use Haiku para agentes read-only

• Combine model e effort estrategicamente

• Documente a razao da escolha de modelo

• Teste com CLAUDE_CODE_SUBAGENT_MODEL antes de fixar

Evite

• Usar Opus para todas as tarefas indiscriminadamente

• Usar Haiku para tarefas que exigem raciocinio

• Ignorar o campo effort (default pode nao ser ideal)

• Fixar model IDs sem necessidade de reprodutibilidade

• Esquecer que env var sobrescreve tudo

Armadilha comum: Definir CLAUDE_CODE_SUBAGENT_MODEL=haiku no .env do projeto e esquecer. Isso forca Haiku em todos os subagentes, incluindo os que precisam de raciocinio complexo. Use essa variavel apenas temporariamente para testes de custo.

Resumo da Decisao

Guia rapido: qual modelo usar?

Tipo de Tarefa Modelo Effort
Buscar arquivos, grep, listar haiku low
Classificar, triar, categorizar haiku low / medium
Implementar codigo sonnet high
Code review sonnet high
Gerar documentacao sonnet medium
Decisao arquitetural opus high / xhigh
Orquestrar multi-agentes opus high
Debug complexo opus xhigh / max
