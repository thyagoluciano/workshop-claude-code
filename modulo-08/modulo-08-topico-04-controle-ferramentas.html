Controle de Subagentes com disallowedTools

Como aplicar o principio de menor privilegio em subagentes, restringindo ferramentas via allowlist e denylist para garantir seguranca e previsibilidade em pipelines multi-agente.

1. Principio de Menor Privilegio

Quando um subagente e invocado via Agent, ele herda todas as ferramentas da conversa principal por padrao - incluindo Write, Edit, Bash e qualquer MCP tool conectada. Em muitos cenarios, isso e excessivo e perigoso.

! Seguranca

Um subagente de pesquisa nao precisa de Write ou Edit. Se o prompt for mal interpretado, ele pode sobrescrever arquivos criticos.

⚙ Previsibilidade

Restringir ferramentas torna o comportamento do subagente determinístico. Ele so pode fazer o que foi explicitamente permitido.

✓ Reducao de Risco

Menos ferramentas = menos superficie de ataque. Subagentes limitados nao podem executar acoes nao intencionais.

Regra fundamental: Cada subagente deve ter acesso apenas as ferramentas estritamente necessarias para completar sua tarefa. Nunca confie no padrao de herdar tudo.

2. Duas Abordagens: allowlist vs denylist

O Claude Code oferece dois mecanismos complementares para controlar quais ferramentas um subagente pode usar. Ambos sao definidos no frontmatter do arquivo .md do agente.

+ tools (allowlist)

Lista apenas as ferramentas permitidas. Tudo que nao esta listado e bloqueado. Abordagem mais restritiva e segura.

- disallowedTools (denylist)

Herda tudo da conversa principal, exceto as ferramentas listadas. Util quando voce quer remover apenas algumas tools especificas.

Allowlist com tools

safe-researcher.md (frontmatter)YAML
---
tools: Read, Grep, Glob
---

# Safe Researcher

Voce e um agente de pesquisa read-only.
Analise o codebase e retorne suas descobertas.
Nunca modifique nenhum arquivo.

Neste exemplo, o subagente so pode usar Read, Grep e Glob. Ferramentas como Write, Edit, Bash e qualquer MCP tool ficam completamente inacessiveis.

Denylist com disallowedTools

no-writes.md (frontmatter)YAML
---
disallowedTools: Write, Edit
---

# No-Writes Agent

Voce pode fazer tudo exceto criar ou modificar arquivos.
Use Bash para executar testes e Grep para buscar padroes.

Aqui o subagente herda todas as ferramentas da conversa principal (incluindo Bash, Read, MCP tools, etc.), mas Write e Edit sao explicitamente removidas.

Quando ambos sao definidos

Ordem de resolucao: Quando tools e disallowedTools estao presentes no mesmo frontmatter, disallowedTools e aplicado primeiro ao pool completo de ferramentas. Depois, tools e resolvido contra o pool restante. Uma ferramenta listada em ambos e sempre removida.
Exemplo: ambos definidosYAML
---
tools: Read, Grep, Bash
disallowedTools: Bash
---

# Resultado: apenas Read e Grep disponiveis
# Bash aparece em tools mas tambem em disallowedTools
# disallowedTools tem precedencia → Bash e removido

3. Fluxo de Resolucao de Ferramentas

Fluxo de resolucao: pool completo → disallowedTools remove → tools filtra → pool final do subagente

4. Ferramentas Disponiveis no Claude Code

O Claude Code expoe um conjunto de ferramentas internas (built-in tools) que todo subagente pode potencialmente herdar. Alem dessas, qualquer MCP server conectado na conversa principal tambem e herdado.

FerramentaTipoDescricao
ReadLeituraLe conteudo de arquivos do filesystem
WriteEscritaCria ou sobrescreve arquivos
EditEscritaAplica diferencas (diffs) em arquivos existentes
BashExecucaoExecuta comandos shell arbitrarios
GlobLeituraBusca arquivos por padrao de nome (glob patterns)
GrepLeituraBusca conteudo dentro de arquivos (regex)
AgentOrquestracaoInvoca subagentes (pode ser restrito por tipo)
SkillEspecialExecuta skills registradas
mcp__*MCPQualquer tool exposta por MCP servers conectados
Heranca de MCP tools: Por padrao, subagentes herdam todas as MCP tools da conversa principal. Isso inclui tools como mcp__github__create_issue, mcp__slack__send_message, etc. Use disallowedTools para remover MCP tools especificas ou tools para excluir todas implicitamente.

5. Restringindo Quais Subagentes Podem Ser Spawned

O campo tools no frontmatter tambem controla quais tipos de subagentes um agente pode invocar. Isso e particularmente relevante quando agentes rodam como main thread via --agent.

Controle granular de Agent typesYAML
---
# Permite APENAS spawnar subagentes "worker" e "researcher"
tools: Read, Grep, Glob, Agent(worker, researcher)
---

# Orchestrator Agent
Delegue tarefas de pesquisa para o researcher
e tarefas de implementacao para o worker.
Sintaxe no frontmatterComportamento
Agent(worker, researcher)So pode spawnar subagentes com esses nomes especificos
AgentPode spawnar qualquer tipo de subagente
(omitido)Nao pode spawnar nenhum subagente
Escopo importante: O controle de Agent(type) na lista de tools so se aplica a agentes rodando como main thread via claude --agent orchestrator.md. Em subagentes comuns, o campo tools controla apenas as ferramentas built-in e MCP.

6. Desabilitando Subagentes Especificos

Alem de usar tools como allowlist, voce pode usar disallowedTools ou permissions.deny para bloquear subagentes especificos pelo nome.

Via permissions.deny no settings

.claude/settings.jsonJSON
{
  "permissions": {
    "deny": [
      "Agent(Explore)",
      "Agent(my-custom-agent)"
    ]
  }
}

Via CLI flag

TerminalShell
# Bloquear um subagente especifico via flag
claude --disallowedTools "Agent(Explore)"

# Bloquear multiplos
claude --disallowedTools "Agent(Explore),Agent(writer)"

Isso e util para desabilitar agentes built-in que voce nao quer que sejam invocados, como o Explore, ou agentes customizados que existem no diretorio .claude/agents/ mas nao devem ser usados em certos contextos.

7. Permission Modes em Subagentes

Subagentes podem ter seu modo de permissao definido no frontmatter via permissionMode. Porem, ha regras de heranca que o agente principal impoe.

ModeDescricaoHeranca
default Pede confirmacao para acoes potencialmente destrutivas Subagente usa seu proprio frontmatter
acceptEdits Aceita edicoes automaticamente, pede confirmacao para Bash Prevalece sobre frontmatter do subagente
auto Aceita tudo automaticamente dentro das allowlists Subagente herda; frontmatter e ignorado
dontAsk Nunca pergunta, mas pula acoes nao permitidas Subagente usa seu proprio frontmatter
bypassPermissions Ignora todas as verificacoes de permissao Prevalece sobre frontmatter do subagente
plan Modo somente leitura - planeja mas nao executa Subagente usa seu proprio frontmatter
Regras de heranca criticas: Se o agente principal usa bypassPermissions ou acceptEdits, esse modo prevalece sobre qualquer configuracao no frontmatter do subagente. Se o parent usa auto mode, o subagente herda auto e seu proprio frontmatter de permissao e completamente ignorado.
Exemplo: permissionMode no frontmatterYAML
---
tools: Read, Grep, Glob, Bash
permissionMode: dontAsk
---

# Test Runner

Execute os testes do projeto e reporte resultados.
Se algum teste falhar, analise o erro e sugira correcoes.

# Comportamento: se Bash nao for permitido,
# o subagente pula silenciosamente sem perguntar.

8. Hooks Condicionais com PreToolUse

Hooks PreToolUse permitem validacao customizada antes que uma ferramenta seja executada. Isso e especialmente poderoso em subagentes que usam Bash para operacoes de banco de dados.

Exemplo: db-reader com validacao SQL

db-reader.mdMarkdown
---
tools: Read, Bash, Grep
hooks:
  PreToolUse:
    - matcher: Bash
      hooks:
        - command: .claude/hooks/validate-sql.sh
---

# Database Reader

Voce consulta o banco de dados PostgreSQL para responder
perguntas sobre dados. Use APENAS queries SELECT.
Nunca execute INSERT, UPDATE, DELETE ou DROP.
.claude/hooks/validate-sql.shShell
#!/bin/bash
# Le o input JSON do hook via stdin
INPUT=$(cat)
COMMAND=$(echo "$INPUT" | jq -r '.tool_input.command')

# Verifica se o comando contem SQL destrutivo
if echo "$COMMAND" | grep -iE '(INSERT|UPDATE|DELETE|DROP|ALTER|TRUNCATE|CREATE)' > /dev/null; then
  # Exit code 2 = bloqueia a execucao da tool
  echo '{"reason": "Query SQL destrutiva detectada. Apenas SELECT e permitido."}' >&2
  exit 2
fi

# Exit code 0 = permite a execucao
exit 0
Exit codes do hook: 0 = permite a execucao normalmente. 2 = bloqueia a ferramenta e retorna a mensagem de erro para o agente. Qualquer outro exit code e tratado como erro do proprio hook.

9. MCP Servers Scoped para Subagentes

O campo mcpServers no frontmatter permite dar acesso a MCP servers exclusivos do subagente, que nao existem na conversa principal. Isso e util para isolar contexto e economizar tokens.

Definicao inline

github-agent.mdYAML
---
tools: Read, Grep
mcpServers:
  github:
    command: npx
    args: ["-y", "@modelcontextprotocol/server-github"]
    env:
      GITHUB_TOKEN: "${GITHUB_TOKEN}"
---

# GitHub Agent

Voce gerencia issues e pull requests no GitHub.
Use as ferramentas MCP do GitHub para suas operacoes.

Referencia por string

Se o MCP server ja esta definido no .claude/settings.json, voce pode referencia-lo por nome:

Referencia a server existenteYAML
---
tools: Read, Grep, Bash
mcpServers:
  - github
  - linear
---

# Issue Tracker Agent
# Herda os servers "github" e "linear" do settings.json
Economia de contexto: Manter MCP tools fora da conversa principal e usa-las apenas em subagentes dedicados reduz significativamente o consumo de contexto. Cada MCP tool adicionada a conversa principal consome tokens de definicao em toda interacao, mesmo quando nao usada.

10. Plugins vs Agents: Limitacoes

Plugins distribuidos via registro de plugins tem limitacoes importantes quando comparados a agentes locais definidos em .claude/agents/.

✓ Agents (.claude/agents/)

Suportam hooks, mcpServers, permissionMode, tools e disallowedTools no frontmatter. Controle total.

✗ Plugins (registro)

NAO suportam hooks, mcpServers ou permissionMode no frontmatter. Funcionalidade limitada.

Workaround: Se voce precisa usar um plugin com hooks ou MCP servers, copie o arquivo .md do plugin para .claude/agents/ (escopo de projeto) ou ~/.claude/agents/ (escopo global). Assim ele passa a ser tratado como um agente local com suporte completo a frontmatter.

  PreToolUse:
    - matcher: Bash
      hooks:
        - command: .claude/hooks/validate-sql.sh

---

Consulte o banco PostgreSQL via psql.
Use APENAS queries SELECT para extrair dados.

## Conexao
```
psql -h localhost -U readonly -d analytics
```

.claude/agents/code-reviewer.mdMarkdown
---

# Code Reviewer

Analise o codigo e execute testes para validar qualidade.

## Workflow
1. Leia os arquivos modificados
2. Execute os testes relevantes com Bash
3. Busque padroes problematicos com Grep
4. Retorne um relatorio com issues encontradas

---

# No-Writes Agent

Voce pode usar Bash, Grep, Read, Glob e qualquer MCP tool
disponivel. Voce NAO pode criar ou modificar arquivos.

Use este agente para diagnostico e analise sem risco
de alteracao acidental do codebase.

Voce coordena tarefas delegando para subagentes especificos:
- **safe-researcher**: para investigar e coletar informacoes

Voce NAO pode escrever ou editar arquivos diretamente.

12. Arquitetura de Restricao em Camadas

As quatro camadas de restricao sao aplicadas sequencialmente: CLI → settings → frontmatter → hooks

13. Tabela de Referencia Rapida

CenarioAbordagem RecomendadaFrontmatter
Agente so leitura allowlist restritiva tools: Read, Grep, Glob
Agente sem escrita denylist seletiva disallowedTools: Write, Edit
Agente de testes allowlist com Bash tools: Read, Grep, Bash
Agente de DB allowlist + hook tools: Read, Bash + hook SQL
Bloquear MCP tool denylist especifica disallowedTools: mcp__slack
Bloquear subagente permissions.deny ou denylist disallowedTools: Agent(Explore)
Orchestrator limitado allowlist com Agent types tools: Read, Agent(worker, reviewer)
MCP exclusivo mcpServers no frontmatter mcpServers: { github: ... }

14. Boas Praticas

1 Prefira allowlist

Use tools em vez de disallowedTools quando possivel. E mais seguro listar o que e permitido do que tentar prever tudo que deve ser bloqueado.

2 Isole MCP servers

Defina MCP servers no mcpServers do frontmatter em vez de na conversa principal. Economiza contexto e reduz superficie de ataque.

3 Use hooks para validacao

Quando Bash e necessario mas perigoso, adicione hooks PreToolUse para validar comandos antes da execucao.

4 Teste com plan mode

Use permissionMode: plan para testar o comportamento de um subagente antes de dar permissoes reais de execucao.
