Como o Claude Delega Tarefas a Subagentes (Agent Tool)

Entenda o mecanismo de delegacao do Claude Code: subagentes especializados que operam em context windows isoladas, com system prompts, ferramentas e permissoes independentes. Do Task tool ao Agent tool, dos built-ins aos personalizados.

O que sao Subagentes

Subagentes sao assistentes de IA especializados que executam tipos especificos de tarefas dentro do Claude Code. Cada subagente opera em sua propria context window, com system prompt customizado, acesso restrito a ferramentas e permissoes independentes da conversa principal.

O principio e simples: quando uma tarefa lateral inundaria a conversa principal com resultados de busca, logs ou conteudo de arquivos que voce nao vai referenciar novamente, o subagente faz esse trabalho em seu proprio contexto e retorna apenas o resumo. A conversa principal permanece limpa e focada.

Quando criar um subagente customizado? Quando voce percebe que esta repetidamente criando o mesmo tipo de worker com as mesmas instrucoes. Em vez de reescrever o prompt toda vez, defina um subagente reutilizavel.

Beneficios concretos

C Preservar contexto

Exploracao e implementacao ficam fora da conversa principal. Apenas o resultado resumido retorna.

R Restringir ferramentas

Limite quais tools o subagente pode usar. Um reviewer nao precisa de Write ou Edit.

$ Controlar custos

Roteie tarefas para modelos mais rapidos e baratos como Haiku quando a tarefa permitir.

S Especializar comportamento

System prompts focados para dominios especificos: review, debug, data analysis, etc.

Nota sobre Renomeacao: Task Tool → Agent Tool

v2.1.63: O Task tool foi renomeado para Agent tool. Referencias existentes como Task(...) em settings e definicoes de agentes continuam funcionando como aliases. Se voce tem configuracoes antigas com "deny": ["Task(Explore)"], isso ainda funciona, mas o formato recomendado agora e "deny": ["Agent(Explore)"].

Na pratica, quando Claude precisa delegar uma tarefa, ele invoca a tool Agent (anteriormente Task) com parametros que incluem o tipo do subagente, a mensagem de delegacao e opcionalmente o modelo a usar. O mecanismo interno e identico; apenas o nome mudou para refletir melhor a natureza do que esta acontecendo.

Subagentes Built-in

O Claude Code inclui subagentes built-in que sao usados automaticamente quando apropriado. Cada um herda as permissoes da conversa pai com restricoes adicionais de ferramentas.

Subagente Modelo Tools Quando e usado
Explore Haiku Read-only (sem Write/Edit) Busca e analise de codebase. Claude especifica nivel de thoroughness: quick, medium, very thorough
Plan Herda do pai Read-only (sem Write/Edit) Pesquisa de codebase durante plan mode. Coleta contexto antes de apresentar o plano
General-purpose Herda do pai Todas as tools Tarefas complexas que exigem exploracao + modificacao, raciocinio complexo, etapas dependentes
statusline-setup Sonnet Especificas Quando voce roda /statusline para configurar a status line
claude-code-guide Haiku Especificas Quando voce pergunta sobre features do Claude Code
Diferenca importante: Explore e Plan pulam os arquivos CLAUDE.md e o git status da sessao pai para manter a pesquisa rapida e barata. Todos os outros subagentes (built-in e custom) carregam ambos normalmente.

Como Criar Subagentes Personalizados

Subagentes sao definidos em arquivos Markdown com YAML frontmatter. O frontmatter configura metadados e restricoes; o corpo do arquivo se torna o system prompt do subagente. Voce pode cria-los manualmente ou usar o comando interativo /agents.

Exemplo completo de definicao

~/.claude/agents/code-reviewer.mdMarkdown + YAML
---
name: code-reviewer
description: Expert code review specialist. Proactively reviews code for quality, security, and maintainability. Use immediately after writing or modifying code.
tools: Read, Grep, Glob, Bash
model: sonnet
memory: user
---

You are a senior code reviewer ensuring high standards of code quality and security.

When invoked:
1. Run git diff to see recent changes
2. Focus on modified files
3. Begin review immediately

Review checklist:
- Code is clear and readable
- Functions and variables are well-named
- No duplicated code
- Proper error handling
- No exposed secrets or API keys
- Input validation implemented
- Good test coverage
- Performance considerations addressed

Provide feedback organized by priority:
- Critical issues (must fix)
- Warnings (should fix)
- Suggestions (consider improving)

Include specific examples of how to fix issues.

Criacao interativa com /agents

O comando /agents abre uma interface com abas para gerenciar subagentes. A aba Running mostra subagentes ativos e permite abrir ou parar. A aba Library permite:

  • Visualizar todos os subagentes disponiveis (built-in, usuario, projeto, plugin)
  • Criar novos subagentes com setup guiado ou geracao via Claude
  • Editar configuracao e acesso a ferramentas de subagentes existentes
  • Deletar subagentes customizados

No fluxo de criacao, voce pode selecionar Generate with Claude e descrever o subagente em linguagem natural. O Claude gera o identificador, description e system prompt automaticamente.

Escopo de subagentes

Subagentes podem ser armazenados em diferentes locais dependendo do escopo desejado. Quando multiplos subagentes compartilham o mesmo nome, o de maior prioridade vence.

Local Escopo Prioridade Como criar
Managed settings Organizacao 1 (mais alta) Deploy via managed settings
--agents CLI flag Sessao atual 2 JSON ao iniciar Claude Code
.claude/agents/ Projeto atual 3 Interativo ou manual
~/.claude/agents/ Todos os projetos do usuario 4 Interativo ou manual
Plugin agents/ Onde o plugin esta habilitado 5 (mais baixa) Instalado via plugins
Dica: Subagentes de projeto (.claude/agents/) sao ideais para subagentes especificos do codebase. Faca commit no version control para que o time inteiro use e aprimore colaborativamente. Para subagentes pessoais disponiveis em todos os seus projetos, use ~/.claude/agents/.

Definicao via CLI (sessao unica)

Terminalbash
claude --agents '{
  "code-reviewer": {
    "description": "Expert code reviewer. Use proactively after code changes.",
    "prompt": "You are a senior code reviewer. Focus on code quality, security, and best practices.",
    "tools": ["Read", "Grep", "Glob", "Bash"],
    "model": "sonnet"
  },
  "debugger": {
    "description": "Debugging specialist for errors and test failures.",
    "prompt": "You are an expert debugger. Analyze errors, identify root causes, and provide fixes."
  }
}'

Os subagentes definidos via --agents existem apenas para aquela sessao e nao sao salvos em disco. Util para testes rapidos e scripts de automacao.

Campos do Frontmatter

A tabela abaixo documenta todos os campos suportados no YAML frontmatter dos arquivos de definicao de subagentes. Apenas name e description sao obrigatorios.

Campo Obrigatorio Descricao
name Sim Identificador unico usando letras minusculas e hifens. Hooks recebem esse valor como agent_type
description Sim Quando o Claude deve delegar a este subagente. Quanto mais clara, melhor a delegacao automatica
tools Nao Tools que o subagente pode usar. Herda todas se omitido. Suporta Agent(worker, researcher) para restringir subagentes que podem ser spawnados
disallowedTools Nao Tools a negar, removidas da lista herdada ou especificada. Aplicado antes de tools
model Nao sonnet, opus, haiku, ID completo (ex: claude-opus-4-7), ou inherit. Default: inherit
permissionMode Nao default, acceptEdits, auto, dontAsk, bypassPermissions, ou plan. Ignorado para plugin subagents
maxTurns Nao Numero maximo de turns agenticos antes do subagente parar
skills Nao Skills para pre-carregar no contexto do subagente no startup. O conteudo completo da skill e injetado
mcpServers Nao MCP servers disponiveis para o subagente. Definicao inline ou referencia por nome. Ignorado para plugins
hooks Nao Lifecycle hooks com escopo neste subagente. Suporta PreToolUse, PostToolUse, Stop. Ignorado para plugins
memory Nao Escopo de memoria persistente: user (~/.claude/agent-memory/), project (.claude/agent-memory/), ou local
background Nao true para sempre rodar como background task. Default: false
effort Nao Nivel de esforco: low, medium, high, xhigh, max. Override do nivel da sessao
isolation Nao worktree para rodar em git worktree temporario com copia isolada do repositorio
color Nao Cor de display: red, blue, green, yellow, purple, orange, pink, cyan
initialPrompt Nao Auto-submetido como primeiro user turn quando rodando como agente principal via --agent. Commands e skills sao processados

Resolucao de modelo

Quando o Claude invoca um subagente, o modelo e resolvido nesta ordem de prioridade:

Ordem de resolucaoprioridade decrescente
1. CLAUDE_CODE_SUBAGENT_MODEL (env var, se definida)
2. Parametro model da invocacao especifica
3. Campo model do frontmatter do subagente
4. Modelo da conversa principal (inherit)

Foreground vs Background

Subagentes podem rodar em dois modos de execucao, e a diferenca e fundamental para o fluxo de trabalho:

F Foreground (Bloqueante)

Bloqueia a conversa principal ate completar. Permission prompts sao repassados ao usuario conforme surgem. Ideal para tarefas que precisam de interacao.

B Background (Concorrente)

Roda em paralelo enquanto voce continua trabalhando. Usa permissoes ja concedidas na sessao e faz auto-deny em qualquer tool call que exigiria prompt. Se precisar de perguntas, o tool call falha mas o subagente continua.

Ctrl+B move um subagente em foreground para background a qualquer momento. Voce tambem pode pedir ao Claude diretamente: "run this in the background". Para desabilitar toda funcionalidade de background tasks, defina CLAUDE_CODE_DISABLE_BACKGROUND_TASKS=1.

O Claude decide automaticamente se roda foreground ou background com base na tarefa, mas voce pode influenciar via campo background: true no frontmatter ou solicitando explicitamente. Se um background subagent falhar por falta de permissoes, inicie um novo em foreground com a mesma tarefa para retry com prompts interativos.

Gestao de Contexto dos Subagentes

O que carrega no startup

Cada subagente inicia com uma context window fresca e isolada. Ele nao ve o historico da sua conversa, as skills ja invocadas, nem os arquivos que o Claude ja leu. O Claude compoe uma mensagem de delegacao que resume a tarefa, e o subagente trabalha a partir dai.

O contexto inicial de um subagente (nao-fork) contem:

Componente O que e Explore/Plan Outros
System prompt Prompt proprio do agente + detalhes de ambiente (nao o system prompt completo do Claude Code) Sim Sim
Task message Prompt de delegacao escrito pelo Claude ao delegar o trabalho Sim Sim
CLAUDE.md + memory Toda a hierarquia de memoria: ~/.claude/CLAUDE.md, project rules, CLAUDE.local.md, managed policy files Nao Sim
Git status Snapshot tirado no inicio da sessao pai Nao Sim
Preloaded skills Conteudo completo de skills listadas no campo skills Nao Sim
Restricao critica: Subagentes NAO podem spawnar outros subagentes. Se seu workflow requer delegacao aninhada, use Skills ou encadeie subagentes a partir da conversa principal.

Auto-compaction

Subagentes suportam compactacao automatica usando a mesma logica da conversa principal. Por padrao, a compactacao dispara em aproximadamente 95% de capacidade. Para disparar mais cedo, defina CLAUDE_AUTOCOMPACT_PCT_OVERRIDE com uma porcentagem menor (por exemplo, 50).

Resumir Subagentes com SendMessage

Cada invocacao de subagente cria uma nova instancia com contexto fresco. Para continuar o trabalho de um subagente existente em vez de comecar do zero, voce pode pedir ao Claude para resumi-lo.

O mecanismo funciona via SendMessage tool com o ID do agente no campo to. O subagente resumido retem todo o historico da conversa anterior, incluindo tool calls, resultados e raciocinio. Ele continua exatamente de onde parou.

Pre-requisito: A funcionalidade de resumir subagentes requer agent teams habilitado via CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1. Sem isso, o SendMessage tool nao esta disponivel.
Exemplo de fluxoConversa interativa
# Primeira invocacao
> Use the code-reviewer subagent to review the authentication module
[Agent completes review]

# Continuacao - Claude resume o mesmo subagente
> Continue that code review and now analyze the authorization logic
[Claude resumes the subagent with full context from previous conversation]

Se um subagente parado receber um SendMessage, ele auto-resume em background sem precisar de uma nova invocacao via Agent. Os transcripts dos subagentes ficam em ~/.claude/projects/{project}/{sessionId}/subagents/agent-{agentId}.jsonl.

Padroes de Uso Efetivos

1 Isolar operacoes volumosas

Rodar testes, buscar docs, processar logs em subagentes. O output verbose fica no contexto do subagente; apenas o resumo retorna.

2 Pesquisa em paralelo

Spawnar multiplos subagentes para investigar modulos independentes simultaneamente. Funciona melhor quando as pesquisas nao dependem uma da outra.

3 Encadear subagentes

Usar subagentes em sequencia: reviewer encontra problemas, optimizer corrige. Cada um completa e retorna resultados ao Claude, que passa contexto ao proximo.

Exemplos de usoPrompts
# Isolar output volumoso
Use a subagent to run the test suite and report only the failing tests with their error messages

# Pesquisa paralela
Research the authentication, database, and API modules in parallel using separate subagents

# Encadeamento
Use the code-reviewer subagent to find performance issues, then use the optimizer subagent to fix them
Cuidado com contexto: Quando subagentes completam, seus resultados retornam a conversa principal. Rodar muitos subagentes que retornam resultados detalhados pode consumir contexto significativo. Para tarefas que precisam de paralelismo sustentado ou excedem a context window, considere agent teams.

Quando Usar Subagente vs Conversa Principal

Cenario Recomendacao Motivo
Tarefa precisa de back-and-forth iterativo Conversa principal Subagentes nao suportam interacao continua facilmente
Multiplas fases compartilham contexto significativo Conversa principal Planejamento → implementacao → teste: o contexto compartilhado e valioso
Mudanca rapida e pontual Conversa principal Overhead de spawnar subagente nao compensa
Latencia importa muito Conversa principal Subagentes comecam do zero e precisam de tempo para coletar contexto
Tarefa produz output verbose desnecessario Subagente O verbose fica isolado; apenas o resumo retorna
Quer enforcar restricoes de ferramentas Subagente Tools e permissoes especificas por subagente
Trabalho autocontido que retorna um resumo Subagente Preserva a context window principal
Pergunta rapida sobre algo ja no contexto /btw Ve seu contexto completo, sem acesso a tools, resposta descartada do historico

Invocacao Explicita de Subagentes

Alem da delegacao automatica, existem tres formas de invocar subagentes explicitamente, com nivel crescente de comprometimento:

1. Linguagem natural

PromptNenhuma sintaxe especial
Use the test-runner subagent to fix failing tests
Have the code-reviewer subagent look at my recent changes

Sem sintaxe especial. Nomeie o subagente e o Claude tipicamente delega.

2. @-mention (garante execucao)

Prompt@ + typeahead
@"code-reviewer (agent)" look at the auth changes

Digite @ e selecione o subagente no typeahead (mesmo mecanismo de @-mencionar arquivos). Isso garante que aquele subagente especifico roda, em vez de deixar a escolha para o Claude.

3. Sessao inteira como subagente (--agent)

Terminalbash
# A sessao inteira usa o system prompt, tools e modelo do subagente
claude --agent code-reviewer

# Para tornar padrao em um projeto:
# .claude/settings.json
{
  "agent": "code-reviewer"
}

O system prompt do subagente substitui inteiramente o system prompt padrao do Claude Code. CLAUDE.md e project memory ainda carregam normalmente. O nome do agente aparece como @<name> no header de startup.

Fork Mode: Herdando a Conversa Inteira

Experimental: Requer Claude Code v2.1.117+ e CLAUDE_CODE_FORK_SUBAGENT=1. Comportamento pode mudar em releases futuras. Funciona em modo interativo, nao-interativo e via SDK.

Um fork e um subagente que herda toda a conversa ate aquele momento em vez de comecar do zero. Ele ve o mesmo system prompt, tools, modelo e historico de mensagens da sessao principal. As tool calls do fork ficam fora da sua conversa; apenas o resultado final retorna.

Use fork quando um subagente nomeado precisaria de muito background para ser util, ou quando voce quer testar varias abordagens em paralelo a partir do mesmo ponto de partida.

Prompt/fork command
/fork draft unit tests for the parser changes so far

Comportamentos do fork mode

Habilitar fork mode muda o Claude Code de tres formas:

  • Claude spawna um fork quando usaria o subagente general-purpose. Subagentes nomeados como Explore continuam normais
  • Todo spawn de subagente roda em background (fork ou nomeado)
  • O comando /fork spawna um fork em vez de agir como alias para /branch

Fork vs Named Subagent

Aspecto Fork Named Subagent
Contexto Historico completo da conversa Contexto fresco com o prompt que voce passa
System prompt + tools Mesmo da sessao principal Do arquivo de definicao do subagente
Modelo Mesmo da sessao principal Do campo model do subagente
Permissoes Prompts aparecem no terminal Auto-denied quando em background
Prompt cache Compartilhado com sessao principal Cache separado
Custo Mais barato (reutiliza cache) Mais caro (novo contexto)
Vantagem do fork: Como o system prompt e as definicoes de tools do fork sao identicas ao pai, o primeiro request reutiliza o prompt cache do pai. Isso torna o forking mais barato que spawnar um subagente fresco para tarefas que precisam do mesmo contexto.

Limitacoes: um fork nao pode spawnar outros forks. O fork pode opcionalmente receber isolation: "worktree" para que edits de arquivo sejam escritos em um git worktree separado.

Hooks para Subagentes

Ha duas formas de configurar hooks que interagem com subagentes:

1. Hooks no frontmatter do subagente

Hooks definidos diretamente no markdown do subagente rodam apenas enquanto aquele subagente esta ativo. Suportam os eventos PreToolUse, PostToolUse e Stop (convertido para SubagentStop em runtime).

Subagente com hooksMarkdown + YAML
---
name: db-reader
description: Execute read-only database queries
tools: Bash
hooks:
  PreToolUse:
    - matcher: "Bash"
      hooks:
        - type: command
          command: "./scripts/validate-readonly-query.sh"
---

You are a database analyst with read-only access.
Execute SELECT queries to answer questions about the data.

2. Hooks no settings.json (nivel de projeto)

Hooks em settings.json respondem a eventos do ciclo de vida de subagentes na sessao principal:

.claude/settings.jsonJSON
{
  "hooks": {
    "SubagentStart": [
      {
        "matcher": "db-agent",
        "hooks": [
          { "type": "command", "command": "./scripts/setup-db-connection.sh" }
        ]
      }
    ],
    "SubagentStop": [
      {
        "hooks": [
          { "type": "command", "command": "./scripts/cleanup-db-connection.sh" }
        ]
      }
    ]
  }
}

Desabilitar Subagentes Especificos

Voce pode impedir o Claude de usar subagentes especificos adicionando-os ao array deny nos settings:

.claude/settings.jsonJSON
{
  "permissions": {
    "deny": ["Agent(Explore)", "Agent(my-custom-agent)"]
  }
}

Funciona para built-in e custom subagentes. Via CLI: claude --disallowedTools "Agent(Explore)".

MCP Servers com Escopo de Subagente

O campo mcpServers permite dar a um subagente acesso a MCP servers que nao estao disponiveis na conversa principal. Servers inline definidos aqui sao conectados quando o subagente inicia e desconectados quando termina:

Subagente com MCPMarkdown + YAML
---
name: browser-tester
description: Tests features in a real browser using Playwright
mcpServers:
  # Definicao inline: com escopo apenas neste subagente
  - playwright:
      type: stdio
      command: npx
      args: ["-y", "@playwright/mcp@latest"]
  # Referencia por nome: reutiliza server ja configurado
  - github
---

Use the Playwright tools to navigate, screenshot, and interact with pages.
Dica de contexto: Para manter um MCP server fora da conversa principal (evitando que tool descriptions consumam contexto la), defina-o inline no subagente. O subagente recebe as tools; a conversa pai nao.

Restricoes de Seguranca para Plugin Subagents

Por razoes de seguranca, subagentes fornecidos por plugins nao suportam os campos hooks, mcpServers ou permissionMode no frontmatter. Esses campos sao ignorados ao carregar agentes de um plugin. Se voce precisa dessas funcionalidades, copie o arquivo do agente para .claude/agents/ ou ~/.claude/agents/.

Memoria Persistente de Subagentes

O campo memory da ao subagente um diretorio persistente que sobrevive entre conversas. O subagente usa esse diretorio para acumular conhecimento ao longo do tempo: padroes do codebase, insights de debugging, decisoes arquiteturais.

Escopo Localizacao Quando usar
user ~/.claude/agent-memory/<name>/ Aprendizados que valem para todos os projetos
project .claude/agent-memory/<name>/ Conhecimento especifico do projeto, compartilhavel via VCS (recomendado)
local .claude/agent-memory-local/<name>/ Especifico do projeto mas nao deve ir para version control

Quando memoria esta habilitada, o system prompt do subagente inclui instrucoes para ler e escrever no diretorio de memoria, e as primeiras 200 linhas ou 25KB do MEMORY.md sao injetadas. Read, Write e Edit sao automaticamente habilitados para gerenciamento da memoria.
