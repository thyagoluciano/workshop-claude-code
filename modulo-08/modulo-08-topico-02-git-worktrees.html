Sessoes Paralelas com Isolamento via Git Worktrees

Como executar multiplas sessoes do Claude Code simultaneamente com total isolamento de arquivos, usando Git worktrees para separar branches e diretorios de trabalho sem conflitos.

O que sao Git Worktrees

Um git worktree e um diretorio de trabalho separado que possui seus proprios arquivos e branch, mas compartilha o mesmo historico e remote do checkout principal. Cada worktree e um checkout independente num branch proprio. Isso significa que voce pode ter dois ou mais branches em estado de "checked out" ao mesmo tempo, cada um em seu proprio diretorio no sistema de arquivos.

Na pratica, em vez de fazer git stash ou git checkout constantemente para alternar entre tarefas, voce simplesmente abre outro terminal apontando para outro worktree. O .git interno e compartilhado -- commits, refs e objects sao os mesmos. Apenas o working directory e o HEAD sao separados.

Conceito fundamental: Worktrees nao sao clones. Eles compartilham o mesmo repositorio .git. Isso significa zero duplicacao de historico, fetch unico e branches sincronizados automaticamente entre todos os worktrees.

Por que usar Worktrees com Claude Code

Quando voce roda Claude Code em modo interativo, o agente le e edita arquivos no diretorio de trabalho atual. Se voce abrir dois terminais no mesmo diretorio, ambas as sessoes vao competir pelos mesmos arquivos -- uma pode sobrescrever mudancas da outra, causando conflitos silenciosos e estado corrompido.

Com worktrees, cada sessao opera num diretorio fisicamente separado. As garantias sao claras:

1 Isolamento total de arquivos

Edicoes em uma sessao nunca tocam arquivos de outra. Cada worktree tem sua propria copia do working directory.

2 Branches independentes

Feature num terminal, bugfix em outro, refactor num terceiro. Sem colisao de checkout, sem stash perdido.

3 Historico compartilhado

Todos os worktrees enxergam os mesmos commits e remotes. Um git fetch em qualquer um atualiza todos.

4 Cleanup automatico

Claude Code gerencia o ciclo de vida dos worktrees -- cria, usa e remove automaticamente quando nao ha mudancas pendentes.

Iniciando com --worktree

A flag --worktree e o ponto de entrada. Ela instrui o Claude Code a criar um worktree dedicado antes de iniciar a sessao interativa.

Com nome explicito

Terminal
$ claude --worktree feature-auth

Esse comando cria o worktree em .claude/worktrees/feature-auth/ e faz checkout num branch chamado worktree-feature-auth. O Claude Code entao inicia a sessao interativa ja dentro desse diretorio.

Com nome aleatorio

Terminal
$ claude --worktree

Sem passar um valor, o Claude gera um nome aleatorio automaticamente. Util para sessoes rapidas de exploracao onde o nome do worktree nao importa.

Combinando com prompt direto

Terminal
$ claude --worktree feature-auth -p "Implemente autenticacao OAuth2 com Google"

A flag --worktree funciona em conjunto com -p (modo non-interactive). O worktree e criado, o prompt e executado, e o resultado fica no branch do worktree.

Importante: Adicione .claude/worktrees/ ao seu .gitignore. Os worktrees gerenciados pelo Claude sao artefatos temporarios e nao devem ser commitados no repositorio.
.gitignore
# Worktrees gerenciados pelo Claude Code
.claude/worktrees/

Escolhendo o Branch Base

Por padrao, novos worktrees sao criados a partir de origin/HEAD. Isso geralmente aponta para main ou master, dependendo da configuracao do repositorio remoto.

Re-sincronizando origin/HEAD

Se o branch padrao do remote mudou (por exemplo, de master para main), voce precisa atualizar a referencia local:

Terminal
# Atualiza origin/HEAD para apontar ao branch padrao do remote
$ git remote set-head origin -a

# Verifica o resultado
$ git symbolic-ref refs/remotes/origin/HEAD
refs/remotes/origin/main

Controle customizado via Hook

Para controle fino sobre qual branch base e usado, voce pode implementar um hook WorktreeCreate. Isso e util quando diferentes worktrees precisam partir de branches diferentes (por exemplo, um worktree de hotfix partindo de production em vez de main).

.claude/settings.json
{
  "hooks": {
    "WorktreeCreate": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "./scripts/setup-worktree.sh \"$WORKTREE_NAME\" \"$WORKTREE_PATH\""
          }
        ]
      }
    ]
  }
}

Copiando Arquivos Gitignored com .worktreeinclude

Worktrees sao checkouts limpos -- eles nao incluem arquivos que estao no .gitignore. Isso significa que arquivos como .env, chaves de configuracao local e outros artefatos ignorados pelo git nao estarao disponiveis no novo worktree.

Para resolver isso, o Claude Code suporta o arquivo .worktreeinclude, que usa a mesma sintaxe do .gitignore para especificar quais arquivos ignorados devem ser copiados para novos worktrees.

.worktreeinclude
# Variaveis de ambiente
.env
.env.local
.env.development

# Configuracoes de segredos locais
config/secrets.json
config/local.yaml

# Certificados de desenvolvimento
certs/*.pem

# Cache de dependencias (opcional, mas acelera setup)
node_modules/
.venv/
Dica pratica: Incluir node_modules/ ou .venv/ no .worktreeinclude evita ter que rodar npm install ou pip install em cada worktree. Mas cuidado com o espaco em disco -- cada copia duplica esses diretorios.

O arquivo .worktreeinclude deve ficar na raiz do repositorio. Ele e lido pelo Claude Code no momento da criacao do worktree.

Isolamento de Subagentes com Worktrees

Subagentes (cobertos no Topico 1 deste modulo) podem ser configurados para operar dentro de worktrees temporarios. Isso garante que um subagente pesquisando e editando arquivos nao interfira no trabalho do agente principal.

Configuracao via frontmatter

No CLAUDE.md, voce define o isolamento na secao de frontmatter do comando de subagente:

CLAUDE.md - frontmatter de subagente
---
isolation: worktree
---

Quando precisar investigar mudancas em multiplos arquivos,
use subagentes com isolamento de worktree para evitar
conflitos com o trabalho em andamento.

Ciclo de vida do worktree de subagente

1. Criacao: Quando o subagente e disparado, Claude Code cria automaticamente um worktree temporario.

2. Execucao: O subagente opera exclusivamente dentro desse worktree, sem afetar o diretorio do agente principal.

3. Resultado: Ao terminar, o subagente retorna seus resultados (texto/dados) para a conversa principal.

4. Cleanup: Se o subagente nao fez nenhuma mudanca nos arquivos, o worktree temporario e removido automaticamente. Se houve mudancas, Claude pergunta se voce quer mante-las.

Vantagem: O subagente pode editar arquivos livremente durante sua investigacao sem risco de corromper o estado do agente principal. Isso e particularmente util para tarefas de pesquisa exploratoria onde o subagente precisa testar hipoteses editando codigo.

Cleanup e Ciclo de Vida

O gerenciamento do ciclo de vida dos worktrees pelo Claude Code segue regras bem definidas:

✓ Sem mudancas

Worktree e removido automaticamente ao final da sessao. Nenhuma acao necessaria do usuario.

? Com mudancas

Claude pergunta se voce deseja manter o worktree. Voce pode revisar as mudancas antes de decidir.

-p Non-interactive

No modo -p (headless), nao ha cleanup automatico. O worktree permanece para inspecao posterior.

↻ Orfaos

Worktrees de subagentes que ficaram orfaos sao removidos no startup do Claude Code apos o periodo definido em cleanupPeriodDays.

Gerenciamento Manual de Worktrees

Embora o Claude Code gerencie worktrees automaticamente com --worktree, voce tambem pode criar e gerenciar worktrees manualmente usando comandos git nativos. Isso e util quando voce quer mais controle ou precisa de configuracoes especificas.

Comandos essenciais

Terminal - Criacao manual
# Criar worktree em diretorio especifico com novo branch
$ git worktree add ../meu-projeto-feature feature-branch

# Criar worktree a partir de um branch remoto
$ git worktree add ../meu-projeto-hotfix origin/production -b hotfix-123

# Criar worktree no padrao do Claude Code
$ git worktree add .claude/worktrees/minha-task -b worktree-minha-task
Terminal - Listagem e inspecao
# Listar todos os worktrees ativos
$ git worktree list
/home/user/meu-projeto           abc1234 [main]
/home/user/meu-projeto-feature   def5678 [feature-branch]
/home/user/meu-projeto-hotfix    ghi9012 [hotfix-123]

# Listar com mais detalhes (formato porcelain para scripts)
$ git worktree list --porcelain
Terminal - Remocao e limpeza
# Remover um worktree especifico
$ git worktree remove ../meu-projeto-feature

# Forcar remocao (se houver mudancas nao commitadas)
$ git worktree remove --force ../meu-projeto-feature

# Limpar referencia de worktrees que foram deletados manualmente
$ git worktree prune
Lembre-se: Ao criar um worktree manual, voce precisa inicializar o ambiente de desenvolvimento. Execute npm install, pip install -r requirements.txt, ou o equivalente do seu projeto. O .worktreeinclude do Claude Code so funciona com worktrees criados via --worktree.

Desktop App: Worktrees Automaticos

O aplicativo desktop do Claude Code adota uma abordagem diferente: ele cria automaticamente um worktree para cada nova sessao. Isso significa que toda sessao iniciada pelo app desktop ja opera em isolamento por padrao, sem necessidade de flags adicionais.

Comportamento do Desktop App

Quando voce inicia uma nova sessao no app desktop:

1. Um worktree e criado automaticamente em .claude/worktrees/

2. Um branch novo e criado para a sessao

3. Arquivos listados em .worktreeinclude sao copiados

4. A sessao opera inteiramente dentro do worktree

Ao final, as mesmas regras de cleanup se aplicam: sem mudancas, o worktree e removido; com mudancas, voce decide o que fazer.

Suporte a VCS Nao-Git

O conceito de worktrees e nativo do Git, mas o Claude Code permite estender o mecanismo para outros sistemas de controle de versao atraves dos hooks WorktreeCreate e WorktreeRemove.

Hooks para VCS alternativos

Os hooks recebem variaveis de ambiente com informacoes sobre o worktree sendo criado ou removido. Voce pode usar essas variaveis para executar operacoes equivalentes no seu VCS.

Exemplo: Hook para SVN
#!/bin/bash
# scripts/svn-worktree-create.sh
# Chamado pelo hook WorktreeCreate

WORKTREE_NAME="$1"
WORKTREE_PATH="$2"

# Criar checkout SVN equivalente
svn checkout "svn://repo/trunk" "$WORKTREE_PATH" \
  --depth infinity \
  --quiet

# Copiar configuracoes locais
cp .env "$WORKTREE_PATH/.env" 2>/dev/null || true

echo "SVN worktree criado em: $WORKTREE_PATH"
Exemplo: Hook para remocao
#!/bin/bash
# scripts/svn-worktree-remove.sh
# Chamado pelo hook WorktreeRemove

WORKTREE_PATH="$1"

# Verificar mudancas pendentes
if svn status "$WORKTREE_PATH" | grep -q '^[ACDMR]'; then
  echo "AVISO: Mudancas pendentes em $WORKTREE_PATH"
  exit 1  # Impede remocao
fi

# Remover checkout
rm -rf "$WORKTREE_PATH"
echo "SVN worktree removido: $WORKTREE_PATH"
.claude/settings.json - Configuracao dos hooks
{
  "hooks": {
    "WorktreeCreate": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "./scripts/svn-worktree-create.sh \"$WORKTREE_NAME\" \"$WORKTREE_PATH\""
          }
        ]
      }
    ],
    "WorktreeRemove": [
      {
        "matcher": "",
        "hooks": [
          {
            "type": "command",
            "command": "./scripts/svn-worktree-remove.sh \"$WORKTREE_PATH\""
          }
        ]
      }
    ]
  }
}

Esses hooks funcionam para qualquer VCS: Perforce, Mercurial, Plastic SCM, etc. A logica e a mesma -- criar um checkout separado no WorktreeCreate e limpar no WorktreeRemove.

Workflow Completo: Tres Tarefas em Paralelo

Veja um exemplo pratico de como usar worktrees para trabalhar em tres tarefas simultaneas:

Terminal 1 - Feature de autenticacao
# Inicia sessao com worktree dedicado
$ claude --worktree feature-auth

# Claude cria .claude/worktrees/feature-auth/
# Branch: worktree-feature-auth (a partir de origin/HEAD)
# Voce interage normalmente:
> Implemente login com OAuth2 usando Google como provider.
> Use passport.js e adicione testes de integracao.
Terminal 2 - Bugfix no endpoint
# Em outro terminal, simultaneamente
$ claude --worktree bugfix-users-500

# Claude cria .claude/worktrees/bugfix-users-500/
# Branch: worktree-bugfix-users-500
> O endpoint GET /api/users retorna 500 quando o campo email e null.
> Investigue e corrija, adicionando validacao adequada.
Terminal 3 - Refatoracao de banco
# Em mais outro terminal
$ claude --worktree refactor-prisma

# Claude cria .claude/worktrees/refactor-prisma/
# Branch: worktree-refactor-prisma
> Migre o acesso ao banco de dados de queries SQL raw para Prisma ORM.
> Mantenha compatibilidade com os testes existentes.

Cada sessao opera em isolamento total. Quando qualquer uma terminar, voce pode revisar as mudancas, fazer merge ou descartar:

Terminal - Pos-sessao
# Ver o que cada worktree produziu
$ git worktree list
/home/user/projeto                         abc1234 [main]
/home/user/projeto/.claude/worktrees/feature-auth     def5678 [worktree-feature-auth]
/home/user/projeto/.claude/worktrees/bugfix-users-500 ghi9012 [worktree-bugfix-users-500]
/home/user/projeto/.claude/worktrees/refactor-prisma  jkl3456 [worktree-refactor-prisma]

# Revisar mudancas de um worktree especifico
$ cd .claude/worktrees/feature-auth
$ git log --oneline main..HEAD
$ git diff main

# Fazer merge quando estiver satisfeito
$ git checkout main
$ git merge worktree-feature-auth

# Limpar worktree apos merge
$ git worktree remove .claude/worktrees/feature-auth
$ git branch -d worktree-feature-auth

Boas Praticas e Armadilhas

✓ Faca

Use .worktreeinclude para copiar .env e configuracoes locais para cada worktree.

✗ Evite

Nao faca checkout manual do mesmo branch em dois worktrees. Git impede isso para evitar corrupcao.

✓ Faca

Adicione .claude/worktrees/ ao .gitignore para nao commitar worktrees temporarios.

✗ Evite

Nao inclua node_modules/ no .worktreeinclude se espaco em disco for limitado -- pode duplicar gigabytes.

✓ Faca

Use isolation: worktree em subagentes que precisam editar arquivos durante pesquisa exploratoria.

✗ Evite

Nao misture worktrees manuais com worktrees gerenciados pelo Claude no mesmo diretorio sem entender o ciclo de cleanup.

Resumo operacional: claude --worktree <nome> e tudo que voce precisa para iniciar uma sessao isolada. O Claude Code cuida da criacao do diretorio, checkout do branch, copia de arquivos gitignored (via .worktreeinclude) e cleanup automatico. Para cenarios avancados, hooks WorktreeCreate e WorktreeRemove dao controle total sobre o processo.
