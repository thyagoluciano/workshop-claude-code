Modulo 8 • Topico 5

Padroes Multi-Agente: Revisor, Planejador, Executor

Arquiteturas de orquestracao com subagentes especializados no Claude Code. Do pesquisador read-only ao agent team com comunicacao bidirecional.

1. Introducao: Orquestracao Multi-Agente

O Claude Code suporta multiplos padroes de orquestracao multi-agente, desde subagentes simples que executam tarefas isoladas ate agent teams experimentais com comunicacao bidirecional. A ideia central e dividir responsabilidades: cada subagente tem um escopo bem definido, ferramentas restritas e, opcionalmente, um modelo diferente.

Subagentes sao definidos como arquivos .md dentro do diretorio .claude/agents/ do projeto. Cada arquivo declara nome, descricao, modelo, ferramentas permitidas e instrucoes de sistema. O agente principal (orchestrator) pode delegar tarefas a qualquer subagente configurado.

Conceito fundamental: Subagentes herdam o contexto do projeto (CLAUDE.md, .claude/settings.json) mas possuem contexto de conversa independente. Isso significa que nao poluem a janela de contexto do agente principal e podem usar modelos mais baratos para tarefas simples.

Estrutura basica de um subagente

.claude/agents/nome-do-agente.md
---
name: nome-do-agente
description: "Descricao curta do que o agente faz"
model: claude-sonnet-4-20250514
tools:
  - Read
  - Grep
  - Glob
---

2. Padrao 1: Pesquisador (Explorer/Researcher)

O pesquisador e um subagente read-only que explora o codebase e retorna findings sem modificar nada. O Claude Code ja inclui um built-in chamado Explore que usa exatamente esse padrao: modelo Haiku, ferramentas limitadas a Read, Grep e Glob.

O caso de uso principal e investigacao sem poluir o contexto principal. Quando voce precisa entender uma parte do codebase antes de decidir o que fazer, spawnar um pesquisador evita gastar tokens do agente principal com leitura exploratoria.

.claude/agents/researcher.md
---
name: researcher
description: "Investiga o codebase e retorna analise detalhada.
  Use para entender arquitetura, encontrar padroes,
  mapear dependencias."
model: claude-haiku-4-20250414
tools:
  - Read
  - Grep
  - Glob
memory: project
---

# Researcher Agent

Voce e um pesquisador de codebase. Sua funcao e:
1. Explorar arquivos e diretorios relevantes
2. Identificar padroes arquiteturais
3. Mapear dependencias entre modulos
4. Retornar um resumo estruturado dos findings

## Regras
- NUNCA sugira mudancas de codigo
- Foque em FATOS observaveis no codigo
- Use Grep para busca ampla, Read para analise profunda
- Consulte sua memoria antes de comecar para evitar
  trabalho duplicado
- Salve aprendizados importantes na memoria ao finalizar
Dica: Com memory: project, o pesquisador acumula conhecimento ao longo do tempo. Na proxima vez que for invocado, ele pode consultar a memoria para evitar re-explorar areas ja mapeadas.

3. Padrao 2: Planejador (Planner)

O planejador analisa o codebase e propoe planos de acao antes de qualquer execucao. O built-in Plan do Claude Code segue esse padrao: e read-only, herda o modelo do agente principal e nao pode editar arquivos. A separacao entre planejamento e execucao e critica para tarefas complexas.

.claude/agents/planner.md
---
name: planner
description: "Analisa tarefas complexas e gera planos de execucao
  detalhados com etapas, riscos e dependencias."
model: claude-sonnet-4-20250514
tools:
  - Read
  - Grep
  - Glob
  - Bash(git log:*, git diff:*, git show:*, wc:*)
---

# Planner Agent

Voce e um planejador tecnico. Receba uma tarefa e produza
um plano de execucao estruturado.

## Formato de saida
1. **Contexto**: O que existe hoje
2. **Objetivo**: O que precisa mudar
3. **Etapas**: Lista ordenada de acoes especificas
4. **Riscos**: O que pode dar errado
5. **Dependencias**: Ordem de execucao entre etapas
6. **Estimativa**: Complexidade (baixa/media/alta)

## Regras
- Leia o codigo antes de planejar
- Cada etapa deve ser acionavel por um agente executor
- Identifique arquivos que serao criados/modificados
- NAO execute nenhuma mudanca, apenas planeje

Casos de uso ideais para o planejador:

  • Dividir tarefas complexas em etapas atomicas
  • Avaliar impacto de mudancas antes de implementar
  • Gerar planos de migracao (framework, versao, API)
  • Estimar escopo e identificar riscos

4. Padrao 3: Executor (Implementer)

O executor e o subagente com permissao para editar codigo. Diferente do pesquisador e planejador, ele tem acesso a ferramentas de escrita: Write, Edit e Bash. Pode opcionalmente usar isolation: worktree para trabalhar em uma branch separada via git worktree.

.claude/agents/implementer.md
---
name: implementer
description: "Implementa mudancas de codigo seguindo planos
  definidos. Cria, edita e testa arquivos."
model: claude-sonnet-4-20250514
tools:
  - Read
  - Write
  - Edit
  - Bash
  - Grep
  - Glob
isolation: worktree
hooks:
  PostToolUse:
    - matcher: Write|Edit
      command: "npx eslint --fix $CLAUDE_FILE_PATH 2>/dev/null || true"
---

# Implementer Agent

Voce e um executor de codigo. Receba um plano e implemente
cada etapa com precisao.

## Regras
1. Siga o plano fornecido sem desvios
2. Implemente UMA etapa por vez
3. Verifique cada mudanca com testes quando possivel
4. Use git diff para confirmar o que foi alterado
5. Reporte progresso apos cada etapa
isolation: worktree cria um git worktree separado para o subagente. Isso significa que ele trabalha em uma branch isolada, sem afetar o working directory principal. Ideal para implementacoes que podem precisar de rollback.

O campo hooks.PostToolUse no exemplo acima executa ESLint automaticamente apos cada escrita ou edicao de arquivo. Isso garante que o codigo gerado pelo subagente ja sai formatado e sem erros de lint.

5. Padrao 4: Revisor (Code Reviewer)

O revisor e um subagente read-only que analisa codigo recentemente modificado. O padrao classico e usar git diff para identificar o que mudou e produzir um code review estruturado. O campo description pode incluir a diretiva de uso proativo.

.claude/agents/code-reviewer.md
---
name: code-reviewer
description: "Revisa mudancas de codigo em busca de bugs,
  problemas de seguranca e melhorias.
  Use proactively after code changes."
model: claude-sonnet-4-20250514
tools:
  - Read
  - Grep
  - Glob
  - Bash(git diff:*, git log:*, git show:*)
---

# Code Reviewer Agent

Voce e um revisor de codigo experiente. Analise as mudancas
recentes e produza um review detalhado.

## Processo
1. Execute `git diff HEAD~1` para ver as mudancas
2. Leia os arquivos completos para ter contexto
3. Analise cada mudanca

## Categorias de Review
- **Bugs**: Erros logicos, null references, edge cases
- **Seguranca**: Injecao, autenticacao, exposicao de dados
- **Performance**: Complexidade, queries N+1, memory leaks
- **Manutenibilidade**: Nomes, duplicacao, acoplamento
- **Testes**: Cobertura, casos de borda, assertions

## Formato de saida
Para cada finding:
- Severidade: critico / alto / medio / baixo
- Arquivo e linha
- Descricao do problema
- Sugestao de correcao
Uso proativo: A descricao "Use proactively after code changes" faz com que o Claude Code sugira automaticamente usar este subagente apos mudancas de codigo. O orquestrador pode decidir spawnar o revisor sem que o usuario peca explicitamente.

6. Padrao 5: Debugger

O debugger e um subagente focado em root cause analysis. Tem acesso a Read, Edit e Bash para poder executar testes, inspecionar logs e aplicar fixes pontuais.

.claude/agents/debugger.md
---
name: debugger
description: "Diagnostica e corrige bugs. Executa testes,
  analisa stack traces e aplica fixes."
model: claude-sonnet-4-20250514
tools:
  - Read
  - Edit
  - Bash
  - Grep
  - Glob
---

# Debugger Agent

Voce e um especialista em debugging. Seu objetivo e
encontrar a causa raiz de bugs e corrigi-los.

## Metodologia
1. **Reproduzir**: Execute o teste/comando que falha
2. **Isolar**: Identifique o modulo/funcao com o bug
3. **Analisar**: Leia o codigo e trace o fluxo de dados
4. **Diagnosticar**: Determine a causa raiz exata
5. **Corrigir**: Aplique o fix minimo necessario
6. **Verificar**: Execute os testes novamente

## Regras
- Sempre reproduza o bug antes de tentar corrigir
- Prefira fixes cirurgicos a refatoracoes amplas
- Documente a causa raiz no output final
- Execute os testes apos cada correcao

7. Padrao 6: Data Scientist

Subagente especializado em analise de dados com SQL, BigQuery ou ferramentas similares. Usa modelo Sonnet para equilibrar custo e capacidade, e tem acesso a Bash, Read e Write para executar queries e gerar relatorios.

.claude/agents/data-scientist.md
---
name: data-scientist
description: "Analisa dados usando SQL/BigQuery. Executa queries,
  gera relatorios e visualizacoes."
model: claude-sonnet-4-20250514
tools:
  - Bash(bq:*, psql:*, sqlite3:*, python3:*)
  - Read
  - Write
---

# Data Scientist Agent

Voce e um analista de dados. Execute queries SQL,
analise resultados e gere relatorios.

## Regras
- Use queries READ-ONLY (SELECT apenas)
- Sempre adicione LIMIT em queries exploratoras
- Documente cada query com comentarios
- Gere resumos estatisticos quando relevante
- Salve resultados em arquivos CSV/JSON quando solicitado

8. Padrao 7: DB Reader com Hooks de Validacao

Um padrao poderoso e combinar subagentes com hooks PreToolUse para validar operacoes antes da execucao. O exemplo classico e um DB Reader que valida se todas as queries SQL sao read-only antes de permitir execucao.

.claude/agents/db-reader.md
---
name: db-reader
description: "Consulta bancos de dados de forma segura.
  Todas as queries sao validadas como read-only."
model: claude-sonnet-4-20250514
tools:
  - Bash(psql:*, mysql:*, sqlite3:*)
  - Read
hooks:
  PreToolUse:
    - matcher: Bash
      command: ".claude/hooks/validate-readonly-sql.sh"
---

# DB Reader Agent

Voce consulta bancos de dados de producao de forma segura.
Todas as queries DEVEM ser SELECT. Qualquer tentativa de
INSERT, UPDATE, DELETE, DROP, ALTER, TRUNCATE sera bloqueada
automaticamente pelo hook de validacao.
.claude/hooks/validate-readonly-sql.sh
#!/bin/bash
# Le o input do hook via stdin (JSON com tool_input)
INPUT=$(cat)
COMMAND=$(echo "$INPUT" | jq -r '.tool_input.command')

# Verifica se contem comandos destrutivos
if echo "$COMMAND" | grep -iE \
  '(INSERT|UPDATE|DELETE|DROP|ALTER|TRUNCATE|CREATE|GRANT|REVOKE)' \
  > /dev/null 2>&1; then

  # Bloqueia a execucao
  echo '{"decision": "block", "reason": "Query nao e read-only. Apenas SELECT e permitido."}' >&2
  exit 2
fi

# Permite a execucao
exit 0
Seguranca: O hook PreToolUse executa antes de cada chamada de ferramenta. Se o exit code for 2, a operacao e bloqueada e a mensagem de erro e retornada ao agente. Isso cria uma camada de protecao programatica independente do prompt.

9. Arquitetura Multi-Agente: Visao Geral

Arquitetura multi-agente tipica: Orchestrator delega tarefas especializadas para subagentes com diferentes permissoes e modelos

10. Combinando Padroes: Chain de Subagentes

Um padrao poderoso e encadear subagentes em sequencia, onde o output de um alimenta o input do proximo. O orquestrador coordena passando contexto relevante entre eles.

Prompt para o orchestrator
Use code-reviewer para encontrar problemas de performance
no modulo de autenticacao. Depois use implementer para
corrigir os problemas identificados. Por fim, use
code-reviewer novamente para validar as correcoes.

O fluxo de execucao e:

1. code-reviewer analisa o modulo e retorna findings

2. O orchestrator extrai os problemas criticos do review

3. implementer recebe os problemas e aplica correcoes

4. code-reviewer valida que as correcoes resolveram os problemas

5. O orchestrator consolida e reporta o resultado final

Passagem de contexto: O orchestrator e responsavel por extrair a informacao relevante do output de cada subagente e passar como instrucao para o proximo. Subagentes nao se comunicam diretamente entre si (exceto no padrao Agent Teams).

11. Pesquisa em Paralelo

Quando investigacoes sao independentes entre si, voce pode spawnar multiplos subagentes simultaneamente. O Claude Code executa os subagentes em paralelo e coleta os resultados.

Prompt para pesquisa paralela
Pesquise em paralelo usando subagentes separados:
1. Use researcher para analisar o modulo de autenticacao
2. Use researcher para mapear o schema do banco de dados
3. Use researcher para investigar a API de pagamentos

Depois consolide os findings em um relatorio unificado.

Cada instancia do researcher recebe uma tarefa diferente e opera em seu proprio contexto. O orchestrator aguarda todos completarem e entao consolida os resultados. Isso e particularmente util em codebases grandes onde a exploracao sequencial seria muito lenta.

Pesquisa paralela: tres instancias do researcher operam simultaneamente em areas independentes

12. Agent Teams (Experimental)

Agent Teams e um recurso experimental que permite sessoes independentes com comunicacao bidirecional via SendMessage. Diferente de subagentes tradicionais, cada membro do team tem seu proprio contexto completo e persistente.

Ativar Agent Teams
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1
Caracteristica Subagentes Agent Teams
Contexto Herdado/limitado Independente e completo
Comunicacao Unidirecional (orchestrator → sub) Bidirecional (SendMessage)
Persistencia Descartado ao terminar Sessao persistente
Modelo Configuravel por agente Configuravel por agente
Caso de uso Tarefas delegadas pontuais Workstreams independentes prolongados
Quando usar Agent Teams: Quando a tarefa pode ser quebrada em workstreams verdadeiramente independentes que precisam rodar por tempo prolongado. Exemplo: um agente trabalhando no frontend enquanto outro cuida do backend, comunicando-se quando precisam alinhar interfaces.

13. Padrao Coordinator

O padrao coordinator define um agente que nao executa tarefas diretamente, mas coordena quais subagentes sao spawnados. Ele tem acesso a tool Agent() para invocar outros agentes.

.claude/agents/coordinator.md
---
name: coordinator
description: "Coordena tarefas complexas delegando para
  subagentes especializados."
model: claude-sonnet-4-20250514
tools:
  - Agent(implementer, researcher, code-reviewer, debugger)
  - Read
  - Bash(git:*)
---

# Coordinator Agent

Voce coordena tarefas complexas delegando para subagentes
especializados. Seu papel e:

1. Analisar a tarefa recebida
2. Quebrar em subtarefas
3. Delegar cada subtarefa ao subagente mais adequado
4. Coletar e consolidar resultados
5. Reportar progresso e resultado final

## Regras de delegacao
- Investigacao/pesquisa -> researcher
- Implementacao de codigo -> implementer
- Review de mudancas -> code-reviewer
- Diagnostico de bugs -> debugger
- NUNCA implemente codigo diretamente

A declaracao tools: Agent(implementer, researcher, code-reviewer, debugger) limita quais subagentes o coordinator pode invocar. Isso cria uma hierarquia controlada de delegacao.

14. Memoria Persistente para Subagentes

O campo memory na configuracao do subagente permite que ele acumule conhecimento ao longo do tempo. Existem tres escopos de memoria:

Escopo Localizacao Compartilhamento
user ~/.claude/memory/ Todos os projetos do usuario
project .claude/memory/ (no repo) Todos os membros do projeto
local .claude/local/memory/ Apenas o usuario local
Subagente com memoria
---
name: architecture-researcher
description: "Pesquisa e documenta decisoes arquiteturais"
model: claude-haiku-4-20250414
tools:
  - Read
  - Grep
  - Glob
memory: project
---

# Architecture Researcher

Antes de comecar qualquer investigacao:
1. Consulte sua memoria para ver o que ja foi documentado
2. Evite re-explorar areas ja mapeadas
3. Foque em lacunas no conhecimento existente

Ao finalizar:
1. Salve novos aprendizados na memoria
2. Atualize informacoes que mudaram
3. Remova informacoes obsoletas
Dicas para uso efetivo de memoria: Nas instrucoes do subagente, peca explicitamente para ele consultar a memoria antes de comecar e salvar aprendizados apos completar. Isso cria um ciclo de acumulacao de conhecimento que torna o subagente mais eficiente ao longo do tempo.

15. Fluxo Completo: Planejar, Executar, Revisar

Fluxo completo: pesquisa, planejamento, execucao e revisao com loop de retentativa
Exemplo de prompt para fluxo completo
Preciso migrar o sistema de autenticacao de JWT para
sessions server-side. Execute o fluxo completo:

1. Use researcher para mapear todos os pontos onde JWT
   e usado no codebase
2. Use planner para criar um plano de migracao detalhado
3. Use implementer para executar cada etapa do plano
4. Use code-reviewer para validar as mudancas
5. Se o review encontrar problemas, use implementer
   para corrigir e code-reviewer para re-validar

16. Boas Praticas

F Foco especializado

Cada subagente deve ter uma unica especialidade bem definida. Evite criar agentes "faz-tudo" com muitas ferramentas e responsabilidades.

D Descricoes detalhadas

Escreva descricoes ricas no campo description. O orchestrator usa essa informacao para decidir quando e como delegar.

T Ferramentas minimas

Limite o acesso a ferramentas ao minimo necessario. Pesquisadores nao precisam de Write. Revisores nao precisam de Edit. Principio do menor privilegio.

V Versione no git

Arquivos em .claude/agents/ devem ser versionados. Isso permite que toda a equipe compartilhe os mesmos subagentes e rastreie mudancas.

Restricao importante: Subagentes nao podem spawnar outros subagentes. A hierarquia e sempre de um nivel: orchestrator → subagente. Se voce precisa de coordenacao em multiplos niveis, use o padrao coordinator explicitamente no agente principal.

Escolhendo o modelo certo por papel

Papel Modelo recomendado Razao
Researcher Haiku Leitura rapida e barata, sem necessidade de raciocinio complexo
Planner Sonnet / Opus Planejamento requer raciocinio estruturado
Implementer Sonnet Bom equilibrio entre qualidade de codigo e custo
Reviewer Sonnet / Opus Review critico requer atencao a detalhes
Debugger Sonnet Debugging requer raciocinio logico solido
Data Scientist Sonnet SQL e analise de dados com custo controlado

17. Quando NAO Usar Subagentes

Subagentes adicionam latencia e custo. Nem toda tarefa se beneficia de orquestracao multi-agente. Evite subagentes nos seguintes cenarios:

X Ida-e-volta frequente

Se a tarefa precisa de interacao constante entre contextos, subagentes criam overhead. Mantenha no agente principal.

X Contexto compartilhado

Se multiplas fases da tarefa dependem do mesmo contexto acumulado, subagentes perdem informacao entre delegacoes.

X Mudanca rapida e pontual

Para uma edicao simples em um arquivo, spawnar um subagente adiciona latencia desnecessaria. Faca direto.

X Latencia importa

Cada subagente adiciona tempo de inicializacao e processamento. Se velocidade e critica, evite orquestracao.

Alternativa leve: Para perguntas rapidas que ja estao no contexto do agente principal, use /btw em vez de spawnar um subagente. O comando /btw envia uma pergunta lateral sem interromper o fluxo principal.

Vamos montar um setup completo com todos os agentes para um projeto real. A estrutura de diretorio ficaria assim:

# Orquestracao Multi-Agente

## Subagentes disponiveis
- **researcher**: Investigacao de codebase (Haiku, read-only)
- **planner**: Planejamento de tarefas (Sonnet, read-only)
- **implementer**: Implementacao de codigo (Sonnet, worktree)
- **code-reviewer**: Revisao de codigo (Sonnet, proativo)
- **debugger**: Diagnostico de bugs (Sonnet, read+edit)
- **db-reader**: Consultas ao banco (Sonnet, SQL read-only)

## Regras de orquestracao
1. Para tarefas complexas: researcher -> planner -> implementer -> code-reviewer
2. Para bugs: debugger (ou researcher -> debugger se causa desconhecida)
3. Para investigacao: researcher (paralelo se multiplas areas)
4. Sempre use code-reviewer apos implementacoes significativas
5. Use db-reader para qualquer consulta ao banco de producao

20. Resumo

Padroes multi-agente no Claude Code permitem dividir tarefas complexas em responsabilidades especializadas, cada uma com seu proprio escopo de ferramentas, modelo e contexto. Os padroes fundamentais sao:

  • Researcher: explora sem modificar (Haiku, read-only)
  • Planner: planeja antes de executar (read-only)
  • Implementer: executa com isolation opcional (worktree)
  • Reviewer: valida mudancas proativamente
  • Debugger: diagnostica e corrige bugs
  • Coordinator: orquestra delegacao entre subagentes

Combine-os em chains sequenciais ou pesquisa paralela. Use memoria persistente para acumular conhecimento. E lembre-se: nem toda tarefa precisa de subagentes. Avalie o tradeoff entre isolamento de contexto e overhead de comunicacao.
