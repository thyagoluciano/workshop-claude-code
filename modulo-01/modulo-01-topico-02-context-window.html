A Context Window: a constraint central e por que ela importa

A context window é a "memória de trabalho" do modelo — finita, preciosa e com retornos decrescentes. Entender essa limitação é o que separa quem usa Claude Code com eficiência de quem luta contra ele.

O que é a Context Window?

A context window é todo o texto que o modelo pode referenciar ao gerar uma resposta, incluindo a própria resposta. Isso é diferente do corpus de dados em que o modelo foi treinado — a context window é a memória de trabalho (working memory) do modelo para a sessão atual.

Tudo que acontece dentro de uma sessão do Claude Code ocupa espaço nessa janela: o system prompt, as instruções do CLAUDE.md, cada mensagem sua, cada resposta do Claude, cada resultado de ferramenta (conteúdo de arquivos lidos, output de comandos bash, resultados de grep), e o pensamento estendido (extended thinking).

Cada turno acumula tokens. A context window engloba input (histórico) + output (resposta atual).

Tamanhos da Context Window por Modelo

Nem todos os modelos Claude têm a mesma capacidade. Os modelos mais recentes da geração 4.6+ oferecem janelas de até 1 milhão de tokens, enquanto gerações anteriores ficam em 200K.

Modelo Context Window Equivalência aproximada
Opus 4.7 / Opus 4.6 1M tokens ~700K palavras / ~3.500 páginas
Sonnet 4.6 1M tokens ~700K palavras / ~3.500 páginas
Sonnet 4.5 200K tokens ~150K palavras / ~700 páginas
Haiku 4.5 200K tokens ~150K palavras / ~700 páginas
Quanto é 1M de tokens na prática? Um token é aproximadamente ¾ de uma palavra em inglês (ou ~½ palavra em português, que usa palavras mais longas em média). 1M de tokens equivale a ler um codebase inteiro de um projeto médio — mas isso não significa que você deva encher a janela toda.

Context Rot: por que mais tokens ≠ melhor

A descoberta mais contra-intuitiva sobre context windows é que performance degrada conforme o contexto cresce, mesmo em tarefas simples. Esse fenômeno é chamado de context rot e foi documentado em um estudo do Chroma Research que avaliou 18 LLMs, incluindo os modelos Claude.

Representação conceitual baseada no estudo "Context Rot" (Chroma Research, 2025). Performance não é linear — degrada conforme tokens aumentam.

Por que isso acontece?

Modelos de linguagem são baseados na arquitetura Transformer, onde cada token "presta atenção" em todos os outros tokens no contexto. Isso cria n² relações para n tokens — uma explosão quadrática que dilui a atenção à medida que o contexto cresce.

🧠 Attention Budget

O modelo tem um "orçamento de atenção" finito. Cada novo token adicionado consome parte desse orçamento, reduzindo a precisão para os demais.

📉 Curva U

Alta precisão para informações no início e final do contexto, mas até 30% menos precisão para informações no meio ("lost in the middle").

📊 Gradual, não Cliff

A degradação é um gradiente, não um penhasco — mas em certo ponto pode ser imprevisível, variando por modelo e tarefa.

Implicação prática para Claude Code: Não é porque o modelo aceita 1M de tokens que você deve encher a janela. Uma sessão com 50K tokens bem curados pode performar melhor que uma com 300K tokens de contexto ruidoso. Mais contexto pode ser pior que menos contexto.

O que consome tokens numa sessão?

Para gerenciar a context window, é fundamental entender o que consome mais espaço. Veja um breakdown típico de uma sessão de desenvolvimento:

Tool Results (conteúdo de arquivos, output de bash) ~45%
45%
Extended Thinking (raciocínio interno do Claude) ~25%
25%
Assistant Responses (texto que o Claude gera) ~15%
15%
User Messages (seus prompts) ~10%
10%
System Prompt (CLAUDE.md, tools, configuração) ~5%
5%

O maior consumidor são os resultados de ferramentas — cada arquivo lido, cada output de comando entra integralmente na janela. Por isso, leituras seletivas são cruciais.

Context Awareness: o Claude sabe quanto falta

Os modelos Sonnet 4.6, Sonnet 4.5 e Haiku 4.5 possuem context awareness — uma capacidade treinada que permite ao modelo rastrear quanto da context window resta. No início da conversa, o modelo recebe o orçamento total:

Token budget injection (automático)
<!-- Injetado automaticamente no início da sessão -->
<budget:token_budget>1000000</budget:token_budget>

<!-- Após cada tool call, o modelo recebe update -->
<system_warning>Token usage: 35000/1000000; 965000 remaining</system_warning>

A analogia da Anthropic é precisa: sem context awareness, o modelo é como um participante de um reality show de culinária sem relógio. Ele não sabe quanto tempo (contexto) resta e pode não conseguir terminar a tarefa. Com awareness, ele sabe exatamente quando precisa economizar ou quando pode investir mais tokens em raciocínio.

Mecanismos de Defesa contra o Esgotamento

O Claude Code implementa e suporta várias estratégias para lidar com a limitação da context window:

1. Auto-compaction

Quando a conversa se aproxima do limite, o Claude Code automaticamente compacta o histórico — resumindo as mensagens anteriores preservando decisões arquiteturais, bugs não resolvidos, e detalhes de implementação, enquanto descarta outputs de ferramentas redundantes.

Customizando compaction no CLAUDE.md
# No seu CLAUDE.md
When compacting, always preserve:
- The full list of modified files
- Any test commands that were run
- Architecture decisions made during this session
- Current branch name and PR context

2. Comando /clear

O comando /clear reseta completamente a context window, mantendo apenas o system prompt e o CLAUDE.md. Use entre tarefas não relacionadas para começar com uma janela limpa.

Regra prática: Se você vai mudar de assunto ("terminei o bug de auth, agora vou trabalhar no layout da dashboard"), faça /clear antes. O contexto do bug anterior só vai atrapalhar a tarefa nova.

3. Comando /btw

Para perguntas rápidas que não precisam ficar no contexto, use /btw. A resposta aparece em um overlay que é descartado e nunca entra no histórico da conversa. Ideal para checar um detalhe sem gastar tokens.

4. Thinking Blocks são descartados entre turnos

Um detalhe importante da arquitetura: os blocos de extended thinking são gerados durante o turno atual, mas são automaticamente removidos do input dos turnos seguintes. Isso significa que o raciocínio interno (que pode ser muito extenso) não acumula na janela.

Implicações Práticas para o dia a dia

❌ Práticas que desperdiçam contexto

Pedir "leia todos os arquivos do projeto"
Manter uma sessão longa sem /clear
Colar logs enormes sem filtrar
Múltiplas tarefas não relacionadas na mesma sessão
CLAUDE.md com informações que o modelo já sabe

✓ Práticas que preservam contexto

Direcionar para arquivos específicos com @
/clear entre tarefas diferentes
cat error.log | tail -50 | claude
Uma tarefa por sessão, bem definida
CLAUDE.md com apenas o que causaria erros se ausente
Exemplos: prompt eficiente vs. wasteful 
// ❌ Wasteful — vai ler TUDO e encher o contexto
"leia o projeto inteiro e me diga o que posso melhorar"

// ✓ Eficiente — direciona e limita o escopo
"olhe src/auth/middleware.ts e os testes em tests/auth/
 — o token refresh está falhando quando o cookie expira.
 Rode npm test -- --grep 'refresh' pra ver o estado atual."

// ❌ Wasteful — log inteiro de 10K linhas
$ cat server.log | claude "o que deu errado?"

// ✓ Eficiente — filtra antes de enviar
$ cat server.log | grep -i error | tail -30 | claude "diagnostique esses erros"

Resumo

1.

Context window = working memory do modelo. Tudo na sessão (prompts, respostas, ferramentas, thinking) compete pelo mesmo espaço.

2.

Mais contexto não é automaticamente melhor. Context rot faz performance degradar com mais tokens — curadoria supera volume.

3.

Tool results são o maior consumidor. Cada arquivo lido e cada output de bash entra inteiro. Leituras seletivas e filtros são essenciais.

4.

Use os mecanismos de defesa. /clear entre tarefas, /btw para perguntas rápidas, compaction customizado no CLAUDE.md.

5.

Thinking blocks não acumulam. A API remove automaticamente entre turnos — uma otimização arquitetural importante.
