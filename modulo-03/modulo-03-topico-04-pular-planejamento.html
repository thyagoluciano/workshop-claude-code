Quando Pular o Planejamento

Nem toda tarefa merece um plano detalhado. Over-planning desperdiça tokens, tempo e paciência. A heurística da Anthropic é direta: se você consegue descrever o diff em uma frase, pule o plano.

O custo oculto do over-planning

Quando você pede ao Claude Code para planejar antes de agir, ele entra em plan mode: lê arquivos, analisa dependências, formula alternativas e apresenta uma proposta antes de escrever qualquer código. Isso é excelente para tarefas complexas — mas para tarefas simples, o planejamento custa mais do que a própria execução.

$ Tokens desperdiçados

O plano consome tokens de input e output que poderiam ser usados na execução real da tarefa.

~ Latência desnecessária

O ciclo planejar-aprovar-executar adiciona roundtrips que atrasam entregas triviais.

? Falsa complexidade

Planejar uma tarefa simples pode fazer o modelo inventar complexidade que não existe.

Regra de ouro (Anthropic): Se você consegue descrever o diff em uma frase, pule o plano. Plan mode é útil, mas também adiciona overhead. Para tarefas de escopo claro e mudança pequena, a execução direta é mais eficiente.

Decision matrix: quando planejar vs. executar direto

A decisão de planejar ou não depende de dois eixos: a complexidade da tarefa (quantos arquivos, quanta lógica nova) e o seu nível de certeza sobre o que precisa ser feito. Quanto mais simples e mais certo, menos planejamento é necessário.

Quanto mais simples e mais certa a tarefa, menos planejamento e necessario. Adapte a abordagem ao contexto.

O espectro: do prompt direto ao plano multi-etapas

Planejamento nao e binario — existe um espectro de niveis entre "executa agora" e "planeja tudo antes". A documentação da Anthropic sugere calibrar o nivel de planejamento de acordo com a complexidade da tarefa. Desenvolva intuição para escolher o nivel certo.

1 Prompt direto Uma frase. Tarefa trivial. O modelo executa sem pensar duas vezes.
2 Prompt + contexto Objetivo + arquivos relevantes + constraints. Clareza sem overhead de plano.
3 Plan mode Pede um plano antes de executar. Ideal para mudancas multi-arquivo.
4 Multi-step plan Plano detalhado, iteracao por etapas, possivel uso de subagentes.

Tarefas que NAO precisam de plano

Para tarefas onde o escopo e claro e a mudanca e pequena, a execucao direta e mais eficiente. Planejar essas tarefas adiciona overhead sem beneficio proporcional. Aqui estão os padroes mais comuns:

A Correcoes pontuais

Corrigir typo, fix lint error, atualizar mensagem de erro, ajustar indentacao. Uma linha, um arquivo, zero ambiguidade.

B Renomear e mover

Renomear variavel, funcao ou arquivo. Mover componente de pasta. O modelo sabe fazer find-and-replace global.

C Adicionar campo simples

Novo campo em um model/schema, novo prop em componente, nova coluna em tabela. Padrão ja existe, so replicar.

D Boilerplate e versoes

Gerar boilerplate (test file, component scaffold), atualizar versao no package.json, adicionar dependency.

Prompts diretos para tarefas simples

Exemplos de prompts diretos — sem plano, sem ceremony Claude Code 
# Correcao pontual
Fix the typo in src/utils/validators.ts line 42 — "recieve" should be "receive"

# Renomear
Rename the function getUserData to fetchUserProfile in src/api/users.ts and update all call sites

# Adicionar campo
Add a "phone" field (optional string) to the User model in src/models/user.ts

# Atualizar versao
Bump the version in package.json to 2.4.1

# Gerar boilerplate
Create a test file for src/services/payment.ts following the pattern in src/services/__tests__/order.test.ts

# Adicionar log
Add a debug log at the start of processOrder in src/services/order.ts showing the orderId
Perceba o padrao: Cada prompt descreve o diff em uma frase. Nao ha ambiguidade sobre o que fazer, onde fazer e como verificar. Esse e o sinal de que voce pode pular o planejamento.

Tarefas que PRECISAM de plano

Quando a tarefa envolve multiplos arquivos, decisões de design ou territorio desconhecido, o planejamento se paga. O plan mode permite que o Claude Code analise o codebase, avalie alternativas e proponha uma abordagem antes de tocar em qualquer arquivo.

1 Refatoracao multi-arquivo

Extrair modulo, mudar arquitetura de pasta, split de componente grande. Afeta imports, testes e dependencias cruzadas.

2 Migracao de schema/ORM

Mudar estrutura do banco, migrar de ORM, alterar relacoes entre entidades. Efeitos em cascata por todo o backend.

3 Nova feature complexa

Feature que envolve frontend + backend + banco + testes. Multiplas decisões de design sobre onde colocar cada parte.

4 Integracao com API externa

API desconhecida, autenticacao complexa, tratamento de erros e retries. Precisa investigar antes de implementar.

Prompts com planejamento para tarefas complexas

Exemplos de prompts que pedem planejamento Claude Code 
# Refatoracao — pedir plano antes
I need to extract the payment logic from src/services/order.ts into its own
module. Before making changes, analyze the current dependencies and create
a plan for the extraction. Consider what tests need to be updated.

# Nova feature complexa — ultrathink para raciocinio profundo
Think hard about how to implement a real-time notification system.
We use Express + PostgreSQL + Redis. I need: database schema for
notifications, a service layer, WebSocket integration, and REST
endpoints. Give me a plan before implementing anything.

# Integracao com API — explorar antes de agir
I need to integrate with the Stripe Connect API for marketplace payouts.
First, read our existing Stripe integration in src/payments/ and plan
how to add Connect support without breaking the current flow.

# Migracao — plano detalhado com etapas
Plan the migration from Sequelize to Prisma for our user-related models.
List all files that will be affected, propose the migration order, and
identify potential breaking changes. Don't start implementing yet.

Calibrando o esforco: thinking budget e modelos

Além de decidir entre planejar ou executar direto, voce pode calibrar quanto esforco computacional o Claude dedica a cada tarefa. A Anthropic oferece dois mecanismos para isso: palavras-chave de thinking e niveis de esforco.

Palavras-chave de thinking budget

No Claude Code, frases específicas mapeiam diretamente para niveis crescentes de budget de raciocínio. Use-as para calibrar a profundidade de analise:

Keyword Budget Quando usar Exemplo de tarefa
think Baixo Tarefas rotineiras que precisam de algum raciocinio Adicionar tratamento de erro em endpoint
think hard Medio Problemas com multiplas abordagens possiveis Otimizar query N+1 em listagem
think harder Alto Decisões de arquitetura, trade-offs complexos Redesenhar sistema de cache
ultrathink Maximo Problemas criticos que exigem analise exaustiva Investigar race condition em prod
Dica pratica: Se o modelo esta "overthinking" em tarefas simples, reduza o nivel de esforco do padrao (high) para medium. Isso diminui o budget de raciocinio e acelera a resposta. Para tarefas triviais, um prompt direto sem keywords de thinking ja e suficiente.

YAGNI aplicado ao planejamento

O principio YAGNI (You Ain't Gonna Need It) se aplica perfeitamente ao planejamento com IA. Da mesma forma que voce nao escreve código para features que ninguém pediu, voce nao deveria planejar o que nao precisa ser planejado. Over-planning e o equivalente agêntico de over-engineering.

X Over-planning (anti-pattern)

Pedir plano detalhado para renomear uma variavel. O modelo vai analisar dependencias, propor alternativas, considerar impacto em testes... tudo para um rename.

OK Right-sizing (best practice)

Prompt direto para o rename, plano detalhado para a refatoracao de arquitetura. Cada tarefa recebe o nivel de planejamento proporcional a sua complexidade.

Anti-pattern vs. Best practice — mesma tarefa, abordagens diferentes Comparacao 
# Anti-pattern: over-planning para tarefa trivial
"Before making any changes, analyze the entire codebase and create a
detailed plan for renaming the variable `usr` to `user` in the auth module.
Consider all possible side effects and propose a rollback strategy."
  → Vai gastar 30s+ planejando um rename que levaria 2s

# Best practice: prompt direto para tarefa trivial
"Rename `usr` to `user` in src/auth/ and update all references"
  → Executa imediatamente, atualiza todos os call sites, pronto

# Anti-pattern: executar direto tarefa complexa
"Migrate our entire database layer from MongoDB to PostgreSQL"
  → Vai comecar a mudar arquivos sem entender dependencias, vai quebrar tudo

# Best practice: planejar antes de tarefa complexa
"Think hard about migrating from MongoDB to PostgreSQL. Read the current
data layer in src/db/ and src/models/. Create a phased migration plan
before making any changes."
  → Analisa, planeja, executa por etapas com verificacao

Desenvolvendo intuicao

A Anthropic reconhece que as recomendacoes de best practices sao pontos de partida, nao regras rigidas. Com o tempo, voce desenvolve intuicao sobre quando planejar e quando executar direto. Essa intuição vem da pratica — e de prestar atencao nos padroes que funcionam para o seu tipo de codebase e fluxo de trabalho.

1 Comece direto

Na duvida, tente o prompt direto primeiro. Se o resultado nao for bom, adicione contexto ou peca um plano no proximo turno.

2 Observe os padroes

Note quais tarefas o Claude Code resolve bem de primeira e quais precisam de iteracao. Ajuste seu threshold de planejamento.

3 Adapte ao contexto

Tarefas exploratorias pedem menos plano. Tarefas em código critico pedem mais. Deixe o contexto guiar, nao uma regra fixa.

A meta final: Com pratica, voce sabe instintivamente quando ser especifico e quando ser aberto, quando planejar e quando explorar, quando limpar contexto e quando deixar acumular. Nenhum guia substitui essa intuicao — ele so acelera o caminho ate la.
