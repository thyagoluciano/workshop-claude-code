Critérios de Verificação: Testes, Screenshots, Outputs Esperados

O Claude Code funciona em um loop agêntico de agir → verificar → corrigir. Sem critérios claros de verificação, esse loop não tem como saber se o trabalho está feito. Definir "o que significa resolvido" é a ação de maior impacto que você pode tomar.

Por que critérios de verificação são essenciais

A documentação oficial de best practices da Anthropic é enfática: incluir testes, screenshots ou outputs esperados para que o Claude possa se auto-verificar é a ação de maior alavancagem que você pode fazer ao usar o Claude Code. O modelo performa dramaticamente melhor quando consegue verificar seu próprio trabalho.

Isso acontece porque o Claude Code opera em um loop de auto-correção: ele implementa, verifica, e se algo falhou, analisa o erro e tenta novamente. Sem critérios de verificação claros, esse loop não funciona — o agente não tem como distinguir "feito" de "feito errado". Agentes que conseguem checar e melhorar seu próprio output são fundamentalmente mais confiáveis: eles pegam erros antes que se acumulem, se auto-corrigem quando desviam e melhoram a cada iteração.

Regra de ouro: Se você não consegue definir como verificar que a tarefa está feita, provavelmente o Claude Code também não vai conseguir. Antes de escrever o prompt, pergunte-se: "como eu saberia se isso está funcionando?" — e inclua essa resposta no prompt.

Tipos de verificação e quando usar cada um

Existem diferentes formas de verificar o trabalho do Claude Code, cada uma adequada a um tipo de tarefa. Na prática, os melhores prompts combinam mais de um tipo de verificação para cobrir diferentes aspectos da mudança.

Diferentes tipos de verificação alimentam o loop de auto-correção do Claude Code. Combine-os para cobertura máxima.

Guia de referência: tipo de verificação × quando usar × como instruir

Tipo Quando usar Como instruir no prompt
Testes unitários Lógica de negócio, funções puras, utils, parsers, validações Rode npm test e garanta que todos passam
Testes de integração Fluxos end-to-end, interação entre módulos, API routes Rode npm run test:integration -- --grep "auth"
Build / Compile Qualquer mudança em código tipado (TS, Rust, Go, Java) Verifique que tsc --noEmit compila sem erros
Linter Padrões de código, formatação, regras do time Rode eslint src/ e corrija qualquer warning
Type check Mudanças em interfaces, tipos, contratos entre módulos Rode mypy src/ e garanta zero erros de tipo
Output esperado APIs, CLIs, scripts, transformações de dados O endpoint GET /users deve retornar status 200
Screenshot / Visual UI, componentes visuais, layouts, CSS Tire um screenshot e compare com o mock
Script customizado Verificações específicas do projeto Rode ./scripts/validate.sh e garanta exit 0

Como instruir o Claude Code a verificar

A chave é ser explícito e concreto. Não basta dizer "teste isso" — diga exatamente qual comando rodar, o que constitui sucesso e o que fazer se falhar. O Claude Code funciona melhor com a melhor forma de feedback: regras claramente definidas para um output, e depois explicação de quais regras falharam e por quê.

Prompt sem critério de verificação (fraco)

Prompt sem verificação Evite 
// O agente não sabe quando parar nem como validar
Adicione um endpoint de health check na API.

Prompt com critérios claros de verificação (forte)

Prompt com verificação explícita Recomendado 
Adicione um endpoint GET /health na API Express.

## Critérios de verificação:
1. Rode `npm test` e garanta que todos os testes passam
2. Rode `curl http://localhost:3000/health` e verifique que retorna:
   { "status": "ok", "timestamp": "<ISO date>" }
3. Verifique que `tsc --noEmit` compila sem erros
4. O endpoint deve responder em menos de 50ms
Dica prática: Cada critério de verificação no prompt se torna um "checkpoint" no loop agêntico. O Claude Code vai executar cada verificação em sequência e, se alguma falhar, vai analisar o erro e voltar para corrigir. Quanto mais concretos os critérios, mais eficiente o loop.

Definição de "resolvido": seja explícito sobre quando a tarefa está feita

Uma das melhores práticas é definir explicitamente o que significa "feito" para cada tarefa. Sem isso, o agente pode parar cedo demais (entregando código que compila mas não funciona) ou continuar iterando além do necessário. A definição de "done" funciona como critério de parada do loop agêntico.

X Definição vaga

"Implemente autenticação JWT."

Quando o agente sabe que terminou? Quando compila? Quando tem testes? Quando o refresh token funciona?

OK Definição explícita

"Implemente autenticação JWT. A tarefa está feita quando:"

1) npm test passa
2) POST /login retorna token
3) Rotas protegidas retornam 401 sem token
4) tsc compila sem erros

Regra prática: Se o seu critério de "feito" não pode ser verificado por um comando no terminal, ele provavelmente é subjetivo demais para o loop agêntico. Transforme-o em algo executável: em vez de "o código deve ser limpo", diga "eslint deve retornar zero warnings".

Test-Driven Development (TDD) com Claude Code

A abordagem de TDD se torna ainda mais poderosa com coding agêntico. O fluxo recomendado pela documentação oficial da Anthropic transforma testes em critérios de verificação nativos do loop:

Prompt TDD completo Fluxo recomendado 
Estamos fazendo TDD. Siga este fluxo rigorosamente:

## Passo 1: Escreva os testes primeiro
Crie testes para um serviço de cálculo de frete:
- Frete grátis para pedidos acima de R$ 200
- Frete fixo R$ 15 para a mesma cidade
- Frete R$ 30 para outras cidades do mesmo estado
- Frete R$ 50 para outros estados

## Passo 2: Confirme que os testes falham
Rode `npm test -- --grep "frete"` e confirme que todos falham
(NÃO crie implementações mock)

## Passo 3: Implemente até todos passarem
Rode os testes após cada mudança. Continue iterando até:
- Todos os testes de frete passam
- `tsc --noEmit` compila sem erros
- `eslint src/services/shipping*` retorna zero warnings

Note como cada passo tem uma verificação concreta. O Claude Code vai iterar naturalmente: escrever testes, confirmar que falham, implementar, rodar testes, ver o que ainda falha, ajustar, e repetir -- geralmente levando algumas iterações até que tudo passe.

Screenshots e verificação visual

Para tarefas que envolvem UI, CSS ou componentes visuais, testes automatizados nem sempre capturam o resultado esperado. A Anthropic recomenda usar verificação visual com screenshots para esses casos. Times internos da Anthropic usam frequentemente screenshots para mostrar ao Claude Code como querem que interfaces fiquem, e iteram com base em feedback visual em vez de descrever funcionalidades em texto.

Fluxo de Verificação Visual
1
Forneça um mock visual

Copie/cole ou arraste uma imagem de referência, ou passe o caminho do arquivo de design.

2
Peça a implementação

O Claude Code cria o componente com base no mock.

3
Tire um screenshot do resultado

Use Puppeteer MCP, Claude in Chrome ou tire manualmente e cole no chat.

4
Itere até match

O Claude compara visualmente e ajusta até o resultado corresponder ao mock.

Prompt com verificação visual UI/CSS 
Implemente o componente de card de produto seguindo o mock em
./designs/product-card.png

## Verificação:
1. Use o Puppeteer MCP para abrir http://localhost:3000/products
2. Tire um screenshot da página
3. Compare com o mock — espaçamentos, cores e tipografia
   devem ser visualmente equivalentes
4. Itere até que o resultado esteja fiel ao design
5. Garanta que `npm test` ainda passa

Verificação por output esperado

Para APIs, CLIs e scripts, a forma mais direta de verificação é definir exatamente o output esperado. Isso dá ao Claude Code um alvo concreto contra o qual comparar o resultado real.

Prompt com output esperado detalhado API endpoint 
Crie um endpoint GET /api/v1/status que retorne informações do sistema.

## Output esperado (curl localhost:3000/api/v1/status):
{
  "status": "healthy",
  "version": "<valor do package.json>",
  "uptime": <number em segundos>,
  "database": {
    "connected": true,
    "latency_ms": <number>
  }
}

## Verificação:
1. Inicie o servidor com `npm run dev`
2. Rode `curl -s localhost:3000/api/v1/status | jq .`
3. Verifique que o response tem status 200
4. Verifique que o JSON contém todos os campos acima
5. Rode `npm test` para garantir que nada quebrou
Prompt com output esperado para CLI Script / CLI 
Crie um script ./scripts/db-migrate.sh que rode as migrações pendentes.

## Output esperado em caso de sucesso:
Running pending migrations...
  ✓ 001_create_users.sql (23ms)
  ✓ 002_add_email_index.sql (12ms)
All migrations completed successfully.

## Output esperado em caso de erro:
Running pending migrations...
  ✓ 001_create_users.sql (23ms)
  ✗ 002_add_email_index.sql — ERROR: column already exists
Migration failed. Run with --rollback to undo.
Exit code: 1

## Verificação:
Rode o script e confirme que o output segue o formato acima.
Teste ambos os cenários (sucesso e erro).

O loop de feedback: verificação falhou, e agora?

O verdadeiro poder dos critérios de verificação aparece quando algo falha. Em vez de parar e pedir ajuda, o Claude Code entra em um ciclo de auto-correção: analisa o erro, entende a causa, implementa a correção e verifica novamente. A melhor forma de feedback para o agente são regras claramente definidas, seguidas de uma explicação de quais regras falharam e por quê.

Quando a verificação falha, o agente analisa o erro, corrige e tenta novamente automaticamente.

Atenção: Se você não fornece critérios de verificação, o Claude Code não tem como entrar nesse loop de auto-correção. Ele vai implementar, "achar" que está pronto e parar. O resultado pode compilar mas ter bugs sutis que só seriam pegos por testes.

Preview: hooks como verificação automática

No Módulo 7 veremos em detalhes como usar hooks do Claude Code para automatizar verificações que hoje você precisa pedir manualmente no prompt. Hooks são comandos shell que executam automaticamente em pontos específicos do ciclo de vida do Claude Code.

Por exemplo, em vez de repetir "rode eslint após cada edição" em todo prompt, você pode configurar um hook PostToolUse que roda o linter automaticamente toda vez que o Claude Code edita um arquivo. Isso garante verificação contínua sem poluir o prompt.

.claude/settings.json — preview de hook de verificação Módulo 7 
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "command": "npx eslint --fix $CLAUDE_FILE_PATH"
      }
    ],
    "Stop": [
      {
        "command": "npm test --silent"
      }
    ]
  }
}
Preview (Módulo 7): O hook Stop roda automaticamente quando o Claude Code acha que terminou. Se os testes falharem, o agente recebe o feedback e volta a corrigir — sem intervenção manual. Isso transforma critérios de verificação em guardrails automáticos.

Exemplos de prompts com critérios de verificação claros

Exemplo 1: Refatoração com segurança Backend 
Extraia a lógica de cálculo de desconto de OrderService para
um novo serviço DiscountService.

## Definição de feito:
1. Rode `npm test` — todos os testes existentes devem passar
   (a refatoração não deve mudar comportamento)
2. Rode `tsc --noEmit` — zero erros de tipo
3. OrderService não deve mais ter nenhum cálculo de desconto
4. DiscountService deve ter testes próprios cobrindo:
   - Desconto por volume (>10 itens: 5%)
   - Desconto por cupom (código válido e inválido)
   - Sem desconto (caso padrão)
Exemplo 2: Nova feature com múltiplas verificações Full-stack 
Adicione paginação ao endpoint GET /api/products.

## Requisitos:
- Query params: ?page=1&limit=20 (defaults)
- Response inclui metadata: total, page, totalPages, hasNext

## Verificação (todos devem passar):
1. `npm test` — testes existentes + novos testes de paginação
2. `curl "localhost:3000/api/products?page=1&limit=2"` retorna
   exatamente 2 items + metadata de paginação
3. `curl "localhost:3000/api/products?page=999"` retorna
   array vazio com hasNext: false
4. `curl "localhost:3000/api/products"` (sem params) usa defaults
5. `tsc --noEmit` compila sem erros
Exemplo 3: Bug fix com teste de regressão Debug 
O cálculo de impostos está errado para produtos com isenção.
Bug reportado: produtos com taxExempt: true ainda incluem ICMS.

## Verificação:
1. Primeiro, escreva um teste que reproduz o bug
   (produto taxExempt com ICMS zero)
2. Confirme que o teste FALHA antes da correção
3. Implemente a correção
4. Confirme que o teste agora PASSA
5. Rode `npm test` — todos os testes devem passar
   (a correção não deve quebrar cálculos normais)

Checklist: verificação eficaz no prompt

1 Comando executável

Inclua o comando exato que o Claude deve rodar: npm test, tsc --noEmit, curl ...

2 Output esperado

Defina o que o comando deve retornar: status code, JSON shape, "zero warnings", exit code 0.

3 Critério de parada

Seja explícito: "A tarefa está feita quando todos os 5 critérios acima passam."

4 Combine verificações

Use testes + build + lint juntos. Cada camada pega um tipo diferente de erro.
