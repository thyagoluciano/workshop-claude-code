Como Forçar o Claude a Ler o Código Antes de Escrever

O erro mais comum ao usar o Claude Code é deixá-lo escrever código sem antes entender o contexto existente. Aprenda técnicas de prompt que forçam o agente a explorar, ler e analisar o codebase antes de tocar em qualquer arquivo.

O problema: código gerado no vácuo

Quando você pede ao Claude Code para implementar algo sem direcionar a leitura do codebase, ele pode gerar código tecnicamente correto mas contextualmente errado: usa convenções diferentes das do projeto, cria duplicações, ignora utilitários existentes ou escolhe um padrão arquitetural incompatível.

A documentação oficial de best practices da Anthropic é direta sobre isso: o Claude Code deve investigar e ler arquivos relevantes ANTES de responder perguntas sobre o codebase. Quando o agente não faz isso, o resultado é código que funciona isolado mas não se integra ao projeto.

Anti-pattern clássico: Copiar e colar código no chat e pedir "melhore isso". Em vez disso, aponte para o arquivo e deixe o agente ler do file system. Ele terá acesso ao contexto completo -- imports, tipos, convenções do projeto -- ao invés de um trecho isolado.

✗ O que acontece sem leitura

Claude cria um controller com Express puro, enquanto o projeto inteiro usa NestJS com decorators. O código funciona, mas quebra todas as convenções.

✓ O que acontece com leitura

Claude lê um controller existente, identifica NestJS + class-validator + Swagger decorators, e gera o novo controller seguindo exatamente o mesmo padrão.

O fluxo correto: Ler → Analisar → Implementar

O Claude Code tem acesso a ferramentas de exploração -- Read, Grep, Glob e Bash -- que permitem navegar o codebase como um desenvolvedor faria. A Anthropic descreve essa abordagem como recuperação just-in-time: em vez de pré-processar tudo, o agente mantém referências leves (caminhos de arquivos, padrões de busca) e carrega dados dinamicamente no contexto conforme necessário.

O agente usa Read, Grep e Glob para entender o codebase antes de escrever uma única linha de código.

Técnicas de prompt para forçar leitura

A documentação oficial da Anthropic recomenda explicitamente pedir ao Claude que leia arquivos relevantes e não escreva código ainda como primeiro passo. Abaixo estão as técnicas mais eficazes, com exemplos concretos.

1. Peça leitura explícita antes de implementar

A forma mais direta: instrua o agente a ler primeiro e só depois agir. Isso ativa as ferramentas de exploração (Read, Grep, Glob) antes de qualquer Edit ou Write.

Prompt ruim Sem direcionamento 
Crie um endpoint POST /api/users
que valide o body e salve no banco.
Prompt bom Força leitura 
Antes de implementar, leia os arquivos
em src/controllers/ e src/schemas/ para
entender as convenções do projeto.

Depois, crie um endpoint POST /api/users
que valide o body e salve no banco,
seguindo os mesmos padrões.

2. Referencie arquivos existentes como padrão

Em vez de descrever o padrão que você quer, aponte para um arquivo que já o implementa. Isso funciona como uma especificação viva -- o agente lê o código real e o replica.

Prompt ruim Descreve o padrão 
Crie um novo service para Payments.
Use injeção de dependência, DTOs com
class-validator, e retorne Result types
com Either pattern.
Prompt bom Aponta para o padrão 
Crie um PaymentService seguindo
exatamente o mesmo padrão de
src/services/OrderService.ts

Leia esse arquivo primeiro para
entender a estrutura, DTOs, error
handling e testes.
Por que isso funciona melhor: Quando você descreve o padrão em texto, o Claude interpreta suas palavras e pode divergir nos detalhes. Quando você aponta para um arquivo, ele lê o código real e reproduz a mesma estrutura. O código existente é a melhor documentação.

3. Use @-mentions para apontar arquivos específicos

O Claude Code suporta referências com @ para incluir arquivos no contexto. A documentação oficial recomenda: referencie arquivos com @ em vez de descrever onde o código está. O agente lê o arquivo antes de responder.

Usando @-mentions no prompt Claude Code CLI 
# O @ faz o Claude ler automaticamente o arquivo antes de agir

Adicione um campo "phone" no formulário de cadastro.
Veja como os outros campos são feitos em @src/components/SignupForm.tsx
e o schema de validação em @src/schemas/signup.schema.ts

# Funciona também com diretórios

Analise a estrutura de @src/modules/auth/ e crie
um novo módulo de payments seguindo o mesmo padrão.
Dica: O @ é mais eficiente que pedir "leia o arquivo X" porque o conteúdo é incluído diretamente no contexto antes do modelo processar seu prompt. O agente já começa a raciocinar com o código disponível.

4. Peça um plano antes da implementação

Pedir um plano força o agente a fazer a fase de exploração e análise explicitamente. A Anthropic recomenda pedir ao Claude que faça um plano de abordagem antes de implementar, especialmente para tarefas complexas.

Prompt ruim Vai direto 
Refatore o módulo de autenticação
para usar JWT ao invés de sessions.
Prompt bom Plano primeiro 
Leia o codebase primeiro e me proponha
um plano antes de implementar.

Quero migrar autenticação de sessions
para JWT. Me mostre:
- Quais arquivos serão afetados
- Qual a ordem das mudanças
- Riscos de breaking changes

Não implemente nada ainda.

5. Use subagentes para exploração pesada

Para investigações que envolvem muitos arquivos, a Anthropic recomenda delegar a pesquisa a subagentes. Eles exploram em um contexto separado, mantendo a conversa principal limpa para a implementação.

Delegando exploração a subagentes Claude Code CLI 
Use subagentes para investigar:

1. Qual ORM o projeto usa e como as queries são feitas
2. Qual o padrão de error handling nos services
3. Como os testes de integração estão estruturados

Depois me apresente um resumo e, com base nisso,
implemente o CRUD de Products.
Por que subagentes: Quando o Claude explora dezenas de arquivos em busca de padrões, o contexto principal enche rápido. Subagentes fazem a exploração em contexto isolado e devolvem apenas o resumo, preservando espaço para a implementação real.

Anti-patterns: o que NÃO fazer

Esses são os erros mais comuns que levam o Claude a escrever código sem contexto suficiente.

Anti-pattern Por que é ruim O que fazer em vez
Colar código no prompt Perde imports, tipos e contexto do projeto inteiro Aponte para o arquivo com @ ou peça para ler
Descrever padrões em texto Descrição ambígua leva a interpretações diferentes Aponte para um arquivo que implementa o padrão
Pedir implementação sem escopo Claude pode explorar centenas de arquivos enchendo o contexto Delimite quais pastas/arquivos importam
"Investigue tudo sobre X" Exploração infinita que consome a context window Seja específico: "Leia os arquivos em src/auth/"
Assumir que o Claude conhece o projeto Cada conversa começa do zero, sem memória Sempre forneça pontos de entrada e referências

Grid comparativo: prompts ruins vs. prompts bons

Cada cenário abaixo mostra o contraste entre um prompt que gera código no vácuo e um que guia o agente a ler o contexto antes de agir.

✗ Prompts que geram código no vácuo

"Crie um middleware de logging"
"Adicione paginação na API de produtos"
"Implemente autenticação OAuth"
"Crie testes para o UserService"
"Refatore esse código: [300 linhas coladas]"
vs

✓ Prompts que forçam leitura primeiro

"Leia os middlewares em src/middleware/ e crie um de logging no mesmo padrão"
"Veja como a paginação funciona em GET /api/orders e aplique o mesmo padrão em /api/products"
"Analise @src/auth/ e adicione OAuth seguindo as convenções de error handling e config"
"Leia os testes em __tests__/services/ para entender o padrão e crie testes para UserService"
"Leia @src/services/UserService.ts e proponha refatorações. Não mude nada ainda."

Exemplo completo: o que acontece por baixo dos panos

Quando você dá um prompt bem estruturado que direciona a leitura, esta é a sequência de tool calls que o Claude Code executa internamente:

Prompt do usuário Input 
Crie um endpoint DELETE /api/products/:id seguindo
o mesmo padrão de @src/controllers/OrderController.ts
Leia os testes existentes em @src/__tests__/controllers/
e crie testes no mesmo formato.
Sequência de tool calls do agente ~12 tool calls 
// Fase 1: EXPLORAR — o agente lê tudo que foi referenciado
Read { file: "src/controllers/OrderController.ts" }
  → identifica: NestJS, decorators, DTO pattern, guard de auth

Glob { pattern: "src/__tests__/controllers/*.spec.ts" }
  → encontra: order.spec.ts, user.spec.ts, health.spec.ts

Read { file: "src/__tests__/controllers/order.spec.ts" }
  → identifica: supertest, factory pattern, cleanup hooks

// Fase 2: ANALISAR — busca dependências relacionadas
Read { file: "src/services/OrderService.ts" }
  → identifica: soft delete pattern, Result type

Grep { pattern: "ProductService|product.service", glob: "**/*.ts" }
  → encontra: ProductService já existe em src/services/

Read { file: "src/services/ProductService.ts" }
  → confirma: já tem findById(), falta delete()

// Fase 3: IMPLEMENTAR — com contexto completo
Edit { file: "src/controllers/ProductController.ts",
       // adiciona o endpoint DELETE seguindo o padrão do OrderController }

Edit { file: "src/services/ProductService.ts",
       // adiciona método delete() com soft delete pattern }

Write { file: "src/__tests__/controllers/product-delete.spec.ts",
       // testes no padrão de order.spec.ts }

// Fase 4: VERIFICAR
Bash { command: "npm test -- --grep product" }
  → ✓ 4 testes passaram
Observe o padrão: Das 12 tool calls, 6 foram de leitura (Read, Grep, Glob) e apenas 3 de escrita (Edit, Write). O agente gastou mais da metade do esforço entendendo o contexto. Esse é o comportamento ideal.

Automatizando com CLAUDE.md

Para não depender de prompts individuais, você pode configurar o comportamento de leitura diretamente no CLAUDE.md do projeto. O Claude Code carrega esse arquivo automaticamente em toda conversa.

CLAUDE.md Seção de regras de comportamento 
## Regras de implementação

- SEMPRE leia os arquivos existentes na pasta relevante ANTES de criar ou editar
- Use os controllers em src/controllers/ como referência para novos endpoints
- Siga o padrão de testes em src/__tests__/ — use factories, supertest, cleanup
- Leia src/shared/types.ts antes de criar novos tipos
- Nunca crie utilitários novos sem antes verificar se já existem em src/utils/

## Padrões do projeto

- Framework: NestJS com TypeORM
- Validação: class-validator com DTOs
- Error handling: Result<T, E> com Either pattern
- Testes: Jest + Supertest para integração, vitest para unit
Dica avançada: Colocar essas instruções no CLAUDE.md elimina a necessidade de repetir "leia antes de escrever" em cada prompt. O agente já começa toda conversa sabendo que deve explorar o codebase antes de agir.

Resumo: os princípios-chave

1 Aponte, não descreva

Referencie arquivos existentes como padrão em vez de descrever convenções em texto.

2 Peça plano antes de código

"Não implemente ainda" força o agente a explorar e analisar antes de escrever.

3 Automatize no CLAUDE.md

Regras de "leia antes de escrever" no CLAUDE.md valem para toda conversa.
