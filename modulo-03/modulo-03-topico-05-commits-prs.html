Commits Descritivos e Criacao de PRs com o Claude

O Claude Code tem acesso nativo ao Git via Bash tool e pode executar o fluxo completo: criar branch, implementar, testar, commitar com mensagens descritivas e abrir Pull Requests — tudo numa unica sessao, sem voce sair do terminal.

Acesso nativo ao Git

O Claude Code nao precisa de plugins especiais para interagir com o Git. Atraves da ferramenta Bash, ele executa qualquer comando Git diretamente — git status, git diff, git log, git commit, git push. Muitos engenheiros da Anthropic usam o Claude para mais de 90% de suas interacoes com Git.

Alem disso, o Claude entende atalhos naturais: quando voce diz "commita isso" ou "cria uma PR", ele sabe exatamente quais comandos executar. Ele analisa o diff das mudancas, o historico recente de commits e o contexto do que foi feito na sessao para compor mensagens descritivas automaticamente.

Dica: Se voce usa GitHub, instale o gh CLI. Com ele disponivel, o Claude pode criar issues, abrir PRs, ler comentarios de review e ate responder a code reviews — tudo direto do terminal.

Configurando convencoes de commit no CLAUDE.md

Para que o Claude gere commits no padrao do seu projeto, a melhor estrategia e documentar a convencao no arquivo CLAUDE.md. O Claude le esse arquivo no inicio de cada sessao e aplica as regras automaticamente em todos os commits que fizer.

CLAUDE.md — secao de convencoes Git Markdown 
# Convencoes de Git

## Commits
- Use Conventional Commits: tipo(escopo): descricao
- Tipos permitidos: feat, fix, refactor, docs, test, chore, ci, perf
- Mensagem do commit em portugues
- Corpo do commit deve explicar o "porque", nao o "o que"
- Referencia issues quando aplicavel: "Closes #123"
- Commits atomicos: uma mudanca logica por commit

## Branches
- Padrao: tipo/descricao-curta (ex: feat/validacao-email)
- Sempre crie branch a partir da main atualizada

## PRs
- Titulo segue o padrao de Conventional Commits
- Descricao deve incluir: contexto, mudancas feitas, como testar
- Sempre vincule a issue relacionada
Por que isso funciona tao bem: O Claude le o CLAUDE.md como parte do seu contexto inicial. Quando voce pede "commita isso", ele ja sabe que deve usar Conventional Commits, escrever em portugues e referenciar issues. Nenhuma instrucao adicional necessaria.

Prompts para commits descritivos

O Claude entende comandos naturais para gerar commits. Ele analisa o git diff, o historico recente e o contexto da sessao para produzir mensagens que capturam tanto o "o que" quanto o "porque" da mudanca.

1 Commit simples

Commita as mudancas com conventional commits

2 Commits atomicos

Crie commits atomicos separando cada mudanca logica

3 Commit com contexto

Commita referenciando a issue #42 e explicando o motivo da mudanca

4 Revisao antes do commit

Mostre o diff, sugira a mensagem de commit e espere minha aprovacao

Exemplo: commit atomico gerado pelo Claude

Output do Claude — commits atomicos git log 
feat(auth): adiciona validacao de email no formulario de cadastro

A validacao anterior aceitava qualquer string como email,
permitindo cadastros com dados invalidos. Agora usamos
z.string().email() do Zod para garantir formato correto
antes de enviar ao backend.

Closes #42

---

test(auth): adiciona testes para validacao de email

Cobre cenarios de email valido, invalido e vazio.
Garante que a mensagem de erro aparece no formulario.

---

refactor(auth): extrai schema de validacao para arquivo dedicado

Move o schema Zod de SignupForm.tsx para signup.schema.ts
para manter consistencia com os demais schemas do projeto.
Note o padrao: Cada commit e uma unidade logica independente. O Claude separou a implementacao, os testes e o refactor em commits distintos — exatamente como um dev experiente faria. E cada mensagem explica o "porque", nao apenas o "o que".

Criando Pull Requests com o Claude

O Claude pode criar PRs completas usando o gh CLI (GitHub CLI) ou via GitHub MCP Server. A grande vantagem: como ele acabou de implementar as mudancas, ele ja sabe exatamente o que mudou, por que mudou e como testar — e usa tudo isso para gerar a descricao automaticamente.

Duas formas de criar PRs

CLI Via gh CLI (recomendado)

O Claude executa gh pr create diretamente via Bash. Funciona out-of-the-box se o gh estiver instalado e autenticado. Abordagem mais simples e direta.

MCP Via GitHub MCP Server

Servidor MCP oficial do GitHub que expoe ferramentas de PR, issues e reviews como tools nativas do Claude. Mais rico em funcionalidades para workflows complexos.

Prompt do usuario Claude Code

implemente a solucao, escreva testes, rode todos os testes

Descricao de PR gerada automaticamente

Uma das maiores vantagens de usar o Claude para criar PRs e que ele gera descricoes ricas e contextuais automaticamente. Como ele acabou de fazer as mudancas, tem todas as informacoes na memoria da sessao: o que mudou, por que mudou, quais arquivos foram alterados, que testes foram adicionados e como testar manualmente.

Anatomia de uma boa descricao de PR
Secao O que contem De onde o Claude tira
Contexto Por que essa mudanca existe, qual problema resolve Issue original + entendimento do problema
Mudancas Lista das alteracoes feitas, arquivos modificados Tool calls de Edit/Write que ele executou
Como testar Passos para verificar manualmente Contexto funcional da implementacao
Testes Quais testes foram adicionados/modificados Arquivos de teste que ele criou/editou
Link com issue Referencia cruzada (Closes #N) Issue que motivou a tarefa

Boas praticas para commits e PRs com o Claude

A Commits atomicos

Peca ao Claude para separar mudancas logicas em commits distintos. Isso facilita review, bisect e eventual revert.

P Mensagens que explicam o "porque"

O diff ja mostra o "o que". A mensagem de commit deve explicar a motivacao, o contexto e as decisoes de design.

R Referencia a issues

Inclua Closes #N ou Relates to #N para manter a rastreabilidade entre codigo e requisitos.

T Teste antes de commitar

Coloque no CLAUDE.md: "Rode os testes antes de todo commit. Nunca commite codigo que quebra testes existentes."

Regra de ouro no CLAUDE.md: Inclua a instrucao Commit and push after every meaningful unit of work. Isso cria um historico granular que serve como safety net — se algo der errado, voce pode reverter facilmente.

Revisao de PR com o Claude Code

Alem de criar PRs, o Claude Code tambem pode revisar PRs existentes — seus ou de colegas. Ele analisa diffs, identifica problemas e sugere melhorias. Voce pode usar isso tanto interativamente quanto via GitHub Actions para review automatizado.

Prompts para revisao

Prompts de revisao de PR Claude Code 
# Revisao geral
Revise a PR #87. Foque em bugs, seguranca e performance.

# Corrigir feedback de review
Leia os comentarios de review na PR #87, corrija o que for
apontado e faca push para a branch.

# Revisao com contexto
Compare a branch feat/validacao-email com main e identifique
potenciais problemas de seguranca nas mudancas.

# Review via GitHub Actions (automatico)
@claude revise esta PR focando em type safety e edge cases
GitHub Actions: A Anthropic oferece uma GitHub Action oficial que permite ao Claude revisar PRs automaticamente a cada push. Com um simples @claude em comentarios de PR, ele analisa o codigo, responde e pode ate implementar correcoes.

Checkpoints e git stash como safety net

Em sessoes longas, onde o Claude faz muitas mudancas, e importante ter mecanismos de recuperacao caso algo saia do caminho esperado.

CP Checkpoints do Claude Code

O Claude Code salva automaticamente o estado do codigo antes de cada mudanca. Use Esc Esc (duplo escape) ou o comando /rewind para voltar a qualquer ponto anterior. Voce pode restaurar o codigo, a conversa ou ambos.

GS Git stash estrategico

Antes de mudancas arriscadas, peca ao Claude para fazer git stash. Isso preserva o estado atual na pilha do Git, permitindo recuperacao mesmo se os checkpoints nao cobrirem a situacao.

Estrategias de recuperacao

Situacao Estrategia Comando/Acao
Claude editou arquivo errado Checkpoint / Rewind Esc Esc ou /rewind
Mudanca grande deu errado Git stash antes + pop depois git stash / git stash pop
Commits ja feitos estao errados Git reset (soft) git reset --soft HEAD~N
Prevenir perdas em sessao longa Commits frequentes Instrucao no CLAUDE.md
Branch inteira comprometida Nova branch a partir da main git checkout main -b novo-attempt
Cuidado com operacoes destrutivas: O Claude Code pede confirmacao antes de executar comandos como git push --force, git reset --hard e amend de commits publicados. Nunca autorize essas operacoes sem entender o que sera perdido.

Automatizando com slash commands

Para workflows de commit e PR que voce repete frequentemente, crie custom slash commands na pasta .claude/commands/. Esses comandos ficam disponiveis para toda a equipe via Git.

.claude/commands/pr.md Custom slash command 
# Fluxo padrao de PR

1. Rode todos os testes: `npm test`
2. Rode lint e typecheck: `npm run lint && npm run typecheck`
3. Se tudo passar, crie commits atomicos com Conventional Commits
4. Faca push para a branch atual
5. Crie uma PR com `gh pr create` incluindo:
   - Titulo no padrao Conventional Commits
   - Descricao com contexto, mudancas e como testar
   - Link para issues relacionadas
6. Se algum teste falhar, corrija antes de prosseguir

Com isso configurado, basta digitar /pr no Claude Code e ele executa todo o workflow de qualidade, commit e criacao de PR automaticamente.
