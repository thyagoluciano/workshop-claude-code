Usando /clear e /compact estrategicamente

O contexto e o recurso mais valioso do Claude Code. Saber quando limpar tudo com /clear e quando comprimir com /compact e a diferenca entre desperdicar tokens e trabalhar com eficiencia maxima.

O custo de NAO limpar o contexto

Cada mensagem que voce envia ao Claude Code inclui todo o historico da conversa como contexto. Isso significa que arquivo lido, cada output de comando e cada tentativa fracassada continua sendo processada (e cobrada) em todas as mensagens seguintes. A documentacao oficial e clara: a context window se enche rapido e a performance degrada conforme ela enche.

Acumulo de tokens ao longo da sessao

~8%
Msg 5

~25%
Msg 10

~85%
Msg 18+

Impacto real: Quando a context window esta cheia, o Claude pode comecar a "esquecer" instrucoes anteriores ou cometer mais erros. Alem disso, cada mensagem subsequente fica mais cara porque tokens acumulados sao reprocessados. Uma sessao limpa com um prompt melhor quase sempre supera uma sessao longa com correcoes acumuladas.

O comando /clear

O que faz

O /clear apaga todo o historico de conversa da sessao atual. Apos executar, o Claude Code volta ao estado inicial: apenas o system prompt e o conteudo dos arquivos CLAUDE.md permanecem no contexto. Nenhuma mensagem, tool call ou resultado anterior sobrevive.

✓ O que permanece

System prompt, CLAUDE.md (projeto, usuario, empresa), configuracoes de modelo, MCP servers conectados, permissoes configuradas

✗ O que desaparece

Todas as mensagens, tool calls e resultados, arquivos lidos, outputs de bash, decisoes tomadas, correcoes feitas, contexto de debugging

Quando usar /clear

1 Troca de tarefa

Terminou o bug fix e vai comecar uma feature nova? /clear. O contexto do bug so atrapalha.

2 Contexto poluido

Se voce corrigiu o Claude mais de 2 vezes no mesmo assunto, o contexto esta cheio de tentativas falhas. Melhor recomecar.

3 Mudanca de modulo/feature

Saiu do backend para o frontend? De testes para documentacao? O contexto anterior nao agrega valor.

4 Sessao muito longa

Se o /usage mostra custo alto e contexto cheio, e hora de salvar o progresso e limpar.

Uso basico do /clear Terminal 
# Terminou a tarefa de corrigir o bug #1234
# Antes de limpar, renomeie a sessao para encontrar depois
/rename fix-bug-1234

# Agora limpa o contexto
/clear

# Comeca a nova tarefa com contexto 100% limpo
Implemente a feature de notificacoes push seguindo o padrao
do modulo de emails existente

O comando /compact

O que faz

O /compact comprime o historico da conversa em um resumo conciso, preservando as informacoes mais importantes e descartando outputs redundantes. Diferente do /clear, voce nao perde tudo — o Claude mantem um sumario das decisoes, arquivos modificados e estado atual do trabalho.

Internamente, o Claude Code passa todo o historico de mensagens para o modelo com instrucoes de sumarizar e comprimir os detalhes mais criticos. O modelo preserva decisoes arquiteturais, bugs nao resolvidos e detalhes de implementacao, enquanto descarta outputs redundantes de ferramentas e mensagens intermediarias.

/compact com instrucoes de foco

O recurso mais poderoso do /compact e aceitar instrucoes de foco que direcionam o que o modelo deve priorizar ao comprimir:

Exemplos de /compact com instrucoes Terminal 
# Foco em codigo e API
/compact Foque nos code samples e uso da API

# Foco em decisoes arquiteturais
/compact Preserve as decisoes de arquitetura e os trade-offs discutidos

# Foco nos testes
/compact Mantenha os comandos de teste e os resultados de falha

# Foco nos arquivos modificados
/compact Preserve a lista completa de arquivos modificados e pendencias

Compactacao personalizada via CLAUDE.md

Voce pode definir instrucoes permanentes de compactacao no seu CLAUDE.md para que toda compactacao (manual ou automatica) siga suas preferencias:

CLAUDE.md Configuracao de compactacao 
# Compact instructions

When you are using compact, please focus on test output and code changes.
Always preserve the full list of modified files and any test commands.
Keep architectural decisions and the rationale behind them.

Compactacao seletiva com /rewind

Para comprimir apenas parte da conversa, voce pode usar Esc + Esc ou /rewind, selecionar um checkpoint e escolher "Summarize from here" (condensa dali em diante) ou "Summarize up to here" (condensa tudo antes daquele ponto).

Quando usar /compact

1 Sessao longa, mesma tarefa

Voce esta no meio de uma refatoracao complexa e precisa de mais espaco, mas nao quer perder o contexto acumulado.

2 Muitas leituras de arquivo

Leu muitos arquivos para explorar o codebase mas agora so precisa manter as conclusoes, nao o conteudo bruto.

3 Debugging extenso

Apos varias rodadas de debug, quer manter apenas o diagnostico final e a solucao, descartando tentativas intermediarias.

4 Antes do auto-compact

Prefere controlar o que e preservado manualmente em vez de deixar o auto-compact decidir por voce.

Workflow: ciclo de sessoes com /clear

Com /clear entre tarefas, cada uma comeca com contexto limpo. Sem /clear, o residuo de tarefas anteriores polui e encarece as seguintes.

Auto-compact: a rede de seguranca

O Claude Code possui um mecanismo automatico de compactacao que dispara quando a conversa atinge aproximadamente 95% da context window. Quando isso acontece, o modelo automaticamente sumariza o historico para liberar espaco e continuar operando.

Auto-compact vs. /compact manual: O auto-compact e uma rede de seguranca — ele garante que a sessao nao trave. Mas voce nao controla o que ele preserva. Usando /compact manualmente com instrucoes de foco, voce decide o que e importante manter. Pense no auto-compact como airbag e no /compact manual como direcao preventiva.

Alem disso, o auto-compact carrega skills invocadas dentro de um budget de tokens: quando a conversa e sumarizada, o Claude Code re-anexa a invocacao mais recente de cada skill apos o resumo, mantendo os primeiros 5.000 tokens de cada, ate um limite combinado de 25.000 tokens.

/clear vs iniciar nova sessao

Ha uma diferenca sutil entre /clear (dentro da sessao) e iniciar uma sessao completamente nova executando claude no terminal:

Opcoes de gerenciamento de sessao Terminal 
# Opcao 1: /clear dentro da sessao atual
# - Mesma sessao, mesmo diretorio, mesmo PID
# - Historico apagado mas sessao continua
/clear

# Opcao 2: Sair e iniciar sessao nova
# - Nova sessao, novo contexto, novo ID
# - Pode voltar a anterior com --resume
claude

# Opcao 3: Continuar a sessao mais recente
# - Retoma de onde parou (com todo o historico)
claude -c

# Opcao 4: Listar e escolher sessao para retomar
claude --resume
Dica pratica: Use /rename antes de /clear para nomear a sessao. Assim voce pode encontra-la depois com claude --resume se precisar revisitar o trabalho anterior. Isso combina o beneficio do contexto limpo com a rastreabilidade do historico.

Heuristicas: quando limpar, quando comprimir

Regras praticas

Use /clear quando...

• Vai trocar completamente de tarefa
• Corrigiu o Claude mais de 2x no mesmo assunto
• Nao consegue resumir o que esta fazendo em uma frase
• O /usage mostra custo alto e contexto cheio
• Mudou de modulo, feature ou area do codebase
• Quer comecar uma abordagem completamente diferente

Use /compact quando...

• Esta no meio de uma tarefa complexa e quer continuar
• Leu muitos arquivos mas ainda precisa do contexto
• Quer liberar espaco sem perder o fio da meada
• Prefere controlar o que e preservado (vs. auto-compact)
• Esta fazendo debugging e quer manter apenas o diagnostico
• A tarefa tem muitas dependencias que nao quer re-explicar

Heuristica de ouro: "Se nao consigo resumir o que estou fazendo em uma frase, e hora de /clear." Quando o contexto esta tao poluido que voce mesmo nao sabe mais o estado da sessao, o Claude tambem nao sabe. Limpe e recomece com um prompt mais claro.

Trade-offs: precisao vs. continuidade

A escolha entre /clear e /compact e sempre um trade-off entre precisao (contexto limpo e focado) e continuidade (manter o historico de decisoes).

Economia real: Use /clear entre tarefas e monitore com /usage. Tokens desperdicados com contexto stale sao cobrados em cada mensagem subsequente. Em sessoes longas sem limpeza, o custo acumulado pode ser significativamente maior do que o necessario. Configure a barra de status para mostrar uso de contexto continuamente com /config.

Workflow de um dia tipico Terminal

$ claude

  ...Claude investiga, corrige, testa...

# Manha — Feature nova

  ...sessao longa, muitos arquivos lidos...
/compact Foque nos arquivos criados e nas decisoes de API design
  ...continua implementando com contexto comprimido...
/rename feat-push-notifications
/clear

# Tarde — Code review
Revise o PR #456 focando em seguranca e performance
  ...review completo...
/rename review-pr-456
/clear

# Tarde — Documentacao
Atualize o README com as instrucoes de setup do modulo de notificacoes
  ...tarefa rapida, sem necessidade de /compact...

Dicas de economia com gerenciamento de contexto

$ Monitore com /usage

Verifique o consumo de tokens regularmente. Configure a status bar para exibir uso de contexto continuamente.

✍ Prompts especificos

Requests vagos como "melhore o codigo" disparam varreduras amplas. Seja especifico para reduzir leituras de arquivos.

⇩ Delegue ao subagent

Operacoes verbosas (testes, logs) podem ser delegadas a subagentes, mantendo apenas o resumo no contexto principal.

Lembre-se: O custo medio por desenvolvedor e cerca de $13 por dia ativo em deploys enterprise. Boa parte desse custo vem de tokens desperdicados com contexto stale. Um habito simples de /clear entre tarefas pode reduzir significativamente esse valor.
