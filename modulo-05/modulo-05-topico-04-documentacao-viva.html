Documentacao Viva: Dev Docs e Living Plans

Use o file system como memoria externa do Claude Code. Arquivos de documentacao persistem entre sessoes, reduzem tokens no contexto e sao versionaveis com git — uma estrategia fundamental de context engineering.

O conceito: file system como memoria externa

A context window do Claude Code e um recurso finito e precioso. Cada arquivo lido, cada output de comando e cada mensagem da conversa consomem tokens. A Anthropic descreve isso como um "attention budget" — quanto mais tokens no contexto, mais a capacidade de atencao do modelo se dilui. O artigo oficial sobre context engineering recomenda buscar o menor conjunto possivel de tokens de alto sinal que maximize a probabilidade do resultado desejado.

A solucao e simples e poderosa: em vez de manter toda a informacao no contexto da conversa, escreva-a em arquivos no file system. Esses arquivos funcionam como uma "memoria de trabalho externa" — o Claude pode consulta-los sob demanda com Read, sem precisar carregar tudo de uma vez. A Anthropic chama isso de structured note-taking (anotacao estruturada), uma das tecnicas centrais para tarefas de longa duracao.

Principio central: A documentacao oficial da Anthropic sobre context engineering destaca que agentes eficazes usam "note-taking strategies for additional persistence" — ou seja, escrevem notas em arquivos externos ao contexto para manter informacao critica disponivel entre compactacoes e sessoes.

O padrao Write → /clear → Read

O workflow mais poderoso para reduzir tokens e o ciclo explorar, documentar, limpar e retomar. O Claude investiga o codebase, escreve suas descobertas em um arquivo .md, voce limpa o contexto com /clear, e entao o Claude le apenas o resumo compacto para continuar o trabalho — sem carregar todo o historico da exploracao.

O padrao Write-Clear-Read: a exploracao consome muitos tokens, mas o resumo escrito em arquivo sobrevive ao /clear e recarrega o contexto de forma compacta.

Exemplo do workflow completo Prompt do usuario 
# Fase 1 — Explorar e documentar
"Explore o codebase e entenda a arquitetura de autenticacao.
 Escreva suas descobertas em docs/ARCHITECTURE.md com:
 - Fluxo de autenticacao passo a passo
 - Arquivos principais envolvidos
 - Dependencias externas
 - Pontos de atencao para mudancas"

# [Claude explora ~30 arquivos, gasta muitos tokens, escreve o .md]
# [Usuario executa /clear para resetar o contexto]

# Fase 2 — Implementar com contexto limpo
"Leia docs/ARCHITECTURE.md e implemente OAuth2 com Google
 seguindo a arquitetura documentada. Rode os testes ao final."

# [Claude le apenas o .md compacto e implementa com foco]
Por que funciona: A exploracao inicial pode consumir 70-90% da context window lendo dezenas de arquivos. Apos o /clear, o Claude recarrega apenas o resumo estruturado (~10% do contexto), ganhando espaco para pensar e implementar com muito mais qualidade.

Tipos de documentacao viva

A documentacao oficial da Anthropic recomenda usar arquivos Markdown como checklists e scratchpads de trabalho. Na pratica, diferentes tipos de arquivo servem a propositos distintos. Aqui estao os mais uteis:

P PLAN.md

Plano de implementacao criado antes de comecar a codar. Lista etapas, decisoes de design, riscos e criterios de sucesso. O Claude consulta durante a execucao.

A ARCHITECTURE.md

Mapa da arquitetura do sistema: componentes principais, fluxos de dados, padroes utilizados. Funciona como "contexto de alto nivel" para qualquer tarefa futura.

D DECISIONS.md

Registro de decisoes tecnicas (ADRs simplificados): o que foi decidido, por que, e quais alternativas foram descartadas. Evita que o Claude re-avalie opcoes ja rejeitadas.

T TODO.md

Checklist de tarefas pendentes. O Claude marca itens conforme completa, permitindo retomar o trabalho apos /clear ou em uma nova sessao sem perder progresso.

S SCRATCH.md

Scratchpad temporario onde o Claude anota descobertas durante exploracao: padroes encontrados, bugs identificados, dependencias mapeadas. Descartavel apos a tarefa.

C CHANGELOG.md

Registro de progresso — o que ja foi feito, o que falhou e por que, metricas parciais. A Anthropic menciona esse padrao como "memoria de longo prazo portavel" do agente.

Living Plans: planeje antes de implementar

A best practice mais recomendada pela Anthropic e separar pesquisa e planejamento da implementacao. A documentacao oficial alerta que deixar o Claude pular direto para o codigo pode resultar em codigo que resolve o problema errado. O PLAN.md e a materializacao dessa pratica: o Claude cria um plano detalhado em arquivo antes de escrever uma unica linha de codigo.

Prompt para gerar um PLAN.md Prompt do usuario 
"Antes de implementar qualquer coisa, crie um arquivo PLAN.md com:

## Objetivo
Migrar o sistema de pagamentos de Stripe v2 para v3

## Analise
- Arquivos afetados
- Breaking changes da API v3
- Testes que precisam ser atualizados

## Plano de execucao
- [ ] Etapa 1: ...
- [ ] Etapa 2: ...

## Riscos
- Pontos de atencao e fallbacks

Nao implemente nada ainda. Apenas o plano."
Exemplo de PLAN.md gerado pelo Claude PLAN.md 
# Plano: Migracao Stripe v2 → v3

## Objetivo
Migrar todas as chamadas de pagamento para a API Stripe v3,
mantendo retrocompatibilidade com webhooks existentes.

## Arquivos afetados
- src/services/payment.service.ts — cliente Stripe principal
- src/webhooks/stripe.handler.ts — handler de webhooks
- src/types/payment.types.ts — tipos TypeScript
- tests/payment.test.ts — testes unitarios

## Breaking changes v3
- charges.create → paymentIntents.create
- Campo source → payment_method
- Webhook charge.succeeded → payment_intent.succeeded

## Plano de execucao
- [ ] 1. Atualizar tipos em payment.types.ts
- [ ] 2. Migrar payment.service.ts para PaymentIntents
- [ ] 3. Atualizar webhook handler para novos eventos
- [ ] 4. Ajustar testes unitarios
- [ ] 5. Rodar suite completa de testes

## Riscos
- Webhooks em transito durante deploy podem usar formato antigo
- Necessario periodo de compatibilidade dual por ~24h
Plan Mode: O Claude Code tem um modo de planejamento nativo (ative com Shift+Tab para alternar para "Plan"). Nesse modo, o Claude le arquivos e responde perguntas sem fazer alteracoes. Use-o para a fase de exploracao e depois converta as descobertas em um PLAN.md antes de executar.

Scratchpad files: anotacoes durante a exploracao

A documentacao oficial da Anthropic recomenda explicitamente que o Claude use arquivos Markdown como "working scratchpad" — especialmente para tarefas grandes com multiplos passos. O scratchpad e diferente do plano: e um rascunho temporal onde o Claude anota o que vai descobrindo durante a investigacao do codebase.

Instruindo o Claude a usar scratchpad Prompt do usuario 
"Investigue todos os erros de lint no projeto.
 Use um arquivo SCRATCH.md como checklist de trabalho:
 1. Rode o linter e escreva todos os erros no SCRATCH.md
 2. Corrija cada erro um por um
 3. Marque como feito no checklist apos verificar
 4. Ao final, delete o SCRATCH.md"

O scratchpad e especialmente util quando multiplas instancias do Claude trabalham em paralelo. A Anthropic menciona que voce pode dar a cada instancia um scratchpad separado, dizendo qual arquivo escrever e qual ler — essa separacao frequentemente produz melhores resultados do que uma unica instancia fazendo tudo.

Scratchpad interno vs. externo: O extended thinking ("think", "think hard", "ultrathink") funciona como um scratchpad interno — o Claude raciocina dentro da propria mente, mas esses pensamentos desaparecem apos o turno. Um arquivo .md funciona como scratchpad externo — persiste entre turnos, entre /clear, e entre sessoes. Para tarefas complexas, use ambos: extended thinking para raciocinar, arquivo para persistir.

Dev docs gerados pelo Claude

Alem de documentacao para consumo do proprio Claude, voce pode pedir que ele gere documentacao util para humanos — e que tambem serve como contexto futuro para o agente. Essa documentacao "de uso dual" e especialmente valiosa porque mantem-se atualizada naturalmente.

Tipo de doc Prompt sugerido Beneficio
Mapa de arquitetura "Documente a arquitetura do projeto em ARCHITECTURE.md" Onboarding de devs + contexto rapido para Claude
API contracts "Gere documentacao dos endpoints em docs/API.md" Referencia para frontend + guia para Claude modificar APIs
Registro de decisoes "Documente a decisao e a razao em DECISIONS.md" Previne re-discussao de opcoes ja descartadas
Guia de contribuicao "Crie CONTRIBUTING.md com padroes e convencoes" Humanos e Claude seguem os mesmos padroes

Ultrathink vs. arquivo: quando usar cada um

O Claude Code tem dois mecanismos de "raciocinio expandido": o extended thinking (interno, dentro do modelo) e os arquivos de documentacao (externos, no file system). Eles sao complementares, nao concorrentes. A escolha depende de voce precisar que a informacao persista ou nao.

Dimensao Extended Thinking Arquivo .md
Persistencia Desaparece apos o turno Persiste entre sessoes
Tokens consumidos Nao conta no contexto visivel Conta quando lido com Read
Compartilhavel Nao — invisivel para o usuario Sim — versionavel com git
Multi-agente Isolado ao agente atual Multiplos agentes podem ler/escrever
Caso de uso Raciocinar sobre um problema pontual Documentar para reutilizacao futura
Como ativar "think hard", "ultrathink" "escreva em PLAN.md"
Regra pratica: Se a informacao e util apenas para este turno, use extended thinking. Se sera util depois do /clear ou em outra sessao, escreva em arquivo. Para tarefas complexas, use os dois: "ultrathink sobre a melhor abordagem, depois escreva seu plano em PLAN.md".

Beneficios da documentacao viva

1 Reduz tokens

Um resumo de 200 linhas substitui 30 arquivos que gastariam milhares de tokens para ler. O contexto fica livre para pensar e agir.

2 Persiste entre sessoes

Diferente do contexto da conversa, arquivos no file system sobrevivem a /clear, a compactacao automatica, e ate ao fim da sessao.

3 Versionavel com git

PLAN.md e DECISIONS.md podem ser commitados. O historico de decisoes tecnicas fica rastreavel e revisavel pela equipe.

4 Multi-agente

Multiplas instancias do Claude podem compartilhar estado via arquivos — um escreve, outro le. A Anthropic recomenda esse padrao explicitamente.

5 Dual use

A documentacao serve tanto para o Claude quanto para humanos. Um ARCHITECTURE.md ajuda no onboarding de novos devs e como contexto para o agente.

6 Evita retrabalho

DECISIONS.md registra por que algo foi decidido. O Claude nao perde tempo re-avaliando alternativas que ja foram descartadas.

Workflow completo: da exploracao a entrega

Combinando todas as tecnicas, aqui esta o workflow recomendado para tarefas complexas:

1

Explorar com Plan Mode

Ative Plan Mode (Shift+Tab). Peca ao Claude para investigar o codebase e entender o contexto. Ele le arquivos sem fazer alteracoes.

2

Documentar descobertas

Peca ao Claude para escrever suas descobertas em arquivos: PLAN.md para o plano, ARCHITECTURE.md para contexto, TODO.md para checklist.

3

Limpar contexto

Execute /clear para resetar o contexto. Todo o conhecimento da exploracao agora esta salvo em arquivos, nao na conversa.

4

Implementar referenciando docs

Na nova sessao limpa, instrua: "Leia PLAN.md e implemente as etapas na ordem. Marque cada item como feito no TODO.md".

5

Iterar e atualizar

Se o contexto ficar pesado novamente, repita o ciclo: atualize os docs com o progresso, /clear, e continue. Os docs evoluem com o projeto.

Instrucao para CLAUDE.md que automatiza o padrao CLAUDE.md 
## Regras de documentacao

# Para tarefas complexas (mais de 5 arquivos afetados):
- Sempre crie um PLAN.md antes de implementar
- Use TODO.md como checklist — marque itens conforme completar
- Registre decisoes tecnicas em DECISIONS.md
- Se o contexto estiver grande, sugira /clear ao usuario

# Para exploracao de codebase:
- Anote descobertas em SCRATCH.md durante a investigacao
- Ao final da exploracao, consolide em ARCHITECTURE.md
- Pode deletar SCRATCH.md apos consolidacao

# Para migracao ou refactoring grande:
- Use CHANGELOG.md para registrar progresso
- Formato: data, o que foi feito, o que falta, o que falhou
Dica avancada: Adicione essas regras no CLAUDE.md do seu projeto. Assim, o Claude adota automaticamente o padrao de documentacao viva em toda tarefa complexa, sem que voce precise instruir a cada vez.

Relacao com compactacao automatica

O Claude Code realiza compactacao automatica quando o contexto se aproxima do limite — ele resume o historico, preservando decisoes e detalhes de implementacao, e descarta outputs de ferramentas redundantes. A documentacao viva complementa essa funcionalidade: enquanto a compactacao e uma medida reativa (acontece quando o contexto esta cheio), a documentacao viva e uma medida proativa (voce escreve antes de precisar).

Atencao: A compactacao automatica pode perder detalhes sutis cuja importancia so se revela depois. Informacoes criticas devem estar em arquivos, nao apenas no historico da conversa. Se algo e importante o suficiente para sobreviver a uma compactacao, escreva em arquivo.
