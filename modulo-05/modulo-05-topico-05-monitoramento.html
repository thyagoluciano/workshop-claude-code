Modulo 5 . Topico 5

Monitorando uso de contexto com status line customizada

Saber quanto contexto resta, quanto custou a sessao e quando agir e a diferenca entre uma sessao produtiva e uma que degrada silenciosamente. O Claude Code oferece ferramentas nativas e customizaveis para manter essa visibilidade.

O comando /usage

O /usage e o comando mais direto para inspecionar o consumo da sessao atual. Ele mostra custo estimado, duracao total (wall clock e API), e linhas de codigo modificadas. O valor em dolar e calculado localmente a partir da contagem de tokens e pode diferir da cobranca real no Console.

claude
$ /usage

Total cost: $0.55
Total duration (API): 6m 19.7s
Total duration (wall): 6h 33m 10.2s
Total code changes: 0 lines added, 0 lines removed

--- Sessao longa com pouca edicao: investigacao / code review ---
Nota para assinantes Pro/Max: O bloco "Session" do /usage mostra consumo de tokens da API e e voltado para usuarios de API. Assinantes de plano ja tem uso incluso na assinatura, entao o custo da sessao nao e relevante para cobranca — mas ainda e util para entender a eficiencia do seu uso de contexto.

Status line customizada: visibilidade continua

Enquanto o /usage e um snapshot sob demanda, a status line e uma barra persistente no rodape do Claude Code que roda qualquer script shell que voce configurar. Ela recebe dados JSON da sessao via stdin e exibe o que seu script imprimir — dando visibilidade continua de contexto, custo, branch git ou qualquer outra metrica.

A forma mais rapida de criar uma status line e usar o proprio Claude Code:

Comando /statusline Claude gera o script para voce 
// O Claude Code gera o script em ~/.claude/ e atualiza settings.json
/statusline show model name and context percentage with a progress bar

// Ou com mais detalhes:
/statusline show model, context %, cost, and git branch with colors

Configurando manualmente via settings.json

Se preferir configurar manualmente, adicione o campo statusLine ao seu ~/.claude/settings.json (configuracao de usuario) ou ao settings do projeto:

~/.claude/settings.json Configuracao da status line 
{
  "statusLine": {
    "type": "command",
    "command": "~/.claude/statusline.sh",
    "padding": 2,
    "refreshInterval": 5   // re-executa a cada N segundos (opcional)
  }
}

Para configuracoes simples, voce pode usar um comando inline com jq direto no settings, sem precisar de um arquivo separado:

~/.claude/settings.json Inline com jq 
{
  "statusLine": {
    "type": "command",
    "command": "jq -r '\"[\\(.model.display_name)] \\(.context_window.used_percentage // 0)% context\"'"
  }
}
Custo zero: A status line roda localmente e nao consome tokens da API. Ela e atualizada apos cada mensagem do assistente, apos /compact, ao trocar modo de permissao ou ao alternar vim mode. Updates sao debounced a 300ms.

Script completo: barra de contexto com cores

Este script combina modelo, barra de progresso com cores por threshold, custo acumulado, duracao e branch git — tudo em duas linhas na status bar:

~/.claude/statusline.sh Bash + jq 
#!/bin/bash
input=$(cat)

MODEL=$(echo "$input" | jq -r '.model.display_name')
DIR=$(echo "$input" | jq -r '.workspace.current_dir')
COST=$(echo "$input" | jq -r '.cost.total_cost_usd // 0')
PCT=$(echo "$input" | jq -r '.context_window.used_percentage // 0' | cut -d. -f1)
DURATION_MS=$(echo "$input" | jq -r '.cost.total_duration_ms // 0')

# Cores ANSI por threshold
CYAN='\033[36m'; GREEN='\033[32m'
YELLOW='\033[33m'; RED='\033[31m'; RESET='\033[0m'

# Cor da barra baseada no uso de contexto
if [ "$PCT" -ge 90 ]; then BAR_COLOR="$RED"
elif [ "$PCT" -ge 70 ]; then BAR_COLOR="$YELLOW"
else BAR_COLOR="$GREEN"; fi

FILLED=$((PCT / 10)); EMPTY=$((10 - FILLED))
printf -v FILL "%${FILLED}s"; printf -v PAD "%${EMPTY}s"
BAR="${FILL// /█}${PAD// /░}"

MINS=$((DURATION_MS / 60000)); SECS=$(((DURATION_MS % 60000) / 1000))

BRANCH=""
git rev-parse --git-dir > /dev/null 2>&1 && \
  BRANCH=" | 🌿 $(git branch --show-current 2>/dev/null)"

echo -e "${CYAN}[$MODEL]${RESET} 📁 ${DIR##*/}$BRANCH"
COST_FMT=$(printf '$%.2f' "$COST")
echo -e "${BAR_COLOR}${BAR}${RESET} ${PCT}% | ${YELLOW}${COST_FMT}${RESET} | ⏱️ ${MINS}m ${SECS}s"

Apos salvar, torne o script executavel com chmod +x ~/.claude/statusline.sh e adicione o caminho ao settings.json. O resultado mostra algo como:

Status line renderizada
[Opus] 📁 meu-projeto | 🌿 feature/auth
███░░░░░░░ 30% | $0.42 | ⏱️ 12m 35s

--- Depois de trabalho intenso: ---
[Opus] 📁 meu-projeto | 🌿 feature/auth
█████████░ 92% | $3.87 | ⏱️ 48m 12s

Dados disponiveis na status line

O Claude Code envia um JSON completo via stdin para o seu script. Estes sao os campos mais relevantes para monitoramento de contexto e custos:

Campo JSON O que contem Por que monitorar
context_window.used_percentage % do contexto utilizado (calculado sobre input tokens) Indicador principal de quando agir (/compact, /clear)
context_window.remaining_percentage % do contexto restante Visao inversa para scripts que preferem "quanto resta"
context_window.context_window_size Tamanho maximo da janela (200K ou 1M com extended context) Saber o teto real do modelo em uso
context_window.total_input_tokens Tokens de input atualmente no contexto (inclui cache) Entender composicao do consumo
cost.total_cost_usd Custo estimado da sessao em USD Controle de gastos em tempo real
cost.total_duration_ms Tempo total wall-clock da sessao Detectar sessoes que se arrastam
model.display_name Nome do modelo em uso (Opus, Sonnet, etc.) Confirmar qual modelo esta ativo apos /model
effort.level Nivel de esforco atual (low, medium, high, max) Verificar se /effort esta ajustado para a tarefa
exceeds_200k_tokens Boolean: se ultrapassou 200K tokens na ultima resposta Alerta de que o contexto esta muito grande

Auto-compact: a rede de seguranca

O Claude Code possui um mecanismo automatico de compactacao que e ativado quando a conversacao se aproxima do limite da context window. Quando disparado, o Claude sumariza mensagens anteriores para liberar espaco, preservando codigo, decisoes importantes e o estado dos arquivos.

A O que o auto-compact preserva

Padroes de codigo, estado dos arquivos, decisoes-chave tomadas durante a sessao, e instrucoes do CLAUDE.md sobre o que manter.

! O que pode ser perdido

Detalhes de conversas antigas, outputs longos de comandos, tentativas que falharam, e nuances de discussoes exploratórias iniciais.

Voce pode personalizar o que o auto-compact preserva adicionando instrucoes ao CLAUDE.md do projeto:

CLAUDE.md Instrucoes de compactacao 
# Compact instructions

When you are using compact, please focus on test output and code changes.
Always preserve the full list of modified files and any test commands used.
Keep track of which API endpoints were discussed and their expected schemas.

Voce tambem pode disparar compactacao manual com instrucoes especificas:

Compactacao manual com foco /compact 
// Compactar preservando contexto especifico
/compact Focus on the API changes and test results

// Usar Esc + Esc ou /rewind para checkpoint mais granular
/rewind   // selecionar ponto de checkpoint e sumarizar ate ali

Controlando custos: /effort e MAX_THINKING_TOKENS

Extended thinking e habilitado por padrao e melhora significativamente a performance em tarefas complexas. Porem, thinking tokens sao cobrados como output tokens e o budget padrao pode ser de dezenas de milhares de tokens por request. Para tarefas simples, voce pode reduzir custos de varias formas:

Metodo Como usar Quando usar
/effort low|medium|high|max Ajusta o nivel de raciocinio do modelo Tarefas simples (low), complexas (high/max)
MAX_THINKING_TOKENS=8000 Variavel de ambiente que limita tokens de thinking Controle granular de custo por sessao
/config (desabilitar thinking) Desativa extended thinking completamente Tarefas triviais onde raciocinio e desnecessario
/model (trocar modelo) Sonnet para tarefas rotineiras, Opus para complexas Otimizar custo-beneficio mid-session
Dica de economia: Para modelos mais recentes (Opus 4.6+, Sonnet 4.6+), a Anthropic recomenda usar o parametro /effort em vez de MAX_THINKING_TOKENS para controlar a profundidade de raciocinio. O budget_tokens esta em processo de descontinuacao nesses modelos.

Integrando monitoramento no workflow

Monitoramento nao e algo que voce faz uma vez e esquece — e um habito que se integra ao fluxo de trabalho. Aqui esta o padrao recomendado:

1 Inicio de sessao

Confirme modelo e effort level na status line. Use /effort para ajustar antes de comecar. Considere /model sonnet para tarefas rotineiras.

2 Durante o trabalho

Olhe a barra de contexto periodicamente. Ao atingir 70%, avalie se vale /compact com instrucoes de foco. Delegue operacoes verbose a subagentes.

3 Troca de tarefa

Use /rename para nomear a sessao antes de /clear. Assim voce pode retomar com claude --resume. Contexto velho desperdicado = tokens jogados fora.

4 Fim de sessao

Cheque /usage para entender o custo total. Identifique padroes: sessoes caras podem indicar prompts vagos ou excesso de exploracao.

Prompts especificos economizam tokens: Pedidos vagos como "melhore esse codebase" disparam scans amplos. Pedidos especificos como "adicione validacao de input na funcao login em auth.ts" permitem ao Claude trabalhar eficientemente com leituras minimas de arquivos.

Estrategias adicionais de economia

M Mova instrucoes para skills

O CLAUDE.md carrega no inicio. Instrucoes especificas (PR review, migrations) consomem tokens mesmo quando irrelevantes. Skills carregam sob demanda.

H Use hooks para filtrar

Um hook PreToolUse pode filtrar output de testes para mostrar so falhas, reduzindo contexto de milhares de tokens para centenas.

S Delegue a subagentes

Testes, logs e docs extensas ficam no contexto do subagente. So o resumo retorna para a conversa principal.

Consumo em background: O Claude Code usa uma pequena quantidade de tokens para funcionalidades em segundo plano, como sumarizacao de conversas para o --resume e processamento de comandos. Isso tipicamente custa menos de $0.04 por sessao.

Resumo do Modulo 5: Gerenciamento de Contexto

Ao longo deste modulo, construimos um arsenal completo de tecnicas para gerenciar a context window — o recurso mais precioso do Claude Code. Aqui esta a progressao logica de tudo que cobrimos:

Estrategias do Modulo 5 — Da teoria a pratica

1

Topico 1 — Sintomas de degradacao de contexto: Entender por que o Claude "esquece" coisas, repete codigo ou perde convencoes. O problema nao e o modelo — e a pressao sobre a context window.

2

Topico 2 — /clear e /compact como ferramentas de higiene: As duas ferramentas fundamentais: /clear para reset total entre tarefas e /compact para sumarizacao seletiva preservando contexto-chave.

3

Topico 3 — Leituras seletivas com Grep, Glob e Read: Nao leia arquivos inteiros quando precisa de uma funcao. Use Grep para encontrar, Read com offset/limit para ler so o trecho, e Glob para mapear a estrutura.

4

Topico 4 — Documentacao viva com CLAUDE.md: Manter um CLAUDE.md enxuto (ate ~200 linhas) que funciona como "memoria externa", reduzindo a necessidade de re-explorar o codebase a cada sessao.

5

Topico 5 — Monitoramento com status line customizada: Fechar o ciclo com visibilidade continua. A status line mostra quando agir, /usage mostra o impacto, e /effort + modelo controlam o custo.

A regra de ouro: Contexto e como RAM — cada byte ocupado com informacao irrelevante e um byte que nao esta disponivel para o que importa. As melhores sessoes com o Claude Code sao aquelas onde voce gerencia ativamente o que entra e sai da context window, usando as ferramentas certas no momento certo.
