Leituras Seletivas de Arquivo em Vez de Exploração Ampla

A context window do Claude Code é um recurso finito e precioso. Ler arquivos inteiros quando você precisa de apenas um trecho é como imprimir um livro para ler uma única frase. Domine as primitivas de leitura cirúrgica para manter o contexto limpo e a performance alta.

O problema: contexto desperdiçado

Cada arquivo que o Claude Code lê entra integralmente na context window. Uma única sessão de debugging ou exploração de codebase pode gerar e consumir dezenas de milhares de tokens. Quando o contexto enche, a performance degrada: o modelo perde o foco, esquece instruções anteriores e começa a cometer erros que não cometeria com um contexto limpo.

O anti-pattern mais comum é pedir ao Claude para "ler todos os arquivos do projeto" ou deixá-lo explorar livremente sem direção. Isso consome contexto com informação irrelevante e reduz a qualidade das respostas subsequentes.

Anti-pattern clássico: "Leia todos os arquivos da pasta src para entender o projeto." Isso pode consumir facilmente 50.000+ tokens de contexto com informação que o Claude não precisava. Em vez disso, descreva o que você quer resolver e deixe o Claude buscar cirurgicamente o que precisa.

Modelo híbrido: upfront + just-in-time

O Claude Code usa um modelo híbrido de carregamento de contexto. O arquivo CLAUDE.md é carregado integralmente no início da conversa (contexto upfront) com informações essenciais sobre o projeto: convenções, stack, estrutura de pastas, comandos de build e teste.

Todo o resto é obtido sob demanda (just-in-time) usando primitivas como Glob, Grep, Read e Bash. O agente mantém identificadores leves (caminhos de arquivo, queries, links) e usa essas referências para carregar dados no contexto dinamicamente, apenas quando precisa deles para a tarefa atual.

Princípio de design: A estrutura de pastas e arquivos do projeto funciona como uma forma de context engineering. Um projeto bem organizado permite ao Claude navegar eficientemente usando glob e grep, sem precisar ler tudo de uma vez.

O funil de busca seletiva

A estratégia ideal segue um funil: comece amplo com Glob para localizar arquivos, refine com Grep para encontrar o trecho exato, e so entao use Read direcionado para carregar apenas o necessário no contexto.

O funil de busca seletiva: cada etapa refina os resultados e minimiza tokens consumidos no contexto.

Grep antes de Read: a regra de ouro

A estratégia mais eficiente para encontrar código relevante é sempre buscar antes de ler. O Grep retorna apenas os nomes dos arquivos que contém o padrão (ou as linhas correspondentes com contexto), consumindo muito menos tokens do que ler arquivos inteiros.

Estratégia: Grep primeiro, Read depois Prompt para o Claude 
// Prompt eficiente — direciona a busca
"Encontre onde a função processPayment é definida
 e corrija o bug de arredondamento no cálculo do total."

// O que o Claude faz internamente:

// Passo 1: Grep para localizar (custo: ~50 tokens)
Grep { pattern: "function processPayment|processPayment =",
       glob: "**/*.ts" }
  → src/services/payment.service.ts:45
  → src/utils/legacy-payment.ts:120

// Passo 2: Read direcionado (custo: ~200 tokens)
Read { file: "src/services/payment.service.ts",
       offset: 40, limit: 30 }
  → Lê apenas as linhas 40-70, onde está a função

// Passo 3: Edit cirúrgico
Edit { file: "src/services/payment.service.ts",
       old: "Math.round(total)",
       new: "Math.round(total * 100) / 100" }
Resultado: Em vez de ler os dois arquivos inteiros (~5000 tokens), o Claude consumiu apenas ~250 tokens para encontrar e corrigir o bug. Isso é 95% menos contexto desperdiçado.

Read com offset e limit: precisão cirúrgica

A ferramenta Read do Claude Code aceita parâmetros offset (linha inicial) e limit (número de linhas a ler). Isso permite ler apenas o trecho necessário de arquivos grandes, sem carregar milhares de linhas irrelevantes no contexto.

Exemplos de Read seletivo Tool calls 
// Ler apenas as primeiras 50 linhas (imports + config)
Read { file: "src/config/database.ts", limit: 50 }

// Ler a função que começa na linha 120
Read { file: "src/services/auth.ts", offset: 120, limit: 40 }

// Ler o final de um arquivo longo (últimas declarações)
Read { file: "src/types/index.ts", offset: 450, limit: 50 }

Você pode instruir o Claude a usar essa estrategia diretamente no prompt:

Prompt direcionado Instrução para o Claude 
"Leia apenas a função validateEmail no arquivo
 src/utils/validators.ts (está por volta da linha 85).
 Não precisa ler o arquivo inteiro."

Bash para análise: head, tail, wc, awk

Quando o Claude encontra arquivos grandes como logs, dumps de banco ou CSVs, ele usa comandos Bash como head, tail, wc e awk para extrair trechos específicos sem carregar o conteúdo completo no contexto. Isso e especialmente útil para análise de dados e debugging.

Comandos Bash para leitura seletiva Exemplos práticos 
# Contar linhas antes de decidir como ler
Bash { command: "wc -l src/data/users.csv" }
  → 45.832 linhas — não vale a pena ler tudo!

# Ver apenas o cabeçalho + primeiras linhas
Bash { command: "head -20 src/data/users.csv" }

# Ver as ultimas entradas de um log
Bash { command: "tail -50 logs/app.log" }

# Filtrar apenas erros de um log grande
Bash { command: "grep -n 'ERROR' logs/app.log | tail -20" }

# Extrair coluna específica de um CSV
Bash { command: "awk -F',' '{print $1, $3}' data/report.csv | head -10" }

# Contar ocorrências de um padrão
Bash { command: "grep -c 'TODO' src/**/*.ts" }
Dica da Anthropic: O Claude Code decide automaticamente como carregar arquivos grandes no contexto usando bash scripts como grep e tail. Mas você pode guiá-lo explicitamente no prompt para garantir que ele não tente ler o arquivo inteiro.

Instruindo leituras seletivas via prompt

O Claude Code já tende a usar estratégias seletivas, mas você pode guiá-lo explicitamente para garantir eficiência máxima. Quanto mais preciso o prompt, menos exploração desnecessária.

✗ Prompts vagos (desperdiçam contexto)

"Olhe o código do projeto e me diga o que ele faz."

"Leia todos os arquivos de services/ para entender a arquitetura."

"Analise o projeto inteiro e sugira melhorias."

✓ Prompts cirúrgicos (economizam contexto)

"Leia apenas a função createUser em src/services/user.ts."

"Busque onde UserRepository é instanciado e mostre o padrão de DI usado."

"Encontre os testes que cobrem o endpoint POST /api/orders."

Dicas para prompts que guiam leituras cirúrgicas

Técnica Exemplo de prompt Por que funciona
Referenciar arquivo com @ @src/auth.ts tem um bug na linha 85 O Claude lê o arquivo antes de responder
Nomear a função Corrija o bug em processPayment() Grep encontra a função diretamente
Indicar a linha Veja a linha 120 de database.ts Read com offset vai direto ao ponto
Limitar o escopo So nos arquivos de src/api/ Glob filtra antes de buscar
Descrever o padrão Busque por TODO ou FIXME no projeto Grep é mais eficiente que Read

Subagentes para exploração pesada

Quando a tarefa realmente exige explorar muitos arquivos (entender a arquitetura de um projeto novo, mapear dependencias, auditar seguranca), o Claude Code pode delegar essa leitura ampla a subagentes. Cada subagente opera em sua própria context window isolada, lê os arquivos necessários, e retorna apenas um resumo condensado para o agente principal.

Delegando exploração para subagentes Estrategia com Task 
// O agente principal delega a exploração
Task { description: "Analise a pasta src/services/ e retorne:
  1. Lista de services e suas responsabilidades
  2. Padrões de DI utilizados
  3. Dependências externas de cada service" }

// O subagente le TODOS os arquivos (na context dele)
// → Consome ~40.000 tokens na context do subagente
// → Retorna ~800 tokens de resumo para o agente principal

// Beneficio: o agente principal mantem a context limpa
// e recebe apenas o que precisa para tomar decisoes
Quando usar subagentes vs. busca direta: Se a informação pode ser encontrada com um ou dois Grep, faça direto — não crie um subagente. Subagentes são úteis para tarefas que exigem ler muitos arquivos onde a maior parte da informação será descartada. O custo de coordenação só vale a pena quando a exploração é genuinamente ampla.

CLAUDE.md: o contexto upfront inteligente

O CLAUDE.md é o único arquivo carregado integralmente por padrão. Use-o estrategicamente para reduzir a necessidade de leituras futuras. Inclua nele informações que o Claude precisaria buscar repetidamente:

📁 Estrutura de pastas

Mapa das principais pastas e o que cada uma contem. Evita que o Claude faca Glob exploratória.

🔧 Comandos essenciais

Build, test, lint, deploy. O Claude não precisa procurar no package.json ou Makefile.

📚 Convenções do projeto

Naming, patterns, onde colocar testes. Reduz leituras exploratórias para entender o padrão.

Exemplo de CLAUDE.md otimizado para leituras seletivas CLAUDE.md 
# Estrutura do projeto
src/
  api/          # Controllers REST (Express)
  services/     # Lógica de negócios
  repositories/ # Acesso a dados (Prisma)
  utils/        # Funções utilitárias puras
  types/        # TypeScript types e interfaces

# Comandos
npm test              # Roda todos os testes
npm test -- --grep X  # Roda testes que matcham X
npm run build         # Compila TypeScript

# Convencoes
- Testes ficam em __tests__/ ao lado do arquivo
- Services recebem dependências via construtor (DI manual)
- Validação usa Zod schemas em src/schemas/

Resumo: estratégias de leitura por cenário

Cenario Estrategia recomendada Custo estimado
Encontrar onde uma função é definida Grep por nome da função ~50 tokens
Ler uma função específica Grep + Read com offset/limit ~250 tokens
Encontrar arquivos de um tipo Glob com padrão ~30 tokens
Analisar arquivo de log grande Bash: tail, grep, awk ~100-300 tokens
Entender um arquivo inteiro Read completo (so se necessario) ~1000-5000 tokens
Mapear arquitetura do projeto Task (subagente) para explorar ~500 tokens no main
"Leia tudo e entenda o projeto" NUNCA. Use CLAUDE.md + buscas seletivas ~50.000+ tokens
Regra prática: Se você pode descrever o que procura em uma frase, o Claude pode encontrar com Grep. Se não consegue descrever, delegue a exploração para um subagente. Nunca peça "leia tudo" — isso é o equivalente a SELECT * FROM tabela_enorme sem WHERE.
